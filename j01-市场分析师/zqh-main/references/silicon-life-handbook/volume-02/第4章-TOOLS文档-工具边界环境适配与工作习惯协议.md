# 第4章｜TOOLS 文档：工具边界、环境适配与工作习惯协议
## ——为什么 TOOLS 不是权限清单，也不是技能说明，而是硅基生命与真实环境对接的“本地操作备注”

> 官方依据：
> - `reference/templates/TOOLS.md`：TOOLS.md 是“本地备注”，用于记录环境中特有的信息；**Skills 定义工具如何工作，TOOLS 只记录你的具体环境**。
> - `concepts/agent-workspace.md`：`TOOLS.md` 不控制工具可用性；它仅提供关于你的本地工具和约定的说明。
>
> 交叉参考（外部实践）：
> - 社区最佳实践普遍强调：**workspace files 是 agent 的运行接口**；其中 `TOOLS.md` 的最佳角色不是“写规则”，而是“把本地基础设施映射成 agent 能稳定调用的认知速查表”。

## 1）这份文档是什么
先把最容易搞混的一点说清楚。

官方对 `TOOLS.md` 的定义非常克制，也非常重要：

> **它是本地备注。**  
> Skills 定义工具如何工作；`TOOLS.md` 只记录你的环境里独有的信息。  
> 例如：摄像头名称、SSH 主机、TTS 首选语音、设备昵称、本地约定。

这意味着：

> **TOOLS.md 不是工具权限清单，不是技能说明书，不是系统 prompt 的替代品，更不是“写了就会有这个工具”。**

训虾派视角下，它的真正定位是：

> **TOOLS.md 是 Agent 与现实环境之间的“接口注释层”。**

也就是说，系统已经有工具，skills 已经定义了调用方法，但 agent 还不知道：
- 你的“home-server” 到底是谁
- 你说的“客厅摄像头” 对应哪个设备
- 你常用的“蓝牙音箱” 在系统里叫什么
- 你团队内部把哪个路径叫“中央知识库”

这些，不应该硬塞进 SOUL、AGENTS 或 Skills；它们最适合写进 `TOOLS.md`。

## 2）核心目的（Purpose）
`TOOLS.md` 的核心目的，不是增加工具能力，而是**减少工具使用中的环境摩擦与误判**。

它解决的是 4 个现实问题：

### 2.1 工具存在，但 agent 不知道该怎么映射到你的环境
系统里有 `message`、`exec`、`read`、`write`、`browser`、`tts` 等工具，但如果你的工作现场有一堆“本地名字”，agent 就很容易误用。

例如：
- 你说“客厅音箱”，它不知道具体对应哪个播放设备
- 你说“阿里云那台”，它不知道 SSH 别名是什么
- 你说“对外素材目录”，它不知道本地路径是哪个

### 2.2 环境信息总是散落在脑子里，无法复训
如果你的本地约定只存在你脑子里：
- 每训练一只新虾都要重新教一遍
- 每次换模型/换 agent 都要重新磨合
- 一旦上下文压缩或会话重置，这些“默契”会消失

### 2.3 同名对象太多，容易误操作
现实世界里最常见的问题是：
- 同时有 3 个服务器
- 同时有多个工作目录
- 同时有多个浏览器配置
- 同时有多个账号/群/频道

TOOLS.md 的作用，就是把这些“本地歧义”消掉。

### 2.4 需要把共享技能与个人环境分开
这也是官方特别强调的地方：

> **Skills 是可共享的；你的环境配置是你自己的。**

如果你把环境细节写进 skill：
- 技能就不可迁移
- 分享 skill 会泄露环境
- 技能升级会覆盖你自己的本地习惯

TOOLS.md 正是为了解决这个问题而存在。

## 3）拟人化映射与仿真价值
如果做拟人化映射，`TOOLS.md` 对应的是：

> **一个成熟助理的“办公环境手册 / 本地现场速查表”。**

一个好的助理不只是知道“怎么做”，他还知道：
- 老板说“老地方”指的是哪里
- “财务那台机子” 是哪台
- “公众号素材库” 在哪个盘
- “会议室大屏” 用哪个投屏名

这些不是价值观（SOUL），也不是工作纪律（AGENTS），更不是技能封装（Skills）。
它们是：

> **环境映射知识。**

### 3.1 它的仿真价值，不在“更聪明”，而在“更像本地员工”
没有 TOOLS.md 的 agent，看起来像远程实习生：懂概念，但不熟现场。  
有 TOOLS.md 的 agent，更像熟悉公司环境的老助理：知道名字、知道别名、知道约定。

### 3.2 它让 Agent 具备“场景贴合度”
很多 AI 的失败不是认知错，而是**环境错位**。  
例如知道“要发文件”，但不知道你公司规定要放 `media/outbound/`；知道“要 SSH 上去”，但不知道用的是 `home-server` 别名。

### 3.3 它决定了工具调用的“最后一公里体验”
Prompt、SOUL、AGENTS 都可以让 agent 更像人；但 TOOLS 决定它是否像一个**熟环境的人**。

## 4）对长期 AI 表现的作用（Impact）
TOOLS.md 对长期表现的影响常被低估，但它在实际系统里非常关键。

### 4.1 提升环境适配稳定性
每次会话开始时，如果 agent 都得重新猜“你说的 X 是什么”，长期表现必然不稳定。TOOLS.md 可以把这些映射固定下来。

### 4.2 降低误操作率
这是最现实的价值。
没有 TOOLS.md，很容易出现：
- 发错目标
- 写错目录
- 连错主机
- 用错设备
- 混淆测试环境与生产环境

### 4.3 加速新虾上手
一只新训练的虾，只要读到 TOOLS.md，就能更快进入现场，不需要你反复教“我们这里怎么叫这个东西”。

### 4.4 提升多 Agent 一致性
当军团里的多只虾都用同一套本地环境术语时，协作的摩擦会显著减少。

### 4.5 让外部系统接入更可解释
一个成熟的系统，不应该靠“记住这次怎么用”来接外部系统，而应该有文档化环境说明。

## 5）和其他文档最容易混淆的边界
这是本章最重要的部分之一，也是你要求我加深的重点。

### 5.1 TOOLS.md vs SOUL.md
#### TOOLS.md 负责：
- 环境映射
- 本地别名
- 设备命名
- 常用路径
- 约定俗成的工具使用方式

#### SOUL.md 负责：
- 人格
- 风格
- 边界
- 价值观

#### 最常见混淆
把“默认使用哪个喇叭、哪个路径、哪个 SSH 别名”写到 SOUL 里。  
结果：SOUL 变臃肿，而且人格协议被环境细节污染。

---

### 5.2 TOOLS.md vs AGENTS.md
#### TOOLS.md 负责：
- 你的环境里“对象叫什么”
- 某类工具在你现场通常如何被称呼

#### AGENTS.md 负责：
- 什么情况下该用工具
- 用完要不要给证据
- 工具失败时怎么办

#### 最常见混淆
把“什么时候用 read/write/message”写到 TOOLS。  
这是 AGENTS 的事情，不是 TOOLS 的事情。

---

### 5.3 TOOLS.md vs Skills
#### TOOLS.md 负责：
- 你的环境信息
- 本地速查表

#### Skills 负责：
- 方法封装
- 工具组合逻辑
- 某一类任务的可复用流程

#### 最常见混淆
把本地路径、私有主机名、账号别名写进 skill。  
结果：
- skill 失去可共享性
- 一升级 skill 就覆盖环境配置
- 还可能泄露本地基础设施

这正是官方模板里强调“为什么要分开”的核心原因。

---

### 5.4 TOOLS.md vs openclaw.json / 权限配置
#### TOOLS.md 不是：
- 工具开关
- 权限系统
- 渠道路由配置
- 插件启停配置

你在 TOOLS 里写：
- “我有 browser”
- “可以访问 Feishu”
- “可以发 Telegram”

**不会让系统凭空获得这些能力。**

这类能力由：
- 配置文件
- 渠道登录
- 插件启用
- 鉴权状态
- 系统权限策略
决定。

这点必须写死，否则初学者最容易误解。

## 6）市面上最容易踩的坑
这一节专门回答你要的“比对容易踩的坑”。

### 坑 1：把 TOOLS.md 写成“我希望你怎么做”
这会和 AGENTS 冲突。

错误写法示例：
- 遇到任务先分析再执行
- 做完必须给证据
- 优先结构化输出

这些不属于 TOOLS，而属于 AGENTS / SOUL。

---

### 坑 2：把 TOOLS.md 写成“工具白名单/权限表”
这会和系统配置冲突。

错误写法示例：
- 允许使用 read / write / exec
- 禁止使用 browser

这些不是 TOOLS 决定的。  
结果是 agent 可能“以为自己可以/不可以”，造成更多混乱。

---

### 坑 3：把所有基础设施细节塞进 Skill
这是非常常见也非常危险的坏实践。

坏处：
- skill 无法复用
- 环境迁移困难
- 版本更新覆盖本地适配
- 共享时泄露主机/账号/路径

---

### 坑 4：只记设备名字，不记使用语义
例如只写：
- `server-a`
- `speaker-2`
- `cam-main`

但不写：
- 它们分别是干什么的
- 哪个场景该用哪个
- 哪个是测试环境，哪个是正式环境

结果：看似有文档，实际上仍然会误用。

---

### 坑 5：不维护，导致工具备注过时
最典型的后果：
- 主机 IP 改了
- 路径迁移了
- 设备换名了
- 默认输出目录换了

但 TOOLS 还是旧的。  
这会让 agent 进入“看起来很懂，实际总是失败”的状态。

---

## 7）应该写什么
一个高质量的 TOOLS.md，至少应覆盖以下 6 类内容。

### 模块一：对象映射
- 服务器别名
- 设备昵称
- 常用目录别名
- 输出目录与用途

### 模块二：场景说明
- 什么情况下用哪个对象
- 哪些对象是默认值
- 哪些对象不能误用

### 模块三：环境差异
- 测试 vs 正式
- 本机 vs 远程
- 不同 workspace 的路径区别

### 模块四：高频约定
- 文件应该放哪里
- 哪个路径是可分享的
- 哪个目录是仅本地内部使用

### 模块五：命名规范
- 内部常用代称是什么
- 用户常用说法与系统对象如何对应

### 模块六：提醒与禁忌
- 哪些对象高风险
- 哪些路径不能直接暴露
- 哪些设备/账号需要特别谨慎

## 8）使用技巧（Tips）

### 技巧 1：用“人话别名 → 系统对象”格式写
例如：
- `home-server → 192.168.1.100, user: admin`
- `客厅音箱 → Living Room HomePod（默认扬声器）`

这种格式比纯技术字段更利于 agent 理解。

### 技巧 2：每条都附带“用途”
例如：
- `media/outbound/ → 对外发送前必须复制到这里`

只有名称没有用途，是半成品。

### 技巧 3：把“高频错用点”单独列出
例如：
- 不要把截图发到内部缓存目录
- 不要把测试服务器当正式服务器
- 不要把 `workspace` 和 `~/.openclaw/` 混为一谈

### 技巧 4：少而精，不要写成资产总表
TOOLS.md 不是 CMDB。  
它只该保留**和 agent 任务最相关、最常误用、最能节省摩擦**的信息。

### 技巧 5：有变更就改，不要攒着改
因为 TOOLS.md 的价值在于“实时环境贴合”。

## 9）入门（最低可用）
> 目标：15 分钟写出一份最小 TOOLS.md，让 agent 至少能理解你的本地别名与高频对象。

### A）最低可用模板（复制即可）
```md
# TOOLS.md - 本地备注

## 常用路径
- outbound-media → ~/.openclaw/media/outbound/（对外发送前必须放这里）
- knowledge-root → ~/.openclaw/workspace/knowledge/

## 常用主机
- home-server → 192.168.1.100, user: admin（家庭/测试环境）

## 常用设备
- living-room-speaker → 客厅默认播放设备

## 高风险提醒
- 不要把正式文件写到临时目录
- 不要把测试环境当生产环境
```

### B）入门验收
- agent 是否开始更少问“这个路径/设备/主机是什么”？
- 是否减少明显的环境映射错误？

## 10）进阶（环境映射版）
> 目标：把 TOOLS.md 从“备注”升级为“环境映射协议”，支持多对象、多场景、多路径的稳定工作。

### A）进阶新增模块
1. **场景分类**：内容输出 / 服务器 / 媒体设备 / 外部账号 / 浏览器配置
2. **默认值**：默认用哪个，不要让 agent 每次猜
3. **风险区分**：正式/测试/只读/可写
4. **对外可见性**：哪些目录适合发送，哪些只适合本地使用

### B）进阶模板（复制即可）
```md
# TOOLS.md（进阶：环境映射版）

## 1. 输出相关
- outbound-media → ~/.openclaw/media/outbound/
  - 用途：对外分享文件前复制到这里
  - 注意：不要直接发送工作区内部路径

## 2. 知识库相关
- knowledge-root → ~/.openclaw/workspace/knowledge/
  - 用途：正式知识沉淀

## 3. 服务器相关
- home-server → 192.168.1.100, user: admin
  - 类型：测试环境
- prod-server → x.x.x.x, user: deploy
  - 类型：正式环境（高风险，谨慎）

## 4. 设备相关
- living-room-speaker → 客厅默认播放设备
- kitchen-speaker → 厨房备用设备

## 5. 高风险提醒
- 生产服务器必须二次确认
- 对外发送只能从 outbound-media 路径走
```

### C）进阶验收
- 不同场景下，agent 是否能自动区分“该用哪个对象”？
- 是否明显减少“测试/正式搞混”的风险？

---

## 11）最佳实践（卓越操作备注版）
> 目标：把 TOOLS.md 升级成“军团级环境接口文档”，既服务单只虾，也服务未来的多 Agent 协作与迁移。

### A）最佳实践新增机制
1. **对象命名标准化**：所有别名遵循统一命名规则。
2. **用途 + 风险 + 默认值 三联结构**：每个对象都清楚说明。
3. **版本变更记录**：重要环境变更要可追溯。
4. **迁移友好**：避免把敏感信息硬编码到可共享文档里。

### B）最佳实践模板（复制即可）
```md
# TOOLS.md（最佳实践：卓越操作备注版）

## Meta
- version: 1.0
- last_updated: YYYY-MM-DD
- scope: local-environment-mapping

## 1. 输出路径
- outbound-media → ~/.openclaw/media/outbound/
  - 用途：对外分享
  - 默认：是
  - 风险：低

## 2. 知识路径
- knowledge-root → ~/.openclaw/workspace/knowledge/
  - 用途：正式知识沉淀
  - 默认：是
  - 风险：中（内部路径，不直接外发）

## 3. 主机映射
- home-server → 192.168.1.100, user: admin
  - 用途：测试
  - 风险：中
- prod-server → x.x.x.x, user: deploy
  - 用途：正式
  - 风险：高（必须谨慎）

## 4. 设备映射
- living-room-speaker → 客厅默认扬声器
  - 用途：默认 TTS 输出
- kitchen-speaker → 厨房设备
  - 用途：备选

## 5. 禁忌
- 不要把 ~/.openclaw/credentials/ 写进这里
- 不要把密钥、密码、token 写进这里
- 不要把环境映射写进共享 skills

## 6. 变更记录
- YYYY-MM-DD：新增 prod-server 风险说明
```

### C）最佳实践验收
- 新训练的 agent 是否能在 1 天内熟悉你的环境对象？
- 多只虾是否能共享同一套环境映射词汇？
- 文档是否做到了“有用但不过度暴露”——既帮助工作，又不泄露敏感基础设施？

## 12）卓越案例：什么样的 TOOLS.md 才算业内最佳实践
这一节专门回答你要求的“出色卓越案例”。

### 案例标准 1：它让新虾上手时间显著下降
如果一份 TOOLS 写得足够好，新来的 agent 不需要你反复解释：
- 这个目录干嘛的
- 那个服务器能不能动
- 哪个路径可对外发送

### 案例标准 2：它能明显减少误操作
真正优秀的 TOOLS，不是越长越好，而是能显著减少：
- 发错位置
- 写错路径
- 连错主机
- 用错设备

### 案例标准 3：它和 Skills 清晰分层
业内最佳实践通常具备一个非常鲜明的特征：

> **Skills 讲方法，TOOLS 讲环境。**

如果一个系统把这两层混掉了，后期一定会出现：
- 迁移困难
- 共享困难
- 更新困难
- 泄露风险

### 案例标准 4：它可以被多个 agent 共同理解
卓越的 TOOLS，不是某一只虾的私有黑话，而是整个系统的公共接口注释。
