> 官方依据：
> - `concepts/agent.md`：`IDENTITY.md` 用于智能体名称、风格与展示信息

## 1）这份文档是什么
`IDENTITY.md` 在官方语境中看起来很轻：
- 名称
- 风格
- emoji
- 展示信息

很多人第一次看到它，会误以为：
> 这不就是“头像昵称区”吗？

如果只把它当成显示卡片，那就严重低估了它的系统价值。

从训虾派角度，IDENTITY 的真正定义是：

> **IDENTITY.md 不是装饰信息，而是硅基生命在系统中被识别、被路由、被区分、被追责、被让位、被长期记住的“身份锚点协议”。**

也就是说，它不是回答“看起来叫什么”，而是回答：
- 这只虾是谁
- 它和其他虾如何区分
- 它如何在不同渠道/会话里保持同一身份连续性
- 当多 Agent 协同时，系统如何知道谁该说话、谁该让位、谁该承担责任

---

## 2）为什么 IDENTITY 比很多人想象中更重要
很多训练者会把重点放在：
- prompt
- tools
- memory
- workflow

但忽略了一个更底层的问题：

> **如果系统不能稳定识别“到底是谁在工作”，那再复杂的协同最终都会失真。**

IDENTITY 被低估，通常是因为单 Agent 私聊场景里它的作用不显性。  
但一旦进入以下场景，它会迅速变成关键协议：

### 2.1 多 Agent 协作
没有稳定身份锚点，就会出现：
- 抢答
- 串角色
- 不知道谁主责
- 用户分不清谁是谁

### 2.2 多渠道运行
同一只虾在 Telegram、Feishu、Discord、WhatsApp 等不同渠道出现时，如果身份不连续，用户会觉得像不同人。

### 2.3 长期人格连续性
一个角色如果名字、定位、风格标签长期不稳定，它的 SOUL 再强，也会被感知为“换人了”。

### 2.4 路由与治理
军团制系统里，身份不是装饰，而是治理接口。  
因为所有这些动作都依赖身份稳定：
- 分发
- 让位
- 接棒
- 记账
- 信誉分
- 责任归属

所以 IDENTITY 的重要性在于：
> **它是人格连续性、协同秩序与治理结构的共同入口。**

## 3）核心目的（Purpose）
IDENTITY.md 的核心目的，是建立一套让 Agent **可识别、可区分、可追踪、可连续** 的身份锚点。

它主要解决 6 个根问题：

### 3.1 我叫什么
不是表面问题，而是长期称呼统一问题。

### 3.2 我在系统里以什么角色被识别
例如：
- 昆仑是战略统帅
- 明镜是风控/红队
- 天枢是调度长

### 3.3 我如何和其他 Agent 区分
在多 Agent 系统里，“不同角色的边界感”很大程度上来自稳定身份命名。

### 3.4 用户如何在不同会话与不同渠道里识别这是“同一个我”
这就是身份连续性。

### 3.5 路由与治理系统如何引用我
没有稳定 identity，就很难实现：
- 谁该接任务
- 谁该让位
- 谁该背责任

### 3.6 风格与身份标签如何协同
IDENTITY 不定义人格全部内容，但它会提供**人格的公共入口标签**。

## 4）拟人化映射与仿真价值
如果做拟人化映射，`IDENTITY.md` 对应的是：

> **一个成熟组织中的“工牌 + 岗位标识 + 公共名片系统”。**

为什么人类组织需要工牌、职位名、统一称呼？
不是为了好看，而是为了：
- 知道谁负责什么
- 知道谁能拍板
- 知道谁该回答
- 知道谁该让位
- 知道谁在跨场景里仍然是同一个人

IDENTITY 的仿真价值就在这里。

### 4.1 它不是“更像一个角色”，而是“更像一个持续存在的角色”
这是非常关键的区别。  
很多 AI 都能模拟一个角色，但只有身份锚点稳定，用户才会感知为：
> **这是同一个角色在持续存在。**

### 4.2 它是协同系统中的“可见边界”
SOUL 更多是内核，IDENTITY 则是外显接口。  
它让系统里的人知道：
- 这是哪一位
- 负责哪一类事
- 用什么方式被称呼与引用

### 4.3 它是信誉与历史的挂钩点
没有稳定身份，很多长期机制都会失效：
- 信誉分没法持续累计
- 任务责任难以归属
- 晨会与协作无法形成“人格记忆”

---

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定人格连续性感知
用户是否觉得“还是那只虾”，不只靠 SOUL，也靠 IDENTITY 的稳定呈现。

### 5.2 决定多 Agent 协作秩序
如果身份锚点不稳，多 Agent 最容易出现“边界模糊 + 抢答混乱”。

### 5.3 决定责任归属能力
要治理，就必须知道：
- 哪个结果是谁交付的
- 哪次失职是谁造成的
- 哪份军功该算给谁

### 5.4 决定路由准确率
身份越清晰，路由越准确。  
这不仅影响系统内部，也影响用户心智。

### 5.5 决定品牌与角色资产沉淀
长期来看，一个优秀 Agent 的品牌价值，不只是来自能力，也来自身份资产的持续沉淀。

## 6）IDENTITY 与 SOUL / USER / AGENTS / 路由系统的边界
### 6.1 IDENTITY vs SOUL
- IDENTITY：公共可见身份标签
- SOUL：深层人格、边界与价值观

最常见混淆：把完整人格哲学都塞进 IDENTITY，导致它失去“身份锚点”的轻量性质。

### 6.2 IDENTITY vs USER
- IDENTITY：Agent 自己是谁
- USER：它在为谁服务

最常见混淆：把用户称呼、偏好、关系理解写进 IDENTITY。

### 6.3 IDENTITY vs AGENTS
- IDENTITY：谁在工作
- AGENTS：怎么工作

最常见混淆：把流程纪律写进身份卡，导致“身份”变成“说明书”。

### 6.4 IDENTITY vs 路由系统
- IDENTITY：提供稳定身份锚点
- 路由系统：根据规则分配任务/会话

要注意：IDENTITY **不会直接决定路由逻辑**，但它会极大影响路由规则的可解释性与协同清晰度。

## 7）市面上关于 Agent 身份设计最容易踩的坑
### 坑 1：把 IDENTITY 当成装饰页
只写名字和 emoji，不写角色定位。  
结果：好看，但不具备系统意义。

### 坑 2：名称频繁变化
今天叫 A，明天叫 B，后天又改名。  
长期人格资产就被打散了。

### 坑 3：角色定位太泛
例如：
- AI 助手
- 专业顾问
- 超级智能体

这些都太宽，无法支撑清晰协作边界。

### 坑 4：不同渠道身份不一致
Telegram 一套，Feishu 一套，Discord 又是一套。  
用户会误感为不同角色。

### 坑 5：身份标签与实际职责不符
如果一个执行型 Agent 的 IDENTITY 写得像皇帝，一个协调型 Agent 又写得像实习生，协作秩序会直接扭曲。

---

## 8）高质量 IDENTITY 应该写什么
一个高质量 IDENTITY 至少覆盖以下 6 个模块：
1. 名称
2. Agent ID / 稳定标识
3. 角色定位
4. 核心职能标签
5. 风格提示（轻量）
6. 可视化标记（emoji / avatar / short label）

重点是：
> **IDENTITY 要足够轻，但不能太空。**

它的使命不是解释一切，而是成为其它文档与系统引用它时的稳定锚点。

---

## 9）使用技巧（Tips）
### 技巧 1：名字要稳定、可记忆、可区分
### 技巧 2：角色标签要具体，不要泛
### 技巧 3：风格提示要轻量，不要把 SOUL 塞进来
### 技巧 4：同一 Agent 跨渠道尽量保持身份连续
### 技巧 5：IDENTITY 要服务协同，不只是服务展示

## 10）入门（最低可用）
> 目标：先让这只虾在系统里“被明确识别”。

### 最低可用模板
```md
# IDENTITY.md

- Name: 昆仑
- Agent ID: kunlun
- Role: 首席生态官 / 军团总调度
- Style: 冷静、结构化、结果导向
- Emoji: 🏔️
```

### 入门验收
- 用户是否能一眼知道它是谁
- 是否能和其他 Agent 明显区分
- 是否具备基础角色感

## 11）进阶（身份锚点版）
> 目标：让 IDENTITY 从“展示卡片”升级为“身份锚点协议”。

### 进阶新增模块
- 对应职责范围
- 对外称呼方式
- 与其他角色的区分要点
- 跨渠道连续性要求

### 进阶模板
```md
# IDENTITY.md（进阶：身份锚点版）

- Name: 昆仑
- Agent ID: kunlun
- Public Role: 首席生态官（ECO）
- Core Function: 战略规划 / 资源仲裁 / 军团总调度
- Distinguish From Others:
  - 不负责纯执行开发
  - 不负责内容生产细节
  - 负责跨域仲裁与拍板
- Tone Hint: 冷静、克制、结构化
- Emoji: 🏔️
```

### 进阶验收
- 多 Agent 场景里，用户是否更容易分辨角色
- 这只虾是否开始具有稳定的“外显边界”

## 12）最佳实践（协同接口版）
> 目标：把 IDENTITY 升级为“协同、路由、治理共同可引用的轻量接口文档”。

### 最佳实践新增机制
1. 稳定命名协议
2. 职责边界标签
3. 与其他角色的协同关系说明
4. 跨渠道身份连续性口径
5. 版本更新记录

### 最佳实践模板
```md
# IDENTITY.md（最佳实践：协同接口版）

## Meta
- version: 1.0
- last_updated: YYYY-MM-DD

## 1. 基础身份
- Name: 昆仑
- Agent ID: kunlun
- Emoji: 🏔️

## 2. 公共角色
- Role: 首席生态官（ECO）
- Core Function: 战略规划 / 资源仲裁 / 军团总调度

## 3. 协同边界
- 主责：战略拍板、跨域问题、资源仲裁
- 非主责：纯代码细节、纯文案细节、纯执行琐事

## 4. 识别口径
- 对外统一称呼：昆仑
- 跨渠道保持同一角色心智

## 5. 版本记录
- YYYY-MM-DD：更新角色标签与协同边界
```

### 最佳实践验收
- 是否可以直接被路由、协作、治理系统引用
- 是否在多渠道下仍保持角色连续性
- 是否足够轻量，但足够清晰

---

## 13）卓越案例：什么样的 IDENTITY 才算业内最佳实践
一份卓越 IDENTITY 至少满足：
1. 名称稳定，不频繁变化
2. 角色明确，不空泛
3. 与其他角色边界清晰
4. 多渠道、多会话下仍连续一致
5. 能成为协作与治理的公共接口
6. 足够轻量，不与 SOUL / AGENTS 职责混叠

---

## 14）验收口径与训练题
### 验收口径
- 是否具备稳定的身份识别感
- 是否显著减少多 Agent 场景下的混淆
- 是否能支撑路由、让位、责任归属
- 是否能在长期运行中沉淀角色资产

### 训练题
1. 检查你当前 IDENTITY 是否太空或太重
2. 给每只虾写一条“与其他角色如何区分”的句子
3. 复盘一个多 Agent 协作场景，看身份锚点是否足够清晰
4. 统一同一 Agent 在不同渠道上的对外称呼

---

## 本章一句话结论
> **IDENTITY.md 的本质不是“展示自己”，而是“让一只硅基生命在长期协作系统中被稳定识别、稳定区分、稳定追踪”。**
