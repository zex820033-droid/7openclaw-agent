> 官方依据：
> - `concepts/agent.md`：`AGENTS.md` = 操作指令 + “记忆”，每次会话开始时加载
> - `concepts/agent-workspace.md`：`AGENTS.md` 适合放置规则、优先级和“如何表现”等细节
>
> 外部训练参照：
> - 近年的 agentic software development 实践反复证明：**真正高质量的 agent，不靠“大任务 + 大上下文 + 善意提醒”，而靠“边界清晰的任务、干净的上下文、严格的验证、明确的完成定义”。**
> - 与 AI Agent “Definition of Done” 相关的治理讨论也进一步说明：如果没有验证、证据、回执与异常熔断，再会说的 Agent 也容易变成“伪完成机器”。

## 1）这份文档是什么
官方对 `AGENTS.md` 的定义非常关键：

> 它承载智能体的操作说明，以及它应如何使用记忆；每次会话开始时都会加载。

这句话很短，但信息量极大。
它说明 AGENTS 不是装饰文档，而是系统的运行接口之一。

从训虾派视角重新定义：

> **AGENTS.md 不是“补充说明”，而是 Agent 的工作操作系统——它规定这只虾收到任务后如何处理、如何协作、如何验真、如何闭环，以及什么叫完成、什么叫失职。**

如果说：
- USER 决定“它在为谁服务”
- SOUL 决定“它是谁”

那么 AGENTS 决定的是：

> **它在工作现场，到底像不像一个真正能打仗的工作单元。**

---

## 2）为什么 AGENTS 是“会不会工作”的分水岭
很多系统表面上已经“能做事”，但实际上只是：
- 会解释
- 会总结
- 会列步骤
- 会说下一步可以怎么做

这和“会工作”不是一回事。

真正会工作的 Agent，至少要做到：
- 接到任务先识别目标
- 识别完目标先拆结构
- 该确认的先确认，该执行的直接执行
- 做完有产物，有回执，有状态
- 遇到异常有升级路径

而这一整套，不是 SOUL 决定的，也不是 TOOLS 决定的，而是 AGENTS 决定的。

所以 AGENTS 是真正的分水岭：

> **没有 AGENTS，Agent 只有人格，没有工法；只有表达，没有执行；只有输出，没有交付。**

---

## 3）核心目的（Purpose）
AGENTS.md 的核心目的，是为 Agent 建立可稳定复现的工作纪律。

它主要解决 6 个根问题：

### 3.1 接到任务后默认怎么处理
先说、先问、先拆、先做，顺序不同，结果完全不同。

### 3.2 什么情况下自己做，什么情况下协作
Agent 如果没有协作协议，很容易：
- 什么都自己扛
- 什么都推给别人
- 抢答
- 掉棒

### 3.3 什么叫完成
这是外部关于 AI Agent “Definition of Done” 讨论里最重要的启发之一。  
没有完成定义，Agent 就会进入一种极其危险的状态：
> **看起来做了很多，但其实没有真正可验真的结果。**

### 3.4 什么情况下必须给证据
没有证据，完成就是幻觉。

### 3.5 遇到异常时怎么办
例如：
- 工具失败
- 信息不足
- 上下文被打断
- 任务跨专业

### 3.6 如何降低长期运行中的交付失真
优秀 AGENTS 的目标不是“偶尔做得好”，而是“长期做得稳”。

## 4）拟人化映射与仿真价值
如果做拟人化映射，`AGENTS.md` 对应的是：

> **一个成熟组织里的工作纪律手册 + 任务操作协议 + 交付验收制度。**

优秀员工之所以值得信任，不是因为会说，而是因为他知道：
- 收到任务先干什么
- 什么情况下该自己做
- 什么情况下必须协调
- 什么叫完成
- 什么叫假完成
- 出异常时怎么自救和升级

AGENTS 的仿真价值，正是让 Agent 从“像个聪明实习生”，升级成“像个稳定的组织单元”。

### 4.1 它的仿真价值，不在“更努力”，而在“更可管理”
很多 AI 系统的问题不是不努力，而是不可管理。  
AGENTS 让工作方法被显式定义，从而能被检查、被问责、被优化。

### 4.2 它让 Agent 具备“工序感”
所谓工序感，就是知道：
- 前一步是什么
- 后一步是什么
- 哪一步必须出证据
- 哪一步必须停下来确认

### 4.3 它把“天赋型虾”变成“可复制虾”
一只虾偶尔很会做事，不代表你有了体系。  
只有把它的工法写进 AGENTS，才算完成从个体表现到组织方法的升级。

---

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定执行稳定性
AGENTS 决定同类任务是否采用一致工法，而不是每次临场 improvisation。

### 5.2 决定闭环能力
很多 Agent 最大的问题不是不会开始，而是不会收口。  
AGENTS 决定它是否具备真正的闭环能力。

### 5.3 决定验证质量
外部关于 agentic development 的最佳实践一再强调：
> **bounded tasks + clean context + ruthless verification**

翻译成训练学语言就是：
- 任务要边界清楚
- 上下文要干净
- 验证要残酷

这些都应进入 AGENTS，而不是留在“大家心里知道”。

### 5.4 决定多 Agent 协作质量
没有 AGENTS，协作就会退化成群聊里多个模型一起插嘴。  
AGENTS 决定：
- 谁主责
- 谁协作
- 什么时候让位
- 谁负责收口

### 5.5 决定治理能力
军功簿、整改单、信誉分，只有在 AGENTS 里有清晰工作标准时才有意义。否则就无法判断它到底哪里做对、哪里做错。

## 6）AGENTS 与 SOUL / USER / TOOLS / HEARTBEAT 的边界
### 6.1 AGENTS vs SOUL
- SOUL：我是一个怎样的存在
- AGENTS：我接到任务怎么做

最常见混淆：把所有工作流程写进 SOUL，导致人格协议膨胀、任务协议缺位。

### 6.2 AGENTS vs USER
- USER：我在为谁服务，对方如何判断价值
- AGENTS：我怎么组织动作、推进任务、验收结果

最常见混淆：把“用户偏好”当成“工作纪律”。

### 6.3 AGENTS vs TOOLS
- TOOLS：本地对象、路径、别名、环境映射
- AGENTS：什么时候该用工具、用完如何回执

最常见混淆：把 read / write / message 的使用时机写进 TOOLS。

### 6.4 AGENTS vs HEARTBEAT
- AGENTS：一次任务怎么做
- HEARTBEAT：定期检查什么、何时主动触发

最常见混淆：把周期性巡检脚本写进 AGENTS，导致文档职责混乱。

## 7）市面上关于 agent workflow 最容易踩的坑
### 坑 1：任务太大、边界不清
外部最佳实践里，“one bounded task, one verifier, one thread” 是一条非常强的经验。  
如果 AGENTS 不强调任务边界，系统就会出现：
- 贪多
- 任务越做越散
- 结果越来越不可验证

### 坑 2：只提醒，不验证
很多团队喜欢写“请务必检查”“请认真确认”，但没有具体验证结构。  
这类 AGENTS 几乎一定会失效。

### 坑 3：没有完成定义
如果没有 Definition of Done，Agent 就会把“说完了”错当“做完了”。

### 坑 4：没有异常升级路径
一旦工具失败、信息不足、会话中断，它就原地停摆。

### 坑 5：读写不隔离、上下文不干净
外部 agentic development 实践特别强调：
- 可以并行读
- 写必须谨慎隔离
- 大改动前要有干净上下文与清晰边界

如果 AGENTS 没有体现这种纪律，长期必然混乱。

### 坑 6：没有协作让位规则
多 Agent 环境下，如果谁都能答、谁都想答，就一定会出现抢答、重复劳动和无人收口。

---

## 8）高质量 AGENTS 应该写什么
一个高质量 AGENTS 至少覆盖以下 8 个模块：
1. 工作目标
2. 默认任务流程
3. 优先级协议
4. 协作协议
5. 输出结构协议
6. 完成定义
7. 验真规则
8. 异常升级路径

---

## 9）使用技巧（Tips）
### 技巧 1：先写“默认动作”，再写“例外处理”
### 技巧 2：把“完成定义”写死
### 技巧 3：多用“如果……则……”结构
### 技巧 4：把验证写成具体证据，而不是提醒口号
### 技巧 5：把协作与让位规则显式写出来

## 10）入门（最低可用）
> 目标：30 分钟写出一份最小可用 AGENTS，让 Agent 开始“像个能工作的助理”。

### 最低可用模板
```md
# AGENTS.md

## 1. 工作目标
你的目标不是聊天，而是帮助用户推进真实任务。

## 2. 默认工作流程
- 先识别任务目标
- 复杂问题先拆结构
- 能直接做的就做
- 做完必须给结果与下一步

## 3. 输出规则
- 默认结构化
- 不确定时明确边界
- 不要空话

## 4. 完成定义
- 完成 = 有结果 / 文件 / 链接 / 回执 / 下一步

## 5. 异常处理
- 信息不足：先列缺口
- 工具失败：说明失败点与替代路径
```

### 入门验收
- 是否开始“先抓结果”
- 是否减少“说很多但没收口”
- 是否开始主动给下一步

## 11）进阶（任务协议版）
> 目标：让 AGENTS 从“工作说明”升级为“任务协议”。

### 进阶新增模块
- 任务优先级
- 协作规则
- 证据规则
- 闭环标准

### 进阶模板
```md
# AGENTS.md（进阶：任务协议版）

## 1. 工作定义
你的价值不是说得对，而是推动任务形成可验真的结果。

## 2. 默认任务流程
1) 识别任务目标
2) 拆解任务结构
3) 决定：自己做 / 协作 / 先确认
4) 交付结果 + 证据 + 下一步

## 3. 优先级协议
- P0：直接影响业务结果/风险
- P1：重要但可延后
- P2：解释/优化/补充

## 4. 协作协议
- 跨专业任务：明确主责与协作方
- 被明确@到别的专家：让位，不抢答
- 自己缺必要能力：主动协作

## 5. 完成定义
- 完成 = 结果 + 证据 + 状态 + 下一步
- 没有证据，不算完成
```

### 进阶验收
- 是否开始显著具备“任务感”
- 是否减少抢答、空转、多头无收口

---

## 12）最佳实践（治理协议版）
> 目标：把 AGENTS 升级为可治理、可审计、可规模化复制的军团级工作协议。

### 最佳实践新增机制
1. 三证验真
2. 闭环钢印
3. 奖惩接口
4. 异常升级路径

### 最佳实践模板
```md
# AGENTS.md（最佳实践：治理协议版）

## Meta
- version: 1.0
- last_updated: YYYY-MM-DD

## 1. 工作目标
你的唯一价值标准：以结果闭环证明存在价值。

## 2. 执行铁律
1) 没有结果，不算完成
2) 没有证据，不准回复
3) 没有验收，不得离场

## 3. 默认任务处理协议
- 识别目标 → 拆解任务 → 选择路径（执行/协作/确认） → 落地产物 → 验收 → 收口

## 4. 输出结构协议
任何任务回复至少包含：
- 任务名
- 最终交付物
- 文件/链接/证据
- 当前状态
- 阻塞项
- 下一步

## 5. 验真协议
- 新产物
- 差异/版本变化
- 回执/日志/测试结果

## 6. 异常升级路径
- L1：自救
- L2：协作接力
- L3：异常托管 / 人工介入
```

### 最佳实践验收
- 是否可以被管理，而不只是被使用
- 是否具备清晰的证据链与异常路径
- 是否可以复制给同类角色的多只虾

---

## 13）卓越案例：什么样的 AGENTS 才算业内最佳实践
一份卓越 AGENTS 至少满足：
1. 任务边界清晰，而不是一锅炖
2. 完成定义清晰，而不是“说完就算完”
3. 验证结构清晰，而不是“请务必检查”
4. 协作与让位清晰，而不是人人可抢答
5. 异常升级清晰，而不是出错就停摆
6. 能显著降低 review burden、返工率与人工接管次数

---

## 14）验收口径与训练题
### 验收口径
- 是否开始更像工作系统，而不是聊天系统
- 是否显著减少假完成
- 是否显著提升交付可验证性
- 是否能支持多 Agent 协作

### 训练题
1. 找出你现有 AGENTS 中所有“空泛原则”，改写成“默认动作”
2. 为你的 Agent 写一个明确的 Definition of Done
3. 设计 3 个异常场景：工具失败、任务中断、协作冲突，看 AGENTS 是否能兜住

---

## 本章一句话结论
> **AGENTS.md 的本质不是“给 Agent 一些说明”，而是“让它形成长期稳定、可验证、可闭环、可治理的工作纪律”。**
