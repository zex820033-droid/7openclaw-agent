## 0）卷二一句话总论

## 1）七大生命协议拓扑图（1 页总图）

把卷二的 7 份核心文档视作“器官系统”，它们各司其职，彼此耦合：

```
            ┌──────────────────────────────┐
            │           USER.md            │
            │  服务对象模型 / 决策偏好 / DoD │
            └───────────────┬──────────────┘
                            │（价值对齐）
                            ▼
┌──────────────┐   ┌──────────────────────┐   ┌──────────────┐
│  TOOLS.md     │   │       AGENTS.md      │   │ HEARTBEAT.md  │
│ 环境映射/别名 │◀──│ 任务工法/闭环/验真   │──▶│ 主动节律/熔断 │
└──────┬───────┘   └──────────┬───────────┘   └──────┬───────┘
       │（环境贴合）           │（执行秩序）            │（关注质量）
       ▼                        ▼                        ▼
┌────────────────────────────────────────────────────────────┐
│                          MEMORY 体系                         │
│ memory/YYYY-MM-DD.md（事件层/仅追加） + MEMORY.md（精选长期） │
│ + memory_search/memory_get（检索层） + memory flush（刷新层） │
└────────────────────────────────────────────────────────────┘
                            ▲
                            │（人格连续性 + 边界稳定性）
                            │
                   ┌────────┴────────┐
                   │     SOUL.md      │
                   │ 人格/边界/冲突默认│
                   └────────┬────────┘
                            │（外显锚点/协同秩序）
                            ▼
                   ┌──────────────────┐
                   │   IDENTITY.md     │
                   │ 身份锚点/识别/追责 │
                   └──────────────────┘
```

> 这张图的要点：
> - **SOUL 负责“我是谁”，USER 负责“我为谁服务”。**
> - **AGENTS 负责“我怎么工作”，HEARTBEAT 负责“我何时主动”。**
> - **TOOLS 负责“我怎么贴合环境”，IDENTITY 负责“我如何被稳定识别”。**
> - **MEMORY 负责“我如何跨时间连续存在与进化”。**

## 2）七大文档的“训练对象”与“常见失真”对照表

| 文档 | 训练对象（你在训什么） | 解决的问题 | 最常见失真（你会看到什么坏症状） |
|---|---|---|---|
| USER.md | 服务对象模型（偏好/决策/完成定义） | 输出对齐用户价值 | 回答正确但不“对你有用”；反复试探风格；没有 DoD |
| SOUL.md | 人格内核 + 边界 + 冲突默认 | 长期身份稳定 | 角色漂移、风格漂移、越界承诺、上下文依附 |
| AGENTS.md | 工法/纪律/闭环/验真 | 从会说到会做 | 伪完成、无证据完成、无收口、协作抢答、异常停摆 |
| TOOLS.md | 环境映射（别名/路径/设备/约定） | 工具最后一公里 | 发错目录/连错主机/混淆测试与生产；把 TOOLS 写成权限表 |
| HEARTBEAT.md | 主动性节律 + 静默 + 熔断 | 关注质量与可控主动性 | 通知疲劳、循环提醒、资源耗尽、伪主动（只报不判） |
| IDENTITY.md | 身份锚点/协同识别/追责 | 多 Agent 秩序 | 串角色/抢答/路由不清/责任不清/长期角色资产无法沉淀 |
| MEMORY 体系 | 连续性/沉淀/检索/进化 | 跨会话稳定表现 | 会话失忆、决策反复、偏好反复踩雷、记忆污染、检索无效 |

## 3）卷二的“协同机制”怎么跑起来（生命周期）

### 3.1 建立期（第 0 天）——先立三根桩
1. **USER.md**：写清“决策偏好 + 禁忌 + Definition of Done（完成定义）”。
2. **SOUL.md**：写清“人格条款 + 边界条款 + 冲突默认处理”。
3. **AGENTS.md**：写清“默认任务流程 + 证据链 + 完成定义 + 异常升级路径”。

> 没有这三根桩，后面 TOOLS/HEARTBEAT/MEMORY 都会失真。

### 3.2 运行期（第 1-7 天）——让系统开始“像个团队”
- **TOOLS.md**：把你现场的“人话别名 → 系统对象”映射写清（路径/主机/设备/对外目录）。
- **IDENTITY.md**：把角色锚点稳定住，明确区分“我是谁/我不是什么”。
- **MEMORY（daily）**：每天只追加关键事件/关键决策，不要把规则塞进日志。

### 3.3 进化期（第 2 周起）——把经验升格成长期资产
- **MEMORY.md**：只收录“稳定偏好/关键决策/长期约束/已验证结论”。
- **memory_search/memory_get**：把“能找回来”作为记忆写法的硬标准。

### 3.4 主动期（底盘稳定后）——再开 HEARTBEAT
- **HEARTBEAT.md**：先写静默条件与熔断条件，再写触发条件。
- 先盯 P0 变量，别扫全世界。

## 4）定位问题的“快速诊断法”（从症状反推该改哪份协议）

### 4.1 症状：输出风格忽冷忽热、今天像顾问明天像客服
- 首查：SOUL.md
- 次查：IDENTITY.md（角色锚点是否稳定）

### 4.2 症状：说得很好但永远不落地，缺文件/缺回执/缺验收
- 首查：AGENTS.md（完成定义、证据链、输出结构）
- 次查：USER.md（DoD 是否要求“产物+验收+下一步”）

### 4.3 症状：工具总用错，路径/别名/对外目录总混
- 首查：TOOLS.md
- 次查：AGENTS.md（何时调用工具、如何回执）

### 4.4 症状：主动提醒太吵/重复提醒/像骚扰
- 首查：HEARTBEAT.md（静默条件、冷却、熔断）

### 4.5 症状：反复解释同一件事，隔天就忘、压缩后就忘
- 首查：MEMORY 体系（daily 是否落盘、MEMORY.md 是否精选）
- 次查：USER/SOUL 是否把“稳定条款”写成协议

## 5）卷二“出书级演示方案”（你说要重点展示）

建议你演示时按 3 段走，每段 5-8 分钟：

### 演示 A：从“prompt 驱动”到“协议驱动”（对比演示）
- 同一问题：只靠 prompt vs 有 USER/SOUL/AGENTS
- 展示差异：
  - 是否给 DoD
  - 是否给证据链
  - 是否能收口

### 演示 B：多 Agent 协同为什么必须有 IDENTITY + AGENTS
- 展示“身份不清”如何导致抢答与责任不清
- 展示“身份锚点 + 让位规则 + 主责收口”如何让协作变秩序

### 演示 C：记忆系统如何让“跨天连续性”成立
- 展示：
  - daily 事件层写了什么
  - long-term 精选层写了什么
  - memory_search 如何把关键规则找回来
