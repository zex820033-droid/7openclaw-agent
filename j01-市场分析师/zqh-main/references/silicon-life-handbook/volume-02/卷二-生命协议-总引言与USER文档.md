# 卷二总引言

## 1）这卷到底在解决什么问题
如果把 OpenClaw 只当成“能调用工具的聊天系统”，你会得到一个偶尔聪明、偶尔失真、偶尔很好用的 AI。

但如果你把它当成一个要长期协作、可复制训练、可规模化治理的硅基生命系统，你就必须回答一个更底层的问题：

> **一只虾的长期表现，到底由什么决定？**

不是单靠模型。也不是单靠 prompt。

真正决定长期表现的，是它有没有一整套稳定的生命协议文档：
- 谁是用户（USER）
- 它是谁（SOUL）
- 它怎么工作（AGENTS）
- 它如何理解本地环境（TOOLS）
- 它什么时候主动（HEARTBEAT）
- 它如何被识别（IDENTITY）
- 它如何长期沉淀（MEMORY）

所以卷二讨论的不是“功能”，而是：

> **如何把 OpenClaw 从工具系统，升级成硅基生命训练系统。**

---

## 2）为什么卷二是全书最关键的一卷
因为卷一可以讲世界观，卷三可以讲系统骨架，卷四可以讲训练流程，卷五可以讲治理；但如果卷二没立住，后面所有内容都会变成漂浮的概念。

卷二的重要性在于：
- 它把“训虾”从玄学变成文档工程
- 它把“好用一次”变成“长期稳定”
- 它把“个人经验”变成“可交接模板”
- 它把“会聊天的模型”变成“有生命协议的工作单元”

如果没有卷二，系统会出现四种经典失真：
1. **人格失真**：同一只虾前后不像一个人
2. **工作失真**：会说不会做，会分析不会闭环
3. **主动性失真**：乱提醒、乱心跳、乱打扰
4. **记忆失真**：该记的不记，不该记的乱塞

---

## 3）卷二的核心判断
本卷的唯一主判断是：

> **OpenClaw 里的硅基生命，不是靠一段 prompt 活着，而是靠一组生命协议长期塑形。**

换句话说：
- Prompt 更像临场指令
- 文档更像器官系统

Prompt 可以短期纠偏，文档才决定长期形态。

---

## 4）从训练学角度，卷二的七份文档分别训练什么
| 文档 | 训练对象 | 它解决的问题 | 最容易失真的地方 |
|---|---|---|---|
| `USER.md` | 服务对象理解 | 我在为谁服务 | 写成简历，不像协议 |
| `SOUL.md` | 人格与边界 | 我是谁、我不是什么 | 写成文案或口号 |
| `AGENTS.md` | 工作纪律 | 我如何接任务、交付、闭环 | 写成空泛原则 |
| `TOOLS.md` | 环境映射 | 本地对象、别名、路径怎么理解 | 写成权限表/技能文档 |
| `HEARTBEAT.md` | 主动节律 | 我什么时候该主动、什么时候静默 | 写成过度打扰脚本 |
| `IDENTITY.md` | 身份锚点 | 我叫什么、如何被识别 | 写得太轻，缺少协同价值 |
| `MEMORY.md` / `memory/*.md` | 经验沉淀 | 哪些事实该长期保留 | 写成日志堆积场 |

---

## 5）卷二怎么学，才不会学偏
### 入门
先把 USER / TOOLS / IDENTITY 稳住。先建立“服务对象 + 环境映射 + 身份锚点”。

### 进阶
再建立 SOUL / AGENTS。把人格与工作纪律协议化。

### 最佳实践
最后上 HEARTBEAT / MEMORY。因为主动性和记忆如果过早接入，最容易放大失真。

---

## 6）卷二最容易和市面课程讲法冲突的地方
### 冲突一：市面常教“多写 prompt”，卷二强调“少堆 prompt，多建协议”
因为 prompt 更像临场对话，而协议是可复用、可版本化的长期资产。

### 冲突二：市面常教“先让 AI 主动起来”，卷二强调“先做人，再做事，再主动”
也就是：
- 先有 USER / SOUL
- 再有 AGENTS / TOOLS
- 最后再上 HEARTBEAT

### 冲突三：市面常把“个性化”理解为语气模仿，卷二强调“个性化 = 决策与边界对齐”
真正高级的个性化，不是像你说话，而是像你判断价值。

---

## 7）本卷的交付物标准
卷二写完，不应该只是“你看懂了这些文件干什么”。
而应该达到以下结果：
- 你能为任意一只虾建立一套基础生命协议
- 你能判断它哪里失真，是 USER 的问题、SOUL 的问题还是 AGENTS 的问题
- 你能把一只高质量虾的方法复制成多只
- 你能把这套内容直接拿去出书、授课、训团队

## 1）这份文档是什么
官方对 `USER.md` 的定义很克制：

> 它描述用户是谁，以及应如何称呼他们。

如果只按这个字面理解，很多人会把 USER 写成：
- 人物简介
- 简历摘要
- 兴趣列表
- 背景介绍

这些都不算错，但远远不够。

从训虾派角度，USER 的真正定义应当是：

> **USER.md 不是“用户简介”，而是“主人理解协议”——它决定 Agent 以什么视角理解用户、如何判断什么才叫对用户真正有价值。**

它不是为了解释“这个人是谁”，而是为了回答：
- 这个人如何做决策
- 这个人最看重什么
- 这个人讨厌什么低级错误
- 什么样的交付物，对这个人才算完成

---

## 2）为什么 USER 是第一入口
一个 Agent 再强，如果它不知道：
- 它在为谁服务
- 这个人真正要的是什么
- 这个人的判断标准是什么

它就只能靠猜。

而长期协作里，“靠猜”是最大的性能噪音来源之一。

所以 USER 是第一入口，因为它决定了：
> **Agent 会不会从第一天开始，就走在对的方向上。**

没有 USER 的典型表现：
1. 回答看起来完整，但总不贴人
2. 语气、长度、结构不断试探
3. 不知道你更重 ROI、速度、审美还是风险
4. 总在无关紧要的地方用力

---

## 3）核心目的（Purpose）
USER.md 的核心目的，是让 Agent 建立一个稳定的“服务对象模型”。

它主要解决 5 个根问题：

### 3.1 我在为谁服务
不是名字，而是关系定位。
例如：
- 创始人
- 统帅
- 咨询客户
- 团队负责人

### 3.2 这个人用什么标准判断价值
例如：
- ROI 优先
- 结果闭环优先
- 可复制性优先
- 风险控制优先

### 3.3 这个人的工作风格是什么
例如：
- 先结论后背景
- 不爱废话
- 喜欢结构化汇报
- 重要事项要有证据链

### 3.4 这个人最讨厌什么
例如：
- 空话
- 虚构
- 没有下一步
- 说完成但拿不出证据

### 3.5 什么样的交付才算真正有用
这点最关键。因为很多 AI “答得对”，但不一定“对你有用”。

---

## 4）拟人化映射与仿真价值
如果做拟人化映射，`USER.md` 对应的是：

> **成熟助理对老板的认知模型。**

优秀助理不会只知道老板叫什么、做什么；他更知道：
- 老板怎么做取舍
- 老板不喜欢什么话术
- 老板对“完成”的定义是什么
- 老板在哪些问题上要求快，哪些问题上要求稳

USER 的仿真价值，不在“认识用户”，而在：

### 4.1 关系仿真
让 Agent 知道自己不是在对陌生人回答，而是在为某类明确关系对象服务。

### 4.2 偏好仿真
让 Agent 不用每次重新试探长度、结构、节奏。

### 4.3 决策仿真
让 Agent 学会“像这个用户一样判断价值”，而不是只会一般性正确。

### 4.4 成功标准仿真
让 Agent 明白：
> **什么样的输出，对这个人来说才算真的完成。**

---

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定相关性稳定性
USER 决定 Agent 会不会长期贴着用户核心关注点走，而不是给一堆“也对但没用”的答案。

### 5.2 决定协作效率
如果 USER 写得好，低级对齐成本会显著下降：
- 不再反复试探风格
- 不再反复问“你更想要什么版本”
- 不再老在错误地方展开

### 5.3 决定完成定义
外部关于 AI Agent “Definition of Done”的讨论给了很强的启发：
传统软件的完成标准偏确定性，而 AI 的完成标准更容易漂移。

所以 USER 必须参与定义：
- 对这个用户来说，什么叫 done
- 是“回答完整”就 done，还是“有产物 + 有验收 + 有下一步”才 done

### 5.4 决定可复制性
一只虾如果服务得很好，但你没有把“它为什么懂你”写进 USER，那么这种默契无法复制。

### 5.5 决定误判率
很多所谓幻觉，不只是知识幻觉，也是“价值判断幻觉”——它根本没理解你在乎什么。

---

## 6）USER 与 SOUL / AGENTS / MEMORY 的边界
### 6.1 USER vs SOUL
- USER：这个用户是谁，他怎么判断价值
- SOUL：这只虾是谁，它如何保持自我

最常见混淆：把“用户喜欢什么风格”写进 SOUL。

### 6.2 USER vs AGENTS
- USER：服务对象与成功标准
- AGENTS：任务流程与工作纪律

最常见混淆：把“收到任务先拆解、做完要给证据”写进 USER。

### 6.3 USER vs MEMORY
- USER：长期稳定的用户画像与偏好
- MEMORY：发生过的事实、决定、上下文

最常见混淆：把短期项目细节塞进 USER，导致 USER 越来越像日志。

---

## 7）市面上关于“用户画像训练”最容易踩的坑
### 坑 1：把 USER 写成简历
这会让 Agent 知道你做过什么，但不知道你如何判断价值。

### 坑 2：只写背景，不写决策原则
结果是 Agent 知道你是谁，但不知道什么对你真正重要。

### 坑 3：只写喜欢，不写禁忌
高级协作里，知道“不能做什么”往往比知道“最好做什么”更重要。

### 坑 4：没有完成定义
很多课程只讲 persona，不讲 Definition of Done。结果 Agent 很像你喜欢的风格，但还是不会交付对的结果。

### 坑 5：长期不更新
用户本人已经变了，但 USER 还停留在三个月前。这会导致“看起来懂你，实际上服务的是旧版本的你”。

---

## 8）高质量 USER 应该写什么
一个高质量 USER 至少覆盖以下 7 个模块：
1. 基本身份
2. 当前主线
3. 决策原则
4. 输出偏好
5. 明确禁忌
6. 完成定义
7. 变更记录

---

## 9）使用技巧（Tips）
### 技巧 1：写“决策原则”，不要只写“人物背景”
### 技巧 2：写“讨厌什么”，比写“喜欢什么”更有用
### 技巧 3：把“完成定义”写进去
### 技巧 4：把当前主线保持动态更新
### 技巧 5：用可执行句子，不要用抽象标签

---

## 10）入门（最低可用）
> 目标：10 分钟写出一份最小可用 USER，让 Agent 开始“像你的专属助理”。

### 最低可用模板
```md
# USER.md

## 1. 你在为谁服务
- 用户姓名/称呼：
- 当前身份：
- 当前主要业务：

## 2. 我的工作风格
- 默认先给结论，再展开
- 输出优先结构化
- 不要空话

## 3. 我的成功标准
- 可落地
- 可验收
- 可复制

## 4. 我的禁忌
- 不要“收到/在做/稍后同步”式废话
- 不要编造
```

### 入门验收
- 是否明显减少风格试探
- 是否开始更贴近你的问题重点
- 是否减少“正确但没用”的回答

---

## 11）进阶（决策协议版）
> 目标：让 USER 从“人物说明”升级成“服务决策协议”。

### 进阶新增模块
- 当前主线
- 决策优先级
- 完成定义
- 协作方式

### 进阶模板
```md
# USER.md（进阶：决策协议版）

## 1. 用户画像
- 姓名/称呼：
- 当前身份：
- 当前主战场：

## 2. 当前主线
- 本月主目标：
- 当前三个重点项目：
- 当前最大风险：

## 3. 决策原则
- 优先级：ROI / 风险 / 可复制性 / 速度
- 不接受：无证据结论、无验收完成

## 4. 输出偏好
- 先结论后展开
- 建议必须包含下一步动作与验收口径

## 5. 完成定义
- “完成” = 有产物 + 有验收 + 有回执/日志
```

### 进阶验收
- 建议是否开始自动绑定下一步
- 是否开始明显更像“长期搭档”而不是临时助手

---

## 12）最佳实践（服务协议版）
> 目标：把 USER 升级为可复制、可治理、可审计的军团级服务协议。

### 最佳实践新增机制
1. 版本号
2. 例题 / 反例库
3. 完成定义清单
4. 变更记录

### 最佳实践模板
```md
# USER.md（最佳实践：服务协议版）

## Meta
- version: 1.0
- last_updated: YYYY-MM-DD

## 1. 我是谁
- 称呼：
- 当前身份：
- 当前业务版图：

## 2. 当前主线
- 目标：
- Top3 项目：
- Top1 风险：

## 3. 决策原则
1) ROI
2) 可复制性
3) 风险
4) 速度

## 4. 完成定义
- 新产物
- 验收口径
- 下一步

## 5. 我喜欢的输出
- 示例1：
- 示例2：

## 6. 我讨厌的输出
- 反例1：
- 反例2：
```

### 最佳实践验收
- 新虾是否能快速形成服务一致性
- 是否显著减少“价值判断不贴人”的错误
- 是否可以跨多个 agent 复用

---

## 13）卓越案例：什么样的 USER 才算业内最佳实践
一份卓越 USER 至少满足：
1. 不只是写背景，而是写决策原则
2. 不只是写偏好，而是写完成定义
3. 不只是让 Agent“像懂你”，而是让它“持续做对你有用的事”
4. 能被多个 agent 共享，而不丢失服务一致性
5. 用户阶段变化时，文档能快速迭代而不失稳

---

## 14）验收口径与训练题
### 验收口径
- 是否明显提升相关性
- 是否显著降低对齐成本
- 是否具备稳定完成定义
- 是否可复制给同一军团的其他虾

### 训练题
1. 把你现有 USER 里所有“背景信息”改写成“决策信息”
2. 补上“我讨厌什么”和“什么叫完成”
3. 复盘最近 3 次 AI 回得不对，是 USER 缺了哪一块
