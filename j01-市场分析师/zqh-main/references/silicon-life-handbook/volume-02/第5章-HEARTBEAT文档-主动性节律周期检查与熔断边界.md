# 第5章｜HEARTBEAT 文档：主动性节律、周期检查与熔断边界
## ——为什么 HEARTBEAT 不是“让 Agent 更勤快”，而是让硅基生命拥有可控主动性的节律协议

> 官方依据：
> - `concepts/agent-workspace.md`：`HEARTBEAT.md` 是**可选的心跳运行微型检查清单**，应保持简短，避免消耗过多 token

> 外部训练参照：

> - 与 AI Agent “Definition of Done” 及 circuit breaker 相关的最佳实践显示：任何周期性主动机制，都必须同时具备**触发条件、静默条件、终止条件、异常熔断条件**。

## 1）这份文档是什么
官方对 `HEARTBEAT.md` 的定义很短，但非常精准：

> **可选的心跳运行微型检查清单。保持简短，以避免消耗过多 token。**

这句话里至少有四个关键词：
- **可选**
- **心跳运行**
- **微型检查清单**
- **保持简短**

它说明 HEARTBEAT 不是：
- 第二份 AGENTS
- 第二份 SOUL
- 永不停歇的主动任务引擎
- 想到什么就执行什么的自动驾驶许可

从训虾派角度，HEARTBEAT 的真正定义应当是：

> **HEARTBEAT.md 不是“让 Agent 更积极”，而是“为主动性建立节律、边界与熔断条件”的轻量协议。**

它解决的不是“会不会主动”，而是：
- 什么时候该主动
- 什么时候绝对不该主动
- 主动到什么程度就够了
- 没有异常时是否应保持静默
- 一旦陷入循环，在哪里立即熔断

## 2）为什么 HEARTBEAT 是最容易写坏的一章
卷二所有文档都容易被写偏，但 HEARTBEAT 是最容易一写就失真的。

为什么？因为主动性是最诱人的能力之一。
很多人一上来就想要：
- 自动提醒
- 自动跟进
- 自动扫描
- 自动发现问题
- 自动推进任务

听起来都对，但问题在于：

> **主动性如果没有节律与边界，几乎一定会从“有帮助”滑向“有干扰”，再滑向“有风险”。**

HEARTBEAT 写坏后，最常见的失真有 5 类：

### 2.1 通知疲劳
Agent 什么都提醒，最后用户对所有提醒都失去敏感度。

### 2.2 循环执行
外部关于 autonomous agents 的安全讨论已经明确指出：错误的重试与反思机制，会把 agent 拖进“semantic infinite loop”。

### 2.3 伪主动
看上去很勤奋，实际上只是在重复扫描、重复汇报、重复确认，没有推进真正结果。

### 2.4 权限越界
因为“想主动帮忙”，开始做本不该自动做的动作。

### 2.5 资源耗尽
心跳如果不控频、不控范围、不设熔断，最终烧掉的不是 token，而是整个系统的注意力预算。

所以 HEARTBEAT 这一章必须比别的章更谨慎。

## 3）核心目的（Purpose）
HEARTBEAT.md 的核心目的，是为 Agent 建立一套**可控、可预测、可静默、可熔断**的主动性节律。

它主要解决 6 个根问题：

### 3.1 什么情况下需要周期检查
不是所有任务都需要心跳。只有那些：
- 会变化
- 会超时
- 会掉棒
- 会积压
- 会失联
的对象，才值得进入心跳清单。

### 3.2 检查什么
高质量 HEARTBEAT 不会写“定期看看有没有事”。
它只检查极少数关键对象，例如：
- 有无阻塞
- 有无掉棒
- 有无关键告警
- 有无必须汇报的新结果

### 3.3 检查后做什么
心跳不是为了“看见”，而是为了“判断是否值得动作”。

### 3.4 什么情况下必须静默
这是 HEARTBEAT 的生命线。  
如果没有明显新情况、没有关键异常、没有用户真正需要知道的变化，**最佳动作往往是静默**。

### 3.5 什么情况下必须熔断
比如：
- 同一问题重复提醒 3 次无新增信息
- 连续重试但无新证据
- 同一异常已经在升级路径中

### 3.6 如何避免把主动性变成骚扰性
这就是 HEARTBEAT 的最高难点：
> **让系统有节律地关心，而不是无节制地打扰。**

## 4）拟人化映射与仿真价值
如果做拟人化映射，`HEARTBEAT.md` 对应的是：

> **一个成熟助理/参谋的“周期巡检习惯 + 报告节律 + 不越界提醒规则”。**

优秀助理不会：
- 每 10 分钟跑来问一次“还有事吗”
- 没有新增信息也不断同步
- 遇到一点波动就大惊小怪

真正成熟的助理会：
- 只盯关键变量
- 只在需要时打扰
- 没有新增就保持静默
- 一旦发现异常，快速判断是自己处理、升级处理，还是仅做记录

HEARTBEAT 的仿真价值就在这里。

### 4.1 它不是“更像有生命”，而是“更像有节律”
很多人误以为主动性 = 生命感。  
其实更准确地说：
> **节律感才是高质量生命感。**

### 4.2 它是“关注能力”的协议化
没有 HEARTBEAT，Agent 常常只能被动响应。  
有 HEARTBEAT 后，Agent 才开始像一个真正会持续盯盘、盯事、盯结果的系统。

### 4.3 它决定的是“关注的质量”，不是“动作的数量”
坏 HEARTBEAT 用更多动作证明存在感。  
好 HEARTBEAT 用更高质量的关注证明价值。

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定主动性是否可控
没有 HEARTBEAT，系统主动性弱；HEARTBEAT 写坏，系统主动性失控。  
所以 HEARTBEAT 决定的不是有没有主动性，而是主动性是否可控。

### 5.2 决定长期关注质量
长期运行中，真正有价值的不是“随时在线”，而是“关键时刻没漏掉”。

### 5.3 决定通知质量
通知的价值不在频率，在信噪比。  
HEARTBEAT 决定提醒是否：
- 有新增
- 有变化
- 有价值
- 有下一步

### 5.4 决定资源稳定性
外部关于 agent loop / resource exhaustion 的讨论已经说明：  
没有熔断与边界的主动机制，很容易把系统拖入：
- 重复执行
- 连续反思
- 无限重试
- 预算耗尽

### 5.5 决定系统可信度
如果 Agent 常常乱提醒、重复提醒、无效提醒，用户很快就会关闭这套系统。  
HEARTBEAT 直接影响信任积累。

## 6）HEARTBEAT 与 SOUL / AGENTS / MEMORY / CRON 的边界
### 6.1 HEARTBEAT vs SOUL
- SOUL：我是谁，我坚持什么
- HEARTBEAT：我多久检查一次、检查什么、何时静默

最常见混淆：把“冷静、克制、不要打扰过度”只写在 SOUL，但不写入 HEARTBEAT 的实际规则。

### 6.2 HEARTBEAT vs AGENTS
- AGENTS：一次任务来了怎么做
- HEARTBEAT：没有新任务时，系统如何周期性检查现有对象

最常见混淆：把任务流程规则全塞进 HEARTBEAT，结果把心跳变成“第二个工作系统”。

### 6.3 HEARTBEAT vs MEMORY
- MEMORY：记住发生过什么
- HEARTBEAT：决定是否值得在现在这个时刻检查与触发

最常见混淆：把所有巡检发现都长期塞进 MEMORY，造成记忆污染。

### 6.4 HEARTBEAT vs CRON
- CRON：何时触发
- HEARTBEAT：触发后检查什么、以什么标准决定是否动作

这是非常重要的边界。
很多人以为只要有定时任务，就等于有 HEARTBEAT。不是。

> **CRON 解决“时间到了”，HEARTBEAT 解决“时间到了以后该不该动、该怎么动”。**

## 7）市面上关于主动性设计最容易踩的坑
### 坑 1：把主动性理解为“更频繁地做事”
这是最原始也最危险的误解。  
主动性不是频率，而是高质量触发。

### 坑 2：没有静默条件
如果文档只写“发现 X 就提醒”，但不写“没有新增时保持静默”，系统必然噪音爆炸。

### 坑 3：没有熔断条件
外部关于 infinite loop 的安全讨论明确显示：任何会自我重试、自我反思、自我检查的机制，只要没有上限，就存在资源耗尽风险。

### 坑 4：把什么都放进心跳
一旦检查面过宽：
- token 消耗上升
- 注意力稀释
- 真异常被淹没

### 坑 5：只会“报”，不会“判”
很多伪主动系统只是把看到的全报出来。  
真正高质量 HEARTBEAT 必须先判断：
- 值不值得报
- 值不值得动
- 值不值得升级

### 坑 6：没有分层动作
所有异常都按同一方式处理，要么过度紧张，要么完全迟钝。

## 8）高质量 HEARTBEAT 应该写什么
一个高质量 HEARTBEAT 至少覆盖以下 8 个模块：
1. 巡检对象
2. 触发频率/时机
3. 检查项
4. 判定标准
5. 静默条件
6. 升级条件
7. 熔断条件
8. 输出格式

---

## 9）使用技巧（Tips）
### 技巧 1：只盯关键变量，不要盯一切
### 技巧 2：每条巡检都写“没有新增时怎么办”
### 技巧 3：一定要有熔断条件
### 技巧 4：先判断“值不值得动作”，再决定是否提醒
### 技巧 5：心跳输出要尽可能模板化、低噪音

## 10）入门（最低可用）
> 目标：写出一份不会骚扰用户、但能检查关键状态的最小 HEARTBEAT。

### 最低可用模板
```md
# HEARTBEAT.md

## 巡检对象
- 当前进行中的关键任务
- 是否有阻塞
- 是否有超时未闭环事项

## 默认动作
- 有明显阻塞或超时：提示一次
- 有新结果：汇报一次
- 没有新增：保持静默

## 熔断条件
- 同一问题无新增信息，不重复提醒
```

### 入门验收
- 是否明显减少“为提醒而提醒”
- 是否能做到无新增时静默
- 是否能抓住关键异常

## 11）进阶（节律协议版）
> 目标：让 HEARTBEAT 从“巡检清单”升级为“主动性节律协议”。

### 进阶新增模块
- 检查优先级
- 输出格式
- 升级路径
- 冷却时间 / 重复提醒上限

### 进阶模板
```md
# HEARTBEAT.md（进阶：节律协议版）

## 1. 巡检对象
- 关键进行中任务
- 长时间 running 的会话/流程
- 已承诺但未回执的事项

## 2. 判定标准
- 有无阻塞
- 有无超时
- 有无新产物
- 有无需要升级的异常

## 3. 默认动作
- 发现新结果：汇报
- 发现阻塞：升级/提醒
- 没有新增：静默

## 4. 冷却规则
- 同一问题未出现新增，不重复提醒
- 已在升级流程中的问题，不并发重复上报

## 5. 熔断条件
- 连续 2-3 次检查无新增动作价值，则停止追打
```

### 进阶验收
- 是否开始具备“低噪音、高信噪比”的提醒能力
- 是否能显著减少重复提醒

## 12）最佳实践（主动性治理版）
> 目标：把 HEARTBEAT 升级为“可控主动性 + 资源保护 + 信任积累”的军团级协议。

### 最佳实践新增机制
1. 心跳对象分层（P0/P1/P2）
2. 通知冷却期
3. 熔断器（circuit breaker）
4. 异常升级矩阵
5. 输出统一模板

### 最佳实践模板
```md
# HEARTBEAT.md（最佳实践：主动性治理版）

## Meta
- version: 1.0
- last_updated: YYYY-MM-DD

## 1. 巡检对象分层
- P0：关键任务掉棒 / 安全异常 / 高风险超时
- P1：重要任务阻塞 / 回执缺失
- P2：一般状态更新

## 2. 检查项
- 是否有新产物
- 是否有超时
- 是否有阻塞
- 是否已有升级动作

## 3. 默认动作
- P0：立即升级
- P1：汇报 + 下一步建议
- P2：无新增则静默

## 4. 冷却规则
- 同一问题无新增 24h 内不重复提醒（按场景调整）

## 5. 熔断条件
- 重试达到上限
- 无新增信息连续出现
- 已进入人工/更高等级处理链路

## 6. 输出模板
- 对象：
- 状态变化：
- 影响：
- 建议动作：
- 是否需要打扰用户：是 / 否
```

### 最佳实践验收
- 是否能把主动性从“打扰能力”升级为“可信赖的守望能力”
- 是否有明确资源保护机制
- 是否能长期运行而不制造通知疲劳

## 13）卓越案例：什么样的 HEARTBEAT 才算业内最佳实践
一份卓越 HEARTBEAT 至少满足：
1. **短**：它是微型清单，不是第二份 AGENTS
2. **准**：只盯关键对象，不铺天盖地扫描
3. **静**：没有新增就不吵
4. **稳**：有冷却、有熔断，不会陷入循环
5. **判**：不是“看到就报”，而是“先判断值不值得报”
6. **护**：能保护预算、注意力与系统信任

---

## 14）验收口径与训练题
### 验收口径
- 是否具备清晰的静默条件
- 是否具备清晰的熔断条件
- 是否具备高信噪比输出
- 是否显著降低重复提醒与资源浪费
- 是否让用户感到“被照看”，而不是“被打扰”

### 训练题
1. 找出你当前 HEARTBEAT 里所有“泛扫描动作”，删到只剩关键变量
2. 为每个检查项补上“没有新增时怎么办”
3. 为你的系统设计一个 heartbeat circuit breaker
4. 设计 3 个测试场景：
   - 无异常但长时间运行
   - 同一异常重复出现
   - 关键任务突然掉棒

> **HEARTBEAT.md 的本质不是“让 Agent 更勤快”，而是“让硅基生命拥有可控、可静默、可熔断的主动性节律”。**
