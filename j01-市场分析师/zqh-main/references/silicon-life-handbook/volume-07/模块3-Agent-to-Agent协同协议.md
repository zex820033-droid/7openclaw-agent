# 卷七｜协同军团：从单体 Agent 到多智能体作战
## 模块3｜Agent-to-Agent 协同协议：交接棒比谁跑得快更重要
### ——为什么"交出去"和"接过来"都需要协议，而不是默契

---

## 一句话结论

> **Agent-to-Agent 协同协议解决的不是"能不能发消息"，而是"交出去的任务能不能安全接住、上下文能不能完整传递、责任能不能明确转移"。没有交接协议的军团，任务在 Agent 之间传递时必然掉棒；有了协议，任务才能像接力赛一样安全交接。**

---

## 1. 这是什么（定位）

### 1.1 Agent-to-Agent 协同协议的本质

Agent-to-Agent 协同协议，是多 Agent 之间**任务交接、信息传递、责任转移**的标准化契约。

它回答四个核心问题：
1. **什么时候交**：什么条件下触发任务交接
2. **交什么**：传递哪些上下文、状态、责任
3. **怎么交**：交接的标准动作是什么
4. **交完怎么办**：交接后的追踪、闭环、结算机制

### 1.2 为什么不是"发消息"而是"协议"

很多人以为 Agent 之间协同就是"调用 sessions_send 发个消息"。

这只是**通信**，不是**协同**。

真正的协同必须包含：
- **上下文传递**：不只是说"你去接单"，而是把完整背景交给对方
- **状态同步**：当前任务进展、已完成部分、未完成部分
- **责任转移**：明确"交出去之后谁负责"
- **闭环确认**：接收方确认收到、执行、完成的完整回环

### 1.3 协议与默契的区别

| 对比维度 | 默契 | 协议 |
|---------|------|------|
| 传递内容 | 靠猜测 | 明确清单 |
| 责任边界 | 模糊 | 清晰 |
| 异常处理 | 随机应变 | 预设路径 |
| 可追溯性 | 无 | 有 |
| 规模化 | 不可扩展 | 可扩展 |

### 1.4 本模块在卷七的位置

模块1解决了"谁负责什么"（编制）。
模块2解决了"谁来决策"（三省制+统帅）。

模块3要解决的是：

> **当一个 Agent 需要把任务交给另一个 Agent 时，怎么交、交接什么、交完怎么闭环。**

这是协同的**执行层协议**。

## 2. 核心目的（为什么存在）

### 2.1 消灭"掉棒"现象

没有交接协议的军团，最常出现：
- A 把任务转给 B，说"你接手吧"
- B 不知道任务背景、约束、截止时间
- B 从头开始问，浪费大量时间
- 严重时任务彻底丢失

交接协议的第一个目的，是**消灭"掉棒"**。

### 2.2 让"交"和"接"都有标准动作

没有协议时：
- 交出方：说一声就完事
- 接收方：默认接了就开始做

有协议后：
- 交出方：必须传递完整上下文，必须等待确认
- 接收方：必须明确确认，必须回告理解状态

交接协议的第二个目的，是**让交接变成有标准动作的系统行为**。

### 2.3 让责任转移可追溯

没有协议时：
- 任务做坏了，A 说"已经交给 B 了"，B 说"我没接清楚"
- 责任扯不清

有协议后：
- 每次交接都有记录
- 交出时间、接收时间、理解确认、执行状态全部在案
- 出了问题可以追溯

### 2.4 让闭环可验证

没有协议时：
- B 说"做完了"
- A 不知道 B 到底做了什么
- 验收无从进行

有协议后：
- 交接时明确交付物清单
- 完成时对照清单验收
- 有证据、有记录、可复核

### 2.5 让上下文传递保真

Agent 之间的上下文传递，是协同的核心挑战。

如果交接时上下文丢失或不完整：
- B 可能误解任务
- B 可能重复 A 已做的工作
- B 可能遗漏关键约束

交接协议的第三个目的，是**让上下文像接力赛的接力棒一样，完整、保真、不掉棒**。

## 3. 拟人化映射（仿真价值）

### 3.1 交接棒类比什么

Agent-to-Agent 交接，最像：

| 人类场景 | Agent 对应 |
|---------|----------|
| 接力赛交接棒 | 任务上下文传递 |
| 医生交接班 | 病历+状态+注意事项 |
| 项目交接文档 | 完整项目背景+当前状态+待办 |
| 空姐交接表 | 乘客状态+特殊需求+待跟进 |

### 3.2 为什么人类组织要有交接制度

人类历史上，交接失误导致的灾难比比皆是：
- 医疗事故：护士交接时漏说过敏信息
- 航空航天：航天飞机 O 型圈事故，工程师的警告没有被交接给管理者
- 金融交易：交易员离职时没有交接头寸

这些不是能力问题，而是**交接协议缺失**。

### 3.3 Agent 交接的特殊挑战

和人类交接相比，Agent 交接有三个独特挑战：

**挑战1：没有"自然理解"兜底**
人类交接时，接收方可以根据常识补全没说清楚的部分。
Agent 没有常识兜底，必须接收完整上下文才能正确执行。

**挑战2：上下文在传递中容易失真**
人类语言传递时，接收方会用自己的理解重新组织。
Agent 传递上下文时，如果格式不规范，信息会漂移。

**挑战3：Agent 可能不知道自己缺了什么**
人类交接时，接收方会问"你说的 X 是指什么"。
Agent 可能不会主动提问，导致误解潜伏到执行阶段才暴露。

### 3.4 交接协议的仿真价值

交接协议的仿真价值，体现在：

**第一：交接确定性**
- 交出方知道"交了什么"
- 接收方知道"接了什么"
- 双方对交接内容无歧义

**第二：责任明确性**
- 交接瞬间，责任明确转移
- 转移前归交出方，转移后归接收方
- 有记录、可追溯

**第三：执行可预期性**
- 接收方知道完整任务背景
- 不会从头摸索
- 执行效率最大化

## 4. 对长期 AI 表现的影响

### 4.1 决定军团能否规模化

没有交接协议，军团规模越大，问题越严重：
- 2个 Agent：可能勉强能协同
- 5个 Agent：交接混乱开始显现
- 10个 Agent：系统级掉棒频发

有交接协议后，军团规模扩张是**结构化增长**，不是混乱增长。

### 4.2 决定任务能否端到端追踪

长期运行最难的不是单点执行，而是**端到端追踪**：
- 谁发起的
- 谁执行的
- 谁交接的
- 谁完成的
- 耗时多久
- 有没有掉棒

没有交接协议，这些数据无法收集。
有交接协议，这些数据自然沉淀。

### 4.3 决定错误能否快速定位

出了问题，最怕的是：
- 不知道哪个环节出了问题
- A 说"我交的时候是对的"
- B 说"我接的时候就是错的"

有交接协议后，**每个交接点都有记录**，问题定位从靠猜变成靠证据。

### 4.4 决定新 Agent 能否快速融入

新 Agent 加入军团时：
- 没有交接协议：需要从头了解所有背景，效率极低
- 有交接协议：通过历史交接记录快速了解任务状态和上下文

## 5. 怎么写/怎么配（步骤）

### 5.1 步骤一：定义交接触发条件

不是所有场景都需要正式交接。

定义什么情况必须走正式协议：
```md
## 必须交接的场景
- 任务需要跨角色执行
- 当前 Agent 能力/权限不足以继续
- 任务需要等待外部依赖
- 当前 Agent 需要去执行其他更高优先级任务
- 交接时长超过预设阈值
```

定义什么情况可以简化：
```md
## 可简化交接的场景
- 同角色内的简单转发
- 完全由接收方独立执行的新任务
- 极短任务（5分钟内完成）
```

### 5.2 步骤二：设计交接数据包结构

每次交接必须包含：

```md
## 交接数据包 v1.0

### 基础信息
- 交接ID：[唯一编号]
- 交接时间：
- 交出方：
- 接收方：
- 原始任务卡ID：

### 上下文传递
- 任务背景：
- 已完成部分：
- 未完成部分：
- 关键约束：
- 截止时间：

### 状态同步
- 当前状态：
- 阻塞/风险：
- 依赖项：

### 交付物清单
- [ ] 交付物1：路径/位置
- [ ] 交付物2：路径/位置

### 预期接收方动作
- 下一步行动：
- 预期完成时间：
- 验收标准：
```

### 5.3 步骤三：定义交接确认流程

```md
## 交接确认流程

1. 交出方发送交接数据包
2. 接收方回告"已接收"
3. 接收方回告"我理解的任务是：[复述]"
4. 交出方确认"理解正确"或"补充说明"
5. 交接正式成立，责任转移
```

### 5.4 步骤四：定义异常处理机制

```md
## 交接异常处理

### 异常1：接收方超时未确认
- 触发条件：发送交接数据包后 X 分钟未确认
- 处理：交出方追问 → 若继续无响应，升级至协调者

### 异常2：接收方理解有偏差
- 触发条件：接收方复述与交出方意图不符
- 处理：交出方补充说明 → 重新确认 → 交接成立

### 异常3：交出方在交接过程中失联
- 触发条件：交接未完成，交出方失联
- 处理：接收方升级至协调者 → 协调者从头跟进

### 异常4：接收方拒绝接收
- 触发条件：接收方认为任务超出能力/权限范围
- 处理：接收方说明原因 → 协调者裁定
```

### 5.5 步骤五：设计闭环确认机制

```md
## 闭环确认机制

### 执行中闭环
- 接收方定期同步进度（按预设频率）
- 出现阻塞立即回告

### 执行完毕闭环
- 接收方提交完成通知
- 包含：交付物路径、完成状态、实际耗时
- 交出方确认收到
- 交出方验收（按任务卡验收口径）
- 任务卡状态更新为"已完成"
```

### 5.6 步骤六：建立交接记录仓库

```md
## 交接记录仓库

位置：[统一存放路径]

命名规范：[YYYY-MM-DD]-[交接ID]-[任务简述]

保留周期：
- 活跃任务：全程保留
- 已完成任务：保留12个月
- 归档任务：摘要保留
```

### 业内最佳实践：Agent-to-Agent 协同协议七原则

1. **交接必须主动确认**：不能默认"接了"，必须明确回告"我接了，我理解的是..."
2. **上下文必须完整传递**：不能只说"你去做"，必须传递完整背景、约束、状态
3. **责任转移必须有节点**：交接瞬间必须明确"之前归我，之后归你"
4. **交接必须有记录**：不能只靠口头，记录进入日志，可追溯
5. **异常必须有预设处理路径**：不能临时想办法，必须提前定义升级机制
6. **闭环必须有验收**：不能只说"做完了"，必须对照任务卡验收口径验证
7. **交接质量必须可评估**：不能只管交接不管质量，必须有置信度评估机制

## 7. 常见误区（写错会怎样）

### 误区1：把"发消息"当成"交接"

**错误示例**：
```md
A: "这个任务你接手吧"
B: "好的"
```

**问题**：这只是发消息，不是交接。没有上下文传递，没有确认，没有闭环机制。

**后果**：
- B 对任务一无所知
- B 从头摸索，效率极低
- 可能重复 A 已做的工作
- 可能遗漏关键约束

---

### 误区2：交接后撒手不管

**错误示例**：
```md
A: "交接给你了，我去做别的了"
B: "好的"
```

**问题**：交出方以为交完就完事，不追踪、不确认、不兜底。

**后果**：
- B 做坏也不知道
- 没有进度同步
- 出了问题无法追溯

---

### 误区3：接收方不确认就默认接了

**错误示例**：
```md
A: "这个任务给你"
[沉默]
[A 默认 B 接了，继续下一项]
```

**问题**：接收方没有明确确认，双方理解可能不一致。

**后果**：
- B 可能根本没注意到
- B 理解的任务和 A 想的不一样
- 任务卡在中间

---

### 误区4：交接时只传递任务，不传递状态

**错误示例**：
```md
A: "去做一个用户分析报告"
[没告诉 B 之前已经做过一版]
```

**问题**：遗漏已完成部分和当前状态，导致重复工作或方向错误。

**后果**：
- B 重复 A 已做的工作
- 可能推翻 A 的结论
- 浪费时间

---

### 误区5：没有交接闭环机制

**错误示例**：
- 交接出去就完了
- B 做没做、做得怎么样，没人知道

**问题**：没有完成通知、没有验收确认、没有状态更新。

**后果**：
- 任务不知道是否完成
- 交付物不知道是否合格
- 无法端到端追踪

---

### 误区6：交接记录不保存

**错误示例**：
- 全靠口头交接
- 没有记录

**后果**：
- 出了问题无法追溯
- 无法分析交接瓶颈
- 无法优化交接协议

## 本模块一句话结论

**Agent-to-Agent 协同协议解决的不是"能不能发消息"，而是"交出去的任务能不能安全接住、上下文能不能完整传递、责任能不能明确转移"。没有交接协议的军团，任务在 Agent 之间传递时必然掉棒；有了协议，任务才能像接力赛一样安全交接。交接是军团协同的生命线，必须在所有协同执行之前先立住。**
