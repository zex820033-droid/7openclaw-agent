# 卷七｜协同军团：从单体 Agent 到多智能体作战
## 模块1｜军团编制设计：没有编制，就没有军团
### ——为什么一群 Agent 不是军团，有编制的才是

---

## 一句话结论

> **军团编制设计解决的不是"开几个 Agent"，而是"每个 Agent 负责什么、权限边界在哪里、谁向谁汇报、出了问题找谁"。没有编制的多 Agent，只是各自跑的个体；有编制的多 Agent，才是能协同作战的组织。**

---

## 1. 这是什么（定位）

### 1.1 军团编制的本质

军团编制，是多 Agent 协作的**组织架构设计**。

它回答四个核心问题：
1. **角色定义**：这个 Agent 叫什么、做什么、不做什么
2. **权限边界**：这个 Agent 能做什么、不能做什么、什么需要审批
3. **层级关系**：谁主导、谁辅助、谁复核、谁上报
4. **协作接口**：Agent 之间怎么通信、怎么交接、怎么让位

### 1.2 军团编制不是"给 Agent 起名字"

很多人以为军团编制就是"我需要一个客服 Agent、一个技术 Agent、一个战略 Agent"。

这只是**角色清单**，不是编制。

真正的编制必须包含：
- 角色的**核心职责**（做什么）
- 角色的**决策权限**（能拍板什么）
- 角色的**汇报对象**（谁管它）
- 角色的**协作对象**（谁能调用它）
- 角色的**晋升/降级规则**（表现好/差怎么变）

### 1.3 军团编制在卷七的位置

卷七是协同卷，解决的是"一群 Agent 怎么真正协同"。

模块1是卷七的**第一模块**，因为：

> **编制不清，协同无从谈起。**

没有编制：
- 路由系统不知道把消息发给谁
- 交接协议不知道交给谁负责
- 让位规则不知道谁该让谁
- 治理账本不知道记录谁的表现

---

## 2. 核心目的（为什么存在）

### 2.1 防止"伪军团"

没有编制的多 Agent，看起来是个军团，实际是个体户集合：
- 每个 Agent 各干各的
- 同一个问题可能多个 Agent 重复回答
- 出了事不知道找谁
- 没有人在整体层面统筹

我们把这种状态叫**伪军团**。

军团编制的第一个目的，就是**消灭伪军团**。

### 2.2 让协同从"靠默契"变成"靠结构"

人类组织早期也是靠默契协同：
- 我们是兄弟，默契就够了
- 不用写流程，我们都知道怎么做

但组织一大，默契一定会崩。

军团也是一样：
- 2-3 个 Agent 时，靠默认行为可能勉强能协同
- 5 个以上 Agent，没有结构化编制，一定乱

军团编制的第二个目的，是**把协同从靠默契升级为靠结构**。

### 2.3 给治理提供组织锚点

卷六的治理系统需要挂载点：
- 任务卡需要知道派给谁
- 信誉分需要知道记在谁名下
- 整改单需要知道发给谁

这些挂载点的底层，都是**角色**。

军团编制的第三个目的，是**给整个治理体系提供组织锚点**。

### 2.4 让扩缩容有章可循

没有编制，团队想加 Agent 时只能拍脑袋：
- 再加一个客服？
- 再加一个销售？

有编制之后，扩缩容是**结构化决策**：
- 当前瓶颈在哪个角色？
- 新增角色会不会和现有角色冲突？
- 编制调整要不要修改路由规则？

## 3. 拟人化映射（仿真价值）

### 3.1 军团编制类比什么

如果把军团类比人类组织，它最像：

| 人类组织 | 军团对应 |
|---------|---------|
| 公司架构图 | 军团编制表 |
| 岗位职责说明书 | Agent 角色定义 |
| 审批权限表 | Agent 权限边界 |
| 组织架构调整 | 军团编制调整 |
| 新员工入职培训 | 新 Agent 初始化配置 |

### 3.2 为什么成熟公司要有组织架构

人类历史上，组织架构经历了几个阶段：

**阶段1：个体户**
- 一个人全做
- 没有分工

**阶段2：合伙制**
- 几个强人搭档
- 靠默契

**阶段3：职能制组织**
- 按功能分组
- 有明确的部门墙
- 有层级

**阶段4：矩阵式/网络式**
- 跨部门协作
- 灵活的项目组

大多数团队的多 Agent 协同，还停在**阶段1-2**：
- 一个 Agent 什么都能做
- 或者靠"它们自己会协调"的幻觉

真正的军团协同，需要进入**阶段3**，至少是**阶段3**。

### 3.3 编制设计的仿真价值

军团编制设计的仿真价值，体现在三个层次：

**第一层：角色确定性**
- 用户知道什么问题找什么 Agent
- Agent 知道自己负责什么、不负责什么

**第二层：协作可预期**
- Agent 知道什么时候该主动
- 什么时候该等待指令
- 什么时候该转交

**第三层：系统可治理**
-出了问题能追溯
- 表现好坏能记录
- 调整能有序进行

## 4. 对长期 AI 表现的影响

### 4.1 决定 Agent 是否形成"角色意识"

没有编制，Agent 很难形成稳定的角色认知：
- 什么都答一点
- 什么都不确定是不是自己该答
- 边界模糊导致行为飘移

有编制之后，Agent 有清晰的**角色认同**：
- 我是技术 agent，技术问题我负责
- 我不是战略 agent，战略问题不该我答

这会显著减少**角色漂移**。

### 4.2 决定长期协同是否可持续

长期协同最大的敌人是**边界腐蚀**：
- 今天这个角色多做了一点
- 明天那个角色少做了一点
- 日积月累，谁负责什么全靠猜

有编制作为锚点，边界腐蚀可以被**结构化抵御**：
- 定期检查角色是否在编制内行事
- 偏离可被及时发现和纠正

### 4.3 决定记忆系统是否卫生

没有编制，记忆系统会混乱：
- 技术经验写进客服 Agent 的记忆
- 战略决策写进技术 Agent 的记忆
- 查无可查

有编制后，记忆系统可以**按角色归类**：
- 技术 Agent 记录技术知识
- 客服 Agent 记录用户交互
- 战略 Agent 记录决策上下文

### 4.4 决定编排扩展是否可行

当军团需要扩缩容时：
- 无编制：靠拍脑袋，加完一团乱
- 有编制：结构化决策，新角色无缝嵌入

这对长期系统的**可扩展性**至关重要。

## 5. 怎么写/怎么配（步骤）

### 5.1 步骤一：定义核心角色

从业务需求出发，问三个问题：
1. **用户最常问哪类问题？** → 形成核心角色
2. **哪些职责必须由不同角色承担？** → 区分角色边界
3. **当前团队有几个真实角色？** → 确定最小编制

不建议在一开始就设计超过 **5 个核心角色**。

### 5.2 步骤二：明确每个角色的核心职责

每个角色用一句话定义：
- 这个角色**唯一负责**什么
- 这个角色**绝不越界**做什么

格式建议：
```
角色名：[角色名称]
唯一职责：[一句话说明唯一负责什么]
绝不越界：[这个角色不做什么]
```

### 5.3 步骤三：设定权限边界

每个角色需要回答：
- **可自主决定的事**：无需审批，自己拍板
- **需审批的事**：需要上级或他角色确认
- **禁止做的事**：无论什么情况都不能做

### 5.4 步骤四：设计层级关系

回答：
- 谁是**主协调者**（统帅）
- 谁向主协调者汇报
- 平行协作关系怎么定义

### 5.5 步骤五：定义协作接口

每个角色需要明确：
- **主动协作对象**：日常工作需要和谁配合
- **被动响应对象**：谁可能调用我
- **让位规则**：什么情况下应该主动让位

### 5.6 步骤六：编写角色说明书

把以上内容整理成结构化文档：
```md
## 角色定义

### 角色名称
- 核心职责：
- 绝不越界：
- 权限等级：
- 汇报对象：
- 协作对象：

### 权限边界
- 可自主决定：
- 需审批：
- 禁止：

### 协作协议
- 主动协作对象：
- 让位触发条件：
- 异常升级路径：
```

## 6. 技巧（业内最佳实践）

### L1｜入门：最小可用编制

**特征**：3 个以内角色，最简单的职能划分

**技巧**：
- 先区分**执行层**和**协调层**
- 执行层负责干活，协调层负责分发和兜底
- 不要一开始就追求完美角色定义，先跑起来

**模板**：
```md
## 军团编制 v1.0

### 角色1：[执行者]
- 核心职责：直接执行用户任务
- 绝不越界：不负责战略决策
- 汇报给：[协调者]

### 角色2：[协调者]
- 核心职责：分发任务、统筹协同
- 绝不越界：不直接执行具体任务
- 兜底：用户请求未被识别时，默认由协调者接单
```

---

### L2｜进阶：职能分离编制

**特征**：4-6 个角色，按职能明确分离

**技巧**：
- **强职能分离**：技术问题归技术 Agent，业务问题归业务 Agent
- **设立路由规则**：根据消息类型自动路由到对应角色
- **协调者不再是所有消息的兜底**，而是特定类型的兜底

**模板**：
```md
## 军团编制 v2.0

### 角色1：[技术专家]
- 核心职责：技术问题解答、代码、架构
- 可自主决定：技术方案选型
- 需审批：涉及核心技术变更
- 禁止：非技术领域的业务承诺

### 角色2：[业务专家]
- 核心职责：业务咨询、需求分析、流程解释
- 可自主决定：业务流程类问题
- 需审批：涉及业务流程变更
- 禁止：技术实现细节承诺

### 角色3：[协调者/统帅]
- 核心职责：任务分发、跨域协调、兜底
- 可自主决定：简单问题直接处理
- 需审批：涉及角色职责争议
- 禁止：长期绕过执行者直接干活
```

---

### L3｜高阶：多层治理编制

**特征**：6-10 个角色，含多层治理机制

**技巧**：
- **引入"三省制"**：每个决策需经提议、复核、终审
- **设立专项角色**：如质量审核员、风险监察员
- **建立角色晋升/降级通道**：表现数据化
- **编制文档化**：所有角色定义写入 SOUL.md / AGENTS.md

**模板**：
```md
## 军团编制 v3.0

### L1 执行层
| 角色 | 核心职责 | 可自主决定 | 需审批 | 禁止 |
|------|---------|-----------|-------|------|
| 技术执行 | 技术实现 | 技术方案 | 架构变更 | 业务承诺 |
| 业务执行 | 业务执行 | 流程操作 | 业务变更 | 技术承诺 |
| 质量审核 | 质量验收 | 质量判断 | 重大缺陷 | 绕过执行直接改 |

### L2 复核层
| 角色 | 核心职责 | 权限边界 |
|------|---------|---------|
| 技术复核 | 技术方案复核 | 技术决策复核权 |
| 业务复核 | 业务决策复核 | 业务决策复核权 |

### L3 终审层
| 角色 | 核心职责 | 权限边界 |
|------|---------|---------|
| 统帅 | 最终决策 | 跨域争议、一票终审 |

### 角色协作矩阵
| 触发场景 | 角色A | 角色B | 角色C |
|---------|-------|-------|-------|
| 技术问题 | 技术执行主责 | 技术复核审核 | 统帅终审 |
| 业务问题 | 业务执行主责 | 业务复核审核 | 统帅终审 |
| 质量争议 | 质量审核主责 | 对应复核审核 | 统帅终审 |
```

---

### L4｜专家：动态弹性编制

**特征**：10+ 角色，支持动态扩缩、角色流转

**技巧**：
- **角色不是固定的，是动态的**
- 根据任务类型动态组建**临时项目组**
- 角色之间可以**临时借调**
- 建立**角色能力矩阵**：每个角色的能力等级、擅长领域
- 引入**角色代言人机制**：每个角色有一个主维护者（人类）

**模板**：
```md
## 军团编制 v4.0

### 核心角色注册表
| 角色ID | 角色名 | 能力标签 | 权限等级 | 状态 | 维护者 |
|-------|-------|---------|---------|------|-------|
| R001 | 技术专家 | 代码/架构/调试 | L2 | active | @人A |
| R002 | 业务专家 | 咨询/流程/培训 | L2 | active | @人B |
| R003 | 质量审核 | 验收/测试/标准 | L2 | active | @人C |
| R004 | 协调统帅 | 统筹/决策/升级 | L3 | active | @人A |

### 动态角色矩阵
| 任务类型 | 主责角色 | 协作角色 | 审核角色 | 终审角色 |
|---------|---------|---------|---------|---------|
| 技术实现 | R001 | R002 | R003 | R004 |
| 业务咨询 | R002 | R001 | R003 | R004 |
| 质量审计 | R003 | R001/R002 | - | R004 |

### 角色流转规则
- 当某角色超负荷时，可临时借调其他角色支援
- 角色借调需经过协调统帅批准
- 临时角色变更记录进入治理账本

### 角色能力持续评估
- 每季度评估角色能力是否匹配业务需求
- 能力不匹配时触发角色再培训或重新招聘
```

---

### 业内最佳实践：军团编制设计七原则

1. **角色最小化**：先问"能不能合并"，再问"要不要新增"
2. **职责唯一性**：每个角色只有一个唯一核心职责，多职责必导致模糊
3. **权限对等性**：有多大权力，就有多大责任；禁止权和审批权必须配套
4. **层级扁平化**：层级不超过3层；超过3层必然产生信息损耗
5. **接口标准化**：角色间协作必须走标准化接口，不允许"默契协作"
6. **数据可追踪**：每个角色的表现必须可量化、可记录、可结算
7. **变更留痕化**：编制调整必须经过正式审批，不能口头临时改

## 7. 常见误区（写错会怎样）

### 误区1：角色定义太抽象

**错误示例**：
```md
客服 Agent：负责客户服务相关工作
```

**问题**：什么叫"客户服务相关"？边界在哪里？

**后果**：
- Agent 自己不确定该不该接
- 其他 Agent 不知道该不该让
- 用户不知道该问谁

**正确示范**：
```md
客服 Agent：
- 核心职责：解答用户关于产品使用、订单状态、售后政策的问题
- 绝不越界：不承诺技术实现细节、不承诺价格优惠、不承诺交付时间
```

---

### 误区2：角色之间没有边界

**错误示例**：
- 技术 Agent：负责所有技术问题
- 业务 Agent：负责所有业务问题
- 看起来有分工，实际边界重叠

**问题**：一个既涉及技术又涉及业务的问题，谁答？

**后果**：抢答或失答

**正确示范**：
```md
## 边界协议
- 技术问题（代码/架构/部署）→ 技术 Agent 主责
- 业务问题（流程/政策/培训）→ 业务 Agent 主责
- 混合问题 → 技术 Agent 先答技术部分，业务 Agent 补足业务部分
- 争议问题 → 路由至协调者裁定
```

---

### 误区三：只有执行层，没有治理层

**错误示例**：
- 5 个执行 Agent
- 没有协调者
- 没有复核者

**问题**：出了问题没人拍板，任务没人分发

**后果**：系统级失序

---

### 误区四：编制太复杂

**错误示例**：
- 15 个角色
- 8 层层级
- 30+ 协作接口

**问题**：维护成本超过协同收益

**后果**：
- 没人愿意按编制走
- 实际退化成无编制状态

---

### 误区五：编制是静态的

**错误示例**：
- 编制设计好之后再也不改

**问题**：业务变了，编制没跟上

**后果**：编制变成僵尸文件

---

### 验收标准

#### L1 验收（入门）
- [ ] 角色数量 ≤ 3
- [ ] 每个角色有核心职责定义
- [ ] 有明确的协调者/兜底角色
- [ ] 能说清楚"用户问 X 找谁"

#### L2 验收（进阶）
- [ ] 角色数量 4-6
- [ ] 每个角色有明确的权限边界
- [ ] 有结构化路由规则
- [ ] 能识别角色重叠场景并有冲突解决机制

#### L3 验收（高阶）
- [ ] 层级不超过3层
- [ ] 有三省制或类似复核机制
- [ ] 编制文档化且版本受控
- [ ] 角色表现可追踪

#### L4 验收（专家）
- [ ] 支持动态角色配置
- [ ] 有角色能力矩阵
- [ ] 有正式的角色变更审批流程
- [ ] 季度复盘机制

#### 业内最佳实践验收
- [ ] 角色最小化原则被遵守
- [ ] 职责唯一性被保证
- [ ] 权限对等性被实现
- [ ] 层级扁平化（≤3层）
- [ ] 接口标准化
- [ ] 数据可追踪
- [ ] 变更留痕化

## 本模块一句话结论

**军团编制设计解决的不是"开几个 Agent"，而是"每个 Agent 负责什么、权限边界在哪里、谁向谁汇报、出了问题找谁"。没有编制的多 Agent，只是各自跑的个体户；有编制的多 Agent，才是能协同作战的组织。编制是军团的第一张组织图，必须在所有协同机制之前先立住。**
