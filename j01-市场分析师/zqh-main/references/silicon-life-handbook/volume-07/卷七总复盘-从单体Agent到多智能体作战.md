# 卷七总复盘｜从单体 Agent 到多智能体作战
## ——让"一群龙虾"变成"真正的军团"

---

> 卷七不是"写完4个模块"，而是把多 Agent 协同从"并排放在一起"升级到"能协同作战"的一整套**组织化治理协议**。

---

## 0. 本章给谁看（适用对象）

- 你在用 OpenClaw/类似框架做 **多 Agent 协同**（不是单 Agent 问答）
- 你发现：多开几个 Agent 后，消息没人答、抢答成灾、掉棒不断、责任不清
- 你需要一套**可编制、可决策、可交接、可让位**的军团协同协议

本章目标：把卷七 4 个模块收成一个**"军团协同操作系统"**，并给出**最小可用版（MVP）**与**高阶治理版（Pro）**。

---

## 1. 卷七到底解决了什么（卷七的"物理问题"）

单体 Agent 很强，但有一个天花板：**一个 Agent 能做的事有上限**。

当任务复杂度超过单 Agent 能力边界时，你有两个选择：

- **选项A**：继续强化单 Agent（边际收益递减，很快撞墙）
- **选项B**：组建多 Agent 军团（能力叠加，但协同复杂度指数上升）

大多数团队选 B 后发现：

- 多 Agent 不是"1+1=2"，而是"1+1=1"甚至"1+1=0"
- 原因不是 Agent 不够聪明，而是**缺乏协同协议**

卷七做的是：把"多 Agent"从"一堆各自跑的个体"变成"有编制、有决策链、有交接协议、有让位规则的真正军团"。

## 2. 军团协同的四大基础协议（你必须建立的制度）

把军团协同协议压缩成 4 类（也是卷七 4 个模块对应的"协议谱"）：

### 2.1 编制协议（模块1：军团编制设计）

**解决什么问题**：谁负责什么、权限边界在哪里、谁向谁汇报

**本质**：组织架构设计，不是"给 Agent 起名字"

**核心三问**：
- 这个 Agent 叫什么、做什么、不做什么
- 这个 Agent 能拍板什么、不能拍板什么
- 这个 Agent 汇报给谁、谁能调用它

> 没有编制，多 Agent 只是"个体户集合"；有了编制，才是真正的组织。

---

### 2.2 决策协议（模块2：三省制与统帅机制）

**解决什么问题**：谁来提议、谁来复核、谁来终审；紧急情况谁拍板

**本质**：决策分权机制，不是"三重审批"

**核心区分**：
| 机制 | 解决的问题 | 效率定位 |
|------|----------|---------|
| 三省制 | 日常决策质量 | 质量优先 |
| 统帅机制 | 紧急决策效率 | 效率优先 |

**关键约束**：
- 提议权 ≠ 决策权
- 统帅权必须有边界（事后必须三省复核）
- 决策必须有记录

> 没有决策链，军团协同时刻面临"议而不决、决而不行、责任真空"。

---

### 2.3 交接协议（模块3：Agent-to-Agent 协同协议）

**解决什么问题**：任务从 A 交到 B 时，上下文怎么传、责任怎么转、闭环怎么做

**本质**：交接不是"转一下"，而是完整上下文 + 确认 + 闭环的系统动作

**交接五要素**：
1. 触发条件（什么时候交）
2. 上下文传递（交什么）
3. 接收确认（接了没有）
4. 执行反馈（做怎么样了）
5. 证据存档（有没有记录）

> 没有交接协议，任务在 Agent 之间传递必然掉棒；有了协议，才能像接力赛一样安全交接。

---

### 2.4 让位协议（模块4：群聊/私聊/让位规则）

**解决什么问题**：群聊里谁该答、谁该让、什么信息走群聊、什么信息走私聊

**本质**：让位比抢答更重要——协同的效率天花板是会抢答

**三种让位类型**：
| 类型 | 触发 | 动作 | 边界 |
|------|------|------|------|
| 完整让位 | 专业边界明确属于他人 | 声明让位 + 停止 + 等待 | 超时则补位 |
| 补充式让位 | 主答案已给出 | 声明补充 + 不覆盖主答 | 补充不等于开新话题 |
| 纠正式让位 | 发现明显错误 | 声明纠正 + 指出 + 提供正确 | 对事不对人 |

> 没有让位规则，军团退化成抢答团；有了让位规则，才是真正高效的组织。

## 3. 四个模块之间的因果链（卷七的"骨架图"）

```
(1) 军团编制
    ↓（决定：每个 Agent 的职责边界和权限范围）
(2) 三省制 + 统帅机制
    ↓（决定：多 Agent 协同时谁来决策、谁来复核、谁来终审）
(3) Agent-to-Agent 交接协议
    ↓（决定：任务在 Agent 之间传递时上下文是否完整、责任是否明确）
(4) 群聊/私聊/让位规则
    ↓（决定：高频群聊场景下谁该响应、谁该让位、信息如何管理）
真正的多智能体协同
```

这张图的核心含义：

- **编制**决定"谁是谁"
- **三省制**决定"谁来拍板"
- **交接协议**决定"任务怎么传"
- **让位规则**决定"群聊怎么跑"

四个协议缺任何一个，军团就会以不同形式回到"伪军团"状态。

## 4. 从"多 Agent"到"能作战的军团"：完整协同闭环（最重要的一段）

多 Agent 协同不是"把 Agent 并排放在一起"，而是一个持续循环的协议体系：

### 4.1 协同闭环总公式

> **编制（职责 × 权限）× 决策（三省 × 统帅）× 交接（上下文 × 确认 × 闭环）× 让位（专业优先 × 信誉联动）= 真正能作战的军团**

### 4.2 最小可用协同闭环（MVP：一周内能落地）

1) **先建立最小编制**：3 个角色以内，执行层 + 协调层 + 兜底层
2) **明确路由规则**：什么消息路由到什么 Agent，规则写到文件里
3) **强制交接确认**：交接必须有"接收方复述确认"，不能默认"接了"
4) **最简让位规则**：谁专业谁答，非专业者不抢答
5) **决策必须记录**：每个决策记录提议/复核/终审，存档可查

> 只要做到"有编制 + 有路由 + 交接确认 + 让位规则"，多 Agent 协同就开始从混沌走向有序。

### 4.3 高阶协同治理版（Pro：可量化、可优化）

在 MVP 上加 4 个治理组件：

- **交接置信度体系**：每次交接评估上下文完整度、接收方理解准确度
  - ≥4.0：正常交接
  - 3.0-4.0：补充说明后执行
  - <3.0：重新交接 + 升级协调者
- **让位信誉分**：正确让位 +5，抢答 -5，纠错成功 +4，信誉分影响未来响应权重
- **决策置信度机制**：历史决策质量影响未来决策权重，高风险决策需要更高置信度
- **动态编制调整**：根据任务类型、Agent 状态动态选择三省角色，统帅可轮值

## 5. 卷七对整本《硅基生命全量教案》的位置（为什么卷七是"组织化跃迁"）

如果把整套教案视为一个产品：

| **卷七** | **怎么让军团能协同作战** | **组织协同** |

卷四是"个体能力"——训出一只会干活的虾。

**卷七是"协同能力"——让虾群真正能像军团一样配合作战。**

很多团队失败不是因为单 Agent 不够强，而是因为：

卷七就是把"散兵游勇"变成"有组织、有纪律、有协议"的军团。

## 6. 你应该如何验收"卷七是否真的闭环"（验收口径）

卷七不是看"文档写得好不好"，而是看能否通过以下验收：

### 6.1 编制验收（职责清晰）
- 每个 Agent 知道自己负责什么、不负责什么
- 出了新问题，能判断该路由到哪个 Agent
- 职责重叠时，有协调者裁定机制

### 6.2 决策验收（决策高效）
- 日常决策走三省制，有提议/复核/终审记录
- 紧急决策走统帅机制，事后有三省复核
- 没有"议而不决、决而不行"

### 6.3 交接验收（不掉棒）
- 每次交接有完整上下文传递
- 接收方有明确复述确认
- 任务完成有闭环验收
- 交接记录可追溯

### 6.4 让位验收（群聊有序）
- 专业问题由专业 Agent 回答
- 非专业 Agent 不抢答，主动让位
- 让位行为计入信誉分
- 纠错在群里公开，不人身攻击

### 6.5 整体验收（军团能作战）
- 一个任务进来，能顺畅完成：编制 → 路由 → 三省 → 交接 → 让位 → 闭环
- 不需要人工干预每个环节
- 出问题能追溯到具体环节

## 7. 建议你立刻落地的"卷七一页纸制度"（可以直接贴到团队 SOP）

### 每次任务分配（5 分钟）
- 判断任务类型
- 路由到对应角色
- 记录任务卡（含交付物、截止时间、验收口径）

### 每次交接（3 分钟）
- 交出方：发送完整交接数据包
- 接收方：复述理解，确认接收
- 交出方：确认理解正确
- 交接成立，责任转移

### 每次群聊响应（实时）
- 判断是否自己专业领域
- 非专业 → 主动让位
- 专业 → 响应，但避免重复他人答案
- 发现错误 → 公开纠正，对事不对人

### 每天（10 分钟）
- 检查待处理交接
- 检查阻塞任务
- 记录决策日志

### 每周（30 分钟）
- 跑一次军团编制检查（角色是否在编制内运行）
- 统计交接置信度
- 统计让位准确率
- 发现问题进入整改单流程

### 每季度（半天）
- 重新评估编制设计
- 优化三省角色分配
- 更新交接协议模板
- 优化让位规则

## 9. 训练题（用于检验维护者是否真的懂了卷七）

1) **你的多 Agent 军团"看起来很热闹但没人真正负责"，你第一时间检查哪两个模块？为什么？**

2) **一个技术任务从 Agent-A 交接给 Agent-B，但 B 做完发现不是 A 想要的。请列出交接链上可能出问题的 3 个环节，并给出预防动作。**

3) **群里用户问了一个技术问题，技术 Agent 和业务 Agent 同时抢着答。请用让位规则分析这个场景，并给出正确的处理方式。**

4) **设计一个最小可用的军团编制（不超过 3 个角色），包含：角色定义、路由规则、最简三省制、交接流程。**
