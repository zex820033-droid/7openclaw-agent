# 卷三｜系统骨架：OpenClaw 官方文档深度解剖
## 模块3｜Session / Context / Queue
### ——长会话为什么会污染、为什么会“越聊越飘”，以及如何用 OpenClaw 的机制治理

> 官方依据（本模块只以官方机制为准，不靠猜）：
> - `concepts/session.md`（会话键模型、生命周期、重置、存储）
> - `concepts/context.md`（上下文构成、/context、窗口成本）
> - `concepts/queue.md`（命令队列、模式、并发与保证）
> - `concepts/messages.md`（入站去重、防抖、队列与流式）
> - `concepts/compaction.md`（压缩：持久化摘要）
> - `concepts/memory.md`（压缩前 memory flush、记忆文件层）
> - `concepts/session-pruning.md`（会话剪枝：仅修剪工具结果、非持久化）
> - `concepts/retry.md`（渠道重试：避免非幂等重复）
>
> 外部交叉验证（仅吸收与上述机制一致的行业共识）：
> - “上下文（Context）≠ 记忆（Memory）”必须制度化，否则训练会退化成玄学
> - 长对话系统必然面对：窗口稀缺、历史膨胀、工具输出堆积、并发冲突与速率限制
> - 队列/防抖/去重属于工程治理，不是体验小细节；没有它，多智能体与重工具必然互相踩踏

## 1）这个模块到底在讲什么
这个模块回答三个训练者必须掌握的问题：

1) **为什么一只虾“越聊越飘/越跑越慢/越跑越贵”？**（本质是上下文窗口与工具堆积）
2) **为什么同样一句话在不同场景（私聊/群聊/定时任务）会表现不同？**（本质是会话键与路由隔离）
3) **为什么你明明在催“继续”，系统却像“卡住/重复/抢答”？**（本质是队列序列化、模式与防抖）

你会发现：卷二写得再好，只要这里没搞清楚，训练就会出现两类致命后果：
- **认知误判**：把“会话/队列/压缩”问题误判成“模型不聪明/人格漂移”
- **工程失控**：并发冲突、重复执行、速率限制、工具结果爆炸、成本失控

## 2）三件事先分清：Session / Context / Memory
这是本模块的第一性原理。

### 2.1 Session（会话）是什么
**会话是 OpenClaw 用来把“对话状态”分桶、持久化、重置与路由的单位。**（见 `concepts/session.md`）

### 2.2 Context（上下文）是什么
**上下文是 OpenClaw 在一次运行中发送给模型的所有内容**，受模型上下文窗口限制。（见 `concepts/context.md`）

上下文包含：
- 系统提示词（工具、skills 列表、运行时元信息、注入的工作区文件）
- 对话历史
- 工具调用与结果、附件
- 压缩摘要、修剪产物

### 2.3 Memory（记忆）是什么
**记忆是工作区里写入磁盘的 Markdown 文件**（`memory/YYYY-MM-DD.md` + 可选 `MEMORY.md`），它是唯一事实来源。（见 `concepts/memory.md`）

### 2.4 三者关系（一句话）
> **Session 负责“状态桶”，Context 负责“当下窗口”，Memory 负责“跨时间资产”。**

这三件事混在一起，你的训练体系就会变成“聊对了就算记住、聊错了就当没发生”。这是行业里最常见的“长期失真源”。

## 3）官方机制A：Session（会话）

### 3.1 主会话与会话键：为什么私聊能连续、群聊要隔离
官方定义：
- 每个智能体的一个直接聊天会话视为主会话。
- 直接聊天折叠为 `agent:<agentId>:<mainKey>`（默认 `main`）。
- 群组/频道聊天获得各自的键。（见 `concepts/session.md`）

训练者要抓住的点：
- **私聊连续性**来自“主会话桶”，不是模型天生记性好。
- **群聊隔离**来自“会话键隔离”，不是“群聊更容易忘”。

### 3.2 dmScope：你到底想要哪种“连续性”
`session.dmScope` 控制私信如何分组（官方给出 main / per-peer / per-channel-peer / per-account-channel-peer）。

训练上的意义：
- 你要的是“一个老板一个大脑”（连续性最大）还是“多用户收件箱隔离”（污染最小）？
- 如果做城市合伙人/多客户，**per-channel-peer / per-account-channel-peer**这种隔离更接近真实业务需求。

### 3.3 会话存储：事实来源在哪里
官方强调：**Gateway 网关是唯一数据源**。会话状态存储在网关主机：
- `~/.openclaw/agents/<agentId>/sessions/sessions.json`
- 对话记录：`~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

训练含义：
- 不要把“客户端看到的历史”当事实来源。
- 复盘问题要以 gateway 存储为准。

### 3.4 生命周期与重置：何时该 /new、何时该 /compact
官方给了多种重置策略：每日重置、空闲重置、按类型覆盖、按渠道覆盖，以及 `/new`、`/reset` 触发。（见 `concepts/session.md`）

训练建议（与官方一致的最小判断）：
- **想彻底换话题/换模型/切断污染** → `/new` 或 `/reset`
- **想保留近期但瘦身旧历史** → `/compact`

（注意：`/compact` 是压缩，不是重置。）

## 4）官方机制B：Context（上下文）

### 4.1 上下文窗口：为什么“越聊越贵/越聊越慢”是必然
官方定义：上下文受模型的窗口 token 限制；所有内容都计入，包括工具 schema。（见 `concepts/context.md`）

训练上的现实：
- 不是你“写得更详细就更好”，而是“窗口稀缺，你必须选择”。
- 大型工具输出、图片/附件、工具 schema 都会占用窗口。

### 4.2 /context 与 /status：训练师必须会用的两把尺
官方给出：
- `/status`：窗口有多满 + 会话设置。
- `/context list`：注入了什么 + 大致大小。
- `/context detail`：更细的分解（含工具 schema、skills 列表成本）。

训练师必须把它们当作“体检仪”：
- 你觉得 agent 变笨了，先看 context 是否被垃圾挤爆。

### 4.3 Compaction（压缩）：持久化摘要，不是“删历史”
官方定义：
- 压缩把较早对话总结成紧凑条目，并**持久化**到 JSONL。
- 保持近期消息不变。
- 可自动触发，也可手动 `/compact`。（见 `concepts/compaction.md`）

训练含义：
- 压缩是让长会话继续跑下去的必要机制。
- 压缩之后你还能“记得什么”，取决于：摘要质量 + 重要信息是否写入 memory。

### 4.4 Memory flush：压缩前的“记忆迁移窗口”
官方定义：当会话接近自动压缩时，OpenClaw 可触发一次**静默记忆刷新**回合，提醒把持久性笔记写入 `memory/YYYY-MM-DD.md`。（见 `concepts/memory.md`）

训练含义：
- 这不是礼貌提示，这是系统给你的“最后一次上岸机会”。
- 你不写，压缩后就只能靠摘要；摘要永远比不上结构化记忆。

### 4.5 Session pruning：只修剪工具结果，不动历史
官方定义：会话剪枝在每次 LLM 调用前修剪旧的工具结果，**不重写磁盘历史**。（见 `concepts/session-pruning.md`）

训练含义：
- 剪枝是“成本治理/缓存治理”手段，不是记忆整理手段。
- 它保护你不被工具结果拖死，但它不会替你做知识沉淀。

## 5）官方机制C：Queue（命令队列）

### 5.1 为什么需要队列：并发冲突是真实存在的
官方解释：入站消息靠进程内队列序列化自动回复运行，避免会话文件、日志、CLI stdin 等共享资源冲突，降低速率限制。（见 `concepts/queue.md`）

训练含义：
- 你看到的“慢”，很多时候不是模型慢，而是系统在保证不会踩踏。

### 5.2 队列保证：每个会话一次只能跑一个
官方说：`runEmbeddedPiAgent` 按会话键入队，保证每个会话只有一个活动运行。（`concepts/queue.md`）

训练含义：
- 同一会话里连续发“继续/快点/再来一段”，如果不做模式控制，就会产生排队与合并行为。

### 5.3 队列模式：steer / followup / collect 等到底改变了什么
官方定义（简化训练版）：
- `collect`（默认）：把排队消息合并成一个后续轮次（更抗刷屏）。
- `followup`：当前轮次结束后逐条后续运行。
- `steer`：尝试“引导当前运行”，在下一个工具边界取消剩余工具调用。
- `steer-backlog`：既引导当前运行，又保留后续。

训练含义：
- **collect** 是“运营友好”的默认：减少重复回复与注意力污染。
- **steer** 适合“及时纠偏”：比如你发现 agent 走错方向，要立刻刹车。

### 5.4 防抖与溢出：debounce / cap / drop 的训练含义
官方提供：
- `debounceMs`：等待静默后再启动后续轮次（防“继续继续”）。
- `cap`：每会话最大排队消息数。
- `drop`：溢出策略（old/new/summarize）。

训练含义：
- 训练师在群里刷指令，最终不是提高效率，而是制造 backlog。
- `drop: summarize` 的设计是“系统替你做降损”，但你不该依赖它当流程。

### 5.5 入站去重与防抖：你以为重复，其实是渠道重投递
官方说明：
- 入站去重：重连后重复投递同一消息会被短期缓存挡掉。（`concepts/messages.md`）
- 入站防抖：同一发送者快速连续消息可合并成一个轮次；媒体/附件不防抖。（`concepts/messages.md`）

训练含义：
- 在“工具链重/网络抖动/多端登录”场景下，这些机制避免系统自我打架。

## 6）长会话失真五连：污染、遗忘、乱序、重复、踩踏
这五个问题在外部课程/论坛里被反复讨论，但在 OpenClaw 里可以被更工程化地解释与治理。

### 6.1 污染（Context Pollution）
表现：越聊越啰嗦、答非所问、抓不住主线。

在 OpenClaw 里通常来自：
- 注入文件太大被截断/噪音太多（用 `/context` 可见）
- 工具结果堆积（可用 pruning 缓解）
- 会话承载过多异质任务（需要 /new 或拆会话策略）

### 6.2 遗忘（Window Forgetting）
表现：隔天就忘、压缩后丢关键偏好。

官方机制提示你：
- Context 不是 Memory。
- 压缩会总结旧历史。
- 压缩前可以 memory flush。

训练解法：
- 关键决策写入 memory 文件，而不是指望聊天。

### 6.3 乱序（Out-of-order）
表现：你发了三条指令，回复顺序不符合直觉。

通常原因是：
- 队列在序列化同一会话运行
- collect 合并后只回一次

训练解法：
- 重要纠偏用 `steer`（或单条明确指令）
- 连续碎片指令要合并成“一个清晰任务卡”

### 6.4 重复（Duplicate Work）
表现：同一个动作执行两次、同一段话重复回复。

官方相关机制：
- 入站去重（防渠道重投递）
- 重试策略（按请求重试，避免非幂等重复）（`concepts/retry.md`）

训练解法：
- 非幂等操作（写入/发送）必须“先确认再执行”，并记录回执。

### 6.5 踩踏（Concurrency Collisions）
表现：多个运行互相抢文件、抢会话、抢 CLI。

官方解法：
- 队列序列化入站自动回复运行（`concepts/queue.md`）
- 每会话一次只有一个活动运行

训练解法：
- 多线程并发是系统能力，但“同会话并发”不是你该追求的效率。

## 7）最佳实践：把“会话治理”写进训练流程
下面这些不是“建议”，而是你要想做出稳定龙虾军团的训练必修课。

### 7.1 训练师三件套（每次异常先做这三步）
1) `/status`：窗口是否逼近上限？
2) `/context list`：注入文件是否截断？工具 schema 是否占主导？
3) 判断是：/new（换会话）还是 /compact（压缩瘦身）还是写 memory（落盘迁移）

### 7.2 任务卡优于碎片指令
行业共识：碎片化指令会让上下文膨胀且难以验收。
OpenClaw 的队列/防抖/collect 已经告诉你：系统会尽力合并碎片，但你更该从源头治理。

### 7.3 重要信息必须落盘
官方明确：记忆的事实来源是 Markdown 文件。
训练上把“聊过”升级为“写下”，是长期稳定的分水岭。

### 7.4 把工具结果当一次性耗材
工具结果的价值是“帮助当前轮决策”，而不是“永久保留”。
这也是官方提供 pruning 的原因：它解决的是工具结果膨胀，而不是知识沉淀。

## 8）标杆案例（训练场景）
以下案例不靠虚构 OpenClaw 行为，全部可用官方机制解释。

### 案例1：群里连发三条“继续”，结果只回一条
- 解释：默认 `collect` 会把排队消息合并成一个后续轮次（`concepts/queue.md`）。
- 训练修复：把“继续”改成“继续写模块3的第X节到第Y节，包含案例与验收口径”。

### 案例2：会话跑久后，agent 变啰嗦、抓不住重点
- 解释：context 膨胀 + 工具结果堆积 + 注入文件截断（`concepts/context.md`）。
- 训练修复：先 `/context detail` 看最大贡献者；必要时 `/compact`；把长期要点写入 memory。

### 案例3：你以为写入失败，其实是队列在等
- 解释：队列序列化 + 会话单运行保证（`concepts/queue.md`）。
- 训练修复：减少同会话并发指令；用更明确的单任务卡；必要时开启详细日志排查。

### 案例4：工具输出巨大导致成本上升
- 解释：工具 schema + 工具结果都计入上下文（`concepts/context.md`）；pruning 只修剪 toolResult，不动历史（`concepts/session-pruning.md`）。
- 训练修复：减少超大工具输出；把需要保留的“结论”写入知识库或 memory，而不是保留原始输出。

## 9）常见误区与反模式（严格对齐官方）

### 9.1 误区：上下文就是记忆
反模式：不写 memory，指望聊天长期可用。
- 官方反证：Context 与 Memory 不是一回事（`concepts/context.md` / `concepts/memory.md`）。

### 9.2 误区：/compact 等于重置
反模式：想换话题却只压缩。
- 官方区别：/compact 是压缩摘要；/new 或 /reset 才是新会话。（`concepts/compaction.md` / `concepts/session.md`）

### 9.3 误区：队列是“卡住”
反模式：看到慢就连续催促，制造 backlog。
- 官方解释：队列序列化避免冲突；collect/防抖为此设计。（`concepts/queue.md` / `concepts/messages.md`）

### 9.4 误区：工具结果应该永久保留
反模式：让工具输出无限堆积，最后压爆窗口。
- 官方机制：pruning 专门修剪 toolResult；压缩为历史总结。（`concepts/session-pruning.md` / `concepts/compaction.md`）

### 9.5 误区：重试会帮你“把整条流程重跑一遍”
反模式：把重试当工作流引擎。
- 官方原则：重试按请求、避免重复执行非幂等操作。（`concepts/retry.md`）

## 10）模板 / 训练题 / 验收口径

### 10.1 最小“会话治理”检查清单（训练师用）
- 我是否能区分：session / context / memory？
- 我是否能使用：/status、/context list、/context detail？
- 我是否知道何时用：/new vs /compact？
- 我是否知道默认 queue mode 以及它对“继续继续”的影响？
- 我是否把关键决策写入 memory 文件，而不是只留在聊天里？

### 10.2 训练题
**题1｜三分法**
把以下问题分别归因到：session / context / queue / memory。
- 群聊里不同人说话导致模型答偏
- 连发三条指令只回一条
- 隔天忘记昨天确定的决策
- 工具输出越来越慢、越来越贵

**题2｜治理动作选择**
给你一个场景：
- 上下文很满
- 但你不想丢掉最近 10 条关键对话
- 同时你需要把“长期偏好”保留下来

你应该做哪三件事？（提示：/context + /compact + memory 写盘）

**题3｜队列模式选择**
解释何时你会选择：collect / followup / steer。

### 10.3 验收口径（DoD）
本模块写完，必须达到：
- 能用官方语言解释 sessionKey、dmScope、会话存储位置与重置策略
- 能解释 context 构成与窗口成本，并能用 /context 工具诊断
- 能解释 queue 的序列化目的、模式差异与防抖/溢出策略
- 能把“长期稳定性问题”从玄学变成可定位、可修复的系统问题

## 本模块一句话结论
> **Session 决定状态桶如何分离，Context 决定当下窗口如何装载，Queue 决定并发如何不踩踏；只有把这三件事制度化，卷二的生命协议才会在长期运行中真正稳定，而不是“短期看着很聪明、长期越跑越飘”。**
