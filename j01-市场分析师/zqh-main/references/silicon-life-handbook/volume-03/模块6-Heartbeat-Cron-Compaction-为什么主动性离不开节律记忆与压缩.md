## 1）这模块到底在解决什么问题
很多人一说“让 agent 主动”，想到的都是：
- 多加几个定时任务
- 每隔几分钟轮询一次
- 看到点什么就推送给主人
- 会话长了就继续堆历史，反正模型会总结

这是典型的“只想要主动性，不想要治理成本”的思路。

OpenClaw 在这一层真正解决的问题不是“怎么让 agent 动起来”，而是：

> **怎么让 agent 在长期运行中既能主动、又不失控；既能记住关键东西、又不被长上下文拖死。**

换句话说，这模块要解决四个基础问题：

1. **什么时候主动看一眼世界** —— Heartbeat
2. **什么时候在精确时间执行任务** —— Cron
3. **上下文快爆了怎么办** —— Compaction
4. **压缩前哪些东西必须落盘，否则就真的忘了** —— Memory Flush

如果这四个层没设计好，所谓“主动型 agent”最后通常只会变成三种东西：
- 烦人的提醒器
- 失忆的长会话 bot
- 越跑越脏的自动化怪物

## 2）一句话总判断
> **Heartbeat 解决“周期性感知”，Cron 解决“精确定时执行”，Compaction 解决“上下文窗口有限”，Memory Flush 解决“压缩前把真正该记住的东西写入磁盘”。四者合在一起，才构成长期稳定运行的主动性系统。**

## 3）Heartbeat、Cron、Compaction、Memory Flush 分别是什么

### Heartbeat
Heartbeat 是 Gateway 中运行在主会话里的**周期性智能体轮次**。默认按固定间隔触发，让 agent 去看看：
- 有没有紧急事项
- 有没有要提醒人类的事
- 有没有后台任务状态变化
- 有没有需要继续推进的事情

它的重点不是“精确”，而是“周期性感知 + 上下文感知”。

### Cron
Cron 是 Gateway 内置调度器里的**精确定时任务系统**。
它适合：
- 每天 7:00 发晨报
- 每周一 9:00 跑复盘
- 20 分钟后提醒
- 独立运行某个分析任务

它的重点不是“理解上下文”，而是“在指定时间可靠执行”。

### Compaction
Compaction 是会话接近上下文窗口上限时的**压缩机制**。OpenClaw 会把较早历史总结成一条紧凑摘要，把近期消息保留，从而继续运行。

它的重点不是“记住全部细节”，而是“在模型窗口有限的前提下，让会话继续活下去”。

### Memory Flush
Memory Flush 是压缩前的**静默记忆刷写轮次**。也就是：在上下文即将被压缩之前，提醒 agent 把那些真正应该长期保留的东西写入 `memory/YYYY-MM-DD.md`。

它的重点不是“多记一点”，而是：

> **不要让真正重要的决策、偏好、状态，在 compaction 时一起被压扁。**

## 4）它们在硅基生命里分别对应什么
如果卷二是“生命协议学”，那这里就是“生命节律学”。

### Heartbeat 对应：周期性巡检
像人类一天里会不断扫视环境：
- 有没有新消息
- 有没有急事
- 有没有没处理完的事情

Heartbeat 就是硅基生命的“短周期感知循环”。

### Cron 对应：日程表与准点执行
像人类知道：
- 9 点开会
- 晚上发日报
- 周一做复盘

Cron 是硅基生命的“外部时钟系统”。

### Compaction 对应：工作记忆整理
人脑不会把几千轮对话一字不漏原样顶在前额叶里，人会做压缩、抽象、保留近因。Compaction 就是这件事的系统实现。

### Memory Flush 对应：把重要事写进笔记本
人在脑子快装不下时，会把关键事项记进外部笔记。Memory Flush 就是在压缩前，把值得长期保留的东西写进工作区里的记忆文件。

## 5）为什么主动性从来不是“想主动就主动”
很多团队对“主动型 agent”的误解，本质上都是把主动性理解成“更频繁地打扰”。

但真正稳定的主动性，至少要满足四个条件：

### 1. 有节律
不是时时刻刻都触发，而是有明确频率、活跃时段、冷却边界。

### 2. 有场景分工
- 周期性感知用 Heartbeat
- 精确调度用 Cron
- 事件驱动补充可以用 Hooks

### 3. 有上下文治理
主动跑得越久，会话越容易脏；如果没有 Compaction，长会话迟早失真。

### 4. 有长期记忆沉淀
如果每次都是“感觉 agent 记得”，但从未写入磁盘，那只是会话幻觉，不是长期记忆。

所以真正的主动性不是一个特性，而是一整套运行纪律。

## 6）官方机制怎么说

### 关于 Heartbeat
官方文档明确把 Heartbeat 定义为：
- 在主会话中运行的周期性智能体轮次
- 目标是让模型在不打扰用户的情况下，提醒需要关注的事项
- 默认提示会要求读取 `HEARTBEAT.md`
- 如果没事，回复 `HEARTBEAT_OK`
- 该 OK 标记会被系统吞掉，不一定显示给用户

这说明 Heartbeat 的设计目标不是“持续发消息”，而是：

> **持续巡检，但默认静默。**

### 关于 Cron
官方文档把 Cron 定义为 Gateway 网关内置调度器：
- 任务持久化存储在 `~/.openclaw/cron/`
- 可使用 `at / every / cron` 三种计划方式
- 可运行在主会话、隔离会话、当前会话、自定义会话中
- 可选择 announce 投递摘要，或 none 仅内部运行

这说明 Cron 的核心不是“提醒”，而是：

> **确定性、可持久化、可精确调度的后台执行。**

### 关于 Compaction
官方文档明确指出：
- 模型上下文窗口有限
- 长会话会累积消息和工具结果
- 自动压缩会把早期历史总结成紧凑摘要，并持久化到会话 JSONL
- `/compact` 可手动触发

也就是说，OpenClaw 不是假装“上下文无限”，而是承认物理约束，然后给出工程治理手段。

### 关于 Memory Flush
官方记忆文档明确写到：
- 自动压缩前，OpenClaw 可以触发静默记忆刷写轮次
- 提醒模型在压缩前写入持久记忆
- 默认提示要求写入 `memory/YYYY-MM-DD.md`
- 若没有内容则回复 `NO_REPLY`

这一步非常关键，因为它承认了一个事实：

> **压缩摘要不是长期记忆文件，真正的长期记忆仍然要写回工作区磁盘。**

## 7）Heartbeat：周期性感知，而不是无脑打扰

### 7.1 Heartbeat 最适合什么
官方给出的推荐场景非常清楚：
- 每 30 分钟看一眼收件箱
- 检查最近 2 小时日历
- 后台任务完成时做摘要
- 白天闲置太久时轻量 check-in

这些任务的共同点是：
- 不要求精确到秒
- 要结合当前上下文判断是否值得提醒
- 多项任务可以批量处理

这就是 Heartbeat 的最佳使用场景：

> **低频、批量、上下文感知的巡视工作。**

### 7.2 HEARTBEAT.md 的价值
`HEARTBEAT.md` 本质上是一个“小而稳定的检查清单”。

比如：
- 快速扫描 inbox 有没有紧急事项
- 如果白天且长时间沉默，轻量问候
- 如果任务阻塞，记下缺什么，下次问 Peter

这意味着 Heartbeat 不是把全部逻辑写死在系统 prompt 里，而是把“巡检规则”文档化、工作区化。

### 7.3 Heartbeat 为什么必须小
官方明确提醒：
- `HEARTBEAT.md` 要保持小巧
- 太大就会增加提示膨胀
- 空文件甚至可以跳过心跳，节省调用

所以最优解不是把 HEARTBEAT.md 写成作战手册，而是：

> **像飞行员起飞前 checklist，一眼能扫完，反复可执行。**

### 7.4 Heartbeat 的边界
Heartbeat 不适合：
- 每天 7:00 准时发报
- 20 分钟后精确提醒
- 长耗时分析任务
- 高频轮询导致的噪声工作

这些事情应交给 Cron，而不是塞给 Heartbeat。

## 8）Cron：精确定时，而不是上下文感知

### 8.1 Cron 最适合什么
Cron 适合一切“要在这个时间点执行”的任务：
- 晨报
- 周报
- 一次性提醒
- 每周深度分析
- 定时跑后台检查

相比 Heartbeat，它最大的优势是：
- 时间更准
- 可用隔离会话
- 可覆盖模型和思维级别
- 可选择是否向外投递

### 8.2 主会话与隔离会话的区别
官方把 Cron 分为两类核心执行模式：

#### 主会话任务
- 通过 `systemEvent` 入队
- 在下一次心跳或立即唤醒时跑
- 共享主会话上下文

适合：
- 和当前主线程强相关的提醒
- 需要 agent 结合最近对话判断的事情

#### 隔离任务
- 在 `cron:<jobId>` 或自定义会话中单独运行
- 不污染主聊天记录
- 可以独立 announce 或 none

适合：
- 晨报、晚报、监控、周分析
- 噪音型、高频型、后台型任务

### 8.3 为什么 Cron 对长期系统这么重要
如果没有 Cron，很多团队会把所有主动任务都塞进 Heartbeat。
结果就是：
- 心跳变重
- 周期性扫描变成时间调度替代品
- 主会话被后台任务污染
- 模型 token 消耗失控

Cron 的价值，就是把“准点执行”从“周期性感知”里拆出来。

## 9）Compaction：长会话不治理，必然污染

### 9.1 压缩不是可选优化，而是生存条件
只要你让 agent 长期跑：
- 会话消息会累积
- 工具结果会累积
- 跟进任务会累积
- 系统事件也会累积

模型上下文窗口是硬上限，不会因为你希望 agent 更聪明就自动变大。

所以 Compaction 的本质不是“锦上添花”，而是：

> **长期运行系统的生存条件。**

### 9.2 OpenClaw 如何压缩
官方文档的要点是：
- 保留近期消息
- 把更早历史总结成一条紧凑摘要
- 摘要持久化进会话 JSONL
- 必要时自动压缩后重试原始请求

这意味着 OpenClaw 采用的是典型的：
- 近因保鲜
- 远因摘要
- 历史持久化

### 9.3 压缩能解决什么，不能解决什么
Compaction 能解决：
- 上下文装不下
- 历史过长导致调用失败
- 近期轮次仍需保留较清晰上下文

Compaction 不能解决：
- 你从来没把重要事实写进记忆文件
- 你把工具脏输出当长期知识
- 你把任务日志当人格设定

也就是说，Compaction 能救上下文窗口，不能替代记忆治理。

## 10）Memory Flush：为什么压缩前刷记忆是关键一步

### 10.1 如果没有 Memory Flush，会发生什么
没有这一步，系统会出现一种很常见的错觉：

> “我以为 agent 记住了，其实它只是还没压缩。”

一旦会话被压缩：
- 细节被抽象
- 口气被压平
- 临时状态可能消失
- 还没落盘的重要结论也可能一起被吞掉

### 10.2 OpenClaw 的做法为什么高级
官方不是把 Compaction 做成黑盒，而是在压缩前插入一个静默记忆刷写轮次，提醒模型：
- 现在接近压缩了
- 该写的 durable memories 现在写
- 写到 `memory/YYYY-MM-DD.md`
- 没有可写就 `NO_REPLY`

这一步本质上是：

> **把“会话内部暂存”转成“工作区磁盘持久化”。**

### 10.3 MEMORY.md 和 daily memory 的边界
根据官方文档：
- `memory/YYYY-MM-DD.md`：每日运行日志，只追加
- `MEMORY.md`：精心整理的长期记忆，只在主要私人会话加载

因此对于 Memory Flush 最稳的策略是：
- 运行过程中的持久笔记，先进入 daily memory
- 真正长期稳定的偏好/决策，再由后续整理进入 `MEMORY.md`

这就避免两种污染：
- 把每日噪声写进长期记忆
- 把长期偏好埋在天天追加的流水账里

## 11）四者联动：主动性、记忆、上下文窗口是一套系统
这一模块最重要的，不是把四个概念背下来，而是理解它们的联动关系。

### 联动链路
1. **Heartbeat** 周期性巡检
2. **Cron** 在精确时间点触发确定性任务
3. 两者都会不断产生活动会话和新上下文
4. 上下文变长后，**Compaction** 负责压缩历史
5. 压缩之前，**Memory Flush** 把 durable notes 写入 daily memory
6. 后续需要时，再通过工作区注入或记忆搜索取回

这是一条完整闭环：

> 感知 → 执行 → 累积 → 压缩 → 落盘 → 再取回

这也是为什么卷三模块6的本质不是“自动化补充知识”，而是：

> **长期运行稳定性的核心机制。**

## 12）常见反模式

### 反模式1：把 Heartbeat 当定时任务系统
表现：
- 早报、晚报、复盘、提醒，全塞进 Heartbeat
- 心跳越来越长
- 时间越来越不准

后果：
- 主会话污染
- token 浪费
- 用户被打扰

### 反模式2：把 Cron 当主动思考系统
表现：
- 每 10 分钟跑一个独立 cron 去“看看有没有事”

后果：
- 噪音爆炸
- 缺乏上下文感知
- 成本比 Heartbeat 高得多

### 反模式3：长会话不压缩
表现：
- 希望系统永远带全量历史跑
- 不做 `/compact`
- 不设自动压缩

后果：
- 上下文超限
- 模型响应质量下降
- 工具结果淹没关键信息

### 反模式4：把 compaction 当记忆系统
表现：
- 从来不写 memory 文件
- 以为会话摘要就是长期记忆

后果：
- 压缩几轮之后，真正重要的偏好和决策丢失

### 反模式5：没有熔断与升级链路
表现：
- 主动任务失败后继续重复
- 高噪声 cron 一直跑
- blocked/aborted/timeout 不当回事

后果：
- 系统越自动越不可信

## 13）最佳实践

### 最佳实践1：Heartbeat 只做轻量巡检
推荐用途：
- inbox / calendar / pending task 扫描
- 轻量提醒
- 后台状态观察

不要用它做重任务。

### 最佳实践2：Cron 专管准点任务和隔离任务
推荐用途：
- 晨报 / 周报 / 复盘
- 定时分析
- 一次性提醒
- 独立后台任务

### 最佳实践3：把 HEARTBEAT.md 保持成 checklist
不要写成长文，不要塞进十几个分支规则。让它始终短、稳、反复可执行。

### 最佳实践4：把 blocked / aborted / timeout 视为未完成
不要把“跑过”当成“交付了”。
主动系统里最危险的不是失败，而是：

> **失败了还假装完成。**

### 最佳实践5：Daily memory 负责沉淀，MEMORY.md 负责提纯
- 今日运行上下文进 `memory/YYYY-MM-DD.md`
- 长期偏好与稳定事实再整理进 `MEMORY.md`

### 最佳实践6：压缩前必须允许写记忆
如果工作区不可写，Memory Flush 会跳过。那就等于承认：这轮压缩前没有外部记忆沉淀能力。

### 最佳实践7：高频自动化一定要有活跃时段和冷却
白天和半夜，心跳策略不应一样；高频 cron 必须能降频、暂停、熔断、切模型。

## 14）模板 / 示例 / 训练题 / 验收口径

### 14.1 一个合格的 HEARTBEAT.md 示例
```md
# Heartbeat checklist

- Quick scan: anything urgent in inboxes?
- Check calendar for events in next 2 hours.
- If a task is blocked, write down what is missing.
- If daytime and quiet for too long, do a lightweight check-in.
```

要求：
- 短
- 稳
- 可重复执行
- 不含密钥

### 14.2 Heartbeat vs Cron 快速选择表

| 场景 | 应该用什么 | 原因 |
|---|---|---|
| 每30分钟检查 inbox / calendar | Heartbeat | 周期性 + 可批处理 + 有上下文 |
| 每天 7:00 发晨报 | Cron（isolated） | 精确定时 |
| 20分钟后提醒我开会 | Cron（at） | 一次性、准点 |
| 每周深度分析 | Cron（isolated + model override） | 重任务、应隔离 |
| 后台项目健康检查 | Heartbeat | 轻量巡检更适合 |

### 14.3 训练题

#### 训练题1
如果你希望 agent 每天早上 7 点准时发《晨间Alpha快报》，同时不污染主会话，应该用什么？

标准答案：
- 用 Cron
- `sessionTarget: isolated`
- 如需发到聊天渠道，用 announce / delivery

#### 训练题2
如果你希望 agent 每 30 分钟看看有没有重要邮件、日历冲突和后台任务完成，应该用什么？

标准答案：
- 用 Heartbeat
- 把检查清单写入 `HEARTBEAT.md`

#### 训练题3
如果会话已经非常长，但某些偏好和决策只存在于当前聊天中、还没写入 memory 文件，这时最大的风险是什么？

标准答案：
- 一旦 compaction，这些信息可能被压缩掉
- 因此需要 Memory Flush 在压缩前写入 `memory/YYYY-MM-DD.md`

### 14.4 验收口径
一个真正理解模块6的人，至少应该能回答清楚下面五个问题：

1. Heartbeat 和 Cron 的分工边界是什么？
2. 为什么“主动性”不能只靠多设几个定时器？
3. Compaction 解决什么，不解决什么？
4. 为什么 Memory Flush 必须发生在压缩之前？
5. daily memory 和 `MEMORY.md` 的边界是什么？

如果这五个问题答不清，就说明还没有真正理解：

> **主动性、记忆与上下文窗口不是三套机制，而是一套联动系统。**

# 本章一句话收口
**Heartbeat 决定 agent 有没有节律地看世界，Cron 决定它能不能在正确时间可靠执行，Compaction 决定长会话能不能继续活下去，Memory Flush 决定压缩前的关键经验会不会真正沉淀为长期记忆。四者缺一，长期稳定的主动型硅基生命都不成立。**
