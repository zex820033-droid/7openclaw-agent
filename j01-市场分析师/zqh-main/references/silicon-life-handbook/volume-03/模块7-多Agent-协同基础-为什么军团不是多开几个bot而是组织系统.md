## 1）这模块到底在解决什么问题
当系统开始从“单只虾”走向“军团作战”时，很多团队会天然以为：
- 那就多建几个 bot
- 每个 bot 一个职责
- 再加一个总控去转发
- 这样自然就能协同了

这几乎一定会失败。

因为多 Agent 真正困难的地方，从来不是“数量”，而是四件事：

1. **边界怎么隔离**
2. **消息怎么路由**
3. **上下文怎么不串线**
4. **谁有首答权，谁只该协同，不该抢答**

如果这四层没设计好，所谓多 Agent 最后通常会退化成：
- 多个 bot 抢答同一个问题
- 谁都能回答，结果谁都不专业
- 记忆、权限、会话互相污染
- 看起来很热闹，实则没有组织秩序

所以模块7真正解决的问题是：

> **为什么军团协同首先是系统架构问题，其次才是提示词和分工口号问题。**

## 2）一句话总判断
> **多 Agent 协同的本质，不是“多开几个模型实例”，而是用隔离的工作区、独立的 agentDir、分离的会话存储、确定性的绑定路由、清晰的会话边界和可控的 agent-to-agent 调度，把多个专长单元组织成一套有秩序的协作系统。**

---

## 3）多 Agent 在 OpenClaw 里到底是什么
官方文档对“一个 agent”的定义很清晰：

一个 agent 不是 prompt，不是 bot 名称，不是 Telegram 账号，而是一个**完全独立作用域的大脑**，拥有自己的：
- 工作区（workspace）
- 状态目录（agentDir）
- 认证配置（auth profiles）
- 会话存储（sessions）
- 本地技能（workspace/skills）
- 引导文件（AGENTS/SOUL/USER/TOOLS/IDENTITY 等）

因此，多 Agent 的本质是：

> **同一个 Gateway 里运行多个相互隔离的“独立人格 / 独立记忆 / 独立上下文 / 独立权限域”。**

这和“同一个 bot 套多个人设 prompt”完全不是一回事。

## 4）它在硅基生命里对应什么
如果单体 agent 对应的是“一只虾的大脑与身体”，那多 agent 对应的就不是“多颗脑细胞”，而是：

### 4.1 职能分化
像组织里有：
- 战略脑
- 执行脑
- 风控脑
- 设计脑
- 技术脑
- 财务脑

每个大脑有自己更适合处理的任务。

### 4.2 角色边界
不是谁都能代表组织发言。
有些角色只能：
- 提供意见
- 出分析
- 出中间产物

而有些角色才拥有：
- 首答权
- 拍板权
- 对外发言权

### 4.3 组织协同
一个成熟组织不是所有人对所有事同时发言，而是：
- 先路由给该答的人
- 其他人补位但不抢位
- 需要跨域时再发起协同

所以多 Agent 的仿真价值不是“更多脑子”，而是：

> **把人格、能力、权限、上下文和组织关系拆开治理。**

---

## 5）为什么多 Agent 不是“越多越强”
外部最佳实践里有一个高频共识：

### Anthropic 的共识
先从简单、可组合的模式开始；不要为了“高级感”一开始就上复杂框架。

### LangChain 的共识
很多任务单 agent + 好工具就够了；只有当上下文管理、团队边界、专业分工成为瓶颈时，多 agent 才真正必要。

### Vellum / Azure 的共识
多 agent 的问题不是“能不能搭起来”，而是：
- 上下文怎么工程化
- handoff 怎么定义
- 失败怎么回退
- 评估怎么做

这意味着：

> **多 Agent 只有在“单体 agent 已经因上下文、边界或治理而接近失控”时，才真正值得引入。**

否则它只会引入新的复杂度：
- 延迟更高
- 成本更高
- 排错更难
- 推理链更长
- 责任更难追

## 6）官方机制怎么说

### 6.1 多 Agent 的三个硬边界
根据 OpenClaw 官方文档，多 agent 至少有三层硬隔离：

#### 工作区隔离
每个 agent 可以有自己的 workspace：
- 各自的 AGENTS.md / SOUL.md / USER.md
- 各自的本地文件和知识
- 各自的 workspace/skills

#### 状态目录隔离
每个 agent 有自己的 `agentDir`：
- 各自 auth-profiles.json
- 各自模型配置与状态

官方明确提醒：
**不要复用 agentDir**，否则会导致认证与会话冲突。

#### 会话存储隔离
每个 agent 的 sessions 存在：
- `~/.openclaw/agents/<agentId>/sessions`

这意味着每个 agent 的对话历史、路由状态、sessionKey 映射本来就该分开。

### 6.2 绑定路由是确定性的
官方明确：绑定路由遵循**最具体优先**。
优先级大致是：
1. peer 匹配
2. guildId / teamId
3. accountId
4. channel 级匹配
5. 默认 agent

这说明“谁该答”不是临场拍脑袋，而是：

> **先由系统路由裁决，再由 agent 行为协议补充约束。**

### 6.3 DM / 群 / 频道的会话模型不同
官方 session 文档明确区分：
- direct chat
- group
- channel
- topic / thread
- cron / hook 等特殊来源

这意味着多 agent 协同不只是“多角色”，而是：

> **不同会话类型 + 不同路由策略 + 不同上下文生命周期 的组合问题。**

## 7）隔离：为什么必须先分脑，再谈协同
这是整个模块最核心的一层。

### 7.1 为什么隔离先于协同
如果不先隔离，你所谓的“多 agent”通常只是：
- 共用一个工作区
- 共用一套记忆
- 共用一个认证态
- 共用一个主会话

那最后只会产生“人格拼盘”。

官方文档的设计非常鲜明：
- 一个 agent = 一个独立作用域大脑
- 多 agent = 多个隔离大脑并行存在

也就是说，协同的前提从来不是“先互通”，而是：

> **先隔离，后通信。**

### 7.2 隔离哪些东西最重要
#### 1）人格隔离
SOUL / IDENTITY / AGENTS 不要互相覆盖。

#### 2）上下文隔离
不要让一个 agent 无限制继承另一个 agent 的主会话上下文。

#### 3）权限隔离
不同 agent 可以配置不同的 sandbox / tools allow / deny。

#### 4）认证隔离
不同人、不同渠道、不同业务线，不应共享同一份 agentDir。

#### 5）技能隔离
通用技能可以共享，但专属技能应该留在各自 workspace/skills 里。

### 7.3 真正的军团不是“全互通”
组织里高效协同不是每个人随时随地看得到一切，而是：
- 该知道的知道
- 不该拿到的拿不到
- 需要时通过路由和 handoff 获得

这也是多 Agent 设计里最容易被忽视的地方。

---

## 8）路由：为什么谁该答是协同系统的第一道关口
模块4讲过路由，但在多 agent 这里，路由的地位更高了。

### 8.1 多 Agent 的协同失败，很多死在第一步
常见失败场景：
- 群里问技术问题，战略 agent 抢答
- 私聊本该去财务 agent，却被默认 agent 吞掉
- 同一渠道多个账号，结果会话串错 agent

这些都不是“模型不聪明”，而是路由错误。

### 8.2 OpenClaw 的设计为什么强
OpenClaw 的绑定路由让你可以按：
- channel
- accountId
- peer
- guildId
- teamId
进行裁决

这意味着你可以把“谁该接这条消息”前置到系统层，而不是每次都靠 prompt 里提醒“请你让位”。

### 8.3 让位规则为什么必须和路由一起看
系统路由解决的是：
- 这条消息先到谁

行为协议解决的是：
- 到了以后该不该答
- 该不该转
- 该不该静默让位

如果没有后一层，你就会得到“技术上路由对了，但行为上仍然抢答”的混乱系统。

## 9）会话：多 Agent 的真正污染源往往不是能力，而是上下文串线
外部最佳实践里反复提到一个点：

> 多 agent 的最大瓶颈之一不是推理能力，而是 context management。

### 9.1 为什么会话串线比技能缺失更可怕
技能缺失，通常只是做不到某件事。
会话串线，则会导致：
- 错人格回答
- 错记忆回答
- 错对象延续上轮语气与状态
- 群和私聊上下文互相污染

### 9.2 OpenClaw 为什么把 sessionKey 做得这么细
OpenClaw 用不同的会话键来区分：
- `agent:<agentId>:<mainKey>`
- `agent:<agentId>:<channel>:group:<id>`
- `agent:<agentId>:<channel>:channel:<id>`
- topic / thread / cron / hook 等

这不是形式主义，而是上下文治理。

### 9.3 DM scope 为什么重要
官方提供了 `session.dmScope`：
- `main`
- `per-peer`
- `per-channel-peer`
- `per-account-channel-peer`

这决定了私聊是：
- 全部折叠进一个主会话
还是
- 按人 / 渠道 / 账户做更细隔离

对于军团或共享收件箱场景，这是极其关键的配置点。

## 10）Skills 与工作区：为什么军团既要共享能力，又要保留人格差异
模块5讲的是能力系统本身，这里讲的是它在军团里的组织方式。

### 10.1 官方的三层 skills 位置很关键
OpenClaw 的 skills 可以来自：
1. 内置 skills
2. `~/.openclaw/skills` 共享 skills
3. `<workspace>/skills` 单 agent skills

这恰好对应了军团能力建设的三层：
- 平台公共能力
- 组织共享能力
- 角色私有能力

### 10.2 为什么不能全放共享目录
如果所有 skills 都放到共享目录：
- 每个 agent 都看到太多不相关能力
- 人格与职责边界被削弱
- token 开销上升
- 误召回概率变大

### 10.3 为什么也不能全放本地目录
如果所有 skills 都只放本地：
- 公共 best practices 无法复用
- 能力治理成本飙升
- skill review / registry 很难统一

所以最优结构通常是：
- 通用基础技能：共享
- 角色专属技能：工作区本地
- 试验技能：局部挂载或 extraDirs

## 11）Sub-agent / sessions / agent-to-agent：协同链路到底怎么理解
这里要把几个容易混的概念拆开。

### 11.1 多 agent 不等于 sub-agent
**多 agent** 是 Gateway 里多个长期存在的独立脑。

**sub-agent** 更像某个主 agent 为了完成当前任务，临时拉出的隔离执行单元。

区别在于：
- 多 agent 偏组织编制
- sub-agent 偏任务编队

### 11.2 sessions_* / sessions_spawn 的意义
OpenClaw 提供的 session 工具链，本质上是在支持：
- 查看其他会话
- 拉起隔离运行
- 给其他会话发消息
- 跨会话调度

这说明 OpenClaw 对协同的理解不是“所有 agent 共用一个对话框”，而是：

> **一个主控会话协调多个独立运行单元。**

### 11.3 agent-to-agent 协同最关键的不是能发消息，而是有边界
很多团队看到“agent 可以给 agent 发消息”就很兴奋。
但真正关键的问题是：
- 谁可以调谁
- 什么时候调
- 调完如何追结果
- 失败谁兜底
- 是否允许越级发言

如果只解决“能通信”，没有解决“通信纪律”，协同系统一定会退化成噪声网络。

## 12）外部最佳实践给了什么启发
这部分是本章和模块5一样要重点展开的地方。

### 12.1 Anthropic：多 agent 不是炫技，而是为复杂任务拆解上下文
Anthropic 的多 agent 研究系统有一个关键启发：
- 主 agent 负责规划
- 然后创建并行 agent 去分别搜索
- 再统一汇总

这和军团协同高度一致。
真正有价值的不是“并行”本身，而是：

> **把复杂任务拆成多个更窄、更干净的上下文子问题。**

### 12.2 Anthropic：context engineering 是多 agent 的核心
Anthropic 2025 的 context engineering 文章其实给了多 agent 一个非常重要的底层逻辑：
- token 是有限资源
- 关键不是说什么 prompt，而是把什么上下文给谁

这正是多 agent 的根本价值之一：
- 不同 agent 只拿自己该拿的上下文
- 不把全部知识和全部任务都塞进一个大脑

### 12.3 LangChain：先单 agent，后多 agent
LangChain 的文章强调：
- 单 agent + 好工具更容易构建、调试、维护
- 只有当上下文和团队边界成为瓶颈时，多 agent 才合理

这个原则非常重要，因为它能防止“为了高级而高级”。

### 12.4 Vellum / Azure：真正的难点是 orchestration
这些资料的共识都不是“多 agent 很酷”，而是：
- handoff 设计
- context passing
- failure handling
- evaluation
- ownership

换句话说：

> **多 agent 的难度不在于启动多个模型，而在于如何组织多个角色之间的秩序。**

## 13）常见反模式

### 反模式1：把多 agent 当多 prompt
表现：
- 共用 workspace
- 共用 session
- 只改名字和 system prompt

后果：
- 假分工
- 真污染

### 反模式2：人人都能首答
表现：
- 群里一个问题，多个 agent 都来答
- 没有让位机制

后果：
- 用户感知混乱
- 组织角色失真

### 反模式3：没有明确 handoff 触发条件
表现：
- 什么时候该转给专家不清楚
- 什么时候该自己兜底也不清楚

后果：
- 不是转晚了，就是乱转

### 反模式4：共享所有记忆与能力
表现：
- 所有 agent 看同一套记忆与技能
- 什么都“共享”

后果：
- 看似协同，实则失去角色边界

### 反模式5：多 agent 却没有治理账本
表现：
- 谁调用了谁没有记录
- 谁超时、谁掉棒没有追责

后果：
- 复杂度上升，责任感下降

## 14）最佳实践

### 最佳实践1：先证明单 agent 已到边界，再上多 agent
只有当这些问题出现时，才值得引入多 agent：
- 单个上下文太肥
- 职责冲突严重
- 团队边界难治理
- 专业域差异太大

### 最佳实践2：先隔离人格、记忆、会话、权限
这是多 agent 的前提，不是优化项。

### 最佳实践3：绑定路由前置，行为让位补充
系统决定“先到谁”，行为协议决定“该不该答”。两层都要有。

### 最佳实践4：共享技能分层治理
- 平台通用技能共享
- 专家技能本地化
- 通过 registry / review 统一治理

### 最佳实践5：协同链路必须有升级与兜底
- 超时怎么办
- 掉棒怎么办
- 失败谁补位
- 是否允许主控兜底

### 最佳实践6：把会话视为协同资产，而不是聊天副产物
多 agent 协同质量，很大程度上取决于 session 设计是否干净。

### 最佳实践7：多 agent 评估不只看结果，还看 handoff 质量
至少要看：
- 是否路由正确
- 是否串线
- 是否重复劳动
- 是否专家真的回答了专家问题
- 是否有明确责任归属

## 15）模板 / 示例 / 训练题 / 验收口径

### 15.1 多 Agent 决策表

| 情况 | 建议 |
|---|---|
| 单 agent + 工具已能稳定完成 | 不要急着拆多 agent |
| 单会话上下文太肥、专业域冲突明显 | 可以考虑多 agent |
| 多角色需要独立人格 / 独立技能 / 独立认证 | 应该上多 agent |
| 只是为了“更高级” | 不建议 |

---

### 15.2 军团设计最小模板

```md
# 军团协同最小模板

## 角色层
- 战略主控：负责拍板、路由、兜底
- 专家执行：负责本专业首答
- 审核风控：负责反驳、验真、放行

## 系统层
- 每个 agent 独立 workspace
- 每个 agent 独立 agentDir
- 每个 agent 独立 sessions
- 绑定路由按 channel/accountId/peer 配置

## 行为层
- 明确首答权
- 明确让位规则
- 明确何时 handoff
- 明确超时/掉棒/兜底机制

## 治理层
- 建立协同台账
- 建立 review 机制
- 建立失职与升级规则
```

### 15.3 训练题

#### 训练题1
为什么多 agent 不是“给同一个 bot 写几套不同 prompt”？

标准答案：
- 因为真正的 agent 是独立作用域大脑
- 需要独立 workspace / agentDir / sessions / auth / skills
- 否则只是伪分工

#### 训练题2
为什么“谁该答”首先是路由问题，而不是礼貌问题？

标准答案：
- 因为消息先到错人，后续全错
- 绑定路由决定了系统的首个裁决点
- 行为让位只是第二层修正

#### 训练题3
一个军团里为什么既要共享 skills，又不能把所有 skills 全共享？

标准答案：
- 因为要兼顾能力复用与角色边界
- 全共享会导致误召回、边界弱化、token 开销上升
- 应按平台层 / 组织层 / 角色层分层治理

#### 训练题4
sub-agent 和长期多 agent 的差别是什么？

标准答案：
- sub-agent 偏任务编队
- 多 agent 偏长期组织编制
- 前者是临时执行单元，后者是长期人格与职责单元

### 15.4 验收口径
如果一个人真的理解了模块7，他至少应该能回答清楚：

1. 多 agent 的最小系统边界有哪些？
2. 为什么协同前必须先隔离？
3. 绑定路由和让位协议分别解决什么问题？
4. 为什么多 agent 的主要难点其实是 context management？
5. 多 agent、sub-agent、session-to-session 调度三者有什么差别？
6. 共享能力与本地人格应该如何分层？
7. 什么情况下不该上多 agent？

如果这些问题答不清，就说明他理解的仍然只是：

> “多 agent = 多开几个 bot。”

而不是：

> **“多 agent = 组织化的、可治理的、上下文有边界的协作系统。”**

# 本章一句话收口
**多 Agent 的真正价值，不是让系统更热闹，而是让不同人格、不同能力、不同权限、不同上下文的硅基单元在同一个 Gateway 下被有秩序地组织起来：该隔离的隔离，该路由的路由，该协同的协同，该让位的让位，该追责的追责。只有这样，军团才不是一群 bot，而是一套能作战的组织系统。**
