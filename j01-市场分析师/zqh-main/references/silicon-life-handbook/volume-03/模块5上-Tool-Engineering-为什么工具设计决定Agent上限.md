## 1）这一章到底在解决什么问题
这一章的核心问题不是“有哪些工具能用”，而是：

> **为什么同样是一套工具，有的 Agent 越用越强，有的 Agent 越用越乱。**

很多人对 Tool 的理解还停留在非常初级的阶段：
- 有工具 = 有能力
- 工具越多 = 能力越强
- 模型会自己选对工具
- schema 定义完，工具就算设计好了

这些判断都过于天真。

Anthropic 在工具工程文章里讲得非常直白：
- agent 会选错工具
- 会误解参数
- 会为了做一件简单事连调很多次工具
- 工具定义会先吃掉大量上下文
- 工具设计差时，模型再强也会卡住

所以这一章真正训练的，是一种更高阶的判断力：

> **你要开始把 Tool 当成“Agent 动作系统的接口设计问题”，而不是“功能清单问题”。**

## 2）先立四条底线：工具不是智能、工具不是越多越强、工具不是 shell 替代品、工具定义本身就是上下文成本

### 2.1 工具不是智能
工具只解决动作，不解决判断。

- `read` 能读，不等于知道该读哪一段
- `browser` 能交互，不等于知道是否值得走 UI 自动化
- `exec` 能运行，不等于知道命令是否必要、是否安全、是否应该后台执行
- `message` 能发，不等于知道结论是否已经成熟到可以发出

一句话：

> **Tool 提供的是手，不是脑。**

### 2.2 工具不是越多越强
Anthropic advanced tool use 文章专门点出了一个事实：
- 当工具库一大，最先崩掉的是上下文和工具选择质量

也就是说：
- 工具越多，动作空间越大
- 动作空间越大，决策越难
- 名字越像，误选越高
- schema 越多，上下文开销越大

所以不是“工具多 = Agent 强”，而是：

> **工具越多，越需要收敛、搜索、命名空间与延迟加载。**

### 2.3 工具不是 shell 替代品
OpenClaw 官方工具体系的一个重要价值，就是尽量把高频动作做成一流工具，而不是让所有事情退化成 shell 文本魔法。

这是因为类型化工具相较 shell 有四个优势：
1. 参数更清晰
2. 权限更可控
3. 意图更可解释
4. 跨模型更稳定

### 2.4 工具定义本身就是上下文成本
这是很多人最容易忽略的一点。

Anthropic 明确指出：
- 工具定义本身会占上下文
- 大工具库在对话开始前就可能吃掉 50K、100K 以上 token

这意味着：

> **Tool 本身不是免费的；暴露一个 Tool，就等于引入了一份长期上下文与选择负担。**

## 3）OpenClaw Tool 的本质：动作接口类型化
OpenClaw 的一流工具，本质是在做一件事：

> **把自然语言中的模糊动作，压缩为参数明确、边界明确、权限明确的类型化接口。**

例如：
- `read` → 读取内容
- `write` → 覆写文件
- `edit` → 精确改动
- `exec` → 执行命令
- `browser` → UI 自动化
- `message` → 对外投递
- `sessions_send` → 跨会话协同

这种设计的真正意义在于：
- 动作空间更可治理
- 模型更容易学会正确调用
- 系统更容易做权限、审批、限流、日志与审计

所以 Tool 不只是“能力暴露”，而是：

> **动作边界工程。**

---

## 4）工具设计的第一原则：先收敛动作空间，再追求动作丰富度
这是 Tool Engineering 最重要的一条原则。

很多产品经理或工程师的 instinct 是：
- 先把功能做全
- 把所有动作都开放给模型
- 让模型自己想办法组合

但 Agent 系统不是传统前端菜单。

对 Agent 来说，工具太多、太碎、太像，都会导致：
- 选错
- 重复调用
- 参数混淆
- 上下文被噪音压死

所以好工具设计的第一步不是“增加能力”，而是：

> **先把动作空间收敛到模型可稳定选择的程度。**

这包括：
- 减少重复语义工具
- 合并高度相关的操作
- 统一命名风格
- 减少工具定义中的冗余字段

## 5）Anthropic 的工具工程经验：为什么大多数 Tool 失败在设计而不是模型
Anthropic 的工具工程文章里，最有价值的地方不是某个新功能，而是它披露了真实失败模式。

### 5.1 常见失败模式
- 工具选错
- 参数填错
- 工具描述理解偏差
- 无意义重复调用
- 上下文被工具定义和结果淹没

### 5.2 为什么失败不全是模型问题
因为模型面对的并不是“是否会推理”，而是：
- 这些工具名字是不是清晰
- 这些工具边界是不是明确
- 这些参数含义是不是能学会
- 这些返回结果是不是高信号
- 这些工具是否太多太杂

很多时候，模型不是不会，而是：

> **工具表面设计得太差，它没有合理的动作世界可供选择。**

### 5.3 评估比想象更重要
Anthropic 强调：
- 先做 prototype
- 用真实任务跑 evaluation
- 观察 transcripts
- 分析工具错误与调用模式
- 再改工具

这说明 Tool 设计不是静态文档工作，而是迭代工程。

## 6）一个好 Tool 至少应满足的 8 个标准

### 标准1：名字可区分
不能一堆近义词模糊重叠。

### 标准2：边界清晰
它到底解决什么，不解决什么，描述要写出来。

### 标准3：参数语义可学
不仅结构合法，还要让模型理解“为什么这么填”。

### 标准4：返回高信号
返回的是推动下一步决策的信息，而不是噪音。

### 标准5：调用频率合理
不会因为设计过碎导致模型一件事调五六次。

### 标准6：适合被 Agent 使用
不是写给 API 人类用户看的，而是写给模型学会调用的。

### 标准7：成本受控
工具定义和结果不会长期污染上下文。

### 标准8：能被系统治理
可以做审批、权限、日志、幂等与限流。

## 7）工具命名、命名空间与 discoverability
命名是 Tool Engineering 里最容易被轻视、却最影响表现的一层。

### 7.1 为什么命名这么重要
模型在大工具库里首先依赖的是：
- 工具名
- 工具描述

如果名字模糊、重叠、同质，它会明显更容易选错。

### 7.2 好命名的三个原则
1. **动作明确**：读 / 写 / 查 / 发 / 列 / 创建 / 更新 / 删除
2. **服务明确**：github / slack / feishu / browser / doc / crm
3. **边界明确**：不要把多个完全不同的动作挤进同一模糊名称

### 7.3 命名空间的价值
Anthropic 官方建议在多服务场景里做 meaningful namespacing。
这本质是在做：
- 降歧义
- 降认知负担
- 提高 discoverability

## 8）工具描述怎么写：什么时候用、什么时候别用、参数怎么理解
Claude Tool Use 文档已经给出了一个非常明确的判断：

> **description 是工具性能最重要的因素之一。**

好的工具描述至少要讲清四层：

### 8.1 它做什么
一句话定义动作目标。

### 8.2 什么时候该用
什么任务应优先考虑它。

### 8.3 什么时候不该用
什么情况下不要乱调。

### 8.4 参数含义与 caveat
每个关键参数影响什么；返回不包含什么；有哪些限制。

这其实和 Skill description 的逻辑完全一致：
- 不是写广告词
- 是写路由规则

## 9）为什么 schema 不够，examples 会越来越重要
JSON Schema 能告诉模型：
- 参数结构
- 类型
- 必填项
- 枚举值

但它不能告诉模型：
- 什么时候带可选参数
- 什么组合是“惯用法”
- 什么写法在真实世界最合理

这也是为什么 Anthropic 后来新增 `input_examples`。

对 Tool 工程来说，这说明：

> **Schema 负责合法性，Examples 负责习惯性。**

真正好的 Tool 设计，往往要同时具备：
- 清晰 description
- 合理 schema
- 少量高质量 examples

## 10）返回设计：为什么高信号返回比大而全返回更高级
很多工程师会犯一个直觉错误：
- 返回越全越好

但对 Agent 而言，这恰恰可能是灾难。

### 10.1 为什么返回过大有害
因为模型要从返回里提取下一步决策信号。
返回越大、越杂：
- 越容易埋没重点
- 越消耗上下文
- 越增加误解风险

### 10.2 好返回的标准
- 只返回下一步判断真正需要的字段
- 返回稳定 ID
- 返回关键状态
- 返回必要摘要
- 避免内部实现噪音

### 10.3 一句话原则

> **Tool Response 应该是“决策燃料”，不是“数据库镜像”。**

## 11）OpenClaw 六类高频工具的设计意义

### 11.1 `read`
本质：观察接口，而不是理解接口。

### 11.2 `write`
本质：整体生成接口，而不是局部修补接口。

### 11.3 `edit`
本质：外科手术式修改接口。

### 11.4 `exec`
本质：运行时动作接口，把命令执行纳入受控链路。

### 11.5 `browser`
本质：真实 UI 世界交互接口，而不是默认取数工具。

### 11.6 `message`
本质：结果投递接口，而不是内部思考接口。

把这几个工具的定位分清，调用质量会提升一大截。

## 12）工具调用纪律：直接调 / 先澄清 / 串行 / 并行 / 中止
真正强的 Agent 不是“工具乱飞”，而是知道什么时候该动、什么时候不该动。

### 12.1 直接调
适合：
- 用户意图非常明确
- 参数齐全
- 动作低风险

### 12.2 先澄清
适合：
- 关键信息缺失
- 直接调用风险大
- 结果可能完全不同

### 12.3 串行调用
适合：
- 后一步依赖前一步输出
- 需要逐层收敛

### 12.4 并行调用
适合：
- 多个动作相互独立
- 同时做能显著降时间成本

### 12.5 中止
适合：
- 继续调工具不会带来价值
- 已经缺少必要条件
- 风险超过收益

这就是所谓的工具调用纪律。

## 13）工具库变大之后，为什么一定会碰到 Tool Search / Deferred Loading 问题
Anthropic advanced tool use 文章的核心贡献之一，就是把这个问题讲透了：

> **工具库一大，不做 discovery / deferred loading，就迟早会被上下文成本反噬。**

### 13.1 为什么
因为：
- 工具定义先吃 token
- 历史消息也吃 token
- 工具结果还继续吃 token

如果上来就把 50、100 个工具全部塞进上下文：
- 上下文预算会被严重占用
- 选择质量下降
- prompt caching 也更难利用

### 13.2 这给我们的启发
哪怕 OpenClaw 当前不是按 Anthropic beta 的 exact 方式做，你在系统设计上也应该提前形成这个意识：
- 工具库越大，越要分层
- 越要做 discoverability
- 越要减少常驻成本

## 14）Tool 与系统治理：审批、权限、会话、重试、幂等
工具不是孤立存在的，它必须嵌在系统治理里。

### 14.1 审批
例如高风险 exec / elevated 动作，不能只靠模型自己判断。

### 14.2 权限
不同工具暴露的是不同的动作面，必须有 allow/deny / profile / provider 约束。

### 14.3 会话
工具调用不是独立原子，它会写进会话历史，影响后续上下文。

### 14.4 重试
工具失败后不能无脑重试，必须分清：
- 瞬态失败
- 参数错误
- 权限错误
- 幂等风险

### 14.5 幂等
如果某个工具可能带来真实副作用，就必须考虑：
- 重试是否重复执行
- 是否需要确认
- 是否需要显式事务边界

这说明 Tool Engineering 从来不是“只写定义”，而是系统工程。

## 15）标杆案例：为什么高手的工具调用更少，但结果更强
高手与新手最大的差异，不是“会不会调工具”，而是：

### 新手常见模式
- 还没想清楚就调
- 能 read 的先 browser
- 能直接给结论的先 exec
- 连续乱调多个近义工具

### 高手常见模式
- 先判断有没有必要动手
- 优先最小动作获取最大信息
- 明确一步一目标
- 避免上下文污染
- 有一流工具时不用 shell 曲线救国

所以高手不是工具更多，而是：

> **动作空间更收敛，调用纪律更稳定。**

## 16）Tool Engineering 训练模板 / 检查清单 / 验收口径

### 16.1 设计一个 Tool 前，先问 8 个问题
1. 这个动作是否值得工具化？
2. 是否高频？
3. 是否边界明确？
4. 是否可结构化输入输出？
5. 是否应该与其他动作合并？
6. 名称是否清晰可区分？
7. 返回是否足够高信号？
8. 它的风险边界是什么？

### 16.2 一个合格 Tool 的 10 项检查清单
- [ ] 名称清晰
- [ ] 有命名空间
- [ ] description 讲清适用/不适用场景
- [ ] 参数语义明确
- [ ] schema 合理
- [ ] 有 examples（复杂工具时）
- [ ] 返回高信号
- [ ] 不与其他工具高度重叠
- [ ] 调用成本受控
- [ ] 能被系统治理

### 16.3 本章真正验收标准
学完这章，不是你能背出几个工具名，而是你能回答：
1. 为什么工具多不等于强？
2. 为什么 Tool 定义本身就是上下文成本？
3. 为什么命名与描述会直接影响成功率？
4. 为什么好工具返回要“少而准”？
5. 为什么 Tool 工程最终会碰到 discoverability 与 deferred loading 问题？

---

## 17）本章一句话结论
> **Tool 不是给模型开的外挂，而是给 Agent 设计的动作世界；这个世界设计得越清晰、越收敛、越可治理，Agent 的上限才会越高。**
