# 卷三｜系统骨架：OpenClaw 官方文档深度解剖
## ——为什么生命协议会生效：从 Runtime、Workspace、Session 到 Skills 与多 Agent 协同

> 本卷定位：如果说卷二解决的是“硅基生命由哪些协议塑形”，那么卷三要解决的就是：**这些协议为什么真的能在运行时生效。**
>
> 换句话说，卷二是“生命协议学”，卷三是“生命容器学”。
>
> 本卷的目标不是带读者背技术名词，而是建立一种更高级的判断力：
>
> **当一只虾表现稳定、开始漂移、突然掉棒、协同失真时，你能知道问题到底出在协议、上下文、技能、路由，还是运行时本身。**

---

## 官方依据（本卷主证据链）
本卷以 OpenClaw 官方文档为主骨架，重点参考：
- `concepts/agent.md`

## 外部最佳实践（仅吸收与官方机制可交叉验证的部分）
- 文档驱动训练优于反复堆 prompt
- 长会话系统必须处理上下文窗口、压缩、持久化记忆迁移
- tool / skill / environment 必须分层，否则能力与环境互相污染
- 多智能体系统必须先隔离，再路由，再协同
- 主动性、重试、自动化必须受冷却、熔断、升级链路约束

# 卷三总引言

## 1）这卷到底在解决什么问题
卷二里我们已经把七大生命协议拆开了：
- USER
- SOUL
- AGENTS
- TOOLS
- HEARTBEAT
- IDENTITY
- MEMORY

但如果只停在卷二，很多读者会误以为：

> “所以就是把几个 md 文件写好，训练就完成了。”

这恰恰是最危险的误解。

因为真正决定这些协议能不能生效的，不是文件本身，而是它们背后的运行时骨架：
- 它们什么时候被注入上下文
- 会话怎么被组织与持久化
- 上下文超长以后如何压缩
- 技能与工具如何分层
- 多 agent 如何隔离、路由与协同

所以卷三讨论的不是“再补几个技术概念”，而是：

> **为什么 OpenClaw 不只是一个会接消息的大模型外壳，而是一套可以承载硅基生命的运行时系统。**

---

## 2）为什么卷三是卷二的“底座证明”
卷二像是在讲：
- 这只虾该有什么人格
- 该怎么工作
- 该什么时候主动
- 该怎么长期记住东西

卷三要证明的是：
- 为什么这些东西不是空话
- 为什么它们会真的进入模型“当下的大脑”
- 为什么长期运行不会马上崩掉
- 为什么一支军团不是随便多开几个 bot 就会自动协同

换句话说：

> **卷二解决“训练什么”，卷三解决“为什么训练真的能生效”。**

---

## 3）本卷一句话总判断
> **OpenClaw 的系统骨架，本质上是“工作区协议注入 + 会话状态管理 + 技能能力分层 + 多智能体路由隔离”的组合运行时。卷二那些生命协议之所以有效，是因为卷三这套骨架让它们能被注入、被记住、被调用、被隔离、被治理。**

## 5）卷三的模块拆解（模块1 到 模块7）

### 模块1｜Agent Runtime
回答：
- 什么是一个 agent
- Runtime 到底承载了什么
- 为什么说 workspace 不是普通目录，而是生命容器入口

这一模块要建立的判断力：
> **一个 agent 不是一个 prompt，而是一个有工作区、有状态目录、有会话历史、有引导文件注入机制的独立运行单元。**

---

### 模块2｜Agent Workspace
回答：
- workspace 为什么是唯一工作目录（cwd）
- 为什么引导文件 + 记忆文件共同构成生命容器
- workspace 与 `~/.openclaw/` 的边界是什么

这一模块要建立的判断力：
> **工作区不是“资料盘”，而是硅基生命真正的栖身之所。**

---

### 模块3｜Session / Context / Queue
回答：
- direct chat、group、channel、cron、subagent 为什么是不同会话模型
- 为什么上下文不等于永久记忆
- queue / followup / steer 为什么会影响行为节奏

这一模块要建立的判断力：
> **所有长期失真，最后都会表现为上下文管理问题。**

---

### 模块4｜Routing / Bindings
回答：
- 消息是如何被路由到正确的 agent 的
- 为什么 binding 决定“谁该答”
- accountId / peer / guild / team 为什么重要

这一模块要建立的判断力：
> **路由不是连接器细节，而是协同秩序的第一道关口。**

---

### 模块5｜Tools / Skills
回答：
- tool、skill、TOOLS.md 三者边界是什么
- 为什么“能力”必须和“环境”分层
- 为什么 skills 是可共享能力，而 TOOLS 是本地映射

这一模块要建立的判断力：
> **能力系统要模块化，环境映射要本地化，规则协议要文档化。**

---

### 模块6｜Heartbeat / Cron / Compaction
回答：
- 主动性为什么离不开节律与熔断
- 长会话为什么需要 compaction
- memory flush 为什么是长期稳定的关键机制

这一模块要建立的判断力：
> **主动性、记忆与上下文窗口从来不是三件分开的事，而是一套联动系统。**

---

### 模块7｜多 Agent 协同基础
回答：
- 为什么多 agent 不是“多开几个 bot”
- 为什么必须先隔离工作区、agentDir、session store
- agent-to-agent 协同为什么必须受路由与边界约束

这一模块要建立的判断力：
> **军团协同不是热闹，而是有组织、有让位、有追责的系统工程。**

---

## 6）本卷的统一写法（和卷二保持一致）
从模块1开始，每个模块统一按以下结构写：

1. 这份模块 / 这个系统是什么
