# 卷三｜系统骨架：OpenClaw 官方文档深度解剖
## ——为什么生命协议会生效：从工作区注入、会话压缩到技能与多智能体路由

> 本卷定位：如果说卷二解决的是“硅基生命由哪些协议塑形”，那么卷三要解决的就是：**这些协议为什么真的能在运行时生效。**
>
> 这不是给开发者看的纯架构说明，也不是给运营看的浅层使用手册，而是面向：
> - **技术负责人**：看懂 OpenClaw 的运行边界、状态来源、路由结构与系统约束
> - **训练师 / 运营**：理解为什么某些训练动作有效、为什么某些写法会失真、为什么某些问题不是“模型笨”，而是“骨架没立住”

---

## 外部最佳实践（只吸收与官方机制可对齐的部分）
- 文档驱动训练优于反复堆 prompt
- 长会话系统必须处理上下文窗口、摘要压缩与持久化记忆迁移
- Skills / tools / environment mapping 需要分层，避免耦合污染
- 多智能体协同必须有身份隔离、会话隔离与明确路由优先级
- 主动性、重试与自动化必须受节律、冷却与熔断约束

## 1）本卷一句话结论
> **OpenClaw 不是“一个能接消息的大模型外壳”，而是“工作区协议注入 + 会话状态管理 + 技能能力分层 + 多智能体路由”共同构成的运行时骨架。卷二那些生命协议之所以有效，是因为卷三这套骨架让它们能被注入、被记住、被调用、被隔离、被治理。**

---

## 2）为什么卷三是卷二的底座证明
卷二讲了七大生命协议：USER、SOUL、AGENTS、TOOLS、HEARTBEAT、IDENTITY、MEMORY。

但如果不回答下面这几个问题，卷二很容易被误解成“写几份 md 就好了”：

1. 这些文件是**什么时候**进入模型上下文的？
2. 为什么有的文件是每次会话都加载，有的不是？
3. 会话变长以后，旧消息怎么处理？压缩了以后，哪些东西还在？
4. 工具、Skills、环境备注，为什么不能混着写？
5. 多个 agent 在一个网关里时，为什么不会自动天然协同？
6. 为什么 direct chat、group、cron、subagent 的会话行为不一样？

所以卷三真正的使命，是把这些底层问题说透：

> **卷二解决“训练什么”，卷三解决“为什么训练真的能生效”。**

如果不分层，训练者会把所有问题都误判成“模型没理解”。

## 4）图B：消息到交付的数据流图

```text
入站消息
   ↓
路由到 agentId + sessionKey
   ↓
加载工作区注入文件（USER/SOUL/AGENTS/...）
   ↓
拼装当前上下文（近期消息 + 历史 + 可选记忆）
   ↓
必要时压缩旧上下文 / 触发 memory flush
   ↓
模型产出决策
   ↓
调用 tools / skills / sessions_* / subagents
   ↓
写文件 / 发消息 / 产生日志 / 更新会话状态
   ↓
交付回执 + 后续状态
```

### 这张图在训练什么
训练的是：
- 为什么“写盘”比“说过一次”更重要
- 为什么上下文不是无限的
- 为什么每次调用工具之后，系统状态可能已经变了
- 为什么同一个任务在 direct chat、group、cron 下行为会不同

## 6）模块一：Workspace 注入——生命协议为什么能进入大脑

### 6.1 官方机制是什么
根据 `concepts/agent.md`，OpenClaw 在工作区中期望以下引导文件：
- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `BOOTSTRAP.md`
- `IDENTITY.md`
- `USER.md`

在**新会话的第一轮**，OpenClaw 会把这些文件内容直接注入智能体上下文。

这件事的重要性在于：

> **这些文件不是“放在那里备查”，而是运行时真正读进去的。**

这就是为什么卷二里每份文档都必须像协议，而不能像散文、备忘录或情绪宣言。

### 6.2 这在硅基生命里对应什么
这相当于给一只虾做“开机注魂”：
- USER = 它服务谁
- SOUL = 它是谁
- AGENTS = 它怎么做事
- TOOLS = 它如何理解现场对象
- IDENTITY = 它如何对外被认出

所以工作区不是普通文件夹，而是：

> **生命容器。**

### 6.3 为什么这比重复堆 prompt 更强
外部社区关于 AI agent 的成熟经验已经反复证明：
- Prompt 更像临时对话修正
- Workspace files 才是长期稳定底盘

因为文件有四个特性：
1. 可版本化
2. 可迁移
3. 可审计
4. 可复制训练

这就是文档驱动训练优于 prompt 驱动的根本原因。

### 6.4 常见误区
1. **把文件写成宣传文案**：模型读到的是漂亮话，不是可执行条款
2. **把 TOOLS 写成权限表**：会污染技能边界
3. **把 USER 写成人物简介**：不提供决策函数
4. **把 SOUL 写成口号**：不提供边界与冲突默认
5. **以为文件缺失无所谓**：实际上缺失会被注入“文件缺失”标记

### 6.5 最佳实践
- 所有引导文件都应该尽量**结构化**、**低歧义**、**可验真**
- 行文优先写“规则 / 边界 / 优先级 / 例外 / 验收”，少写抽象形容词
- 让每份文件只做一件事，不跨层抢职能

## 7）模块二：Session / Context / Compaction——长对话为什么不会直接崩掉

### 7.1 官方机制是什么
根据 `concepts/session.md`：
- 每个智能体的一个直接聊天会话可以视为主会话
- direct chat、group、channel、cron、hook、node 会映射为不同的 session key
- 会话状态由 Gateway 拥有，JSONL 记录持久化在 `~/.openclaw/agents/<agentId>/sessions/`

根据 `concepts/compaction.md`：
- 长会话会触发**自动压缩**
- 压缩会把较早对话总结成紧凑摘要，并持久化到 JSONL
- 压缩前可运行**静默 memory flush**，把应该长期保留的东西写入磁盘

### 7.2 这在训练里意味着什么
训练者必须接受一个事实：

> **上下文窗口是稀缺资源，不是无限记忆。**

所以一个成熟训练体系必须同时管理三件事：
1. 当前上下文中保留什么
2. 哪些旧内容要压缩成摘要
3. 哪些真正重要的东西要迁移到持久记忆

### 7.3 compaction 的真正价值
很多人把 compaction 误解成“削弱上下文”。
其实恰恰相反：

> **compaction 是长会话能继续跑下去的必要条件。**

没有它：
- 工具输出越积越多
- 历史消息越堆越厚
- 模型上下文会被旧信息挤爆

有了它：
- 近期消息保持原样
- 较早消息变成结构化摘要
- 关键长期信息有机会经 memory flush 写盘

### 7.4 为什么 session 和记忆不能混为一谈
- **session** = 当前会话上下文管理
- **memory** = 跨会话持久记忆管理

session 解决“这轮上下文怎么活下去”，memory 解决“跨时间怎么不失忆”。

很多训练者把所有东西都留在会话里，不做记忆迁移，结果就是：
- 当下聊得很顺
- 一压缩就丢
- 隔天再问就像换脑

### 7.5 最佳实践
- 把 session 看成工作台，把 memory 看成档案库
- 重要决策不要只存在聊天里，要写盘
- 对“会长期影响行为”的内容，优先升格为 MEMORY.md 或 daily memory
- 用 `/compact` 不是认输，而是主动整理战场

### 7.6 常见误区
1. 误以为 JSONL 历史 = 永久可用上下文
2. 误以为压缩后所有细节还会原样留在窗口里
3. 不理解 memory flush 的价值，导致长期偏好不落盘
4. 用同一个主会话承载过多异质任务，造成上下文污染

## 8）模块三：Skills / Tools——能力系统为什么必须分层

### 8.1 官方机制是什么
根据 `concepts/agent.md`：
- 核心工具始终可用，受工具策略约束
- `TOOLS.md` **不控制**哪些工具存在
- Skills 从内置、`~/.openclaw/skills`、工作区 `skills/` 等位置加载

这意味着 OpenClaw 的能力系统至少有三层：
1. **Runtime tools**：read / write / edit / exec / browser / message …
2. **Skills**：把某领域能力封成可复用模块
3. **TOOLS.md**：记录你本地环境里独有的“对象映射”

### 8.2 为什么一定要分层
因为这三层如果混了，就会出现灾难：
- 把环境写进 skills → 技能不可迁移、容易泄露
- 把技能说明写进 TOOLS → 污染本地备注，边界模糊
- 把所有流程都塞进 AGENTS → 工作协议与能力协议缠死

### 8.3 在训练中怎么理解这三层
- **Tool** = 手
- **Skill** = 技法
- **TOOLS.md** = 现场地图

你不能因为一个木匠会用锤子，就把整个工地地图写进锤子说明书里。

### 8.4 标杆实践启发
外部成熟社区里，优秀框架几乎都强调：
- 能力要模块化
- 环境要本地化
- 规则要协议化

OpenClaw 这套设计正好允许训练师把：
- 通用能力沉淀成 skills
- 团队约定保留在 TOOLS.md
- 工作规则沉淀在 AGENTS.md

### 8.5 常见误区
1. 想靠 TOOLS.md 增加不存在的工具
2. 想让一个 skill 携带所有组织特定环境信息
3. 把工具层、流程层、环境层混成一锅粥
4. 认为技能越多越强，忽视上下文负担与维护成本

### 8.6 最佳实践
- 共享的能力写成 skill
- 本地的对象写进 TOOLS.md
- 工作流程写在 AGENTS.md
- 人格、边界、偏好不要塞进能力层

## 9）模块四：Multi-Agent——军团为什么不会天然协同

### 9.1 官方机制是什么
根据 `concepts/multi-agent.md`：
- 一个 agent 是一个**完全独立作用域的大脑**
- 每个 agent 有自己的 workspace、agentDir、session store
- Gateway 通过 bindings 把入站消息路由到不同 agent
- 多个渠道账户（如两个 WhatsApp）也可映射到不同 agent

这意味着：

> **多 agent 不是“同一个脑子换不同名字”，而是多个彼此隔离的生命体。**

### 9.2 为什么这对军团训练极其关键
很多团队一说多智能体，就默认：
- 让几个 agent 同时在群里说话
- 谁想到什么谁先答
- 大家共享同一套记忆

这是最危险的误解。

OpenClaw 官方骨架真正强调的是：
1. **先隔离**（工作区、会话、认证、状态）
2. **再路由**（binding / accountId / peer / guild/team）
3. **再协同**（会话工具、subagent、sessions_send、让位规则）

没有隔离，根本没有稳定协同。

### 9.3 协同为什么不等于“都能互发消息”
根据 `concepts/session-tool.md`：
- `sessions_list / sessions_history / sessions_send / sessions_spawn` 提供跨会话能力
- 但这些工具本身是受边界约束的，不是无限制广播

这说明：

> **协同是一种被约束的系统能力，不是自由聊天。**

### 9.4 在龙虾军团里怎么落地
对你这种军团式训练，卷三的关键不是“怎么多开几个 agent”，而是：
- 让每个 agent 拥有独立人格与工作区
- 用 bindings 保证消息进入对的脑子
- 用身份锚点保证让位与追责
- 用会话工具做受控协同，而不是杂音式群殴

### 9.5 常见误区
1. 多个 agent 共享同一个 workspace
2. 复用同一个 agentDir，导致认证与会话冲突
3. 不设 binding，全靠默认回退智能体
4. 群聊里谁都能答，没有主责与让位规则
5. 把多 agent 当“多窗口”，而不是多生命体

### 9.6 最佳实践
- 一 agent 一 workspace 一 session store
- 先定义身份与职责，再开协同
- 用 bindings 保证入站就被正确路由
- 协同必须绑定主责、接棒、验收与收口

## 10）从入门到最佳实践：系统骨架三阶段升级法

### 阶段一：能跑起来
- 有 workspace
- 有基础引导文件
- 能接消息
- 能调用核心工具

### 阶段二：能长期稳定
- 理解 session / compaction / memory flush
- 重要信息写盘，不只留在聊天
- skills / tools / environment 边界清晰

### 阶段三：能军团化协同
- 多 agent 隔离清楚
- bindings 路由清楚
- 身份与协同协议清楚
- 让位、接棒、验真、收口清楚

## 11）常见误区与反模式

### 11.1 误区：工作区只是文件夹
反模式：把 workspace 当普通资料盘，不理解它是运行时注入入口。  
后果：卷二那些协议全部失去“开机注魂”价值。

### 11.2 误区：长会话靠模型硬扛
反模式：不理解 compaction，不做记忆迁移。  
后果：上下文污染、压缩失忆、长期漂移。

### 11.3 误区：skills 越多越强
反模式：无节制装技能，不做边界分层。  
后果：能力杂糅、调用混乱、上下文负担过重。

### 11.4 误区：多 agent 天然会协同
反模式：几个 agent 同时放群里自由发言。  
后果：抢答、串角色、责任不清、用户困惑。

### 11.5 误区：所有问题都是 prompt 问题
反模式：一出错就补 prompt。  
后果：越补越厚，越厚越漂，系统问题始终没修。

### 11.6 误区：记忆 = 聊天记录
反模式：什么都不写盘，什么都指望上下文记住。  
后果：会话性失忆、决策反复、经验无法沉淀。

### 11.7 误区：把治理问题当技术问题
反模式：明明是身份、路由、主责问题，却只调模型。  
后果：技术越强，组织噪音越大。

## 12）验收口径（DoD）
本卷写完，不是“读懂概念”就算完成，而是达到以下标准：

### 12.1 技术负责人验收口径
- 能说清 workspace 注入、session、compaction、skills、多 agent 的边界关系
- 能解释为什么卷二的生命协议不是“文案”，而是运行时输入
- 能定位一个问题到底出在协议层、运行时层、能力层还是协同层

### 12.2 训练师/运营验收口径
- 能判断为什么一只虾长期跑偏
- 能设计“先做人、再做事、最后主动”的训练顺序
