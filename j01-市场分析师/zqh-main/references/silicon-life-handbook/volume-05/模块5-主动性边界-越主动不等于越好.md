# 卷五｜长期表现：为什么有些 Agent 越跑越稳，有些越跑越差
## 模块5｜主动性边界：越主动不等于越好
### ——为什么长期 Agent 真正的难题，不是“如何让它主动”，而是“如何让它只在对的时候主动”

> 官方依据：
> - `automation/cron-vs-heartbeat.md`：Heartbeat 适合周期性监测与批处理检查，Cron 适合精确时间点任务与独立会话，Lobster 适合多步骤、需审批的工作流。三者的职责边界、会话模型、token 成本、触发方式完全不同。
> - `concepts/agent-workspace.md`：HEARTBEAT.md 是 agent 的节律检查清单，内容应保持短小，避免 token 烧光在巡检本身上。
> - `concepts/context.md`：所有主动性动作都会占用 context；过度主动会导致噪音堆积，反噬判断质量。
> - `gateway/heartbeat.md`：Heartbeat 默认 30 分钟一次；如果巡检内容过多或触发逻辑失控，会迅速消耗资源并制造不必要的消息流。
> - `concepts/compaction.md`：长期 agent 在压缩历史后，如果此前主动行为没有被清晰提炼，重要承诺与决策容易被噪音淹没。
> - `concepts/memory.md`：主动行为产生的决策与承诺，必须写入 memory 才能成为长期可追溯的锚点。
>
> 外部训练参照：
> - 行业关于 **Agent Autonomy Levels / Guardrails / Proactive vs Reactive / Rate Limits / Human Approval Flows** 的工程共识越来越清晰：**真正成熟的系统不是“能主动就让它主动”，而是“在什么层级上允许主动、什么动作必须审批、什么边界必须熔断”。**
> - Levels of Autonomy 框架明确 L1（仅建议）、L2（有限执行）、L3（有监督自治）、L4（完全自治）的分级；越高层级越需要更重的 guardrails。
> - Autonomous agents 的治理经验强调：**多层防护（identity control、progressive rollout、rate/cost limits、hard ceilings）** 是防止过度主动引发事故的关键。
> - Proactive vs Reactive 的治理经验指出：**proactive 并不天然优于 reactive**；错误场景下过度主动会制造噪音、打断用户、破坏信任。

## 1）这份机制是什么
很多人一听“主动性边界”，会理解成：
- 不要太勤快
- 不要太啰嗦
- 不要乱发消息

如果只看到这些，就会把问题想得太浅。

在长期 agent 系统里，真正要回答的问题是：

> **这只虾在什么层级上允许自己决定动作、什么动作必须用户确认、什么边界绝对不能逾越、什么情况下必须主动停止。**

从训虾派视角重新定义：

> **主动性边界，不是让 agent 更努力或更克制，而是为它建立一套分级授权、触发条件、停止条件、熔断条件和审计机制。**

这一章不是讨论“要不要主动”，
而是讨论：

- 怎样区分“有价值的主动”与“制造噪音的主动”
- 怎样区分“监测型主动”与“执行型主动”
- 怎样区分“建议型主动”与“自治型主动”
- 怎样区分“单次提醒”与“长期巡检”
- 怎样设计“主动行为的发生边界、频率边界、成本边界、权限边界”

OpenClaw 提供的底层机制非常清晰：
- **Heartbeat**：周期性、低开销、批处理监测
- **Cron**：精确时间点、可隔离会话、可切模型
- **Reminder / Message**：一次性或按需触发
- **Lobster**：多步骤、需审批、可暂停可恢复

这些机制本身是中性的；
真正决定“越主动越好还是越主动越乱”的，是**你怎么为 agent 设定主动性边界**。

## 2）为什么主动性边界比很多人想象中更关键
主动性边界被低估，是因为很多人会直觉地认为：

> “能主动当然是好事。”

这是一个非常典型的错觉。

### 2.1 过度主动会在三个维度同时制造问题
#### 第一维度：噪音
频繁消息、频繁提醒、频繁检查，会迅速让用户产生疲劳与屏蔽反应。

#### 第二维度：成本
每一次主动行为都会消耗：
- Token
- API 调用
- 会话空间
- 审计资源

#### 第三维度：风险
越主动的 agent，越容易：
- 越界行动
- 误判时机
- 错误执行

### 2.2 错误类型的主动，比不主动更致命
不主动的 agent，最多是“被动等待”。
过度主动的 agent，会：
- 打断工作流
- 淹没重要信息
- 破坏信任
- 制造事故

### 2.3 行业上“autonomy levels”经验已经非常明确
外部关于 AI Agent 自治层级的研究，区分了：
- L1：仅建议，不执行
- L2：有限执行，关键动作需审批
- L3：有监督自治，带 guardrails
- L4：完全自治，需最高级治理机制

每一层对应的主动性边界、审批流、熔断条件完全不同。

### 2.4 OpenClaw 本身的机制也在强调“边界”
官方 `cron-vs-heartbeat.md` 讲得非常清楚：
- Heartbeat：适合周期性监测与批处理，但不要把所有事情都塞进去
- Cron：适合精确时间点，但要考虑是否需要隔离会话
- Reminder / Message：适合一次性触发
- Lobster：适合多步骤、需审批的复杂工作流

这些机制的设计本身就是“边界”的体现。
真正的问题不是“哪个功能更强”，而是：

> **你有没有为这只虾设计清晰的主动性分级与边界条件？**

## 3）核心目的（Purpose）
模块5的核心目的，不是“少主动一点”，而是建立一套让 agent 的主动性：
- 有层级
- 有边界
- 有条件
- 有审计

的系统。

它主要解决 7 个根问题：

### 3.1 什么叫“有价值的主动”
不是所有主动都是噪音。
必须明确：
- 监测到关键风险并预警 → 有价值
- 周期性检查项目健康 → 有价值
- 在用户明确授权后执行 → 有价值
- 没事就问“要不要帮你” → 噪音

### 3.2 什么叫“越界的主动”
- 未授权执行
- 频率过高
- 非必要打断
- 错误时机触发

### 3.3 如何区分“建议型”和“执行型”
- 建议型：可以更主动，因为最终由人拍板
- 执行型：必须更克制，因为直接产生现实后果

### 3.4 如何设计“停止条件”和“熔断条件”
- 停止条件：什么情况下当前巡检/任务应停止
- 熔断条件：什么情况下整个主动性机制应暂停或升级

### 3.5 如何让主动行为产生可追溯记录
每一次主动行为，都应进入：
- 日志
- 审计
- Memory（如果是重要决策/承诺）

### 3.6 如何避免“主动性污染主会话”
Heartbeat vs Isolated Cron 的选择，本质上是在回答这个问题。

### 3.7 如何与成本控制联动
主动行为必须与：
- Rate limit
- Cost limit
- Token limit
联动，否则会失控。

## 4）拟人化映射与仿真价值
如果做拟人化映射，主动性边界最像什么？

> **像一个组织里的员工，什么时候可以自己决定、什么时候必须请示、什么时候可以行动、什么时候必须停手。**

### 4.1 没有主动性边界的 agent，像一个“不会停手的实习生”
- 永远在动
- 永远在问
- 永远在提醒
- 永远不知道什么时候该安静

### 4.2 有主动性边界的 agent，像一个“成熟的专业人士”
- 知道什么时候该主动
- 知道什么时候该等待
- 知道什么时候该请示
- 知道什么时候该停手

### 4.3 主动性边界不是“懒惰”，而是“节律感”
真正高质量的人类工作状态，不是永远不主动，
而是：

> **在对的时候主动、在错的时候克制。**

这正是高水平 agent 需要仿真的能力。

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定信任的存续
过度主动会迅速消耗信任；
恰到好处的主动会持续积累信任。

### 5.2 决定成本的可控性
没有主动性边界，token 和 API 调用一定会爆炸。

### 5.3 决定噪音与信号的比
主动行为越多，不代表信息越多；
很多时候只是噪音越多。

### 5.4 决定安全性
执行型主动如果没有边界，一定会发生事故。

### 5.5 决定可审计性
每一次主动行为，都应能回溯“为什么触发、做了什么、结果如何”。

### 5.6 决定长期可持续性
没有边界的主动系统，最终要么被用户关掉，要么被成本压垮。

## 6）Heartbeat / Cron / Reminder / Message / Session 的职责边界
这一节必须讲透，因为 OpenClaw 提供了多种主动性机制，
但很多人没有意识到它们的职责边界完全不同。

### 6.1 Heartbeat：周期性监测与批处理
- 默认每 30 分钟
- 在主会话运行
- 适合：检查 inbox、calendar、notifications、project health 等多任务
- 不适合：精确时间点任务、重分析、需要隔离上下文的任务
- 官方建议：HEARTBEAT.md 保持短小，避免 token 烧光在巡检本身

### 6.2 Cron：精确时间点与隔离会话
- 支持 5/6 位 cron 表达式 + timezone
- 可选择 main / isolated / custom session
- 可切换 model / thinking
- 适合：每日固定时间报告、每周深度分析、一次性提醒
- 不适合：需要主会话上下文的监测

### 6.3 Reminder / Message：一次性或按需触发
- 适合：“20 分钟后提醒我”“现在发一条消息给某人”
- 不适合：周期性巡检

### 6.4 Lobster：多步骤工作流 + 审批
- 适合：固定管道、多工具调用、需要人工审批、可暂停可恢复
- 不适合：简单单轮任务

### 6.5 Session 选择：Main vs Isolated
- Main：需要共享历史与上下文
- Isolated：干净上下文、独立历史、可切模型

### 6.6 五者边界一句话总结
- **Heartbeat**：周期性、批处理、共享上下文
- **Cron**：精确时间点、可隔离、可切模型
- **Reminder / Message**：一次性触发
- **Lobster**：多步骤工作流 + 审批
- **Session**：决定是否共享历史与上下文

真正高质量的主动性设计，必须知道自己在用哪一个。

## 7）市面上关于 Agent 主动性最容易踩的坑
### 坑 1：把所有主动行为都塞进 Heartbeat
这会导致：
- HEARTBEAT.md 膨胀
- Token 消耗在巡检本身
- 真正重要的信号被淹没

### 坑 2：在不需要精确时间点的地方用 Cron
比如“每 30 分钟检查一次 inbox”，这其实更适合 Heartbeat。

### 坑 3：在需要审批的地方直接执行
这是安全性问题，不是效率问题。

### 坑 4：没有 Rate / Cost / Token Limits
过度主动的 agent 会迅速消耗资源。

### 坑 5：没有审计与回溯
出了问题不知道是哪个主动性动作引发的。

### 坑 6：把“建议型”和“执行型”混淆
建议型可以更自由；执行型必须更克制。

### 坑 7：没有熔断机制
一旦主动行为开始失控，系统没有自动刹车。

### 坑 8：没有与用户确认“何时该主动、何时该沉默”
不同用户对主动性的容忍度完全不同。

## 8）高质量主动性边界系统应该设计什么
一个高质量的主动性边界系统，至少要设计 9 个部分。

### 8.1 主动性层级
- L1：监测与建议
- L2：有限执行（非关键动作）
- L3：有监督执行（带 guardrails）
- L4：完全自治（最高治理要求）

### 8.2 触发条件
- 时间触发
- 事件触发
- 状态触发
- 阈值触发

### 8.3 停止条件
- 任务完成
- 异常检测
- 用户明确停止
- 资源上限

### 8.4 熔断条件
- 连续失败
- 成本超限
- 用户投诉
- 系统异常

### 8.5 频率与成本限制
- Rate limit
- Token limit
- Cost limit
- 并发 limit

### 8.6 审批流
- 哪些动作需要审批
- 审批超时如何处理
- 审批拒绝如何回滚

### 8.7 审计记录
- 每次主动行为的原因
- 执行内容
- 结果
- 用户反馈

### 8.8 与 Memory 的联动
重要主动行为产生的承诺与决策，必须写入 memory。

### 8.9 用户配置接口
让用户可以调整：
- 主动性层级
- 频率
- 边界
- 熔断条件

## 9）使用技巧（Tips）
### 技巧 1：先问“这是监测型还是执行型”
监测型可以更自由；执行型必须更谨慎。

### 技巧 2：先问“这是建议型还是自治型”
建议型可以更主动；自治型必须有 guardrails。

### 技巧 3：Heartbeat 适合批处理，不适合精确时间点
精确时间点用 Cron。

### 技巧 4：Isolated Session 适合重分析，不适合需要上下文的任务
需要上下文的任务用 Main Session 或 Heartbeat。

### 技巧 5：Lobster 适合多步骤 + 审批，不适合简单单轮任务
简单任务不要上 Lobster。

### 技巧 6：Rate / Cost / Token Limits 是底线，不是建议
没有这些限制，主动性一定会失控。

### 技巧 7：每次主动行为都要问“这次主动是否创造了比噪音更多的价值”
这是最关键的判断标准。

## 10）入门（最低可用）
一个最低可用的主动性边界体系，至少要做到这五件事：

### 10.1 区分 Heartbeat 与 Cron 的职责
- 周期性监测 → Heartbeat
- 精确时间点 → Cron

### 10.2 明确“建议型”和“执行型”
- 建议型：主动权在人
- 执行型：必须有审批或 guardrails

### 10.3 设置 Rate / Cost / Token Limits
至少要有硬性上限。

### 10.4 记录主动行为日志
每次主动行为都要有简单记录。

### 10.5 给用户一个“暂停主动性”的开关
这是最后的熔断手段。

## 11）进阶（分级授权版）
进阶版不只是“少惹事”，而是建立真正的主动性分级治理。

### 11.1 建立 L1-L4 主动性层级
- L1：监测 + 建议
- L2：有限执行
- L3：有监督执行
- L4：完全自治

### 11.2 为每个层级设计触发条件、停止条件、熔断条件
不同层级对应不同的边界。

### 11.3 为每个层级设计审批流
L1 可能不需要审批；L3/L4 必须有。

### 11.4 建立成本与审计联动
成本超限 → 自动降级或暂停

### 11.5 建立用户可配置接口
让用户调整主动性层级与边界。

---

## 12）最佳实践（分层治理版）
高水平系统不会只靠“感觉”控制主动性，
而是把主动性也纳入治理体系。

### 12.1 主动性策略文档化
比如：
- 监测策略
- 提醒策略
- 执行策略
- 熔断策略

### 12.2 主动性行为版本化
每次调整主动性策略，都应有版本记录。

### 12.3 主动性回归测试
用典型场景测试：
- 什么时候会主动
- 什么时候不会主动
- 什么边界会触发熔断

### 12.4 主动性审计与分析
定期分析：
- 哪些主动行为有价值
- 哪些是噪音
- 用户反馈如何

### 12.5 主动性成本优化
不断调整：
- 频率
- 触发条件
- 上下文消耗

## 13）卓越案例：什么样的主动性边界才算业内最佳实践
一个业内最佳实践级别的主动性边界系统，通常具备五个特征：

### 13.1 它不追求“越主动越好”
它知道主动性与噪音之间只有一条细线。

### 13.2 它有清晰的主动性层级
不是一刀切，而是分级授权。

### 13.3 它有完整的审批与熔断机制
越高层级的主动性，越需要重的 guardrails。

### 13.4 它有成本与审计联动
主动性不会在成本或治理上失控。

### 13.5 它能从用户反馈中持续优化
不是一次性设计，而是持续调优。

## 14）验收口径与训练题
### 验收口径
判断一个 Agent 的主动性边界体系是否合格，看 10 条：

1. 是否区分 Heartbeat / Cron / Reminder / Message / Lobster 的职责
2. 是否区分“监测型”和“执行型”
3. 是否区分“建议型”和“自治型”
4. 是否有 Rate / Cost / Token Limits
5. 是否有停止条件与熔断条件
6. 是否有审批流（针对执行型/自治型）
7. 是否有审计记录
8. 是否与 Memory 联动（重要决策/承诺）
9. 是否有用户可配置接口
10. 是否定期分析主动行为的价值/噪音比

## 本章一句话结论
**主动性边界不是“让 agent 少动一点”，而是让它的每一次主动都发生在对的时间、对的层级、对的边界内。OpenClaw 的 Heartbeat / Cron / Reminder / Message / Lobster 机制已经给出完整的底层工具，真正决定 agent 是越主动越有价值还是越主动越制造噪音的，是你是否为它建立了清晰的主动性层级、触发条件、停止条件、熔断条件、审批流、成本限制与审计机制。会设计主动性边界，agent 才能既不懒惰也不过度，真正成为可信任、可持续、可治理的长期助手。**
