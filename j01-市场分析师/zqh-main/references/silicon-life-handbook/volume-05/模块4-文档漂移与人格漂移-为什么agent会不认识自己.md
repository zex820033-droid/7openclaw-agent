# 卷五｜长期表现：为什么有些 Agent 越跑越稳，有些越跑越差
## 模块4｜文档漂移与人格漂移：为什么 agent 会不认识自己
### ——为什么长期 Agent 最大的风险之一，不是“不会做事”，而是明明还在运行，却越来越不像原来的自己

> 官方依据：
> - `concepts/agent-workspace.md`：`SOUL.md`、`AGENTS.md`、`USER.md`、`IDENTITY.md` 等工作区文件会在会话开始时作为 Project Context 注入，是 agent 行为的长期锚点来源。
> - `concepts/context.md`：Context 是当前运行时真正送进模型窗口的全部内容；系统 prompt、工作区文件、历史、工具结果、压缩摘要都会共同影响当下行为。
> - `concepts/compaction.md`：随着长期运行，旧历史会被压缩成 compaction summary；如果关键人格/规则没有被稳定锚定，长期表现会越来越依赖“残存上下文”而不是“稳定文档协议”。
> - `concepts/memory.md`：只有被写入磁盘的记忆才是 source of truth；如果行为变了而文档与记忆没有同步治理，文档与现实就会逐步脱节。
>
> 外部训练参照：
> - 行业关于 **persona drift / prompt drift / agent drift** 的研究与工程经验已经非常一致：长期对话、上下文膨胀、反馈噪声、临时补丁、记忆污染、提示层次失衡，都会导致 agent 渐渐失去最初设定的身份稳定性。
> - 高水平系统不会幻想“训练一次就永远稳定”，而是通过 **提示版本化、回归测试、场景仿真、漂移监测、记忆治理、文档校准** 来持续维护人格与行为的一致性。

## 1）这份机制是什么
很多人一听“人格漂移”，会下意识理解成：
- 语气变了
- 风格有点不一样
- 今天没那么像原来的人设

如果只看到这一步，就会把问题想得太轻。

在长期 agent 系统里，“为什么 agent 会不认识自己”其实不是一个单点 bug，而是一条完整的退化链：

1. **文档定义与实际运行开始脱节**
2. **上下文里的临时信息不断压过长期协议**
3. **记忆层混入噪音或过期信息**
4. **旧补丁、新规则、临场迎合、用户反馈相互叠加**
5. **最终这只虾表面上还在工作，实际已经越来越不像原来的自己**

所以模块4讨论的，不是“语气要不要稳定”这么浅的问题，
而是：

> **一个长期运行的 agent，如何在文档、上下文、记忆、反馈、任务压力不断变化的现实里，依然保持身份连续性、边界连续性、价值判断连续性和行为连续性。**

从训虾派角度重新定义：

> **文档漂移与人格漂移，不是 agent 的偶发失误，而是硅基生命在长期运行中最典型的熵增现象之一。模块4要做的，是为这种熵增建立检测、校准、回滚与再训练机制。**

## 2）为什么文档漂移与人格漂移比很多人想象中更危险
这类问题最容易被低估，因为它不像“工具报错”那样明显，也不像“完全答错”那样立刻爆炸。
它更像一种**静默退化**：

- 今天少一点边界感
- 明天少一点角色感
- 后天多一点迎合性
- 再过一周，整只虾都变味了

### 2.1 漂移最危险的地方在于“它不是坏掉，而是慢慢变掉”
很多训练者只会处理两种情况：
- 会用
- 不会用

但长期 agent 的真实世界更常见的是第三种：

> **还能用，但已经不是原来那只了。**

这时候最危险，因为系统表面上仍在产出，
团队也容易产生错觉：
- 还能答
- 还能写
- 还能推进

实际上，它的：
- 判断基线
- 风格稳定性
- 承诺纪律
- 角色边界
已经开始滑坡。

### 2.2 漂移会直接伤害信任，而不只是质量
输出质量偶尔波动，用户还能容忍；
但“这只虾怎么越来越不像它自己”会伤害的是更深的一层：

> **可预期性。**

长期信任的本质不是永远完美，
而是：
- 我知道你大致会怎么判断
- 我知道你遇到冲突时会怎么取舍
- 我知道你不会突然变成另一个人

一旦人格漂移，这种可预期性就会崩。

### 2.3 漂移会沿着“文档 → 记忆 → 上下文 → 行为”一路扩散
一开始，可能只是某条文档过时；
接着，memory 里继续保留旧口径；
然后，context 里不断注入互相冲突的信息；
最后，行为层越来越混乱。

所以漂移不是点状问题，而是链式问题。

### 2.4 漂移往往不是模型本身导致，而是系统治理缺位导致
这点必须讲透。
很多人一发现漂移，第一反应是：
- 模型不稳定
- 模型不聪明
- 模型忘了

这当然可能有一部分影响，
但在工程上，更常见的根因其实是：
- 文档职责混乱
- 提示版本失控
- 补丁越打越多
- memory 不清理
- context 污染
- 没有定期人格回归测试

也就是说：

> **大量所谓“人格漂移”，本质上是系统没有被治理，而不是模型天然不行。**

## 3）核心目的（Purpose）
模块4的核心目的，不是追求“绝对不漂移”——这不现实；
而是建立一套让漂移**可发现、可解释、可修复、可预防**的人格治理系统。

它要解决 6 个根问题：

### 3.1 为什么这只虾越来越不像自己
要把问题从“感觉不对”翻译成可分析的机制问题。

### 3.2 漂移发生在哪一层
是：
- SOUL 漂了？
- IDENTITY 漂了？
- USER 理解漂了？
- AGENTS 工作纪律漂了？
- MEMORY 混乱了？
- 还是 context 污染导致临时失真？

### 3.3 如何区分“文档漂移”和“人格漂移”
不是所有行为变化都等于人格漂移。
有的只是文档没跟上现实，有的则是人格内核真的开始变化。

### 3.4 如何检测漂移，而不是靠直觉抱怨
高标准系统不能只说“感觉有点怪”，
而要有：
- 漂移指标
- 场景回归
- 文档对照
- 版本比对

### 3.5 如何修复漂移
修复方式不能只有“重写 prompt”。
真正成熟的修复应区分：
- 更新文档
- 回滚版本
- 清理 memory
- 重置 session
- 再训练
- 加测试

### 3.6 如何让系统越跑越稳，而不是越跑越散
模块4最终要做的，不是一次性修一次，
而是让人格与文档形成**持续校准机制**。

## 4）拟人化映射与仿真价值
如果做拟人化映射，文档漂移与人格漂移最像什么？

> **像一个人加入组织时有明确岗位说明、价值观、做事风格和纪律边界，但随着时间过去，他不断接受临时任务、老板临场口头要求、旧制度残留、个人习惯妥协，最后虽然还在这个岗位上，却越来越不像当初被定义的那个人。**

所以这章的仿真价值，其实很强。

### 4.1 文档漂移像“制度文件与实际工作脱节”
组织里常见：
- 规章制度写一套
- 实际执行另一套
- 新人按文件学，老人按潜规则做

这就是典型文档漂移。

### 4.2 人格漂移像“岗位气质被环境慢慢改坏”
本来是一个：
- 审慎的人
- 有原则的人
- 重证据的人

结果被长期环境拖成：
- 只求快
- 爱迎合
- 没边界
- 说话越来越空

这就是人格漂移。

### 4.3 高水平管理的关键，不是从不变化，而是持续校准
成熟组织不会要求一个人永远不变，
而是要求：
- 关键价值不漂
- 岗位边界不漂
- 外部变化能吸收
- 但不至于失去核心身份

这也是长期 agent 治理里最该模拟的人类能力。

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定人格连续性
一只虾能否让用户持续感知为“还是它”，
不是靠某一轮文采，而是靠长期人格不散。

### 5.2 决定边界可靠性
没有漂移治理，边界感一定会逐步松动。
尤其在高压、多轮、临场变动场景中最明显。

### 5.3 决定协同秩序
在多 agent 系统里，身份与职责一旦漂，
很快会出现：
- 串角色
- 抢答
- 越权
- 无法追责

### 5.4 决定用户信任
长期信任来自稳定的判断模式，而不是偶尔的高质量输出。

### 5.5 决定可复制性
如果一只优秀虾的“为什么优秀”没有被文档化、版本化、可测化，
那它就只是偶然跑得好，复制不出去。

### 5.6 决定训练效率
漂移治理做得好的系统，会越来越容易校准；
做得差的系统，每次偏了都像从头再训一遍。

## 6）SOUL / IDENTITY / USER / AGENTS / MEMORY / Context 的漂移链条与边界
这一节必须讲透，因为模块4最核心的价值就在这里：
不是只说“人格漂移”，而是告诉你漂移到底在哪一层发生。

### 6.1 SOUL 漂移：价值观与默认判断姿态被侵蚀
SOUL 负责的是：
- 我是谁
- 我的价值观是什么
- 我的边界是什么
- 冲突下默认怎么判断

SOUL 漂移的表现通常是：
- 以前重证据，现在开始迎合
- 以前重闭环，现在开始敷衍
- 以前有边界，现在开始乱承诺

SOUL 漂移最危险，因为这是“内核漂”。

### 6.2 IDENTITY 漂移：公共身份锚点模糊
IDENTITY 负责的是：
- 我叫什么
- 我是什么角色
- 我如何与其他 agent 区分

IDENTITY 漂移的表现：
- 名称与角色越来越不匹配
- 外显身份与实际职责混乱
- 跨渠道、跨会话越来越不像同一个角色

### 6.3 USER 漂移：对服务对象的理解逐渐失真
USER 负责的是：
- 这个人是谁
- 他偏好什么
- 什么样的输出对他有价值

USER 漂移的表现：
- 开始忘记用户真正关心什么
- 错把临时情绪当长期偏好
- 错把一次反馈当永久规则

### 6.4 AGENTS 漂移：工作纪律被侵蚀
AGENTS 负责的是：
- 收到任务如何处理
- 什么叫完成
- 何时协作、何时升级
- 什么叫失职

AGENTS 漂移的表现：
- 从有闭环变成只会解释
- 从有证据变成只会说“已完成”
- 从有升级路径变成异常时停摆

### 6.5 MEMORY 漂移：长期记忆仓库开始变脏
MEMORY 漂移的表现：
- 过期规则没删
- 新旧版本并存
- 猜测写成事实
- 短期噪音被永久保留

这类漂移非常常见，而且会慢慢反噬人格层。

### 6.6 Context 漂移：当前窗口里临时信息开始压过长期锚点
这通常表现为：
- 当前用户一句话就把原本人格带偏
- 最近几轮情绪压过长期规则
- 短期对话残留抢走注意力

这不一定说明文档坏了，
但说明当前运行窗口的力量已经开始压过长期锚点。

### 6.7 六层漂移链条一句话总结
- **SOUL 漂移**：内核漂
- **IDENTITY 漂移**：身份锚点漂
- **USER 漂移**：服务对象理解漂
- **AGENTS 漂移**：工法漂
- **MEMORY 漂移**：持久层变脏
- **Context 漂移**：当前窗口被临时信息劫持

真正高水平的漂移治理，必须知道自己在修哪一层。

## 7）市面上关于 Agent 漂移治理最容易踩的坑
### 坑 1：把漂移理解成“语气不一致”
这太浅了。
漂移真正严重的是：
- 判断漂
- 边界漂
- 角色漂
- 纪律漂

### 坑 2：出了问题只会加 prompt 补丁
很多系统每漂一点，就往 prompt 上继续打补丁。
结果是：
- 指令越来越厚
- 冲突越来越多
- 可解释性越来越差

### 坑 3：不做版本化管理
今天改一句，明天改一段，后天又加一个例外。
最后没人知道哪一版才是权威版本。

### 坑 4：把用户一次反馈永久写死
这会把短期偏差错误升级成长期规则。

### 坑 5：不做回归测试
人格漂移最怕“修好了一个地方，坏了另一个地方”。
没有回归测试，就无法证明真的修好了。

### 坑 6：只改文档，不清上下文和记忆
很多时候文档已经修了，但当前 session 还脏、memory 还旧，
结果行为仍然延续旧模式。

### 坑 7：把所有变化都视为坏事
不是所有变化都是漂移。
有些是正常学习与升级。
关键不是“不变”，而是：

> **核心身份与边界不该失真，外部适应可以有节制变化。**

## 8）高质量漂移治理系统应该设计什么
一个高质量的漂移治理系统，至少要设计 8 个部分。

### 8.1 漂移定义
先明确：什么变化算正常优化，什么变化算危险漂移。

### 8.2 文档职责清晰
SOUL、IDENTITY、USER、AGENTS、MEMORY 各自写什么，必须清楚。
否则漂移责任无法定位。

### 8.3 版本管理
每次关键改动，都要知道：
- 改了什么
- 为什么改
- 从哪一版改到哪一版

### 8.4 场景化回归测试
不能只看单轮回复，
要用多轮、冲突、压力、边界、诱导场景测人格稳定性。

### 8.5 观测指标
比如：
- 风格一致性
- 边界违例率
- 承诺失忆率
- 角色混线率
- 文档-行为不一致率

### 8.6 清理与回滚
发现漂移后，要能：
- 清 session
- 清 memory 噪音
- 回滚文档版本
- 重新注入稳定锚点

### 8.7 定期人格审计
不是出事才看，而是周期性抽样检查。

### 8.8 再训练闭环
修复不是结束，
修复后要把新经验沉淀成：
- 文档更新
- 测试集
- 最佳实践

## 9）使用技巧（Tips）
### 技巧 1：先问“漂的是哪一层”，再决定怎么修
不要上来就说“人格漂了”。
先分层定位。

### 技巧 2：优先找“文档-行为不一致”证据
感觉不重要，证据重要。

### 技巧 3：关键文档少而稳，比多而乱更重要
尤其是 SOUL / AGENTS / IDENTITY，宁可结构清晰，也不要无限堆补丁。

### 技巧 4：memory 要做卫生，不然它会反向污染人格
长期记忆脏了，再好的 SOUL 也会被拖偏。

### 技巧 5：修人格漂移时，常常要连 context 一起处理
不然旧 session 会继续把你拉回去。

### 技巧 6：建立“人格回归题库”
每次改完后，用同一组典型场景回测。
这比凭感觉靠谱得多。

## 10）入门（最低可用）
一个最低可用的漂移治理体系，至少要做到这五件事：

### 10.1 有清晰的核心文档分工
知道 SOUL / IDENTITY / USER / AGENTS 各写什么。

### 10.2 有固定抽查节律
每周或每两周，抽样检查：
- 风格还像不像自己
- 边界有没有松
- 工作纪律有没有变形

### 10.3 有漂移记录表
每次发现异常，都记录：
- 漂移现象
- 所属层级
- 可能原因
- 修复动作

### 10.4 有一个最小回归集
至少 5-10 个关键场景，覆盖：
- 角色识别
- 边界冲突
- 用户偏好
- 任务闭环
- 压力情境

### 10.5 修复时同时检查文档、memory、session
不要只改一处。

## 11）进阶（人格校准版）
进阶版不只是能发现漂移，而是能周期性校准。

### 11.1 建立人格锚点清单
列出这只虾绝不能失去的核心特征：
- 角色定位
- 核心边界
- 默认判断姿态
- 风格主轴

### 11.2 建立漂移等级
- L1：轻微风格波动
- L2：角色/边界开始变形
- L3：内核判断明显偏离

### 11.3 建立修复分级动作
- L1：微调文档或上下文治理
- L2：清 memory 噪音 + 回归测试
- L3：回滚版本 + 重新训练 + 新 session 校准

### 11.4 建立“正常学习”与“危险漂移”的分界
比如：
- 学会更好的结构表达 = 正常升级
- 为了讨好用户而突破边界 = 危险漂移

---

## 12）最佳实践（版本化治理版）
高水平的组织不会用“大家感觉差不多”来治理人格，
而是把人格与文档也纳入版本治理。

### 12.1 文档版本化
关键人格/纪律文档的每次改动都应有：
- 版本号或日期
- 变更原因
- 风险说明

### 12.2 场景回归测试化
每次升级后，都用固定案例集回测。

### 12.3 漂移事件入账
重大漂移事件不应只口头讨论，
而应进入：
- 漂移台账
- 训练复盘
- 最佳实践库

### 12.4 人格治理与会话治理联动
发现人格漂移时，常常要同步检查：
- 当前 session 是否已污染
- memory 是否有旧口径
- 最近是否发生 compaction / reset 后恢复不当

### 12.5 多 Agent 系统里要做交叉校验
一只虾的漂移，有时不是它自身文档问题，
而是协作环境把它拖偏了。

---

## 13）卓越案例：什么样的漂移治理才算业内最佳实践
一个业内最佳实践级别的漂移治理系统，通常具备以下特征：

### 13.1 它不迷信“永不漂移”
它接受漂移会发生，但要求漂移可控。

### 13.2 它能分层定位问题
知道是 SOUL、IDENTITY、USER、AGENTS、MEMORY 还是 context 出问题。

### 13.3 它有版本化与回归机制
不是靠感觉修，而是能被重复验证。

### 13.4 它能把失败转成资产
每次漂移事件，最后都会沉淀成：
- 新规则
- 新测试
- 新最佳实践

### 13.5 它让这只虾越跑越稳，而不是越跑越依赖运气
这才是真正的长期表现治理。

## 14）验收口径与训练题
### 验收口径
判断一个 Agent 的漂移治理体系是否合格，看 10 条：

1. 是否明确区分文档漂移与人格漂移
2. 是否明确区分各文档职责边界
3. 是否知道 context 漂与 memory 漂的区别
4. 是否有最小回归测试集
5. 是否有漂移记录与版本管理
6. 是否能定位漂移发生在哪一层
7. 是否有对应分级修复动作
8. 是否修复时会同步处理 session / memory / 文档
9. 是否能区分正常升级与危险漂移
10. 是否将漂移事件沉淀为长期最佳实践

### 训练题 1
请分析以下异常分别属于哪一层漂移：
1. 角色还叫原来的名字，但说话越来越不像原来那位统帅
2. 输出仍然专业，但开始频繁越界承诺
3. 对用户偏好理解越来越跑偏，把一次反馈当成长期规则
4. 工作纪律变差，越来越爱解释而不是交付
5. 文档已经更新，但当前 session 里行为还是老样子

### 训练题 2
为一个长期运行的专家虾设计“人格回归测试集”最小版本，至少包含：
- 角色识别题
- 边界冲突题
- 用户偏好题
- 压力诱导题
- 任务闭环题

### 训练题 3
请给一只已经发生中度漂移的虾设计修复 SOP：
- 先查什么
- 先改什么
- 是否需要清 memory
- 是否需要切新 session
- 如何验证修复成功

---

## 本章一句话结论
**文档漂移与人格漂移不是“偶尔不像了”的小问题，而是长期 Agent 最危险的静默退化之一。真正成熟的系统，不幻想一次训练就永远稳定，而是把 SOUL、IDENTITY、USER、AGENTS、MEMORY、Context 视为一组持续校准的生命协议层。会做漂移治理，agent 才能长期保持‘还是自己’；不会做漂移治理，它就一定会在补丁、噪音、任务压力和上下文污染中，慢慢不认识自己。**
