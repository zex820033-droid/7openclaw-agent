# 卷五｜长期表现：为什么有些 Agent 越跑越稳，有些越跑越差
## 模块1｜记忆管理：为什么 agent 会“忘记”你是谁
### ——从 OpenClaw 的 memory 机制，到长时 agent 的第一性工程：不是让它记更多，而是让它记得对、记得久、记得可重建

---

## 一句话结论

> **记忆管理不是“做不做笔记”的习惯问题，而是长期 agent 能不能跨会话、跨 compaction、跨任务切换继续保持连续性的基础设施问题。**

换句话说：

> **如果你不把记忆工程化，agent 不是“偶尔忘”，而是必然退化成一次性对话机器。**

所以模块一讨论的不是“要不要记忆”，而是：

- **哪些信息必须升格为 durable memory**
- **哪些信息只能留在当前 context**
- **什么时候该写入 daily log，什么时候该沉淀到 curated memory**
- **如何让记忆跨 compaction 之后仍可恢复，而不是被摘要吞掉**

---

## 这段到底在解决什么

在真实运行里，用户最常见的抱怨不是“它不会回答”，而是：

- 它昨天还知道，今天就不知道了
- 它明明答应过，下一轮像没这回事
- 它认识我三个月了，结果新会话还像第一次见面
- 它做长项目时，前面的关键决策越跑越丢

这类问题表面看叫“失忆”，本质上其实是三种东西没有分清：

1. **Context**：这一轮模型窗口里临时可见的东西
2. **Transcript / Session history**：当前会话的持久转录与 compaction 摘要
3. **Memory files**：被显式写回 workspace、未来可再加载的持久记忆

如果你把三者混为一谈，你就一定会误判：
- 以为“说过了”就等于“记住了”
- 以为“历史还在 transcript 里”就等于“下轮一定能稳定调出来”
- 以为“上下文窗口很大”就等于“长期记忆没问题”

这正是大量市面 agent 课程最容易写肤浅的地方。

它们会说：
- 给 agent 加记忆
- 做 memory
- 做知识库

问题是，这些词太粗。

真正高标准的解读必须继续往下压：

> **长期表现中的记忆，不是知识库存储问题，而是运行态信息如何被筛选、升级、压缩、落盘、再加载的问题。**

---

## 先立 OpenClaw 官方口径：Memory 的 source of truth 是磁盘，不是模型脑子

OpenClaw 官方 memory 文档里最关键的一句话，可以翻译成我们课程里的标准口径：

> **模型不会天然“记住”，真正的 source of truth 是被写入 workspace 的 Markdown 文件。**

这句话的含义非常深。

因为它直接推翻了很多人对“大模型记忆”的想象。

### 误区一：把“模型还记得一点”当成记忆系统
不是。
那只是当前上下文窗口里还残留着一些信息。
只要：
- 会话变长
- 工具结果变多
- context 接近上限
- 触发 compaction

这些“还记得一点”的内容，就可能迅速丢失或弱化。

### 误区二：把“会话 transcript 还在”当成稳态记忆
也不是。
Transcript 的作用，是保存会话历史；
但长会话运行时，真正喂给模型的，是经过 context assembly 之后的**当前窗口**。
一旦被 compaction，总是要做信息压缩的。
压缩就意味着有取舍。

### 误区三：把“MEMORY.md 存在”当成记忆工程已经做好
更不是。
是否有文件，只是开始。
真正的问题是：
- 写进去的内容是否正确分层
- 是否太多噪音
- 是否可追溯
- 是否能在未来会话中被稳定恢复

所以从 OpenClaw 官方机制推出来的第一条铁律是：

> **凡是你希望它未来还能稳定知道的内容，不能只留在对话里，必须写回文件系统。**

这就是“memory as files, not vibes”。

## OpenClaw 的双层记忆结构，为什么是对的

OpenClaw 官方 memory 文档实际上给了一个非常成熟、也非常适合“训虾体系”的分层：

### 第一层：`memory/YYYY-MM-DD.md`
这是**每日运行日志层**。
它的几个关键性质：
- append-only
- 一天一个文件
- session start 时通常会读今天和昨天
- 更适合记 running context、当天决策、阶段性变化、现场状态

这层不是“长期原则”，而是“当日现场”。

### 第二层：`MEMORY.md`
这是**精选长期记忆层**。
它的关键性质：
- curated
- 不是流水账
- 更适合放 durable facts、用户偏好、长期协议、长期项目状态、长期关系结构
- 在 OpenClaw 官方设计里，它只应在 main / private session 加载，不应在 group 场景无脑注入

这层不是“今天发生了什么”，而是“未来一段时间无论何时开局，都应该被快速找回的东西”。

---

## 为什么这套双层结构很像高水平的人类认知系统

这正是你课程体系里特别适合做“拟人化映射”的地方。

### daily log 像什么？
像人的**工作日志 / 临时脑 / 现场笔记**。

- 今天和谁聊过
- 今天做了什么决定
- 哪个项目推进到哪一步
- 当前阻塞是什么

这类东西如果不记，就会快速蒸发；
但如果全都写进“人生信条”，又会把长期记忆搞得臃肿。

### curated memory 像什么？
像人的**长期认知卡片 / 常识性关系图 / 稳定个人档案**。

- 这个人是谁
- 我们之间的长期约定是什么
- 我是谁、我的边界是什么
- 哪些事情是以后一直成立的

所以 OpenClaw 的 memory 设计，不只是工程便利，实际上非常符合认知分层。

在训虾语言里，这就是：

> **一只稳的虾，不是把所有东西都记进脑子，而是把“今天发生的”和“以后一直有效的”分开存。**

---

## 记忆管理为什么不是“越多越好”

这件事必须反复强调，因为它和很多初学者直觉完全相反。

很多人一上来就会做两件错事：
1. 什么都往 MEMORY.md 里写
2. 希望 memory 越大越聪明

这两件事都会把 agent 跑废。

### 原因一：长期记忆一旦噪音过多，真正重要的事实就被稀释
一个 bloated 的 MEMORY.md，和没有长期记忆差别并不大。
因为在运行时：
- 注入有 token 成本
- 检索有注意力成本
- 组织不清有理解成本

也就是说，你不是在给它“更多记忆”，你是在给它“更多噪音”。

### 原因二：长期记忆如果充满短期现场，会迅速过时
比如：
- 今天下午 3 点开会
- 某个临时 blocker
- 某次一次性指令

这些更适合 daily log，而不应该永久抬升到 MEMORY.md。

### 原因三：长期记忆越乱，漂移风险越高
很多 agent 的人格漂移，不是从 SOUL.md 开始坏的，
而是从 MEMORY.md 里混入大量未经筛选、互相矛盾、过期未删的信息开始坏的。

所以从工程角度讲，memory 的目标不是容量最大化，而是：

> **关键事实的保真率最大化。**

这和近一年的 context compression / long-running agents 实测结论其实高度一致：

> 结构化压缩后的高密度信息，通常比无控制保留全部历史更有价值。

---

## 从行业最佳实践看，memory 和 context 不是替代关系，而是接力关系

这是模块一最关键的交叉验证点之一。

### OpenAI 一侧的经验
OpenAI 在 session memory / short-term memory management 的工程示例里强调的重点，不是“无限保存历史”，而是：

- 把要求、决策、约束、理由等高价值信息**紧凑保留**
- 让长期会话仍能延续 commitments
- 用结构化记忆来跨越短期窗口限制

### Anthropic 一侧的经验
Anthropic 关于 long-running agents/context engineering 的工程文章，以及 Claude memory tool 的实践，也在重复同一个逻辑：

- compaction 负责让活动上下文保持可管理
- memory 负责让关键事实跨 compaction 边界仍然存在
- domain memory 和 current context 是协同而不是替代

### 训虾体系里的翻译
翻译成你的课程语言，就是一句非常硬的话：

> **Context 决定它这一步看见什么，memory 决定它下一次开局时还是谁。**

如果没有 context，当前任务做不好；
如果没有 memory，长期身份保不住。

所以真正成熟的 agent，不是二选一，而是：

- **运行时靠 context 保当前聚焦**
- **跨会话靠 memory 保长期连续性**

## OpenClaw 的 pre-compaction memory flush，为什么是长期表现的关键细节

很多人读官方文档会略过这个点，但在我们这套课程里，这其实是一个非常值得高亮的“行业级最佳实践细节”。

OpenClaw 在 session 接近 auto-compaction 时，会触发一个**silent memory flush turn**：

它不是让 agent 和用户继续对话，
而是提醒 agent：

> **在上下文马上要被压缩之前，先把 durable memory 写回磁盘。**

这一点极其重要。

因为 compaction 的本质，是摘要旧历史。
只要是摘要，就一定不是 1:1 保真。
如果关键事实还没被从上下文/历史里抽出来、写回持久层，
它就可能在摘要之后变得模糊、降级，甚至直接消失。

所以这个 memory flush 的设计，本质上是在做一件很成熟的事：

> **在“上下文即将失真”之前，先把长期必须保真的东西抽取出来，转入持久层。**

这是长期 agent 和记流水账聊天机器人之间非常大的分水岭。

对课程来说，这意味着：

### 训虾铁律一
**在任何大幅压缩、重置、切会话、跨阶段推进之前，先做 durable memory 提取。**

如果不做这一步，后面很多“它为什么突然不一样了”的问题，其实都不是模型问题，而是你在机制上没有先落盘。

## 记忆到底该写什么：从“重要”改成“可持续影响未来运行”

这是模块一最需要拉高标准的地方。

很多人问“什么值得写进记忆”，回答通常会很泛：
- 重要的
- 关键的
- 用户偏好

这些都不够。

更高标准的判断应该是：

> **这条信息，会不会持续影响未来的行为选择、任务判断、身份一致性、协同口径？**

如果会，就有资格进入 durable memory 候选区。

按这个标准，可以分成五类。

### 第一类：用户稳定偏好
例如：
- 喜欢什么风格的输出
- 不喜欢什么表达方式
- 对节奏、格式、细节深度的固定偏好

这类信息影响的是未来几乎所有回复。
所以必须进入长期层。

### 第二类：长期身份与边界
例如：
- 我是谁
- 我不该做什么
- 我的职责与默认姿态是什么

这类信息直接决定角色稳定性。
不写，漂移概率会显著增加。

### 第三类：持续中的项目状态
不是所有项目现场都写长期层，
但**跨多天、多周、会在未来反复接续的项目主状态**，必须沉淀。

比如：
- 项目当前阶段
- 已拍板的关键路径
- 未决问题
- 明确的下一步

### 第四类：明确决策与制度口径
例如：
- 已确定采用哪个方案
- 哪个版本作废
- 以后遇到类似情况按什么口径处理

这类是典型的 durable operational memory。

### 第五类：需要被兑现的承诺
例如：
- 用户明确要求记住的事
- 已经答应未来执行的动作
- 某个时间点前必须回收的动作

因为这些信息一旦丢失，伤害的不是“智能感”，而是**信任**。

## 哪些不该写进 MEMORY.md

高标准的课程不仅要讲“该写什么”，更要讲“绝对不要乱写什么”。

### 不该写的第一类：纯短期现场噪音
如：
- 某次临时提醒
- 当天一次性小阻塞
- 当前窗口里已经非常清楚的小上下文

这些更适合留在 session 或 daily log。

### 不该写的第二类：未经验证的猜测
这点尤其重要。

你刚才要求“不能自己猜测”，这条在模块一里必须写死。
因为一旦把猜测写成 memory，就等于：

> **把不确定输入升级成长期行为依据。**

这会直接导致后续的系统性偏航。

### 不该写的第三类：情绪性描述或评价性噪音
比如：
- “今天感觉这个用户有点怎样”
- “我觉得这个项目可能不行”

如果没有事实基础和未来行为价值，不应升格成 durable memory。

### 不该写的第四类：彼此冲突但未合并的旧版本
长期记忆最怕的不是少，而是冲突。
一旦旧规则和新规则并存，后续 agent 会被拉裂。

所以长期记忆必须有**版本意识和清理意识**。

## 记忆管理的行业最高标准，不是“能存”，而是“四可”

如果按行业最高标准来定义，一个成熟的记忆系统必须满足四件事：

### 1）可写入
需要写的时候，agent 知道写到哪里、怎么写、何时写。

### 2）可检索
未来需要的时候，能通过 memory_search / memory_get 或结构化方式把它找回来。

### 3）可审计
知道是谁写的、什么时候写的、为什么写。

### 4）可清理
过时的、冲突的、错误的内容能被修正，而不是只增不减。

如果缺任何一项，长期表现都会出现问题：
- 只可写不可检索 → 写了等于白写
- 只可检索不可审计 → 找到也不敢信
- 只可累积不可清理 → 越跑越脏

## 结合我们课程体系，模块一最后要落到什么动作

这一模块不能停在理解，必须落到动作。

### 动作1：建立双层记忆规则
明确：
- daily log 写什么
- MEMORY.md 写什么
- 什么情况下必须从 daily log 升格到 MEMORY.md

### 动作2：建立 durable memory 触发器
例如：
- 用户说“记住这个”
- 形成明确决策
- 形成长期约束
- 形成未来承诺
- 即将 compaction / reset / 切阶段

出现这些信号，就应触发 memory write。

### 动作3：建立 weekly memory review
每周至少一次：
- 检查 MEMORY.md 是否臃肿
- 检查是否有过时信息
- 检查是否有冲突版本
- 检查最近一周是否有该沉淀但没沉淀的信息

### 动作4：建立 memory hygiene 标准
也就是：
- 不写猜测
- 不写噪音
- 不保留冲突旧版
- 重要变更必须更新而不是叠加混乱描述

|      |      |          |      |            |

|------|----------|------------|------------|
|      |          |            |            |

| 日期 | 承诺事项 | 触发条件 | 当前状态 |

这个模板比普通 MEMORY 模板更高标准的地方在于：
- 把“是否仍有效”显式化
- 把“适用范围”显式化
- 把“待清理/待升级”显式化

因为长期记忆最怕的不是没有，而是**不加治理地一直堆**。

## 常见误区（高标准版）

### 误区1：把 transcript 当 memory
错。
Transcript 是历史，不等于 durable memory。

### 误区2：把 MEMORY.md 当流水账
错。
MEMORY.md 是 curated layer，不是日记。

### 误区3：等到忘了再补写
错。
高标准做法是在决策形成、承诺形成、即将 compaction 之前就先落盘。

### 误区4：把未经验证的判断写成长期事实
这是最危险的一种。
因为它会把猜测升级成未来行为依据。

### 误区5：只写不审
长期记忆不审计，就会慢慢变成噪音仓库。

## 这一模块对长期表现的影响

如果模块一真正做对，会直接带来五个变化：

1. **跨会话连续性提高**：不会每次都像第一次见面
2. **跨 compaction 保真度提高**：关键事实不会轻易被摘要吞掉
3. **承诺兑现率提高**：因为未来行为依据被明确保留
4. **人格稳定性提高**：因为身份和边界有稳定锚点
5. **后续模块全部更稳**：心跳、清理、漂移检测都有了可靠的持久层基础

反过来，如果模块一做错，后面所有高级机制都可能失效。

因为：
- 没有 durable memory，heartbeat 只能围着短期上下文打转
- 没有 durable memory，会话清理会丢关键事实
- 没有 durable memory，漂移修复没有锚点

所以从课程结构上看，模块一不是“基础中的一个模块”，而是：

> **卷五的地基。**

## 本章一句话收口

**记忆管理真正要解决的，不是“让 agent 尽量多记一点”，而是把未来持续影响行为的关键信息，从易失的 context 里及时提炼出来，分层写回 durable memory，并通过清理、审计、检索机制保持其长期有效。OpenClaw 官方机制、OpenAI/Anthropic 的长时 agent 工程经验，以及我们自己的训虾体系，在这一点上高度一致：长期 agent 不是靠“更大脑子”活得久，而是靠“更好的记忆工程”活得稳。**
