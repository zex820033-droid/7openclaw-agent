# 卷五｜长期表现：为什么有些 Agent 越跑越稳，有些越跑越差
## 模块3｜会话污染与清理：为什么 agent 会越说越乱
### ——为什么长期 Agent 的问题，往往不是“能力不够”，而是 session、context、tool results、compaction summary 一起把语境拖脏了

> 官方依据：
> - `concepts/context.md`：Context 是模型当前这一次真正看到的全部内容，不等于 memory。
> - `concepts/compaction.md`：Compaction 会把旧历史总结为持久化摘要，并写入 session history。
> - `concepts/session-pruning.md`：Pruning 只会在每次请求前临时裁剪旧 tool results，不会改写磁盘 transcript。
> - `concepts/session.md`：Session 是 OpenClaw 的真实会话桶，包含 main/direct/group/thread 等隔离规则；`/new`、`/reset` 会创建新的 session id。
> - `reference/session-management-compaction.md`：Pre-compaction memory flush、session transcript、context overflow recovery、store/transcript 双层持久化机制，决定了“长期会话为什么会变脏、什么时候该压缩、什么时候该切新会话”。
>
> 外部训练参照：
> - 长时 agent engineering 的行业共识越来越明确：**长期退化首先不是“模型笨了”，而是上下文被污染、历史被稀释、工具输出堆积、话题边界混乱后，agent 的注意力结构被破坏了。**
> - 高水平系统不会迷信“永远保留全部历史”，而是通过 compaction、摘要、分会话、外部记忆、隔离执行，把“可追溯”与“可运行”分开治理。

## 1）这份机制是什么
很多人一提“会话污染”，脑子里想到的是：
- 聊多了
- 上下文太长
- 有点乱

这几个说法都不够精确。

在 OpenClaw 的真实运行机制里，会话污染不是单一现象，而是以下几层问题叠在一起：

1. **当前 context 里进入了太多低价值内容**
2. **旧 tool results 占用了模型注意力**
3. **多个话题、多个任务、多个时间阶段混在同一 session 桶里**
4. **compaction 之后关键信息没有被清晰保真，只剩模糊摘要**
5. **该转 memory 的东西没转，该开新 session 的时候没开**

所以，会话污染不是“聊天太久”这么简单。
它的本质是：

> **运行中的语境结构被破坏了。**

从训虾派角度，模块3真正要讨论的是：

> **如何让一只长期运行的虾，在历史越来越长、工具结果越来越多、任务越来越复杂的情况下，依然保持当前语境清晰、重点突出、身份稳定、任务边界可辨。**

也就是说，这一章不是在讲“删历史”，而是在讲：

- 怎么判断历史已经开始伤害当前表现
- 怎么区分该 prune、该 compact、该 reset、该写 memory
- 怎么让“可追溯历史”与“可运行上下文”分层治理

## 2）为什么会话污染比很多人想象中更致命
会话污染被低估，是因为很多人会把 long context 理解成“越多越聪明”。

这是一种非常典型、也非常危险的错觉。

### 2.1 不是“保留越多”，而是“当前可用越清晰”
OpenClaw 官方 context 文档已经把问题讲得很明白：

> Context 是模型当前一次运行真正能看到的内容。

这意味着，真正影响当前表现的不是“历史总量”，而是：

- 当前窗口里塞进了什么
- 哪些东西占据了注意力
- 哪些内容在竞争模型的解释资源

如果这个窗口被大量：
- 旧 tool outputs
- 无关寒暄
- 已完成任务残留
- 模糊摘要
- 多话题串线
所占据，那么模型即使“理论上知道很多”，实际也会开始变得：
- 回答发虚
- 抓不住重点
- 语气漂移
- 行为不连贯

### 2.2 污染的最坏后果不是“答错”，而是“越来越不像同一只虾”
很多人把会话污染理解得太轻，只看到：
- 有时答偏
- 有时忘记

真正更严重的是：

> **被污染的 session 会逐渐侵蚀这只虾的行为连续性。**

你会观察到：
- 它开始不像之前那么聚焦
- 它开始不像之前那么懂当前任务
- 它开始不像之前那么像“这个角色”

这时候问题已经不是某一轮回答好不好，
而是：

> **整个长期人格与任务执行状态，开始被脏上下文拖坏。**

### 2.3 污染会吞噬三种最重要的东西
#### 第一种：当前任务边界
它不知道这轮到底该做什么。

#### 第二种：长期身份稳定性
它越来越像“普通模型”，不像“训练好的硅基生命”。

#### 第三种：承诺与决策保真度
前面拍板过的东西被噪音淹没，后面越跑越像没发生过。

所以模块3的价值，不是清洁癖，而是长期表现的生命线。

## 3）核心目的（Purpose）
会话清理机制的核心目的，不是追求“历史最短”，而是建立一套让 agent 在长期运行中始终保持可运行、可追溯、可恢复的上下文治理系统。

它主要解决 6 个根问题：

### 3.1 当前运行窗口应该承载什么，不该承载什么
也就是：
- 哪些必须留在当前 context
- 哪些应该被 summary 接管
- 哪些应该被 memory 接管
- 哪些应该直接离开当前运行窗口

### 3.2 旧工具输出如何不继续污染当前推理
OpenClaw 官方 pruning 文档强调得很明确：
- pruning 的对象主要是旧 `toolResult`
- 是 in-memory 裁剪，不会改写 transcript

这说明官方也承认：

> **长时运行里，tool results 是最典型、最容易膨胀、最需要治理的污染源之一。**

### 3.3 会话何时应该 compact，而不是硬拖
当 context 接近上限时，官方机制会触发 compaction。
这说明长期系统的正确思路不是“永远不压缩”，而是：

> **在压缩时尽量保真，而不是死扛原始历史。**

### 3.4 何时该 reset / new，而不是继续在脏 session 上勉强工作
很多人不敢开新会话，觉得那是“丢历史”。
其实很多时候，不开新会话才是在持续积累系统性混乱。

### 3.5 什么该外化到 memory
如果关键信息没有在切会话前抽出，
那清理就会变成“真丢失”。

### 3.6 如何建立“会话卫生”而不是事后救火
高水平系统不会等完全乱了再抢救，
而是日常就有 session hygiene 机制。

## 4）拟人化映射与仿真价值
如果做拟人化映射，会话污染最像什么？

> **像一个人桌上文件越堆越多、会前记录和旧报告混在一起、上一阶段的废稿还摊着、会议纪要和待办没有归档，最后导致当前开会时已经抓不住重点。**

所以模块3本质上在模拟的，不是“删除聊天记录”，
而是一个成熟专业人士的三种能力：

### 4.1 清桌能力
什么该留桌面，什么该归档，什么该扔掉。

### 4.2 切会能力
一个话题结束了，要不要继续在同一会议室里聊下一个？
很多时候不该。

### 4.3 做纪要能力
重要结论不应该只留在现场噪音里，
而应该被提炼出来，进入长期可追溯层。

这就是 OpenClaw 里：
- current context
- compaction summary
- transcript
- memory
- new session
这些机制背后的认知仿真价值。

所以高水平 agent 的“会话清理”，不应该被理解成粗暴抹去历史，
而应理解成：

> **把‘现场’、‘纪要’、‘档案’、‘新议题’分别放回它们应该待的位置。**

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定任务聚焦度
清理系统做得越好，当前任务越容易被模型准确抓住。

### 5.2 决定人格连续性
脏 session 最容易让 agent 从“有角色感”滑向“普通 AI 风格”。

### 5.3 决定承诺兑现率
如果前面承诺、决策、边界都淹没在脏上下文里，后面失约几乎是必然的。

### 5.4 决定成本结构
旧 tool outputs、无意义上下文、重复历史，都会持续烧 context 成本。

### 5.5 决定长项目可持续性
长期项目最怕的不是任务难，而是：
- 多轮后越来越散
- 历史越来越臃肿
- 每次续上都越来越费劲

### 5.6 决定组织级可靠性
在多 agent / 多项目环境里，如果 session hygiene 不做，系统很快会出现：
- 项目串线
- 人格串线
- 责任串线
- 指令串线

## 6）Session / Context / Pruning / Compaction / Memory 的边界
这一节必须讲透，否则模块3必然被写浅。

### 6.1 Session 是什么
Session 是 OpenClaw 的**会话桶**。
它决定哪一组消息被视为同一条连续对话轨道。

官方 `session.md` 已明确：
- direct chat 默认可能复用主 session
- group / thread 有自己的 key
- `/new`、`/reset` 会切到新的 session id

所以“会话污染”首先发生在 **session 桶里的历史延续方式** 上。

### 6.2 Context 是什么
Context 是模型这一轮真正看到的东西。
不是全部历史本身，而是运行时拼出来的有效窗口。

这意味着：

> **一个 session 很长，不等于这一轮所有历史都被完整看见；
> 但 session 越脏，context assembly 越容易把脏东西持续带进来。**

### 6.3 Pruning 是什么
Pruning 不是删历史。
根据官方文档：
- 它只修剪旧 `toolResult`
- 是 request 前的 in-memory 裁剪
- 不改 `.jsonl` transcript

一句话：

> **Pruning 是“这轮少带点旧工具垃圾进去”，不是“历史真的没了”。**

### 6.4 Compaction 是什么
Compaction 是把旧历史压成摘要，且这个摘要会持久化进 transcript。

一句话：

> **Compaction 是“历史变纪要”，不是“历史还原样摆在那里”。**

### 6.5 Memory 是什么
Memory 不是会话历史本身，而是：
- 被明确提炼出来
- 写回文件系统
- 未来可再次检索、注入、恢复
的持久知识层。

### 6.6 五者边界一句话总结
- **Session**：会话桶
- **Context**：当前运行窗口
- **Pruning**：临时修掉旧 tool outputs
- **Compaction**：把旧历史压成持久摘要
- **Memory**：把必须长期保留的事实抽离成独立持久层

模块3的真正难点，就在于知道：

> **什么时候该修窗口，什么时候该压历史，什么时候该切新桶，什么时候该先写 memory。**

## 7）市面上关于“会话越长越好”最容易踩的坑
### 坑 1：把“可追溯”误解成“必须永远把所有历史带在身上”
这是最大误区。
可追溯不等于当前轮次必须原样背全部历史。

### 坑 2：把 compaction 当成万能药
Compaction 很重要，但它不是万能修复。
如果旧历史本来就混乱、关键事实没提炼、多个议题没分层，压出来的摘要也可能继续脏。

### 坑 3：完全不管 tool results
很多系统的问题，根本不是用户对话，而是过去大量读文件、执行命令、抓网页的结果残留在上下文里。

### 坑 4：不敢开新 session
很多训练者迷信“连续性”，于是一个会话硬撑几周。
最后 continuity 没保住，clarity 先死了。

### 坑 5：清理之前不先提炼 durable memory
这会把清理变成真正的信息流失。

### 坑 6：把不同项目硬塞进同一 session
这是组织型系统最容易出事故的地方。

## 8）高质量会话清理系统应该设计什么
一个高质量的 session hygiene 系统，至少要设计 7 个部分。

### 8.1 污染识别信号
必须明确：什么现象出现，说明当前 session 已经影响表现。

典型信号：
- 回答开始发虚
- 当前任务边界感下降
- 开始拿旧话题干扰新问题
- 频繁答非所问
- 角色风格变淡
- 成本快速抬升

### 8.2 清理动作分层
不是只有一种“清理”。
至少要区分：
- pruning
- compaction
- write memory
- new/reset
- project split

### 8.3 关键信息提炼规则
清理前先问：
- 哪些决策必须留
- 哪些承诺必须留
- 哪些项目状态必须留
- 哪些约束必须留

### 8.4 会话分桶策略
不同项目、不同人、不同渠道，是否应该共用 session？
这个问题如果不提前设计，后面一定串线。

### 8.5 成本治理规则
什么时候该用：
- pruning
- `/compact`
- new session
- isolated run

### 8.6 恢复规则
清理后怎么恢复：
- 从哪里恢复身份
- 从哪里恢复项目状态
- 从哪里恢复下一步动作

### 8.7 审计记录
什么时候清过、为什么清、清前保留了什么。

## 9）使用技巧（Tips）
### 技巧 1：不要把“会话历史”当成“当前工作台”
历史是档案，不是桌面。

### 技巧 2：工具输出要特别警惕
长时运行里，最先膨胀的往往不是用户消息，而是旧 tool results。

### 技巧 3：项目切换时要问一句——还适合在这个 session 里继续吗？
这个问题值千金。

### 技巧 4：压缩前先提炼
任何 compaction / reset / 分桶之前，先做 durable memory 抽取。

### 技巧 5：不要把 pruning 和 compaction 混为一谈
前者是临时减负，后者是历史重组。

### 技巧 6：如果当前会话已经“闻起来不对”，不要硬扛
一旦出现明显脏感，很多时候切新 session 比继续补 prompt 更有效。

## 10）入门（最低可用）
一个最低可用的会话清理系统，至少要做到这五件事：

### 10.1 能识别污染
至少知道这几种信号：
- 答非所问
- 角色变淡
- 任务边界模糊
- 历史干扰当前问题

### 10.2 知道 `/compact` 什么时候用
适合：
- 会话变长
- 明显 bloated
- 但仍属于同一条主线任务

### 10.3 知道 `/new` 或 `/reset` 什么时候用
适合：
- 议题已经切换
- 当前 session 已明显混线
- 你需要一个真正干净的新轨道

### 10.4 清理前能先记忆提炼
把：
- 决策
- 承诺
- 项目阶段
- 关键边界
写回 memory。

### 10.5 不把所有项目塞进同一个连续会话
这是最低可用的 session hygiene 常识。

## 11）进阶（长期稳定版）
进阶版系统不只是“出问题会清”，而是“平时就不让它脏”。

### 11.1 建立会话污染等级
- L1：轻微臃肿，可继续
- L2：开始影响聚焦，建议 compact
- L3：明显串线/角色淡化，建议切新 session

### 11.2 建立项目分会话机制
例如：
- 每个长期项目一条独立 session 轨道
- 群聊和私聊不混主线
- 高污染任务隔离运行

### 11.3 建立 compaction 前提炼制度
让 pre-compaction memory flush 不只是官方底层机制，也成为训练自觉。

### 11.4 建立 tool-result 卫生意识
任何高输出工具都要意识到：
- 当前能用，不代表以后该一直带着
- 长工具链一定要考虑 pruning / 隔离 / 归档

### 11.5 建立恢复模板
新 session 开启后，必须有一套极短恢复口径：
- 角色是谁
- 当前任务是什么
- 已拍板什么
- 下一步是什么

## 12）最佳实践（组织治理版）
一旦进入组织型运行，会话清理就不只是个人助手优化，而是系统治理动作。

### 场景一：多项目并行
最佳实践不是“把大家都塞在主会话里”，
而是按项目、按责任、按阶段拆轨。

### 场景二：多 Agent 协同
不同 agent 的主线任务最好不要互相拖脏。
需要时：
- 用新 session
- 用 isolated 任务
- 用 handoff，而不是强塞在同一脏上下文里

### 场景三：管理层会话与执行层会话分离
战略讨论与执行细节，如果长期混在一条 session 里，后面一定会互相污染。

### 场景四：异常后恢复
真正成熟的系统，不怕清会话，
因为它已经把：
- memory
- 决策
- SOP
- 项目状态
分层保存好了。

所以它可以：

> **勇敢切会话，而不恐惧失忆。**

## 13）卓越案例：什么样的会话清理才算业内最佳实践
一个业内最佳实践级别的会话清理系统，通常具备五个特征：

### 13.1 它不迷信长会话
它知道 continuity 重要，但 clarity 更重要。

### 13.2 它会区分不同层次的问题
- tool output 太多 → pruning
- 历史太长但主线还在 → compaction
- 议题换了/会话脏了 → new/reset
- 关键事实要保留 → memory

### 13.3 它有提炼能力
不会把“清理”做成“遗忘”。

### 13.4 它有分桶意识
不同任务、不同人、不同项目、不同阶段，不会无脑混在一起。

### 13.5 它有恢复能力
清理之后能迅速续上，而不是清完就失忆。

做到这五点，才说明这不是简单删聊天记录，而是真正的长期上下文治理系统。

## 14）验收口径与训练题
### 验收口径
判断一个 agent 的会话清理体系是否合格，看 10 条：

1. 是否区分 session 与 context
2. 是否区分 pruning 与 compaction
3. 是否知道 `/new` / `/reset` 的真实作用
4. 是否有污染识别信号
5. 是否清理前先提炼 durable memory
6. 是否意识到旧 tool results 是高危污染源
7. 是否有项目分桶/分轨意识
8. 是否有清理后恢复口径
9. 是否把“可追溯”与“可运行”分层治理
10. 是否避免把所有历史永远拖进当前窗口

### 训练题 1
请判断以下场景该用什么动作：
1. 会话很长，但还是同一个项目主线，最近回答开始发虚
2. 过去读了大量文件和命令输出，当前问题很简单但仍频繁跑偏
3. 原来的会话在聊 A 项目，现在开始聊完全不同的 B 项目
4. 准备 reset 之前，需要保留哪些信息到 memory

### 训练题 2
为一个“长期项目管理型 agent”设计 Session Hygiene SOP：
- 污染识别信号
- 何时 compact
- 何时 new session
- 清理前提炼项
- 清理后恢复项

### 训练题 3
把下面一句错误认知改写成正确口径：
> “为了不丢历史，所有内容都应该尽量保留在同一个会话里。”

## 本章一句话结论
**会话清理不是删除历史，而是治理语境。长期 Agent 真正的挑战，从来不是“历史会不会保留”，而是“当前运行窗口是否仍然清晰、当前任务是否仍然聚焦、关键信息是否已被正确提炼”。OpenClaw 的 session、context、pruning、compaction、memory 五层机制已经给出一套成熟答案：不是让所有历史永远挂在当前脑海里，而是把现场、摘要、档案和新轨道分层治理。会这样治理，一只虾才会越跑越稳；不会这样治理，它就一定会越说越乱。**
