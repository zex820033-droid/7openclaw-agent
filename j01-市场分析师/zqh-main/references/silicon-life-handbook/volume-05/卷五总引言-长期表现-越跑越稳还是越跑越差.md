# 卷五｜长期表现：为什么有些 Agent 越跑越稳，有些越跑越差
## ——从“训出来”到“跑得久”，长期表现不是玄学，而是上下文工程、记忆工程与运行纪律的综合结果

---

## 一句话结论

> **长期表现不是模型天赋问题，而是三套系统是否被持续维护的问题：上下文系统、记忆系统、节律系统。**

再说得更硬一点：

> **卷四解决的是“怎么把一只虾训出来”，卷五解决的是“为什么同样训练出来的虾，三周之后会走向完全不同的命运”。**

有的越跑越稳，越来越像“组织成员”；
有的越跑越散，越来越像“新开会话的普通 AI”。

差别不在于它最初会不会答，
而在于它有没有被放进一套**长期表现管理系统**里。

---

## 这一卷到底在解决什么

如果卷四是“训练学”，那卷五就是“维护学”。

训练完成之后，很多人会误以为最难的部分已经过去了。
但在真实运行里，真正难的不是第一次训成，而是：

1. **它能不能持续记住关键事实，而不是每次都像第一次见你**
2. **它能不能在长会话里保持聚焦，而不是越聊越散**
3. **它能不能跨 compaction 继续保持决策连续性，而不是摘要以后人格失真**
4. **它能不能有节律地自检、自纠偏，而不是被动等人催**
5. **它能不能在高频任务、多人协同、长周期项目中保持角色稳定，而不是风格漂移、边界失控、承诺失忆**

所以卷五讨论的不是“功能”，而是**寿命**。

不是“它会不会”，而是“它能稳定多久”。

## 先立一个最核心的区分：Context 不是 Memory

这是长期表现里最容易混淆、也是最致命的一个点。

OpenClaw 官方文档已经把它说得很清楚：

- **Context**：是当前一次运行时，模型真正能看到的全部内容
- **Memory**：是写在磁盘上的 Markdown，之后可以重新加载的持久信息

也就是说：

### Context 的本质
是“这一轮模型眼前能看到什么”。
它受上下文窗口限制，会膨胀，会截断，会 compact，会被工具结果污染。

### Memory 的本质
是“你希望未来它还能知道什么”。
它不天然在模型脑子里，而是必须被写进 workspace 里的文件，作为下次重建上下文的材料。

这个区分非常重要，因为很多人做长期 agent 时会犯一个错误：

> **以为只要当前会话说过，模型以后就会记得。**

这是错误的。

OpenClaw 的 memory 文档给出的结论非常直接：

> **source of truth 不是模型临时记住了什么，而是磁盘上最终写下了什么。**

所以卷五的一切，实际上都围绕一个根问题展开：

> **如何把会丢失的上下文，转化为可重建的长期记忆；如何把会污染的会话，变成可压缩、可清理、可恢复的长期运行结构。**

## 长期表现为什么会崩：三种最常见的误判

### 误判一：把“单次表现好”当成“长期表现稳”
很多 agent 第一次看起来都不错。
问题在于：
- 第一次它是在干净上下文里运行
- 第一次它还没被历史噪音污染
- 第一次它还没经历 compaction
- 第一次它还没暴露节律问题

所以第一次好，不说明它长期稳。

### 误判二：把“上下文大”当成“记忆能力强”
上下文大，只意味着这一轮能塞进更多东西。
不意味着：
- 它会自动保留真正重要的信息
- 它会自动清理噪音
- 它会自动在下一轮延续正确状态

如果没有 compaction、memory flush、daily log、curated memory，
再大的 context window 也只是**更大的临时脑袋**，不是长期记忆系统。

### 误判三：把“会主动”当成“长期健康”
很多 agent 一开始很主动，但后来会出现三种退化：
- 主动性消失：变成等指令机器
- 主动性失控：过度打扰、越界推进
- 主动性空转：有动作但没价值

所以主动性不是越多越好，而是要有**节律、边界、检查、日志**。

## 从官方机制看，长期表现的三大底座是什么

结合 OpenClaw 官方文档，这一卷真正依托的是三套底座。

### 底座一：记忆底座（Memory as files, not vibes）
官方明确规定：
- `memory/YYYY-MM-DD.md`：append-only 的日记层
- `MEMORY.md`：精选的长期记忆层
- session 接近 compaction 时，可触发 silent memory flush，把 durable memory 先写回磁盘

这套设计本质上在做一件事：

> **把“会丢”的临时上下文，转化为“可重建”的持久记忆。**

### 底座二：上下文底座（Context as a bounded runtime resource）
官方 context/compaction 文档明确说明：
- context window 是硬上限
- 长会话会积累消息、工具结果、附件和系统注入文件
- OpenClaw 会通过 compaction 总结旧历史，只保留摘要 + 近期消息
- pruning 与 compaction 不是一回事：前者是运行态裁剪，后者是持久化摘要

这意味着：

> **长期表现不是“保持全部历史不动”，而是“在压缩和保真之间持续做工程化平衡”。**

### 底座三：节律底座（Heartbeat as cadence, not noise）
官方 heartbeat 文档真正有价值的地方，不是“可以定时发消息”，而是它把心跳定义为：

> **periodic agent turns in the main session, so the model can surface anything that needs attention without spamming you**

也就是说，heartbeat 的本意不是刷存在感，而是：
- 周期性自检
- 周期性上报
- 周期性发现需要关注的事
- 在“无事发生”和“必须提醒”之间建立契约

这就是长期表现中的节律系统。

## 从行业最佳实践看，长期表现的核心不是“多记”，而是“会压缩、会提炼、会再加载”

这一点，OpenClaw 官方文档和外部最佳实践其实高度一致。

### OpenAI 一侧的启发
OpenAI 在 session memory / context engineering 的示例里反复强调一个工程现实：

> 长时 agent 不能把所有历史机械塞进窗口里，而要把 requirements、decisions、rationales 以更紧凑的形式保留下来。

这和 OpenClaw 的 memory flush + compaction 逻辑，是同一类思想。

### Anthropic 一侧的启发
Anthropic 在 long-running agent/context engineering 的工程文章里，核心强调也是：
- compaction 是长时 coherence 的第一杠杆
- memory 负责跨 compaction 边界保留关键事实
- domain memory 与 step-by-step context 不是替代关系，而是协同关系

这和 OpenClaw 官方 memory/context/compaction 三文档给出的机制，也是高度同构的。

### 工程实测一侧的启发
近一年的 context compression 实测越来越一致地显示：

> **结构化摘要，往往比无控制地保留全部历史更有效。**

这件事在训虾体系里的翻译就是：

> **一只长期稳定的虾，不是记得更多，而是记得更对。**

## 卷五的课程结构，为什么这样排

这一卷不按“概念分类”排，而按“长期退化链条”来排。

### 模块1：记忆管理
先解决“该记什么、写到哪里、如何跨会话保留”的问题。
因为没有 memory layer，后面都空谈。

### 模块2：心跳接入策略
再解决“如何给 agent 建立节律”的问题。
因为没有 cadence，就没有持续维护。

### 模块3：会话污染与清理
然后解决“长会话为什么会越跑越散”。
因为上下文不是静态资产，而是会腐败的运行资源。

### 模块4：文档漂移与人格漂移
再处理“为什么训练好的 agent 逐渐不像自己”。
因为长期运行一定会漂，关键是检测与修复。

### 模块5：主动性边界
再解决“为什么有的主动性变成生产力，有的主动性变成事故源”。
因为主动不是越多越好，而是要被框进边界。

### 模块6：长期稳定性检查清单
最后把前五个模块压缩成一个体检系统。
因为真正能被组织持续执行的，不是理念，而是检查表。

如果说：
- 卷二定义了硅基生命的文档协议

- 卷四定义了训练流程

> **训练之后，如何让一只硅基生命不退化。**

这也是你整个课程最容易拉开行业差距的地方。

因为大多数市面课程讲到这里就停了：
- prompt 怎么写
- workflow 怎么搭
- agent 怎么接工具

但真正能让客户感受到“这不是 demo，而是生产力”的，不是它第一次答得多像人，
而是：

- 七天后它还稳不稳
- 三十天后它还认不认人
- 长项目之后它会不会失忆
- 多轮协作之后它会不会漂移
- 被压缩、被重置、被打断之后它还能不能续上

这才是“长期表现”。

**卷四教你把一只虾训出来，卷五教你别把它跑废。长期表现的本质，不是让 agent 永远记得一切，而是让它在 context 有限、会话会脏、历史会被压缩、任务会持续变动的现实里，仍然保持关键事实不丢、角色不漂、节律不断、行为不散。做到这一点，靠的不是模型天赋，而是记忆工程、上下文工程和节律工程。**
