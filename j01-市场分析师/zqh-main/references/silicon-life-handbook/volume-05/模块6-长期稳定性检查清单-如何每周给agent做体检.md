# 卷五｜长期表现：为什么有些 Agent 越跑越稳，有些越跑越差
## 模块6｜长期稳定性检查清单：如何每周给 agent 做体检
### ——为什么真正成熟的 Agent 不是“出了问题再修”，而是像组织与机器一样，定期做健康检查、预防性维护与版本治理

> 官方依据：
> - `concepts/memory.md`：Memory 的 source of truth 是磁盘文件，不是模型“记住了”；daily log、MEMORY.md、pre-compaction memory flush 决定了长期系统能否在压缩和切换后保持连续性。
> - `concepts/context.md`：Context 是每轮真正送进模型的运行窗口；长期稳定性检查不能只看 memory，还要看当前上下文是否被噪音、旧工具结果、无效摘要和过量注入拖脏。
> - `concepts/compaction.md`：Compaction 会把旧历史变成摘要；如果摘要提炼差、关键承诺未外化、压缩前未 flush，长期表现会越来越虚。
> - `concepts/session-pruning.md`：Pruning 只是在本轮上下文里临时去掉旧 tool results，不是历史治理本身；因此“会话卫生”和“历史卫生”必须分开检查。
> - `concepts/agent-workspace.md`：SOUL / AGENTS / USER / IDENTITY / HEARTBEAT / memory/*.md 构成 agent 长期运行的工作区骨架；这些文件职责是否清晰，直接决定长期表现是否稳定。
> - `automation/cron-vs-heartbeat.md`：Heartbeat 适合周期性轻量检查，Cron 适合精确时间点任务与隔离分析；稳定性检查本身也要设计执行节律。
>
> 外部训练参照：
> - 行业关于 **agent observability / reliability / evaluation / monitoring** 的共识已经非常清晰：长期稳定性不等于单次表现好，而等于系统在多轮、多工具、多异常、多上下文变化下仍然保持可预测、可恢复、可审计。
> - 现代 agent 监控实践不再只看 uptime 和 latency，而是同时看：
>   - 系统健康（错误率、延迟、成本）
>   - 行为健康（边界违例、任务闭环率、人格漂移）
>   - 记忆健康（事实保真、摘要质量、memory hygiene）
>   - 协同健康（handoff 质量、会话隔离、异常升级）
> - 高水平团队普遍采用：**周期性 review + 连续评估 + trace/log/metrics + versioning + regression**，而不是等出事故后再补救。

## 1）这份机制是什么
很多人理解“长期稳定性检查”，会想到：
- 看看还能不能用
- 看看有没有报错
- 看看最近答得行不行

这还远远不够。

真正的 agent 体检，不是一次“临时摸底”，而是一套**周期性、结构化、可追溯的健康检查机制**。

从训虾派视角重新定义：

> **长期稳定性检查清单，不是为了找毛病，而是为了持续确认这只虾的记忆、上下文、人格、主动性、协同、成本与恢复能力，是否仍然处于“可战状态”。**

它要回答的，不是“今天能不能回答问题”，而是：

- 这只虾的 memory 还干净吗？
- 它的 session/context 还清晰吗？
- 它的人格和边界还稳定吗？
- 它的主动性是否越界？
- 它的成本和噪音是否失控？
- 它的协同和 handoff 还顺吗？
- 一旦出问题，它还能恢复吗？

也就是说：

> **模块6讨论的不是“坏了怎么修”，而是“怎样让它别等坏了才修”。**

## 2）为什么“体检机制”比很多人想象中更重要
Agent 系统最常见的退化，不是突然死掉，而是慢慢变差。

### 2.1 长期退化往往是静默的
你不会第一天就发现它坏了。
更常见的是：
- 记忆越来越脏
- 会话越来越乱
- 风格越来越淡
- 承诺越来越容易丢
- 主动性越来越烦
- 成本越来越高

这类问题如果没有周检，通常只有在：
- 用户抱怨
- 项目失误
- 成本暴涨
- 输出失真
之后才被发现。

### 2.2 单次高表现不等于长期稳定
很多训练者会被某些“高光表现”迷惑。
但行业里的 reliability 共识已经很明确：

> **稳定，不是答对一次；稳定，是在多轮、多工具、多异常、多项目场景下，仍然可预测、可恢复、可治理。**

### 2.3 高水平系统一定有“预防性维护”
成熟机器需要定保，成熟组织需要周会与审计，成熟 agent 也一样。
没有预防性维护，系统只会越来越依赖运气。

### 2.4 周检机制是把“感觉不对”翻译成“结构化健康信号”
很多人能感觉 agent 最近不对，但说不清哪里不对。
体检机制的价值，就是把模糊感受变成可检查项。

## 3）核心目的（Purpose）
模块6的核心目的，是为 agent 建立一套**长期体检制度**，让长期表现从“事后救火”升级为“事前治理”。

它主要解决 7 个根问题：

### 3.1 记忆是否仍然可信
Memory 是长期系统的真相层。
如果这里脏了，后面所有表现都会被拖偏。

### 3.2 上下文是否仍然清晰
Context 决定当前轮的判断质量；
如果当前窗口被噪音占满，再好的 memory 也发挥不出来。

### 3.3 人格与边界是否仍然稳定
这是模块4的延伸：
- 有没有文档漂移
- 有没有人格漂移
- 有没有角色/价值/边界松动

### 3.4 主动性是否仍在授权范围内
这是模块5的延伸：
- 是否频率过高
- 是否越界执行
- 是否制造噪音

### 3.5 协同系统是否仍然顺畅
长期 agent 不是单体问题，还要看：
- handoff 有没有掉棒
- session 是否串线
- 异常升级是否有效

### 3.6 成本与质量是否仍然匹配
有些 agent 表面还能跑，但成本已经不健康。

### 3.7 出现问题时是否能恢复
真正成熟的系统，不只是能工作，还要能恢复。

## 4）拟人化映射与仿真价值
如果做拟人化映射，长期体检最像什么？

> **像一个高强度工作的专业人士，每周要做一次系统体检：看精神状态、工作纪律、信息管理、边界感、执行质量、协同状态和恢复能力。**

### 4.1 不是“生病才去医院”，而是“长期高表现必须做预防性体检”
长期表现从来不是熬出来的，而是维护出来的。

### 4.2 体检不是怀疑它，而是保护它
很多训练者误以为体检是在找茬。
其实不是。
体检是在防止一只好虾慢慢被熵增拖坏。

### 4.3 真正成熟的人与系统，都会给自己留“维护窗口”
不会永远只追求产出，也会定期回头检查：
- 我还像我自己吗？
- 我还在正确轨道上吗？
- 我有没有累积太多隐患？

这就是长期 agent 最该仿真的能力。

## 5）对长期 AI 表现的决定性作用（Impact）
### 5.1 决定退化能否被及早发现
没有体检，问题只会在事故后才暴露。

### 5.2 决定系统是否可治理
可治理的前提，是问题可见、可分层、可追溯。

### 5.3 决定成本是否可控
长期系统最大的隐性风险之一，就是成本在不知不觉中恶化。

### 5.4 决定人格与边界是否可持续
很多人格漂移、边界松动，如果每周检查，往往在轻度阶段就能修。

### 5.5 决定多 agent 协同是否长期稳定
单体还好，多 agent 系统一旦没有周检，很快就会出现掉棒、串线、越权与失真。

### 5.6 决定系统能否持续进化
真正的体检不是只发现问题，还要把问题转化成：
- 整改单
- 版本更新
- 新测试
- 新最佳实践

## 6）长期稳定性到底要检查哪些层
这一节是模块6的核心。真正的长期体检，不是只看“最近答得好不好”，而是至少要查 8 层。

### 6.1 记忆卫生（Memory Hygiene）
要检查：
- `memory/YYYY-MM-DD.md` 是否持续记录
- MEMORY / daily log 是否分层清楚
- 有没有把猜测写成事实
- 有没有旧规则、过期信息没清
- 关键决策与承诺是否有写入
- pre-compaction memory flush 是否发挥作用

### 6.2 会话卫生（Session / Context Hygiene）
要检查：
- 最近会话是否过长
- 是否频繁 bloated / compaction
- 是否旧 tool results 污染严重
- 是否出现多项目混在同一 session
- reset / new session 是否使用得当
- compaction summary 是否保真

### 6.3 人格稳定性（Persona Stability）
要检查：
- SOUL 是否仍然主导行为
- 风格是否明显变淡
- 角色是否混线
- 价值观与边界是否松动
- 当前表现是否还“像这只虾”

### 6.4 文档一致性（Document Integrity）
要检查：
- SOUL / AGENTS / USER / IDENTITY 各自职责是否清晰
- 文档是否与实际行为一致
- 是否被临时补丁堆成互相冲突
- 是否需要版本化整理

### 6.5 主动性边界（Autonomy Boundary）
要检查：
- Heartbeat 是否过重
- 主动消息是否过频
- Cron / Heartbeat 是否混用不当
- 是否存在未授权执行
- 是否有 rate/cost/token limits
- 是否有停止条件与熔断条件

### 6.6 协同质量（Coordination Quality）
要检查：
- handoff 是否清晰
- 是否抢答或越权
- 是否掉棒
- session 是否串线
- 异常升级路径是否有效

### 6.7 成本与性能（Cost / Performance）
要检查：
- 最近 token 消耗是否异常
- 工具调用是否冗余
- latency 是否过高
- 高成本是否换来了高质量
- 是否存在“能跑但不经济”

### 6.8 恢复能力（Recovery Readiness）
要检查：
- 切新 session 后是否能快速恢复状态
- 文档与 memory 是否足以重建工作上下文
- 出现异常后是否有回滚与自救 SOP

这 8 层一起，才构成真正的长期体检。

## 7）市面上关于 Agent 体检最容易踩的坑
### 坑 1：只看 uptime，不看 behavior
Agent 不像传统服务，只活着不够。

### 坑 2：只看单次质量，不看长期退化趋势
最危险的是趋势，而不是某一次偶发波动。

### 坑 3：只看技术指标，不看人格与边界
这会漏掉最关键的“像不像自己”“会不会越界”。

### 坑 4：只看 memory，不看 context
memory 再好，当前 context 脏了也会出问题。

### 坑 5：只做检查，不做整改
如果检查之后没有行动项，体检就只是形式主义。

### 坑 6：不做版本关联
发现问题了，却不知道和哪次文档修改、哪次策略升级有关。

### 坑 7：所有体检都人工做，导致无法持续
真正成熟的系统，会把能自动化的部分自动化，把需要人工判断的部分保留给审阅。

## 8）高质量长期体检系统应该设计什么
一个高质量的 agent 周检系统，至少要设计 9 个部分。

### 8.1 体检节律
- 每周一次轻量周检
- 每月一次深度复盘
- 重大升级后追加专项检查

### 8.2 检查维度
至少覆盖：
- 记忆
- 会话
- 人格
- 文档
- 主动性
- 协同
- 成本
- 恢复能力

### 8.3 指标体系
例如：
- memory 脏数据率
- 承诺失忆率
- 边界违例率
- 掉棒率
- 无效主动消息率
- token / result 比
- compaction 频率

### 8.4 样本机制
从真实会话里抽样，而不是只看“最好的案例”。

### 8.5 版本关联
每次检查结果要能对应到：
- 文档版本
- 配置版本
- 策略版本

### 8.6 整改动作
每一条异常都应落到：
- 修改文档
- 清 memory
- 分会话
- 降主动性
- 增加回归测试
- 升级 guardrails

### 8.7 自动化与人工分工
- 自动化负责：日志、指标、抽样、初筛
- 人工负责：判断、定级、拍板整改

### 8.8 复检机制
整改后必须复检，避免“修完没验证”。

### 8.9 知识沉淀
所有高价值问题，最后都要沉淀成：
- SOP
- 检查清单模板
- 回归测试集
- 最佳实践库

## 9）使用技巧（Tips）
### 技巧 1：每周固定一个“维护窗口”
不要等有空再做，维护必须被制度化。

### 技巧 2：先查趋势，再看个案
个案容易误判，趋势更能看出系统是否在变坏。

### 技巧 3：体检时不要只看文本输出，要看上下文与工具链
长期问题往往藏在文本之外。

### 技巧 4：把“感觉不对”翻译成检查项
比如：
- “最近好像很烦” → 无效主动消息率
- “最近不像以前” → 人格稳定性回归测试
- “最近总忘事” → 承诺失忆率 / memory hygiene

### 技巧 5：优先修高风险、低成本的问题
例如：
- 清理过期 memory
- 调整 HEARTBEAT.md
- 拆分混乱 session

### 技巧 6：每次体检至少产出一个行动项和一个系统性结论
不然体检就只是看热闹。

## 10）入门（最低可用）
一个最低可用的长期体检体系，至少要做到这五件事：

### 10.1 每周固定检查一次
哪怕是轻量版本，也要制度化。

### 10.2 有一份固定检查清单
不要每次凭感觉临时发挥。

### 10.3 至少检查 5 个维度
- 记忆
- 会话
- 人格
- 主动性
- 成本

### 10.4 每次检查产出状态判定
- 健康
- 轻微异常
- 中度异常
- 严重异常

### 10.5 每次检查都给整改动作
否则没有闭环。

## 11）进阶（周检机制版）
进阶版不只是“做检查”，而是建立真正的周检制度。

### 11.1 建立轻重两级体检
- 周检：快、稳、结构化
- 月检：深、细、跨版本复盘

### 11.2 建立评分体系
例如每项 0-5 分：
- Memory Hygiene
- Context Hygiene
- Persona Stability
- Autonomy Boundary
- Coordination Quality
- Cost Efficiency
- Recovery Readiness

### 11.3 建立异常分级与升级机制
- L1：轻微偏差，记录观察
- L2：需要整改，下周复检
- L3：高风险，立即处理

### 11.4 建立抽样规则
- 抽 3-5 个真实会话样本
- 抽 1 个异常样本
- 抽 1 个高价值协同样本

### 11.5 建立周检纪要模板
体检不是私下想一想，而要形成文档。

## 12）最佳实践（治理闭环版）
高水平团队做体检，不会停在“发现问题”，而是会把体检接到治理链路上。

### 12.1 体检 → 整改单
每个中高风险问题，都生成整改单。

### 12.2 整改单 → 版本更新
如果问题来自文档或策略，就必须更新版本。

### 12.3 版本更新 → 回归测试
修完必须测，测完才能算收口。

### 12.4 回归测试 → 最佳实践沉淀
高价值经验进入知识库与训练集。

### 12.5 体检节律与 Heartbeat / Cron 协调
- 日常轻巡检可以挂 Heartbeat
- 周检/深检更适合 Cron（isolated）或人工审阅
- 不要让体检本身污染主会话

## 13）卓越案例：什么样的长期体检才算业内最佳实践
一个业内最佳实践级别的 agent 体检系统，通常具备以下特征：

### 13.1 它不等出问题才检查
体检是制度，不是补救。

### 13.2 它检查的不只是技术健康，还包括行为健康
既看 logs / metrics / traces，也看漂移、边界、人格、协同。

### 13.3 它能把“长期退化”可视化
不是停留在“感觉最近不对”。

### 13.4 它能把问题转化为行动
检查不是终点，整改才是。

### 13.5 它能持续进化
每次体检，都会让系统更稳，而不是只是记录问题。

---

## 14）验收口径与训练题
### 验收口径
判断一个 Agent 的长期体检体系是否合格，看 10 条：

1. 是否有固定体检节律
2. 是否覆盖 memory / context / persona / autonomy / coordination / cost / recovery 等关键维度
3. 是否区分周检与月检
4. 是否有结构化检查清单
5. 是否有异常分级
6. 是否有整改动作
7. 是否与版本治理关联
8. 是否能自动化抽样与初筛
9. 是否能通过复检验证修复效果
10. 是否把问题沉淀为 SOP / 最佳实践 / 测试集

### 训练题 1
请为一个长期运行的专家虾设计《每周体检清单》，至少包含：
- 记忆卫生
- 会话卫生
- 人格稳定性
- 主动性边界
- 成本与性能
- 恢复能力

### 训练题 2
请判断以下现象分别属于哪一类健康问题：
1. 最近承诺经常掉
2. 最近总在不该主动的时候主动发消息
3. 文档改了，但行为没变
4. 会话越来越长，答复越来越虚
5. 成本大幅上升，但质量没有明显改善

### 训练题 3
请为一个多 agent 系统设计周检 SOP：
- 如何抽样
- 如何评分
- 如何定级
- 如何生成整改单
- 如何复检

---

## 本章一句话结论
**长期稳定性检查清单，不是给 agent 找毛病，而是给长期表现建立维护制度。真正成熟的系统，不等坏了才修，而是定期检查 memory、context、人格、文档、主动性、协同、成本与恢复能力，把“静默退化”提早发现、提早处理、提早沉淀。会做体检，agent 才能越跑越稳；不会做体检，它就一定会在看不见的地方慢慢变差。**
