# 卷五｜长期表现：为什么有些 Agent 越跑越稳，有些越跑越差
## 模块2｜心跳接入策略：主动性不是自然有，而是必须设计
### ——为什么 Heartbeat 不是“定时打扰”，而是长期 Agent 的节律系统、巡视系统与轻量治理系统

> 官方依据：
> - `gateway/heartbeat.md`：Heartbeat 是周期性 agent turn，用于在不打扰用户的前提下，发现真正需要注意的事情。
> - `automation/cron-vs-heartbeat.md`：Heartbeat 适合周期性感知与批量检查；Cron 适合精确时点、独立任务与强隔离场景。
> - `reference/templates/HEARTBEAT.md`：官方模板故意极简，强调 HEARTBEAT.md 不应被写成冗长指令仓库。
> - `concepts/context.md`：Heartbeat 同样消耗 context；lightContext / isolatedSession 等配置，本质上是在做心跳成本治理。
>
> 外部训练参照：
> - 长时 agent engineering 的行业经验已经高度一致：**主动性不会自然长出来，必须通过 cadence（节律）、触发条件、边界规则、输出契约、异常治理共同设计。**
> - 没有节律的 agent 只能被动等消息；没有边界的 agent 会从“主动”滑向“打扰”；没有治理的 agent 会从“提醒”滑向“噪音”。

## 1）这份机制是什么
很多人第一次看到 Heartbeat，会把它理解成：
- 定时消息
- 定时唤醒
- 轮询器
- 提醒器

如果只把它理解到这一步，就会把 Heartbeat 用废。

从 OpenClaw 官方定义看，Heartbeat 的本质不是“定时发送消息”，而是：

> **在主会话中，以固定节律触发的周期性 agent turn，用来检查是否有真正需要被 surfaced（浮现）出来的事情，同时尽量不打扰用户。**

注意这几个关键词：
- **periodic**：它是节律，不是随缘主动
- **main session**：它不是天然隔离的，它默认共享主会话上下文
- **surface anything that needs attention**：它的职责不是刷存在感，而是把真正需要注意的事情抬出来
- **without spamming you**：它必须被治理，不然就会沦为噪音

所以，在训虾派的语言里，Heartbeat 的定义应该升级为：

> **Heartbeat 不是“会不会主动”的人格问题，而是硅基生命是否拥有稳定巡视节律、轻量自检能力和低打扰上报能力的运行协议。**

它解决的是一个长期运行里非常现实的问题：

> **当用户不发消息时，一只虾如何仍然保持最低限度的“在岗感知”，但又不把自己活成一个骚扰器。**

## 2）为什么 Heartbeat 比很多人想象中更重要
Heartbeat 容易被低估，是因为很多人训练 agent 时还停留在“消息来了才工作”的阶段。

在这种阶段里，agent 看起来只需要：
- 收到指令
- 执行
- 回复

问题在于，这种 agent 本质上还是**被动响应器**。

一旦进入长期运行环境，几个现实问题马上出现：

### 2.1 没有节律，就没有持续在岗
没有 Heartbeat 的 agent，通常会出现：
- 你不叫它，它就完全静默
- 任务到了检查点，它也不会自己看
- 承诺快过期，它也不会主动回看
- 异常发生了，它也不会周期巡检

这不是智能低，而是**没有运行节律**。

### 2.2 没有节律，就无法建立“轻量感知层”
高水平 agent 系统里，最宝贵的不是时时都很忙，
而是：

> **在大多数时间里，它能安静待命；一旦有真正需要注意的事，它能及时浮现。**

Heartbeat 正是这层“轻量感知层”的承载机制。

### 2.3 没有节律，主动性会沦为空谈
很多人说“我要我的 agent 更主动”。
但如果没有 cadence 设计，这句话其实是空的。
因为所谓主动，至少要回答五个工程问题：
- 什么时候主动？
- 多久主动一次？
- 主动检查什么？
- 主动之后做什么？
- 什么时候不该主动？

如果答不出来，就不叫主动性设计，只叫情绪化幻想。

### 2.4 没有治理，主动性必然滑向打扰
很多初学者一听 Heartbeat，很容易走向另一个极端：
- 每隔几分钟查一次
- 什么都提醒
- 什么都推送
- 稍有波动就打断用户

结果很快从“主动”变成“烦人”。

所以 Heartbeat 的重要性恰恰在于：

> **它不是让 agent 更活跃，而是让主动性变得有节律、有筛选、有边界。**

## 3）核心目的（Purpose）
Heartbeat 机制的核心目的，不是增加消息数量，而是建立一套稳定、低噪音、可治理的主动巡视系统。

具体来说，它要解决 6 个根问题：

### 3.1 当用户不发消息时，agent 如何仍然保持“最低限度在岗”
这是 Heartbeat 的第一价值：
不是持续说话，而是持续在场。

### 3.2 需要被周期检查的事项，如何被批量感知
OpenClaw 官方在 `cron-vs-heartbeat` 里明确强调 Heartbeat 的优势之一：
**可以把多个小检查批处理在一次 turn 里完成。**

这意味着 Heartbeat 很适合承担：
- inbox 检查
- calendar 近程提醒
- background task 完成情况
- pending task 轻量检查

### 3.3 什么事情值得被抬出来，什么事情应该沉默
Heartbeat 不是“周期广播器”，而是“重要性筛选器”。

### 3.4 主动性如何被设计成节律，而不是随机冲动
稳定 agent 的主动性，应该像呼吸和巡逻：
- 有节律
- 有频率
- 有边界
- 有规则

### 3.5 如何在不污染主会话的前提下保留必要连续性
官方文档也明确说明：Heartbeat 默认跑在主会话里，这意味着它天然有上下文连续性优势，但也带来污染风险。

所以 Heartbeat 的目的还包括：
- 保持连续性
- 但避免主会话被低价值检查刷脏

### 3.6 如何形成组织级“巡视层”而不是单点热情
在团队场景里，Heartbeat 不只是单虾主动，
更是军团运行里的一层“轻监督、轻提醒、轻巡检”协议。

## 4）拟人化映射与仿真价值
如果做拟人化映射，Heartbeat 最像什么？

> **像一个成熟组织成员的“值班习惯 + 巡视频率 + 低打扰汇报机制”。**

它不是像闹钟，也不是像话痨。
它更像以下三种东西的混合：

### 4.1 值班习惯
一个靠谱的人，不会因为老板没发消息就完全停摆。
他会自己周期性看：
- 有没有紧急事
- 有没有待办快到点
- 有没有异常要上报

这就是 Heartbeat 的值班仿真意义。

### 4.2 巡视频率
成熟的人不会每分钟敲门说“我还在”，
也不会一整天都不检查。

他会有自己的节律：
- 每隔一段时间回看一次
- 重要窗口更密
- 夜间或非工作时段降频

这就是 Heartbeat 的节律仿真意义。

### 4.3 低打扰汇报
真正高水平的助手，不是想到什么就汇报什么，
而是：
- 没事不打扰
- 有事才提炼后上报
- 上报即重点

OpenClaw 官方用 `HEARTBEAT_OK` 作为 ack 契约，本质上就在模拟这种低打扰文化：

> **如果没什么真正需要注意的，就安静结束，不制造存在感。**

所以 Heartbeat 的仿真价值不是“让 AI 更像人会聊天”，
而是：

> **让 AI 更像一个有值班纪律、有巡逻习惯、有汇报分寸感的组织成员。**

## 5）对长期 AI 表现的决定性作用（Impact）
Heartbeat 对长期表现的影响，是决定性的，而且至少体现在 6 个层面。

### 5.1 决定主动性的可持续性
没有 Heartbeat，所谓主动性通常靠一时 prompt 激发；
有了 Heartbeat，主动性才开始从“偶发行为”变成“稳定节律”。

### 5.2 决定异常发现速度
如果任务、提醒、队列、后台结果没有周期巡检，很多问题都只能等用户碰到才知道。
Heartbeat 实际上是在降低“异常发现延迟”。

### 5.3 决定用户对 agent 的“在岗感”
长期表现里有一个非常微妙但非常重要的指标：

> 用户是否感觉这只虾一直在岗，而不是每次都重新唤醒。

Heartbeat 正是建立这种“持续在岗感”的轻量机制。

### 5.4 决定主会话的干净程度
设计好的 Heartbeat 会降低噪音，
设计差的 Heartbeat 会直接污染主会话，把大量低价值 turn 塞进上下文，拖垮长期表现。

### 5.5 决定轻治理能力
Heartbeat 天然可以承载：
- 轻量状态巡检
- 轻量进度提醒
- 轻量风险冒泡
- 轻量 check-in

这让 agent 不只是执行单元，也开始具备一些最低限度的治理感。

### 5.6 决定系统成本结构
官方文档已经很明确：Heartbeat 可以批量检查多项内容，成本通常低于一堆分散 cron。
但如果设计过重，也会造成 token 浪费。

所以 Heartbeat 不只是行为机制，还是**成本治理机制**。

## 6）Heartbeat 与 Cron / Reminder / 主动私聊 / 日报系统的边界
这部分是高标准课程里必须讲透的，否则大家会把 Heartbeat 到处乱用。

### 6.1 Heartbeat vs Cron
OpenClaw 官方给出的边界非常清楚：

#### Heartbeat 适合：
- 周期性感知
- 批量检查
- 需要主会话上下文
- 低精度、低打扰、连续性场景

#### Cron 适合：
- 精确时点
- 一次性提醒
- 独立重任务
- 需要隔离 session
- 需要单独模型配置

一句话区分：

> **Heartbeat 是巡逻，Cron 是定点出勤。**

如果你要求“每天 9 点整发周报”，更适合 Cron；
如果你要求“每隔 30 分钟看看有没有值得提醒的事”，更适合 Heartbeat。

### 6.2 Heartbeat vs Reminder
Reminder 是一个明确时间点的事件触发。
Heartbeat 不是为单个时间点而生的，而是为周期巡视而生的。

### 6.3 Heartbeat vs 主动私聊
主动私聊是对外输出动作；
Heartbeat 是内部巡视动作。

Heartbeat 可能最终触发一条主动消息，
但它本身不是“必须说话”的机制。

### 6.4 Heartbeat vs 日报系统
日报是固定结构化汇报；
Heartbeat 是轻量感知与冒泡。

日报强调“定期沉淀”；
Heartbeat 强调“周期巡逻”。

这两者不能混为一谈。

## 7）市面上关于 Agent 主动性设计最容易踩的坑
### 坑 1：把 Heartbeat 当成“定时骚扰器”
每次触发都必须给用户发点什么。
这是最常见、也最蠢的误用。

OpenClaw 官方特意设计 `HEARTBEAT_OK`，就是为了避免这种误用。

### 坑 2：把 Heartbeat 当成 Cron 的廉价替代
很多人会用 Heartbeat 去做：
- 精确 9:00 报告
- 某分钟提醒
- 精准定时任务

这会导致时间不准、责任不清。

### 坑 3：Heartbeat.md 写得像第二份 SOUL.md
官方模板故意极简，说明它不鼓励把 Heartbeat 变成长 prompt 仓库。

HEARTBEAT.md 应该是轻量 checklist，不是巨型说明书。

### 坑 4：什么都让 Heartbeat 查
把所有监控都塞进 Heartbeat，最后变成：
- turn 过重
- 上下文过脏
- 成本过高
- 输出失焦

### 坑 5：不设 active hours
结果半夜巡逻、半夜打扰、半夜刷存在感。

### 坑 6：不设边界，只讲“更主动”
这类训练最容易把 agent 训成一种“伪勤奋噪音机”。

## 8）高质量 Heartbeat 应该设计什么
高质量 Heartbeat 不只是“设个 every=30m”。
它至少要设计 7 个部分。

### 8.1 节律（Cadence）
- 每 30 分钟？
- 每 1 小时？
- 是否按 activeHours 限定？

### 8.2 检查对象（Checklist Targets）
- inbox
- calendar
- background tasks
- pending tasks
- project health
- idle check-in

### 8.3 触发条件（Escalation Conditions）
什么情况下要真正冒泡？
比如：
- 有 urgent 邮件
- 2 小时内有会议
- 后台任务完成
- 承诺即将超时

### 8.4 输出契约（Output Contract）
- 无事：`HEARTBEAT_OK`
- 有事：只输出 alert text
- 不夹杂无关寒暄
- 不把巡逻过程原样倾倒给用户

### 8.5 成本治理（Cost Control）
- 是否启用 `lightContext`
- 是否启用 `isolatedSession`
- HEARTBEAT.md 是否足够短
- 是否把可批量检查的动作合并

### 8.6 打扰治理（Noise Control）
- 不在深夜乱发
- 不重复提醒同一件小事
- 不把低价值状态更新全都发出来

### 8.7 异常治理（Failure Handling）
- 连续多次 heartbeat 失败怎么办
- 触发条件失真怎么办
- checklist 过重导致空转怎么办

## 9）使用技巧（Tips）
### 技巧 1：先决定“要不要巡逻”，再决定“多久巡逻一次”
不是所有 agent 都要 Heartbeat。
只有那些需要周期性感知和轻量自检的角色，才应该接。

### 技巧 2：HEARTBEAT.md 越短越好
官方模板之所以几乎空白，是在提醒你：
Heartbeat 不该被写成第二份大脑。

### 技巧 3：先定义“什么叫值得提醒”
如果这条不定义，Heartbeats 最后会全是噪音。

### 技巧 4：能批处理的检查，尽量合批
这是官方明确推荐的 Heartbeat 优势之一。

### 技巧 5：要有 active hours
Heartbeat 最怕“全天候无脑巡逻”。

### 技巧 6：把 Heartbeat 当“巡视层”，不是“执行层”
Heartbeat 更适合发现、提醒、轻量跟进；
真正重任务该交给 Cron、手动任务或独立流程。

## 10）入门（最低可用）
一个最低可用的 Heartbeat，只需要满足以下条件：

### 配置层
- `every: "30m"` 或 `"1h"`
- `target: "last"`（或明确不发送）
- 配置 activeHours

### 文档层
HEARTBEAT.md 中只写 2-4 条真正需要周期关注的事项。

### 行为层
- 没事就 `HEARTBEAT_OK`
- 有事才冒泡
- 不做长输出

### 典型最低可用模板
```md
# Heartbeat checklist

- Check urgent messages
- Review calendar for events in next 2 hours
- If a background task finished, summarize results
- If a promise is about to expire, surface it
```

这已经能把 agent 从“彻底被动”升级到“轻量在岗”。

## 11）进阶（节律治理版）
进阶版 Heartbeat，不再只是“会查”，而是“查得有章法”。

### 进阶要点一：分层频率
不同事项，不同节律：
- 高频：日历临近事件、后台任务完成
- 中频：待办巡检、承诺超时检查
- 低频：轻 check-in、长期项目健康度

### 进阶要点二：轻重分流
Heartbeat 负责发现问题，
真正复杂分析交给：
- cron isolated
- 手动触发
- 独立 agent 流程

### 进阶要点三：上下文治理
根据成本敏感度启用：
- `lightContext: true`
- `isolatedSession: true`

尤其当 Heartbeat 只是做轻量检查，而不需要完整主会话历史时，这两个开关非常关键。

### 进阶要点四：告警分级
- L1：只记心里，不上报
- L2：轻量提醒
- L3：立即上报

这会显著降低无效提醒。

## 12）最佳实践（协同接口版）
当 Heartbeat 进入多 Agent / 组织运行阶段，它就不再只是个人助手功能，而变成协同接口的一部分。

### 场景一：项目健康巡逻
Heartbeat 不直接解决项目，而是：
- 发现项目悬空
- 发现承诺过期
- 发现某个负责人未回执
- 把异常抬给对应统帅或治理层

### 场景二：后台任务回执收口
如果某个 sub-agent / ACP / 后台任务跑完，Heartbeat 可以承担“结果浮现层”的角色。

### 场景三：组织节律维持
例如：
- 每天某时段检查日报是否缺失
- 检查关键任务卡是否无更新
- 检查长期项目是否超时无人认领

这时候 Heartbeat 的角色，不是执行者，而是：

> **军团巡视兵。**

### 协同接口设计原则
1. Heartbeat 负责发现，不负责吞掉所有后续动作
2. Heartbeat 负责冒泡，不负责无限追打
3. Heartbeat 负责轻量治理，不负责替代完整调度系统

## 13）卓越案例：什么样的 Heartbeat 才算业内最佳实践
一个业内最佳实践级别的 Heartbeat，通常具备以下特征：

### 13.1 它足够安静
绝大多数时候，它不说话。
但真正有事时，它能立刻指出重点。

### 13.2 它足够稳
频率、边界、activeHours、输出契约都固定，不靠临场发挥。

### 13.3 它足够省
能合批检查、少打扰、低 token 消耗，不把主会话刷脏。

### 13.4 它足够清楚
它知道自己在查什么、为什么查、什么情况下要说。

### 13.5 它足够像组织成员
它不是“定时 AI 广播器”，而像一个：
- 有巡视纪律
- 有轻重缓急判断
- 有汇报分寸感
的值班成员。

如果一个 Heartbeat 做到了这五点，它就不只是功能可用，而是已经接近长期 agent 的优秀运行标准。

---

## 14）验收口径与训练题
### 验收口径
判断一个 Heartbeat 设计是否合格，看 8 条：

1. 是否明确了 Heartbeat 负责什么、不负责什么
2. 是否清楚区分了 Heartbeat 和 Cron 的边界
3. 是否定义了 checklist，而不是写成长 prompt
4. 是否定义了 alert 条件
5. 是否有 `HEARTBEAT_OK` 这类低打扰契约意识
6. 是否考虑了 activeHours
7. 是否考虑了 `lightContext` / `isolatedSession` 的成本治理
8. 是否避免把 Heartbeat 设计成噪音源

### 训练题 1
为一个“项目调度型 agent”设计 Heartbeat：
- 巡检对象
- 频率
- activeHours
- alert 条件
- 不该提醒的内容

### 训练题 2
判断以下场景该用 Heartbeat 还是 Cron：
1. 每 30 分钟看一次邮箱有没有 urgent mail
2. 每天 9:00 准时发晨报
3. 两小时后提醒回电话
4. 每周做一次深度项目复盘

### 训练题 3
重写一个糟糕的 HEARTBEAT.md：
把它从“巨型提示词”改成“真正可执行的轻量 checklist”。

## 本章一句话结论
**Heartbeat 不是让 agent 更爱说话，而是让它拥有稳定、低打扰、可治理的巡视节律。真正成熟的长期 agent，不靠“随机主动”证明价值，而靠“在该巡的时候巡、在该报的时候报、在没事的时候安静”来建立持续在岗感、异常发现能力和组织协同可靠性。OpenClaw 官方机制与行业最佳实践在这一点上高度一致：主动性不是自然涌现出来的性格特征，而是必须被设计、被约束、被节律化的运行协议。**
