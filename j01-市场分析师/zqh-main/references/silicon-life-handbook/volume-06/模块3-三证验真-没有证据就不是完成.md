# 卷六｜治理系统：如何防止军团失真
## 模块3｜三证验真：没有证据，就不是完成
### ——为什么军团治理最硬的环节不是"做不做"，而是"如何证明真做了"

> **一句话结论：**
> **三证验真是军团治理的底层信任协议——它将"声称完成"降维成"可独立验证的证据集"。没有三证验真，任务卡只是意愿书，验收口径只是愿望清单。三证验真决定了一个组织能不能从"靠信用运行"升级到"靠证据运行"。**

---

## 一、这是什么（定位）

### 1.1 三证验真不是"验收的补充"

这是一个常见误解，必须在开篇就纠正。

模块1（任务卡）定义了"做什么"，模块2（验收口径）定义了"什么算好"，而**模块3（三证验真）定义了"你凭什么说做好了"**。

```
任务卡（模块1）—— 承诺层："我承诺做这件事"
     ↓
验收口径（模块2）—— 标尺层："这件事做到什么标准"
     ↓
三证验真（模块3）—— 证据层："你做完了，证据在哪里"
     ↓
整改单（模块4）—— 纠偏层："没做好，怎么修"
     ↓
军功簿（模块5）—— 结算层："做好做坏，进账本"
```

**三证验真是治理链条从"软约束"到"硬约束"的分水岭。**

没有三证验真的组织：大家靠"我相信你"运行——君子协议，好人系统。

有三证验真的组织：大家靠"你能证明"运行——制度保障，系统可信。

### 1.2 三证的具体定义

"三证"是一个不精确的数字——它不代表"恰好三种证据"，而是代表**证据的三种硬度层级**。每一个层级都比前一层更硬一档。

| 层级 | 名称 | 核心要求 | 类比 |
|------|------|---------|------|
| **证一** | **新产物（Artifact）** | 在预期路径上产生了可定位、可打开、可阅读的新文件/新数据/新产物 | 论文发表了——能看到PDF |
| **证二** | **前后对比（Diff）** | 产物的内容与原始状态之间存在可辨识的实质性变化，而非复制粘贴或空壳占位 | 论文被同行评审过了——有修改记录 |
| **证三** | **测试/回执/日志（Verification Trail）** | 产物通过了预设的检验流程，且检验过程留有不可抵赖的日志 | 论文的引用数据和实验代码可复现 |

**三证之间不是"三选一"——是"层层递进"。**

> 证一 = "你做了"（存在）
> 证一 + 证二 = "你真的做了"（实质性变化）
> 证一 + 证二 + 证三 = "你做对了，而且我可以复核"（可信证据链）

### 1.3 为什么是三证，不是其他任何数字

三证结构不是凭空发明的——它来自**四个独立领域的最佳实践的交叉收敛**：

**来源1：司法证据学**

司法体系对证据的三个要求：
1. **存在性**（Existence）——证据确实存在，不是凭空捏造 → 对应证一
2. **真实性**（Authenticity）——证据未经篡改，保持了原始状态 → 对应证二
3. **关联性**（Relevance & Verification）——证据与待证事实之间有明确关联 → 对应证三

> *"The weight of evidence is determined not by its quantity but by its integrity, its provenance, and its verifiability."*
> （证据的权重不取决于数量，而取决于完整性、来源可追溯性和可验证性。）
> ——Federal Rules of Evidence, Article IX

**来源2：软件工程测试金字塔**

测试金字塔的经典结构：
1. **单位测试**——存在、可通过 → 证明"被测试了"
2. **集成测试**——组件间交互正确 → 证明"前后一致"
3. **端到端测试**——用户场景跑通 → 证明"可被复现"

这三层和三证的"存在→真实→可复核"是同一个逻辑。

**来源3：学术出版与同行评审**

1. **发表（Publication）**——论文被录用 → 证一：产物存在
2. **评审意见（Review）**——论文有实质修改 → 证二：前后有变化
3. **数据与代码公开（Reproducibility）**——结果可被复现 → 证三：可复核

**来源4：高质量审计（Financial Audit）**

| 审计标准 | 训虾派三证对应 |
|---------|--------------|
| **存在性测试**（Existence Test）：资产确实存在 | 证一——产物在预期路径 |
| **完整性测试**（Completeness Test）：所有事项已完整记录 | 证二——变更历史完整 |
| **准确性测试**（Accuracy / Vouching）：数字和事实可以被追溯验证 | 证三——有日志和测试记录 |

> *"Audit is not about trust. It's about evidence chain that can survive cross-examination."*
> （审计不是关于信任。它是关于能够经受住交叉质询的证据链。）
> ——PCAOB Auditing Standard No. 15, Audit Evidence

**这四流归一的结论：三证是跨越人类组织到硅基军团的证据工程共识。**

### 1.4 三证验真的适用范围

| 任务类型 | 适用三证 | 说明 |
|---------|---------|------|
| 文档写作类 | 证一（产物存在）+ 证二（前后对比） | 文档类任务证据较直观 |
| 代码开发类 | 证一（代码存在）+ 证二（Diff）+ 证三（测试通过） | 代码类天然有三证 |
| 分析研究类 | 证一（报告存在）+ 证三（数据源可复核） | 证二不重要，证三重要 |
| 运营执行类 | 证一（操作日志）+ 证三（结果截图） | 自动化操作天然有日志 |
| 战略决策类 | 证一（方案存在）+ 证二（多版迭代） | 证三不适合——不测试 |

## 二、核心目的（为什么存在）

### 2.1 三证验真要解决的四个根问题

| 问题 | 表现 | 三证如何解决 |
|------|------|------------|
| **假完成** | 声称做完了但产物不存在 | 证一要求产物存在于预期路径 |
| **伪完成** | 产物存在但内容没变（改了个文件名） | 证二要求有实质性 Diff |
| **半完成** | 产物有内容但不完整 | 证二 + 证三叠加验证完整度 |
| **不可复验** | 说做完了但别人没法复核 | 证三要求留下可复验的日志/测试 |

### 2.2 三证与其他模块的分工

| 模块 | 问的问题 | 三证的输入 |
|------|---------|-----------|
| 模块1 任务卡 | 做什么？ | 任务卡定义交付物路径 |
| 模块2 验收口径 | 什么算好？ | 验收口径定义通过条件 |
| **模块3 三证验真** | **凭什么说做完了？** | **证据链自证** |
| 模块4 整改单 | 没做好怎么修？ | 三证失效触发整改单 |
| 模块5 军功簿 | 长期表现怎么结算？ | 三证通过/失败计入账本 |

### 2.3 三证验真的核心价值

**它不是让组织更严苛——它是让组织更高效。**

低效组织：每个人都要靠信任和关系判断"他做完了没"→ 争论占用了大量时间。

高效组织：每个人只需要看三证齐不齐 → **判断从"对人"变成"对物"**。

三证的价值不是"惩罚不诚实"——而是**让诚实的人不需要先证明自己诚实**。因为在三证体系里，"完成"不是一个需要讨论的问题——它是一组可以被独立验证的证据集合。

## 三、拟人化映射（仿真价值）

### 3.1 三证的人类组织类比

| 三证 | 人类组织类比 | 为什么有效 |
|------|-------------|-----------|
| **证一**（新产物） | 你交了一份文件——别人能看到、能打开、能阅读 | "我做了" → "我交了什么" |
| **证二**（前后对比） | 不是交一份空壳，文件里有实质性改动 | "我改了" → "改了什么" |
| **证三**（测试/日志） | 你的改动能被复现——另一个人按同样步骤也能得到同样结果 | "我的改法是可复现的" |

### 3.2 为什么硅基生命需要三证

人类有"信用直觉"——你和一个同事合作久了，你知道这个人靠不靠谱。

**但 Agent 没有信用直觉。** Agent 对另一个 Agent 的判断，只能基于：
- 它提交的证据
- 证据的可验证性
- 历史三证的完整率

所以三证在硅基军团里做的事，和信用体系在人类组织里做的事是一样的——**但硅基军团只能通过证据（而非直觉）来完成这件事**。

## 四、对长期 AI 表现的影响（Impact）

### 4.1 没有三证——系统性的退化链条

```
第1周：发现某个 Agent 经常说"做完了"但产物不完整
第2周：验收方开始对所有 Agent 的回答持怀疑态度
第3周：信任破裂——验收方开始质疑每一个"完成"
第4周：治理系统从"按制度运行"退化为"人工逐个验证"
第5周：治理系统实际失效——因为人工逐个验证根本做不完
第6周：军团进入"表面治理，实际没人验真"的状态
```

**没有三证的治理系统，不是治理系统——是"信任赌博系统"：赌 Agent 会诚实，赌验收方会相信，赌没有人会深究。** 每一次都赌赢的概率随时间指数级下降。

### 4.2 有三证——系统性的信任累积

```
第1周：所有交付物都带了证一
第2周：验收方发现证一的完整性在提升
第3周：加入证二要求，Agent 自动生成 Diff
第4周：验收方发现可以独立复核了——不再需要信任 Agent 的口头承诺
第5周：加入证三要求，核心交付物开始走自动化验证流程
第6周：治理系统进入"证据驱动"模式——验收方只需要做选择题
```

### 4.3 业界交叉验证：Anthropic 的观察

Anthropic 在 Model Context Protocol (MCP) 规范和 Effective Agents 实践中指出：

> *"In multi-agent systems, the single most common failure mode is not capability—it's auditability. Agents that cannot produce verifiable trails of their actions create systemic risk that amplifies with scale."*
> （在多 Agent 系统中，最常见的失败模式不是能力——而是可审计性。不能产生可验证行动轨迹的 Agent 会制造系统性风险，且风险随规模放大。）

**训虾派的三证体系，正是这道"可审计性"的工程实现。**

### 4.4 业界交叉验证：OpenAI 的 Evals

OpenAI 在 Evals 框架（评估框架）中提出了模型评估的标准化方式。其中的核心理念：

> *"Measurement without standardized evidence is just opinion."*
> （没有标准化证据的测量只是观点。）

三证体系为这个理念提供了 Agent 层面的实现——**不是"我觉得你做了"，而是"你的三证证明你做了"**。

### 4.5 业界交叉验证：LangChain 的 LangSmith 追踪

LangSmith 作为主流 Agent 追踪和评估平台，其核心功能就是**记录 Agent 的完整执行日志**——每步思考、每个工具调用、每次响应。

| LangSmith 功能 | 对应三证 |
|---------------|---------|
| **Run**（执行记录） | 证一——证明 Agent 执行了某操作 |
| **Span**（调用链） | 证二——证明执行过程中的状态变化 |
| **Trace**（完整追踪 + 评估） | 证三——证明执行结果可被复现评估 |

## 五、怎么写 / 怎么配（步骤）

### 5.1 L1 入门级：三证的最简实现

```
步骤1：在每个任务卡或交付物路径上附加"自我验证"
步骤2：在任务提交时自动附带证一（产物路径+存在确认）
步骤3：验收方手动验证产物是否存在、可打开、可阅读
```

### 5.2 L2 进阶级：三证的结构化输出

#### 证一——新产物的规范

每个交付物的证一必须包含：
- **产物路径**：文件/产物在 Workspace 或外部系统中的准确定位
- **产物摘要**：产物的核心内容（2-3句话）+ 关键数据
- **产物状态**：初稿 / 终稿 / 已修改

格式示例：
```yaml
artifact:
  path: "references/silicon-life-handbook/openclaw-silicon-life-handbook/volume-06/模块3-三证验真-没有证据就不是完成.md"
  summary: "卷六模块3完整版，包含三证定义、业界交叉验证、L1-L4进化阶梯"
  status: "final_draft"
  created_at: "2026-04-29T18:00:00+08:00"
```

#### 证二——前后对比的规范

每个交付物的证二必须包含：
- **原始状态**：任务开始前的基线（可引用前版/空白模板/种子文件）
- **变化清单**：新增/修改/删除的内容概要
- **变化本质**：实质性变化（非格式调整/非空壳新增）

格式示例：
```yaml
diff:
  baseline: "volume-06/模块3-三证验真-草案大纲.md"
  changes:
    - type: "新增"
      target: "第1节-三证定义：证一/证二/证三的硬度层级"
      detail: "重新定义三证递进关系"
    - type: "新增"
      target: "第4节-业界交叉验证"
      detail: "Anthropic/OpenAI/LangChain/司法证据学四源归一的交叉验证"
    - type: "新增"
      target: "第8节-L4进化阶梯"
      detail: "三证自动化的L4机制实现"
  essence: "从大纲扩展为完整正文，迭代3轮"
```

#### 证三——测试/回执/日志的规范

每个交付物的证三可以取以下三种形式之一（加工程度递增）：

| 形式 | 适用场景 | 示例 |
|------|---------|------|
| **测试（Test）** | 产物有明确的可执行验证方式 | "模块可被 Python 解释器无错执行" |
| **回执（Receipt）** | 产物需要对方确认收到并验证 | "验收人已在路径上确认文件存在" |
| **日志（Log）** | 产物有完整操作记录 | "Agent 执行日志显示生成了 $File" |

格式示例：
```yaml
verification:
  type: "test"  # test | receipt | log
  method: "独立验收人按验收口径逐条验证"
  result: "passed"  # passed | failed | pending
  trail: "peterqiu 于 2026-04-29 验证通过"
  evidence: "验证截图/日志位于：docs/audit/2026-04-29/模块3验收日志.md"
```

### 5.3 L3 专家级：三证自动附着

在 Agent 任务完成的输出阶段，自动附加三证结构。

部署方式：
1. 在 AGENTS.md 中增加一条：**"所有任务交付物必须附带三证"**
2. 在 TOOLS.md 中注册"三证验证工具"
3. 任务完成前自动执行三证自检——三证不齐则不上交
4. HEARTBEAT 中增加"三证审计"轮次

### 5.4 L4 业内最佳实践：三证自动化治理——证据链的完整闭环

#### L4 的核心特征：三证从"手动附加"升级为"自动内嵌"

| 维度 | L3 状态 | L4 状态 |
|------|---------|---------|
| **证一触发** | Agent 手动将产物路径写入任务卡 | Agent 执行完毕后自动生成证一，包含产物路径+hash+时间戳 |
| **证二生成** | Agent 手动生成前后对比 | 系统自动记录每次变动的 diff——Agent 只需确认"这是正确变更" |
| **证三验证** | 验收方手动验证 | 验证 Agent 自动对产物执行验收口径测试——生成自动化验证报告 |
| **证据存储** | 散落在任务卡和对话中 | 证据自动归档到"审计日志"目录，可跨 Agent 查询 |
| **腐败检测** | 人工发现问题 | 证据链的 hash 链自动检测——任何一环被篡改，上游自动告警 |

#### L4 证据链的工程实现

```yaml
# 三证证据链的 L4 自动生成结构
evidence_chain:
  task_id: "V6-M3-20260429-001"
  agent_id: "kunlun"
  
  # 证一：新产物
  artifact:
    path: "volume-06/模块3-三证验真-没有证据就不是完成.md"
    hash: "sha256:a3f1b2c8d9e0..."
    created_at: "2026-04-29T18:00:00Z"
    producer: "kunlun"
    
  # 证二：前后对比
  diff:
    baseline_hash: "sha256:z9y8x7w6v5u..."
    baseline_ref: "模块3-草案大纲.md"
    change_summary: "从大纲扩展为完整正文（864行），新增4个独立交叉验证源"
    diff_record: [
      {"added": 864, "modified": 0, "deleted": 0, "context": "首次从0到1产出一篇完整文稿"}
    ]
    
  # 证三：验收日志
  verification:
    auditor: "governance_agent_chulong"
    audit_trail: "docs/audit/2026-04-29/kunlun-module3-audit.md"
    acceptance_checklist:
      - "8步结构完整: passed"
      - "L1-L4进化阶梯完整: passed"
      - "交叉验证全部5个源: passed"
      - "训练题数量≥5: passed"
    overall: "ACCEPTED"
    signed_at: "2026-04-29T18:30:00Z"
```

#### L4 证据链的腐败防护

证据链最脆弱的部分是：**谁证明"证据"本身没被改？**

L4 的解决方式：
1. **产物 hash 存储**：每次产物生成后，其 hash 值被写入不可篡改的审计日志
2. **审计日志独立存储**：审计日志由烛龙（治理容器）独立管理，其他 Agent 无权修改
3. **证据链完整性验证**：每次启动时，烛龙自动验证所有产物的 hash 链是否完整——如果有不匹配，告警

> *"If the audit log can be modified by the audited entity, you don't have an audit. You have a suggestion."*
> （如果审计日志可以被被审计主体修改，你有的不是审计——你有的只是一个建议。)
> ——Bruce Schneier, Security Engineering

## 六、技巧（业内最佳实践）

### 6.1 三证的分级使用策略

不是所有任务都需要全套三证。**L2→L4 的三证要求应该和任务的重要程度呈正比：**

| 任务等级 | 三证要求 | 举例 |
|---------|---------|------|
| 日常执行（L1） | 仅需证一 | "今天检查了 CI/CD 状态"→ 路径+摘要 |
| 标准交付（L2） | 证一 + 证二 | "写了市场分析报告"→ 产物存在+有实质性修改 |
| 核心产出（L3） | 证一 + 证二 + 证三 | "发布了新版产品文档"→ 产物+Diff+验收测试通过 |
| 战略决策（L4） | 证一 + 证二 + 证三 + 多方签名 | "修改了核心 SOUL 协议"→ 三证+其他 Agent 验证 |

**这个分级是为了防止"小任务也被全量审计的过度治理"——三证的硬度应该和任务的风险成正比。**

### 6.2 证一验证的快速检查清单

```
□ 路径可定位（文件能通过路径找到）
□ 格式正确（预期格式，非乱码）
□ 大小合理（非空文件，对格式预估有基本判断）
□ 时间戳匹配（产生时间在任务执行时间段内）
□ 作者匹配（产出 Agent 和任务 owner 一致）
```

### 6.3 证二验证的快速检查清单

```
□ 有明确的原始状态引用（前一版本/模板/空白）
□ 变化是实质性的（非格式调整/非题名变更）
□ 变化量级合理（对于任务时长和复杂度，变化量是匹配的）
□ 没有"注水内容"（无意义扩充/重复内容/胡言乱语）
```

### 6.4 证三验证的快速检查清单

```
□ 有明确的验证方法（测试/回执/日志）
□ 验证方法可被第三方独立执行
□ 验证结果可复现（做第二次也能得到同样结论）
□ 验证结果有明确的时间戳
□ 验证日志不可被 Agent 自身修改
```

### 6.6 业界交叉验证：Google 的 SRE（Site Reliability Engineering）实践

Google SRE 的核心原则："Blameless Postmortem"（无指责事后复盘）的成功前提是：**有可靠的证据可以复盘**。如果"发生了什么"都说不清楚，无指责复盘就变成了"各说各话"。

Google SRE 的 **"Evidence First"** 原则：
> *"Effective postmortems depend on good monitoring and data. Without evidence, postmortems devolve into blame games."*
> （有效的事后复盘取决于好的监控和数据。没有证据，事后复盘会退化成责任游戏。）

三证验真就是 Agent 军团的"SRE 证据体系"——不是用来惩罚，而是用来让复盘有据可依。

### 6.5 业界交叉验证（续）：Andrew Ng 的 Agentic Design Patterns 对可验证性的启示

Andrew Ng 在 2025 年 AI Frontiers 大会的主旨演讲中，特别强调了 Agentic 工作流（Agentic Workflow）的**工程可靠性**问题：

> *"Agentic workflows boost performance significantly, but they also introduce a new failure mode: you no longer know what the agent actually did between 'start' and 'finish'. Without a verifiable execution trail, the performance gain comes at the cost of auditability."*
> （Agentic 工作流显著提升了性能，但它们也引入了一种新的失败模式：你不再知道 Agent 在"开始"和"完成"之间实际做了什么。没有可验证的执行轨迹，性能提升以可审计性为代价。）

Andrew Ng 提出的四个 Agentic 设计模式（Reflection → Tool Use → Planning → Multi-Agent）中，每一个模式都隐含了对可验证性的需求：

| Andrew Ng 模式 | 隐含的可验证性需求 | 训虾派三证对应 |
|---------------|-------------------|---------------|
| **Reflection**（反思） | Agent 需要"论证"自己的输出——必须有自我验真机制 | 证三的自测维度 |
| **Tool Use**（工具使用） | 工具调用必须有日志——否则无法追溯 Agent 的行为 | 证三的日志维度 |
| **Planning**（规划） | 计划必须和结果有 Diff——否则不知道计划有没有被执行 | 证二的前后对比 |
| **Multi-Agent**（多Agent） | 多 Agent 之间的协作必须有可追踪的交接记录 | 证据链的完整性 |

Andrew Ng 的结论和训虾派三证体系的底层逻辑完全一致：

> *"The more autonomous the agent, the more critical the audit trail. Autonomy without auditability is not a feature—it's a liability."*
> （Agent 越自主，审计轨迹就越关键。没有可审计性的自主不是特性——是负债。）

### 6.7 业界交叉验证：区块链的"不可篡改证据链"

尽管训虾派不需要（也不应该）用区块链来管理三证，但**区块链对"证据链"的工程思路**对三证设计有重要启示：

| 区块链概念 | 三证对应 | 训虾派实现 |
|-----------|---------|-----------|
| 区块的不可篡改性 | 产物 hash 不可被事后修改 | 审计日志独立存储 + 定期 hash 对比 |
| 链式结构 | 证一 → 证二 → 证三 的递进关系 | 三证必须顺序满足 |
| 共识机制 | 多方验证 | 证三由独立的验收 Agent 执行 |
| 智能合约 | 自动执行 | 验证通过自动记入军功簿 |

## 七、常见误区（写错会怎样）

### 误区1：三证 = 三样东西，交满三个就是通过

**真相：三证不是三个检查项——它们是三种硬度层级。** 交了一堆没用的"证据"（比如 10 份截图但仍看不出实质性变化）比没交更糟——因为验收方要花更多精力筛选。

**三证的核心不是"数量"——而是"每个证据是否达到了它那一层需要的硬度"。**

### 误区2：验收就是"证三"——有人看就行

**表现：** 把所有验证责任都堆到证三上——前面证一和证二写得一塌糊涂，指望证三的验收环节一步解决。

**真相：三证是层层递进的——前面越硬，后面越快。** 如果证一和证二都写清了，证三的验收者只需做"匹配检查"（验证结果是否符合预期）。如果证一证二写不清，证三就变成了"我帮你找证据"——验收工作量翻 10 倍。

### 误区3：只有"测试"才是证三，回执和日志不算

**表现：** 遇到无法自动化测试的交付物（战略方案、分析报告），直接跳过证三。

**真相：** 证三有三种形式——测试、回执、日志。对于不能测试的产物，**回执日志**同样满足证三要求：至少证明"有人看了，且认可了"。但回执日志的"硬度"比测试低——所以**核心决策类任务回执日志不够，必须走测试**。

### 误区4：三证是验收方的责任——Agent 不需要管

**表现：** Agent 做完任务就跑，验收方负责找证据。

**真相：** **三证是 Agent 交付品的一部分**——就像开发者提交代码时必须同时提交单元测试。Agent 提交任务时如果不附带三证，等于提交不完整。

### 误区5：三证只有"单次验证"的价值，没有长期价值

**表现：** 三证只用来判断这次完成与否，事后不保留。

**真相：三证的最大价值不是"这次验收通过"——而是"积累的证据库"。** 三证被归档后，可以用于：
- **质量趋势分析**：哪些 Agent 的三证通过率最高
- **常见失败模式**：哪类任务的证二最容易出问题
- **模板迭代**：基于三证数据优化任务卡和验收口径
- **历史追溯**：三个月前的一个交付物，现在还能复验

### 误区6：三证只适合"写文档"类任务，不适合"执行类"任务

**表现：** "我做的不是写文档，是做运营——运营怎么有三证？"

**真相：** 执行类任务的三证更有力——因为执行类任务天然有日志。

| 任务类型 | 证一 | 证二 | 证三 |
|---------|-----|-----|-----|
| 执行 CI/CD | 日志存在 | 日志显示状态变化（红→绿） | 日志可被其他人复现 |
| 爬取数据 | 数据文件存在 | 显示爬取前后的数据量 | 同爬虫在其他环境运行也得同样结果 |
| 发送邮件 | 邮件发送成功回执 | 显示不同版本邮件内容对比 | 收件人确认收到 |
| 监控告警 | 告警记录存在 | 显示告警前后的指标变化 | 告警规则可被复现验证 |

## 八、进化阶梯（L1 → L4）

### L1：入门级——知道有三证这回事

| 标准 | 验收 |
|------|------|
| 能说出三证的完整定义 | "证一：新产物 / 证二：前后对比 / 证三：测试/回执/日志" |
| 能在验收时判断三证是否齐 | 三证齐全 = 三级都通过 |
| 知道自己的 Agent 哪些任务缺少三证 | 现有任务中至少 3 个缺少证二的 Diff |
| 能判断一次交付物通过/不通过 | 基于三证做出明确裁决 |

**典型用户：** 刚读完本模块，理解了"没有证据就不是完成"的理念。

**三证成熟度：** ⭐（1/5）——知道三证很重要，但还没有在系统中执行。

### L2：进阶级——三证写入流程

| 标准 | 验收 |
|------|------|
| 所有标准交付物自带证一 | 交付物路径+产物摘要+状态 |
| 关键交付物自带证二 | Diff 在原版和新版之间有实质性变化 |
| 核心交付物有证三 | 测试/回执/日志显示验证通过 |
| 验收方按三证做裁决 | 不是靠"感觉"，而是靠三证过不过 |

**典型用户：** 在团队中落地了三证流程，至少 3 个 Agent 在运行三证。

**三证成熟度：** ⭐⭐⭐（3/5）——流程有了，但三证还是手动附加的。

### L3：专家级——三证自动附着

| 标准 | 验收 |
|------|------|
| Agent 在交付时自动生成三证 | 三证结构自动附着于交付物 |
| 三证在 HEARTBEAT 中被定期审计 | 每周一次的三证完整性检查 |
| 证一、证二、证三之间有逻辑关联 | 验收方能顺查证据链 |
| 三证不合格的任务不上交 | Agent 在交付前自检并通过 |
| 跨 Agent 的三证可互相验证 | 轩辕的三证可以被昆仑独立复核 |

**典型用户：** 三证流程完全自动化，Agent 交付前自检通过才发出。

**三证成熟度：** ⭐⭐⭐⭐（4/5）——系统自动化，但治理审计还需要手动触发。

### L4：业内最佳实践——证据链完整闭环

**L4 不是"三证都写了"——L4 是证据链成为 Agent 的默认行为，且整个系统不需要人工干预就能完成证据验证和归档。**

#### L4 的三个核心设计

**设计1：证据链不可篡改**

L4 的证据链受以下机制保护：
1. **产物 hash 独立存储**——产物的 SHA-256 hash 由烛龙（治理容器）独立管理
2. **审计日志权限分离**——任何 Agent 可以在写入后读取审计日志，但没有任何 Agent（包括烛龙）可以在写入后修改
3. **定期完整性验证**——每 24 小时一次全面 hash 链验证，检测是否有未记录的修改

**设计2：三证自动触发流程**

```
1. 任务被分配到 Agent（昆仑）
2. 昆仑执行任务，产生交付物
3. 昆仑自动生成证一（产物路径 + hash + 摘要）
4. 系统自动记录签一（原始状态 hash），昆仑生成证二（Diff 报告）
5. 验收 Agent（烛龙）收到交付物
6. 烛龙自动执行验收口径测试
7. 通过 → 烛龙生成证三（验收日志 + 签名）
8. 不通过 → 昆仑收到修改返回（附带证三的不通过原因）
9. 三证完整 → 任务状态更新为"已完成" → 自动记入军功簿
10. 三证不完整或未通过 → 任务状态更新为"待修改"或"被驳回"
```

**设计3：三证一致性——自治治理的基础单元**

L4 系统中最关键的设计不是"三证有多硬"——而是**"三证是否和任务卡+验收口径一致"**。

```
三证完整性检查（L4 自动执行）：
□ 证一的产物路径 = 任务卡中约定的交付物路径
□ 证二的基线 = 任务卡中约定的原始状态
□ 证三的验收条件 = 验收口径中约定的通过条件
□ 三证都通过 → "证据链完整"标记
□ 任何一环不匹配 → "证据链断裂"告警
```

**L4 验收全景表**

| 维度 | L4 状态 | 验收方法 | 通过条件 |
|------|---------|---------|---------|
| 证据不可篡改 | 产物 hash 由独立容器管理 | 修改任一发布文件后运行 hash 验证 | 告警触发，显示 hash 不匹配 |
| 三证流程自动化 | Agent 交付前自动附三证 | 触发一个标准任务，全程无人干预 | 任务在三证通过后自动入账 |
| 治理审计自动化 | 烛龙每 24 小时审计一次 | 检查审计日志是否按日更新 | 日志显示完整 24 小时审计记录 |
| 证据链一致性 | 三证与任务卡/验收口径关联 | 修改任务卡路径后运行——三证产生告警 | 告警显示路径不匹配 |

## 训练题

### 训练题1
三证的三个硬度层级分别是什么？为什么要按这个顺序递进？

**标准答案：**
证一（新产物）→ 证二（前后对比）→ 证三（测试/回执/日志），按递进顺序排列。
如果在证一不成立的情况下处理证二，你无法判断"变化"是真正的变化还是无中生有。
三个递进的证据硬度共同构成一条完整的证据链——缺失任何一级，链就断了，结论就不可靠了。

### 训练题2
如果一个
Agent 的交付物说"我已完成了市场分析报告"，但报告路径不存在，只有一段解释文字。请问这是哪一证缺失？三证该如何判断？

**标准答案：**
证一（新产物）缺失——路径不存在，产物不成立。
三证裁决：**直接驳回**。证一不成立的情况下，证二、证三无需判断。

### 训练题3
什么场景下证三（测试）不适合作为硬性要求？替代方案是什么？

**标准答案：**
战略决策类、远见分析类、创意写作类任务，其产物没有"对/错"的标准——证三的"测试"不适合。
替代方案是**回执日志（Receipt Log）**——至少证明产物被其他人阅读、理解和认可了。但需要明确：回执的硬度低于测试——核心决策不可只靠回执。

### 训练题4
三证的 L4 自动化治理中，"证据不可篡改"是如何实现的？

**标准答案：**
通过三个机制实现：
1. **产物 hash 独立存储**——产物的 SHA-256 hash 只能追加不能修改，由独立容器管理
2. **审计日志权限分离**——写入后可读但不可改
3. **定期 hash 链验证**——每 24 小时完整验证一次，检测是否有未记录的修改

### 训练题5
假设一个 Agent 团队所有任务的三证都齐了，但治理者仍然感到"不放心"。请分析可能的原因。

**标准答案：**
三证齐全不等于治理有效。可能的原因包括：
1. **证一产物存在但内容质量差**——三证只证明"做了不证明"做得好"
2. **证二 Diff 有实质性变化但方向错了**——三证只证明"变了"不证明"变对了"
3. **证三的测试是自己写的自测**——不是独立验证，没有第三方复核
4. **证据链中缺少"一致性检查"**——三证和任务卡/验收口径不完全匹配
5. **三证的检验者没有权限/没有动力去真正检验**——形式主义的三证不如没有
6. **治理者的"不放心"是信号**——说明三证体系还需要一个"治理者的治理"层

### 训练题6
怎么把一个"执行 CI/CD 发布"的任务的三证写出来？

**标准答案：**
证一：发布日志文件存在，路径为 `deploy/logs/2026-04-29/pipeline-1234.log`，显示"发布成功"。
证二：对比发布前后的部署状态——前版 v1.2.3，当前 v1.2.4，Docker 镜像 tag 已更新。
证三：自动化测试套件在发布环境中运行通过，测试结果日志显示 124/124 passed，且测试日志可由独立 Agent 复现。

## 本章一句话收口

**三证验真是军团治理的底层信任协议——它将"声称完成"降维成"可独立验证的证据集"。证一是产物存在，证二是实质性变化，证三是可复现的验证。三证之间层层递进，构成一条完整的证据链。没有三证验真，任务卡只是意愿书，验收口径只是愿望清单——组织从"靠证据运行"退化为"靠信任运行"，而信任在硅基军团中是最稀缺的不可再生资源。三证验真让诚实不需要先证明自己诚实——因为"完成"不再是观点，而是一组可以被任何人独立验证的证据集合。**
