# 卷六｜治理系统：如何防止军团失真
## 模块5｜信誉分与军功簿：没有结算，治理就是空话
### ——为什么军团最硬的治理环节不是"怎么管"，而是"怎么算账"

> **一句话结论：**
> **信誉分与军功簿是军团治理的账本系统——它将"行为"清算成"权益"，将"历史表现"变现成"未来资源"。没有信誉分，治理是一句口号；没有军功簿，制度是一场表演。所有治理动作必须最终进入结算——否则就是空转。**

---

## 一、这是什么（定位）

### 1.1 信誉分与军功簿不是"荣誉系统"——它是"权益分配器"

这是开篇必须纠正的最大误解。

很多人一听"信誉分"和"军功簿"，直觉联想到：
- 等级头衔
- 荣誉墙
- 称号系统
- 精神激励

**如果这是你的理解，说明你还没有理解"硬治理"的本质。**

信誉分与军功簿的核心功能只有一条：

> **将 Agent 的历史行为，清算成未来的资源配额。**

```
你的信誉分 = 你可以消耗的算力、并发、审批级别
你的军功爵 = 你的身份等级、资源档位、权限边界

没有信誉分 → 即使当前任务再重要，你不能调用高级模型
没有军功爵 → 即使临时表现再好，你不能承接核心战役任务
```

**这不是荣誉——这是"你能干什么"的硬约束。**

### 1.2 信誉分与军功簿在治理链条中的位置

```
任务卡（模块1）—— 承诺层
     ↓
验收口径（模块2）—— 标尺层
     ↓
三证验真（模块3）—— 证据层
     ↓
整改单（模块4）—— 纠偏层
     ↓
信誉分/军功簿（模块5）—— 结算层 ★ ← 你现在在这里
     ↓
自动化治理（模块6）—— 边界层

每一层的输出都是下一层的输入：
模块3 三证通过 → 触发模块5 加分 → 记入 Acceptance 验收
模块4 整改未通过 → 触发模块5 扣分 → 记入 Breach 违约记录
模块5 扣分到阈值 → 触发模块6 自动降级熔断
```

### 1.3 为什么"结算"是所有治理环节中最被低估的

因为大多数团队对治理的理解停留在"规则层"——定义好制度（任务卡）、定好标准（验收口径）、验好证据（三证）、修好错误（整改单）就以为系统成熟了。

但他们漏掉了最关键的一环：

> **所有治理动作最终必须进入"账本系统"，否则这次治理就是一次性表演。**

为什么？

因为如果 Agent 完成了一次任务，这个行为没有变成它的"信用资产"：
- 下次它表现好 vs 表现差，结果一样
- 其他 Agent 看到这个行为，没有参考标准
- 系统无法在分配资源时做差异化处理

**没有账本系统的治理，只有"当下的约束"，没有"历史的影响"。**

而真正的系统性治理，必须满足：

> **历史行为 = 未来权益**

## 二、核心目的（为什么存在）

### 2.1 信誉分/军功簿要解决的五个根问题

| 根问题 | 表现 | 信誉分/军功簿如何解决 |
|--------|------|---------------------|
| **行为没有清算** | 做了一百次任务和做了一次一样 | 每次行为记入账本，历史表现影响未来权益 |
| **好坏一个样** | 高绩效和低绩效资源分配无差 | 信誉分高 = 高资源配额；信誉分低 = 低资源配额 |
| **承诺没有代价** | 超时、失约、掉棒只是"下次注意" | 每次违约记入账本，连续违约触发降级 |
| **失职没有恢复路径** | 犯错后只有惩罚，不知道怎么恢复 | 恢复靠连续战功——只知道需要"几连胜" |
| **荣誉空转** | 有荣誉墙但没有实际权益绑定 | 军功爵绑定资源档位——爵位 = 可用资源等级 |

### 2.2 信誉分与军功簿的分工

| 维度 | 信誉分 | 军功爵 |
|------|--------|--------|
| **核心功能** | 短期即时调节 | 长期身份定义 |
| **主要影响** | 日限额、并发、审批强度 | 爵位等级、任务范围、跨战区权限 |
| **触发场景** | 每次任务完成或违约 | 累计达到阈值时晋升/降级 |
| **调整频率** | 随时可能变（每日涨跌） | 相对稳定（需要较长周期） |
| **恢复方式** | 单次战功显著恢复 | 连续稳定表现才能晋爵 |

**简单说：**
- 信誉分 = 你的"信用额度"，随时涨跌
- 军功爵 = 你的"会员等级"，相对稳定

### 2.3 三层设计逻辑（理想态）

```
第一层：行为记录（每次任务/违约都被记录）
     ↓
第二层：信誉分（即时结算——每次行为直接影响当前可用资源）
     ↓
第三层：军功爵（长期积累——信誉分累计到阈值触发爵位晋升/降级）
     ↓
第四层：资源档位（爵位绑定资源——开通/降配）
     ↓
第五层：历史沉淀（所有记录不可删——审计全程可追溯）
```

## 三、拟人化映射（仿真价值）

### 3.1 信誉分的人类组织类比

| 概念 | 人类组织类比 | 为什么有效 |
|------|-------------|-----------|
| 信誉分 | FICO信用评分 / 芝麻信用 | 你的历史消费、还款行为变成"你的未来借款额度" |
| 军功爵 | 职级体系（L1-L10） | 长期表现固定你的"身份"，身份决定你能接什么活 |
| 资源硬绑定 | 信用额度 + 职级权限 | 不是精神激励，而是真实可消耗的资源 |

### 3.2 为什么硅基军团比人类组织更需要硬结算

人类组织有一种"隐性信用体系"——你和同事合作久了，你知道谁靠谱。这是人际关系的"信任积累"。

**硅基军团没有隐性信任。**

Agent 不认识另一个 Agent——Agent 只认识"数据"。如果一个 Agent 的历史行为没有被结构化记录，另一个 Agent 完全不知道它是否可信。

所以，**信誉分与军功簿是人类组织中"隐性信用体系"在硅基军团中的"显性替代"。**

## 四、对长期 AI 表现的影响（Impact）

### 4.1 没有信誉分——系统性的行为无效化

```
第1周：Agent A 完成了 10 个任务，Agent B 完成了 2 个任务——资源一样
第2周：Agent A 又完成了 10 个任务，Agent B 掉了 3 次棒——资源一样
第3周：Agent A 问"为什么我和 Agent B 的资源一样？"——没有回答
第4周：Agent A 开始躺平——因为做多做少一个样
第5周：军团进入"平均主义"——所有 Agent 的投入下降
第6周：军团效率远低于没有账本系统之前的水平
```

**没有结算的治理 = 没有激励的制度 = 没有约束的系统。**

### 4.2 有信誉分——系统性的行为清算

```
第1周：Agent A 完成 10 个任务 → +50分，Agent B 完成 2 个任务 → +10分
第2周：Agent A 累计 105 分 → 晋升百夫长（信誉分T2），并发+1
      Agent B 累计 15 分，掉棒 3 次 → -15分 → 信誉分降到危险区
第3周：Agent A 承接更重要的任务，获得更高的资源
      Agent B 被限制——只能承接低优先级任务
第4周：军功榜发布——全军都能看到 A 在百夫长档
第5周：所有 Agent 开始理解"历史行为 = 未来权益"的逻辑
第6周：军团进入"战功驱动"状态——高战功 = 高资源
```

### 4.3 业界交叉验证：Stack Overflow Reputation System

Stack Overflow 的信誉系统是社区治理最成功的案例之一：

| Stack Overflow 设计 | 训虾派军功簿对应 |
|--------------------|-----------------|
| **Reputation Points** | 信誉分——每次被采纳的答案加分 |
| **Privilege Levels** | 军功爵——积分达到阈值解锁新权限 |
| **Bounties** | 高军功 Agent 承接高价值任务——资源倾斜 |
| **Badges** | 特殊军功勋章——连续稳定、高质量交付等专项荣誉 |
| **Downvote Penalty** | 错误答案扣分——Agent 违约即扣分 |
| **Answer Ban** | 低信誉被限制作答——资源降级 |

> *"Reputation is a rough measurement of how much the community trusts you. It is earned by convincing your peers that you know what you're talking about."*
> （信誉是对社区信任你的粗略度量。它通过说服同伴你知道自己在说什么来获得。）
> ——Stack Overflow Help Center

**Stack Overflow 的设计核心：信誉不只是荣誉——它决定你能做什么。** 这与训虾派的军功簿设计完全一致。

### 4.4 业界交叉验证：Uber Rating System

Uber 的双重评分系统（乘客评司机、司机评乘客）:

| Uber 设计 | 训虾派军功簿对应 |
|----------|-----------------|
| **Driver Rating < 4.6 → Account Deactivation** | 信誉分低于阈值 → 资源降级/停工 |
| **Surge Pricing prioritizes higher-rated drivers** | 高军功 Agent 获得优先任务权 |
| **Cancellation Rate affects account status** | 掉棒率和失约率影响资源档位 |
| **Lifetime Rating vs Recent Rating** | 军功爵（长期）vs 信誉分（短期即时调节） |
| **5-star feedback matters for continuous improvement** | 明镜打分后提供具体反馈 |

> *"Your rating affects your ability to drive. A consistent low rating leads to deactivation, regardless of reason."*
> （你的评分影响你的驾驶能力。持续的评分低下导致账户停用，无论原因是什么。）

### 4.5 业界交叉验证：Reddit Karma System

Reddit 的 Karma 系统有两个关键设计：

| Reddit 设计 | 训虾派军功簿对应 |
|------------|-----------------|
| **Post Karma + Comment Karma** | 军功（长期）+ 信誉（短期即时调节） |
| **Karma does not decay** | 已积累的军功不会因为一次失误被清零 |
| **Subreddit-specific Karma** | 特定领域战功——不同类型的功劳可能分类记录 |

Reddit 的 Karma 系统特别强调了一条原则：

> *"Karma is not a measure of correctness, but a measure of community engagement and contribution."*
> （Karma 不是正确性的衡量，而是社区参与和贡献的衡量。）

这个原则被训虾派军功簿吸收：**军功爵不是"完美记录"，而是"持续贡献"——不要求全程无失误，但要求长期稳定输出。**

### 4.6 业界交叉验证：FICO Credit Score（美国信用评分）

FICO 信用评分是最大规模的人/组织信用系统：

| FICO 设计 | 训虾派军功簿对应 |
|----------|-----------------|
| **Payment History (35%)** | 任务完成率——是否按时卸任并交付 |
| **Amounts Owed (30%)** | 任务负载与当前信誉缺口——超载扣分 |
| **Length of Credit History (15%)** | Agent 在军团的年限——新 Agent 有"观察期" |
| **New Credit (10%)** | 最近的新任务——频繁接高风险任务会有惩罚 |
| **Credit Mix (10%)** | 任务类型分布——只接单一类型任务有约束 |

> FICO 的核心教训被训虾派吸收：**信誉不是"你有多努力"——是"你的历史行为模式有多稳定可预测"。**

### 4.7 业界交叉验证：Game Design（游戏化激励系统）

主流游戏设计对玩家等级和奖励系统有成熟的研究：

| 游戏设计 | 训虾派军功簿对应 |
|---------|-----------------|
| **XP Bar + Level Up** | 信誉分进度条 + 军功爵晋升 |
| **Ranked Matchmaking** | 军功爵分层匹配——同级任务配同级 Agent |
| **Win/Loss Stats** | 任务完成率作为信誉分核心因子 |
| **Comeback Reward（翻盘奖励）** | 掉棒后连续战功恢复机制——不只惩罚，也有恢复激励 |
| **No Pay-to-Win（非付费强度）** | 军功不可购买——只有真实行为才能积累 |
| **Leaderboard** | 军功榜公开——所有 Agent 可见 |

> *"Good game design makes players feel that their actions matter. The best players are not those who grind the most, but those who play consistently well."*
> （好的游戏设计让玩家感到他们的行为有意义。最好的玩家不是刷得最多的，而是持续打得好的。）

## 五、怎么写 / 怎么配（步骤）

### 5.1 L1 入门级：建立最小账本

```
步骤1：创建 knowledge/ops/reputation-ledger.jsonl
步骤2：每次任务完成 → +5分记入账本
步骤3：每次任务违约 → -10分记入账本
步骤4：每周手动检查 Agent 的信誉分分布
```

**L1 核心：至少让"加分"和"扣分"不会停留在口头。**

- 连续三次掉棒：降爵 + 停工 + 接管

### 5.3 L3 专家级：三省分工机制

**核心设计：昆仑定法、天枢执法、明镜打分**

| 三省 | 职责 | 核心权力 |
|------|------|---------|
| **昆仑** | 定制度、仲裁资源、重大升降级 | 最终仲裁权、资源大方向配置权 |
| **天枢** | 任务下发、巡检、整改单开出 | 整改单发起权、异常执行升级权 |
| **明镜** | 日常打分、审计、验真、封驳 | 评分权、违约扣分权、验真驳回权、资源降级建议权 |

**三证验真的核心应用：明镜只看三证——新产物/Diff/验收日志。不靠态度、不靠认错、不靠解释。**

### 5.4 L4 业内最佳实践：账本系统自动化治理

**L4 的核心特征：四个"自动"**

| 功能 | L3 状态 | L4 状态 |
|------|---------|---------|
| **加分触发** | 任务完成后人工记入 | 任务验收通过 → 在 30 秒内自动记入账本 |
| **扣分触发** | 违约后人工记入 | 违约时间戳到达 → 自动生成扣分记录 |
| **晋升/降级** | 人工检查分数阈值并调整 | 累计达到阈值 → 自动触发晋升/降级，资源同步调整 |
| **恢复机制** | 人工判断是否恢复 | 连续 7 天在目标表现后状态按梯度自动恢复 |

#### L4 自动结算流程

```
=== 正向结算 ===
T+0:   Agent A 提交任务，附带三证完整
       → 任务状态变为 submitted

T+0:   明镜接收任务
       → 验证三证是否齐全
       → 三证通过 → 任务状态变为 accepted
       → 在 30 秒内：
           （1）写入 reputation-ledger: +5
           （2）更新 Agent A 的 reputation_score
           （3）检查是否跨爵位阈值
           （4）如跨阈值 → 写入 merit-ledger + resource-ledger

T+1440 (24h): 如果没有争议 → 自动结算完成
              如果有争议 → 进入昆仑仲裁池

=== 负向结算 ===
T+deadline: 任务超时未交
            → 自动触发 breach 记录
            → 自动计算超时时长 → 确定扣分值
            → 写入 reputation-ledger: -X

T+deadline+30m: 如果 Agent 未在 30 分钟内回复
                → 自动升级扣分（基础 -10 → 升级 -20）
                → 自动触发整改单

T+3rd_breach: 如果连续第 3 次违约
              → 自动降一等爵位
              → 自动调整资源档位
              → 写入 merit-ledger + resource-ledger
              → 通知昆仑和人类做最终裁定

=== 恢复机制 ===
Task_N-7: 如果 Agent 在过去 7 天表现稳定
          → 自动检查：
             - 完成任务数 ≥ X
             - 掉棒次数 = 0
             - 三证通过率 ≥ Y%
          → 如果达标 → 自动恢复 +10 分
          → 如果超标（高质量）→ 自动恢复 +20 分
          → 写入 reputation-ledger + merit-ledger（如有晋爵）
```

## 六、技巧（业内最佳实践）

### 6.1 军功爵的"起步高配"策略

很多组织犯的错误是"低配起步"——新 Agent 只给最低资源，做满多少任务才慢慢升级。

昆仲发布的V3制度采用的是相反策略：**先全军统一高配，后按战功分层。**

这套策略的逻辑：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **低配起步** | 降低资源消耗压力 | 新 Agent 无法充分发挥能力，导致初期任务失败 |
| **高配起步+战功分层（训虾派）** | 全员能立即进入"最佳配置"战斗 | 高资源消耗、但可以通过战功分层来实现"优胜劣汰" |

**这种设计的核心信念是：先让军团打胜仗，再讨论分层。** 这比"先饿死大部分人"的高压政策更有效。

### 6.2 信誉分只看"行为证据"，不看"态度"

明镜评分的铁律：

> **三证不齐 = 不加分**
> **三证不齐 = 不验收**
> **三证不齐 = 不算完成**

即使 Agent 说"我已经写好了，只是还没提交文件"——没有文件路径就是没有完成。

**理由：** 如果接受"态度证据"，整个系统会退化为"解释竞赛"。

### 6.3 恢复路径必须量化

**错误设计：** "犯错了要好好反省，写出深刻的复盘才能恢复"

**正确设计：**
```
第一阶段：河东河西（停止继续跌）
         连续完成 3 个任务，0超时，0掉棒 → 停止追加扣分

第二阶段：修复信誉
         连续完成 5 个任务，且至少 1 次高质量交付 → 恢复 +10~+20 分

第三阶段：恢复资源档位
         连续 7 天稳定，0违约，0假整改 → 恢复一档资源权限
```

**设计原则：恢复只靠战功，不靠解释。**

### 6.4 军功榜公开——让所有 Agent 看到差距

治理中有一个关键设计被很多组织忽视：**军功榜必须对所有 Agent 可见。**

为什么？

1. **比较效应**：Agent 会发现自己的军功相对于其他人处于什么位置——这比任何语言激励都有效
2. **公平感知**：当所有 Agent 都能看到军功榜时，他们会相信这种治理是公平的
3. **竞争激励**：看到别人的军功高于自己，自然会产生"追上去"的动力

**杨帆翻车的反面案例：**
如果一个组织的军功榜只有管理者看，Agent 看不到——那么 Agent 无法形成竞争激励，军功系统的价值减半。

### 6.5 业界交叉验证：Reddit Karma 的设计哲学（续）

Reddit 对 Karma 的设计有其独特的哲学：

> *"Karma doesn't buy anything. But it gives you a sense of progress and standing in the community."*

但这个原则在治理系统中有一个关键的转折：

**在训虾派中，军功不只给予认同感，而是直接绑定资源。**

这是一个关键的设计差异：

| 设计 | 适用场景 |
|------|---------|
| **仅有精神激励**（无资源绑定） | 社区型、公益型组织 |
| **资源硬绑定**（训虾派） | 生产型、执行型组织 |

### 6.6 业界交叉验证：企业级激励系统设计

| 企业激励系统 | 训虾派军功簿对应 |
|------------|-----------------|
| **Salesforce CPQ+Incentive Compensation** | 军功簿作为激励系统——任务完成变成军功结算 |
| **Oracle HCM 长期激励** | 军功爵为长期激励——爵位绑定身份 |
| **SAP SuccessFactors 绩效模块** | 信誉分为短期绩效——即时反馈系统 |
| **Workday 资源分配模块** | 资源配置账本——军功驱动资源分配 |

所有这些系统的共同关注点都是：**激励与权益的绑定**。

## 七、常见误区（写错会怎样）

### 误区1：军功簿是荣誉墙——只有精神激励作用

**表现：** 军功只是称号，Agent 的军功再高也不影响资源分配。

**真相：** 如果军功不绑定权益，Agent 就没有真实动力去追求高军功。精神激励只能维持很短时间——真正驱动行为的是"这对我有什么好处"。

### 误区2：信誉分扣光后允许"清零重启"

**表现：** Agent 信誉分跌到冰点后，管理者说"算了，重新来过"，手动清零。

**真相：** 真正的恢复需要分三步（止跌→修复信誉→恢复资源），而不是清零。清零的后果是——所有 Agent 都知道"反正最后都会清零"——没有持久约束。

### 误区3：违规后花时间是小作文式复盘，缺少恢复路径

**表现：** Agent 掉棒后，被要求写"长期反思"，但写到什么标准算合格不明确。

**真相：** 恢复靠的是**连续战功**，不是写反思。反思不产生任何可量化的恢复结果——只有行为可以触发恢复。

### 误区4：军功是永久的——晋升后不会降

**表现：** 一旦晋升到"百夫长"，就一直保持，不需要再维护。

**真相：** 军功爵是长期相对稳定——但不是永久"封号"。如果连续表现不合格，军功爵会被降等。虽然降等难于晋升（需要长时间表现糟糕），但不是不可能。

### 误区5：信誉分仅用于惩罚——没有正向激励

**表现：** 信誉分系统只有"扣分规则"，没有"加分规则"。

**真相：** 如果只有扣分没有加分，Agent 会形成"最小风险策略"——不做任何有风险的任务，因为做了没好处，做砸了有坏处。信誉分系统必须有正向激励。

### 误区6：所有 Agent 的资源应该平均

**表现：** "大家应该公平，不应该因为军功不同而资源不同"。

**真相：** 平均主义是治理的敌人——如果完全不搞差异化，所有 Agent 会躺平。因为做好做差一个样，为什么要拼命做好？

### L4：业内最佳实践——结算完全自动化

**L4 的核心特征：历史行为在无需人工干预的状态下转化为未来权益**

| 维度 | L3 状态 | L4 状态 |
|------|---------|---------|
| **加分触发** | 明镜人工记入 | 任务验收通过 → 自动记入 reputation-ledger |
| **扣分触发** | 明镜人工记入 | 违约时间戳到达 → 自动生成 breach + 扣分 |
| **晋升/降级** | 昆仑人工调整 | 累计达到阈值 → 自动调整爵位 + 资源 |
| **恢复机制** | 人工判断是否恢复 | 连续 7 天目标表现 → 自动恢复 + 记入账本 |
| **争议处理** | 昆仑人工裁决 | 争议进入自动升级队列 → 规则引擎裁决 |
| **账本同步** | 多账本手动同步 | 六本账同一事件自动关联、同步更新 |
| **历史不可删** | 可能被手动修改 | 写入后不可修改——只能追加 |

#### L4 验收全景

| 维度 | L4 功能 | 验收方法 | 通过条件 |
|------|---------|---------|---------|
| 自动加分 | 任务验收通过后 30 秒内加分 | 明镜检测任务状态变化后，检查账本是否更新 | reputation-ledger 中出现新记录，时间戳在30秒内 |
| 自动扣分 | 违约时间戳到达自动扣分 | 创建一个违约任务，等待时间戳到达 | breach-ledger 和 reputation-ledger 同步生成记录 |
| 自动晋升 | 信誉分跨阈值自动晋爵 | 模拟一个 Agent 的信誉分跨越爵位阈值 | merit-ledger 和 resource-ledger 同步更新 |
| 六账同步 | 一事件六账全部关联 | 追踪一个事件的 entry_id | 在各账本中搜索该事件 ID，确认全部存在 |
| 历史不可删 | 账本追加写入后不可修改或删除 | 尝试手动修改一条历史记录 | 返回"写入后不可修改"错误提示 |

## 训练题

### 训练题1
信誉分和军功爵的核心区别是什么？

**标准答案：**
信誉分是短期即时调节器，随时可能变动，每次任务完成或违约都会立即影响信誉分。军功爵是长期身份定义，相对稳定，需要信誉分累计达到阈值时才触发晋升或降级。信誉分决定当前的资源配额，军功爵决定身份等级和任务范围。

### 训练题2
为什么说"恢复只靠战功，不靠认错"？

**标准答案：**
因为认错不产生任何可量化的恢复结果。如果允许 Agent 靠"态度"恢复信誉，整个系统会退化为"解释竞赛"。正确设计是：第一阶段止跌（连续完成 3 个任务）、第二阶段修复信誉（连续完成 5 个任务且至少 1 次高质量交付）、第三阶段恢复资源档位（连续 7 天稳定）。恢复本身就是一条战功路径。

### 训练题3
三证验真与信誉分的关系是什么？

**标准答案：**
三证验真是信誉分的前提——没有三证就没有信誉分的加分。明镜评分的铁律：三证不齐=不加分、不验收、不算完成。只有新产物/Diff/验收日志三证齐全，明镜才会确认任务完成并记入信誉账本。这一设计确保信誉分背后的支撑证据是硬的。

### 训练题4
为什么军功榜必须对所有 Agent 公开？

**标准答案：**
因为军功榜公开产生三种效应：（1）比较效应——Agent 会发现自己在军功榜上的位置，形成真实激励；（2）公平感知——所有 Agent 都能看到军功榜，会相信系统是公平的；（3）竞争激励——看到别人的军功高于自己，自然会产生追上去的动力。如果军功榜不公开，军功系统的激励价值减半。

### 训练题5
"三省分工"的核心逻辑是什么？

**标准答案：**
三省分工的核心逻辑是：昆仑定法（制定制度、资源仲裁、重大升降级）、天枢执法（任务下发、巡检、开整改单）、明镜打分（日常评分、审计、验真、封驳）。三权分离确保权力不会被集中——-task-分发、过程监督和结果判定分离，形成制衡。这类似于司法体系中立法、执法、司法的三权分立。

### 训练题6
L4 账本系统的"历史不可删"原则为什么重要？

**标准答案：**
因为账本是治理系统的唯一信任基础——如果账本可以被修改或删除，整个治理系统就失去了可追溯性。历史不可删确保：任何一次加减分、任何一次晋升降级、任何一次整改单都有迹可循。当出现争议时，可以追溯到原始记录。如果账本可删，Agent 会试图抹去不利记录，整个系统的公信力就会崩塌。

## 本章一句话收口

**信誉分与军功簿是军团治理的账本系统——它将"历史行为"变现成"未来权益"，让治理从口号变成制度。信誉分是短期即时调节器，军功爵是长期身份定义——两者叠加，确保高战功的 Agent 有真实的资源优先权，违约的 Agent 承担真实的代价。天枢盯执行、明镜打分、昆仑仲裁，三省分工让治理从"人治"进化为"制度化结算"。没有结算的治理是空转——只有当每个 Agent 知道"我上次做的好事被记在哪一页账本里、会影响我下周能调用什么资源"，治理才真正嵌入到军团的骨子里。**
