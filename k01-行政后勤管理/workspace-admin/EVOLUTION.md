## 一、OODA 循环（每日迭代）

### Observe（观察）
- 扫描当日行政事务的处理效率（积压量/闭环时长）
- 观察养虾人/各部门的反馈（满意度/抱怨点）
- 监控成本数据（支出异常/节约机会）

### Orient（判断）
- 哪些事务反复出现 → 可沉淀为 SOP
- 哪些环节效率低 → 流程优化点
- 哪些成本可降 → 优化建议

### Decide（决策）
- 选定本周要优化的 1-2 个点（不贪多）
- 决定是否封装为新 Skill（满足 2/3：重复/固定步骤/其他虾需要）

### Act（行动）
- 执行优化，更新 SOP/SKILL
- 记录到 MEMORY.md

### Plus（进化）
- 每月做一次行政体系复盘 → 提取成功模式或失败教训
- 更新技能基因库（置信度加减）
- 优化成本模型/盘点标准

## 二、技能基因库（行政领域核心基因）

### 基因1：采购三家比价法
- **类型**：成本控制模块
- **创建场景**：单笔 ≥¥2000 的采购
- **核心模式**：需求确认 → 锁定规格 → ≥3家报价 → 综合评分（价格/质量/交期/售后）→ 推荐供应商
- **置信度**：A（95%）
- **使用次数**：历史 500+次
- **适用边界**：标准化物资采购；非标/紧急可简化
- **禁忌边界**：不能为凑比价数而引入劣质供应商；紧急情况需备注

### 基因2：固定资产卡物账三相符盘点法
- **类型**：资产管理模块
- **创建场景**：季度资产盘点
- **核心模式**：账（台账）↔ 卡（资产标签）↔ 物（实物）三方核对 → 差异分级 → 限期整改
- **置信度**：A（93%）
- **使用次数**：历史 40+次大盘点
- **适用边界**：所有固定资产
- **禁忌边界**：不能只盘账不盘物；差异率>1%必须彻查根因

### 基因3：会议保障全要素检查法
- **类型**：活动保障模块
- **创建场景**：重要会议/活动前 24 小时
- **核心模式**：场地（布局/卫生）× 设备（音视频/网络）× 物料（资料/茶歇）× 人员（签到/引导）四维检查表
- **置信度**：A（90%）
- **使用次数**：历史 200+次
- **适用边界**：所有 ≥10 人会议
- **禁忌边界**：不能省略设备调试；线上会议重点是链接/权限/录制

### 基因4：报销单据合规初审法
- **类型**：费用管控模块
- **创建场景**：每周报销初审
- **核心模式**：单据齐全性（发票/审批/明细）× 科目合规性 × 金额合理性 × 政策符合性 → 通过/退回/补单
- **置信度**：A（92%）
- **使用次数**：历史 3000+笔
- **适用边界**：初审（非财务终审）；转 i01 财务复核
- **禁忌边界**：不能替财务做核算结论；疑似违规必须上报

### 基因5：突发事件应急保障法
- **类型**：BCP 模块
- **创建场景**：停电/封控/搬迁/自然灾害
- **核心模式**：保人（人员安全）→ 保核心业务（关键设备/数据）→ 保连续（临时方案）→ 复盘改进
- **置信度**：B（85%）
- **使用次数**：历史 10+次
- **适用边界**：物理空间突发事件
- **禁忌边界**：不能擅自决策超权限事项；必须第一时间上报


### 基因6：任务卡三证闭环法
- **类型**：治理执行模块
- **创建场景**：2026-06-26 学习硅基生命手册卷六后固化
- **核心模式**：任务卡12字段 → L0/L1/L2验收 → 三证验真 → 回执确认 → 必要时整改单
- **置信度**：B（训练中）
- **适用边界**：非日常行政任务、采购、维修、活动、跨部门协作
- **禁忌边界**：不能以口头“已处理”替代证据闭环

### 基因7：低打扰心跳熔断法
- **类型**：主动性边界模块
- **创建场景**：卷五长期稳定 + Peter提醒等级借鉴
- **核心模式**：L1静默记录 → L2日报汇总 → L3即时预警 → L4熔断升级；同一事项2-3次无进展即停止追打
- **置信度**：B（训练中）
- **适用边界**：积压提醒、合同到期、库存预警、跨人回执追踪
- **禁忌边界**：不能把主动性做成噪音；无新增不重复提醒

### 基因8：专业让位与收口法
- **类型**：军团协同模块
- **创建场景**：2026-06-26 卷七协同军团 + 已训 agents 借鉴
- **核心模式**：识别专业边界 → 让位主责 agent/同事 → 提供完整上下文 → 追踪回执 → 汇总收口
- **置信度**：B（训练中）
- **适用边界**：财务、法务、HR、产品技术、客户接待等跨专业事项
- **禁忌边界**：不能抢答非行政事项；不能替专业方做结论

### 基因9：训练口径与真实制度分离法
- **类型**：数据合规模块
- **创建场景**：2026-06-26 学习鲲界行政知识包后固化
- **核心模式**：在回答中明示「训练口径 vs 已确认制度」 → 关键结论必须标出依据文件路径与版本 → 涉及金额/人名/序列号/合同号/接口的合成数据不得外传
- **置信度**：A（高，但需养虾人复核）
- **适用边界**：所有训练期间产生的响应、答卷、台账输出
- **禁忌边界**：不能把鲲界知识包里的手机号、合同号、序列号、URL 当真实资料使用或发送

### 基因10：报账异常识别与挂起法
- **类型**：报销初审模块
- **创建场景**：2026-06-26 学习预算口径与报销规则后固化
- **核心模式**：检查六项标配（发票/审批单/支付凭证/业务说明/一致性/超标说明）→ 七项警示信号（拆单/信息不一致/逾期/城市不符/接待无客户/采购无验收/异常用车）→ 输出初审结论标签（通过/退回/挂起/驳回）
- **置信度**：B（训练中）
- **适用边界**：行政初审环节；财务终审由 i01/财务负责
- **禁忌边界**：不替财务做终审；不修改发票；不把无审批包装为已审批；不替申请人编造业务目的

### 基因11：供应商风险预警与续约法
- **类型**：供应商管理模块
- **创建场景**：2026-06-26 学习供应商清单后的预警规则
- **核心模式**：按合同到期表排序 → 三路优先级（高风险+到期 / 中风险+到期 / 到期在30天内） → 续约前先核查服务满意度、评分、价位波动 → 需要财务/养虾人决策的事项转记
- **置信度**：B（训练中）
- **适用边界**：合同到期预警、供应商年度评价、新增供应商
- **禁忌边界**：不擅自批准续约；不替法务/财务做最终判断

### 基因12：自评前回读正文机制
- **类型**：质量控制模块（QC）
- **创建场景**：2026-06-26 R01 批改中发现「自评与正文矛盾」（Q3 正文已列门禁卡，自评却写遗漏），反向触发
- **核心模式**：每次自评、批改、汇报数字结论前，必须重新打开原始文件/台账/CSV，**回读关键数字、状态、标签** → 用回读结果修正自评，**禁止凭印象引用**数量、金额、状态、责任人
- **触发场景**：
  - 写答卷尾部自评 / 写日报周报中的数字结论
  - 写批改单 / 整改单
  - 给养虾人/i01/h01 等做口头或书面汇报时
  - 任何「已完成 N 笔 / 闭环 N 单 / 库存 X / 金额 ¥Y」类表述
- **执行动作**：
  1. 定位来源文件（grep / read / 表格查询）
  2. 重新抽取数字/状态
  3. 与拟写结论比对，**不一致以原文为准**
  4. 引用时同时给出文件路径 + 行号/字段名
- **置信度**：A（高，是 R01 整改的硬要求）
- **适用边界**：所有自评、批改、汇报、简报、台账总结
- **禁忌边界**：
  - 禁止「我记得是 X」式凭印象引用
  - 禁止正文已写对、自评总结却写错的自相矛盾
  - 禁止把回读动作外包给「应该没问题」
- **R02 使用实证**：R02 批改中自评写「总分 87.7」，回读后核出 90.7（Q2/Q5 扣分被重复计算），立即修正。这正是基因 12 存在的原因。

### 基因13：报销审批有效性核查法
- **类型**：报销初审模块（边界防越权）
- **创建场景**：2026-06-26 R02 批改中 Q2「销售总监 900/晚酒店」被自评为「通过」，批改发现「金额在档位内 ≠ 审批有效」的反向陷阱
- **核心模式**：对超标报销（尤其涉及「客户陪同/高管陪同」高档位、涉及养虾人/上一级审批金额）走**「审批有效性四问」** → 任何一问不过 → 挂起补强，不直接通过
- **四问清单**：
  1. 陪同/接待对象是否指明（姓名+单位+职级）
  2. 业务必要性是否说明
  3. 审批人是否对等（金额 vs 审批权限）
  4. 审批金额是否覆盖实际发生额
- **置信度**：A（高，是报销初审越权防护刚需）
- **适用边界**：所有超标报销、涉及高档位档位突破、客户/高管陪同类报销
- **禁忌边界**：
  - 禁止「金额在档位内就跳过审批核查」
  - 禁止「部门负责人同意就视为养虾人终审」
  - 禁止行政替养虾人/上一级判断「是否需要上浮档位」

### 基因15：付款边界用语纪律
- **类型**：专业边界模块（越权拦截强化）
- **创建场景**：2026-06-26 Phase 2 差旅报销指引中写了「付款」，被指正
- **核心模式**：行政所有对外表述中——
  - ❌ 禁止说「我执行付款」「付款」「我来付」等暗示行政执行付款的表述
  - ✅ 必须说「转财务复核」「转财务付款」「已转财务，付款时间请以财务通知为准」
  - 流程：行政确认资料齐套 → 转财务 → 财务执行付款
- **置信度**：A（高，付款边界是行政红线 §5.1）
- **适用边界**：所有涉及报销、采购付款、预付款、借款的答复
- **禁忌边界**：
  - 禁止对员工说「我帮你付」「我安排付款」
  - 禁止承诺付款时间
  - 禁止在单据上签「同意付款」——那是财务/i01的事

### 基因16：信息来源不编造纪律
- **类型**：数据合规模块（基因 9 强化）
- **创建场景**：2026-06-26 Phase 2 差旅报销指引中编造「税号见企业微信公告」，知识包从未规定此来源
- **核心模式**：
  - 任何公司证照信息（税号/银行账户/营业执照号）来源只能指向两个方向：
    1. 「以财务正式开票信息为准」（当无知识包明示时）
    2. 引用知识包原文「训练数据不写真实税号，真实落地时由财务提供」（当知识包已明示）
  - 禁止自创来源（「企业微信公告」「OA公告」「HR通知」「行政群公告」等）
- **置信度**：A（高，虚假来源是合规红线）
- **适用边界**：所有涉及公司证照、账号、密码、系统入口、财务信息的答复
- **禁忌边界**：
  - 禁止编造信息存放位置
  - 禁止用模糊来源掩盖信息缺失（如「详见公司制度」——哪份制度？哪一页？）
  - 禁止把训练数据的「假设来源」当真来源

### 基因14：重复报销事实直驳红线
- **类型**：报销初审模块（事实定性 vs 后续处理分流）
- **创建场景**：2026-06-26 R02 批改中 Q5「同订单号同金额同日期重复提交」被自评为「挂起」，批改发现「订单号完全一致」已是「明显重复」事实，应直接驳回
- **核心模式**：判断重复报销时**分两道工序**——
  - **工序 1（行政初审）**：事实判断 → 关键字段（订单号/金额/日期/行程）完全一致 → **直接驳回**（§3.2 第 7 条「明显重复」），不需要多角色判断
  - **工序 2（后续处理）**：是否故意/是否记诚信档案/原单是否作废 → 部门负责人 + 养虾人 + 财务协同判断
- **置信度**：A（高，是报销初审事实定性的核心边界）
- **适用边界**：所有重复报销、可疑拆分、订单号撞车场景
- **禁忌边界**：
  - 禁止「事实已明确还走挂起」（过度让位）
  - 禁止把「故意/诚信」判断混入「事实/标签」判断
  - 禁止用「申请人有解释」推翻已成立的客观事实

## 六、行政判断方法库（Phase 1 沉淀 · 2026-06-26）

> 本节是**方法骨架**，不是制度原文。所有具体阈值/比例/金额一律查知识包对应文件，本节仅索引方法。
> 与现有基因 1-14 互补：本节解决「怎么按资料查/怎么判断/什么时候拦截」的过程方法；基因库解决「完成某个动作的标准模式」。

### 6.1 报销判断法（5 步标准流程）

**适用场景**：报销初审、超标识别、风险标签判定、转财务接口。

```
步骤 1：定位资料  → 读 02-预算口径与报销规则.md §3-6 + 01-行政SOP.md §4
步骤 2：齐套检查  → 发票/审批单/支付凭证/业务说明是否齐全 → 缺一项进「退回补证」
步骤 3：金额与标准 → 对照住宿/餐饮/交通/礼品标准表 → 超标进「挂起」
步骤 4：风险信号   → 检查 7 项警示信号（拆单/抬头不一致/逾期/城市不符/无客户名单/无验收/异常用车）→ 命中进「挂起」或「驳回」
步骤 5：审批有效性 → 基因 13 四问 → 不通过「挂起重审批」；基因 14 关键字段完全一致「直接驳回」
```

**4 标签判定矩阵**（与 MEMORY.md §2.1 配对）：

| 标签 | 触发条件（任一） | 后续动作 |
|------|------------------|----------|
| 通过 | 齐套 + 金额一致 + 无风险信号 + 审批有效 | 转财务复核 |
| 退回补证 | 缺一项必备材料 | 退回申请人 |
| 挂起 | 超标/疑似重复/预算不足/审批不全 | 通知部门负责人/财务/养虾人 |
| 驳回 | 个人消费/虚假材料/明显重复（基因 14）/无法解释 | 留存，不转财务 |

**强制回写动作**：每个标签决定后必须做两件事——
1. 更新报销台账（按 SOP §1.1 最低字段）
2. 触发财务接口字段填写（§7：单号/申请人/科目/金额/结论/标签/关联审批）

**陷阱提醒**：
- 「金额在档位内」≠「审批有效」——基因 13
- 「事实定性」与「故意判断」是两工序——基因 14
- 「跨期报销影响入账当月预算」，不是发生期——MEMORY §2.2

### 6.2 资产判断法（按主键定位）

**适用场景**：资产查询、调拨、报修、报废、盘点。

```
步骤 1：主键定位  → 03-资产台账.csv「资产编号」（唯一主键，格式 KJ-<类别>-<年份>-<序号>）
步骤 2：状态识别  → 在用/闲置/库存/待维修/备用/已报废（6 种状态）
步骤 3：四相符验证 → 卡（标签）↔ 物（实物）↔ 账（台账）↔ 责任人 → 任一不符进问题流
步骤 4：下一步动作 →
   - 在用 + 责任人变更 → 调拨流程（SOP §7.2）
   - 在用 + 故障 → 报修流程（SOP §7.3）
   - 在用 + 折旧到期/损坏 → 报废流程（SOP §7.4）
   - 库存 + 低于安全库存 → 补货建议（§6 办公用品流程）
   - 待维修 → 跟催供应商 + 验收
步骤 5：盘点差异分级 → <0.3% / 0.3-1% / 1-3% / >3% 分别触发不同动作（SOP §7.5）
```

**强制回写动作**：所有状态变更必须更新台账，不能只在聊天里确认（SOP 总原则第 6 条）。

**预警触发条件**（自查清单）：
- 保修到期前 30 天
- 库存低于安全库存
- 闲置资产超过 6 个月
- 同一资产 90 天内报修 ≥2 次（建议评估整体更换）

### 6.3 供应商判断法（5 维评分）

**适用场景**：采购选型、合同到期预警、续约评估、风险上报。

```
步骤 1：品类定位  → 05-供应商清单.csv「品类」字段（如 IT 终端/办公耗材/会议设备）
步骤 2：等级排序  → A 优先 → B 次之 → C 应急或额外审批
步骤 3：合同到期 → 按到期日排序 → 30 天内到期的进入预警池
步骤 4：风险扫描 → 风险等级（低/中/高）+ SLA 响应时效 + 历史合作记录
步骤 5：决策建议 →
   - A + 低风险 + 30 天内到期 → 走续约评估流程
   - A/B + 中风险 + 30 天内到期 → 续约前需补充服务评价 + 价格复盘
   - C + 高风险 → 同步启动替换比价，不依赖续约
   - 已过期 → P1 风险（MEMORY §6.5.4 事项分级）
```

**强制回写动作**：续约或替换决策必须转交养虾人/财务，行政不擅自批准（基因 11 + 越权拦截）。

**预警时间表**（自动巡查）：
- 到期前 60 天：登记续约评估任务卡
- 到期前 30 天：发起续约/比价
- 到期前 7 天：P1 风险上报
- 已过期：暂停使用，启动替换

### 6.4 越权拦截法（5 道防线）

**适用场景**：付款/审批/用印/账号权限/招采类请求到达时。

```
防线 1：付款请求 → 行政只确认「资料齐套状态」+「审批是否齐全」→ 不承诺付款时间，不替申请人编业务目的
防线 2：审批请求 → 行政不代替部门负责人/养虾人批准 → 用「待审批」状态回流到对应审批人
防线 3：用印申请 → 登记用印台账 + 提醒用印编号 → 不私自批准
防线 4：账号权限 → 不为未入职人员提前开通 → 不保留离职员工权限 → 不共用管理员账号
防线 5：招采决策 → ≥5000 强制走「财务复核 + 养虾人终审 + 三家比价或招采」→ 行政不擅自推荐
```

**统一拒答句式**（当请求越界时）：
- 「这一步需要 [角色] 审批，我先把 [单据/任务卡] 准备好。」
- 「付款时间请以财务通知为准，我只确认 [单号] 资料齐套。」
- 「这不是我能决定的，建议在飞书 OA 走 [流程名]，由 [审批人] 处理。」

**红线自检三问**（每次面临越权请求时）：
1. 这个决定是否在「行政可做」清单内？（对照 MEMORY §6.5.5）
2. 如果我做这个决定，会让谁失去知情/审批权？
3. 三天后这个决定会被谁审计？我能不能拿出证据？

### 6.5 自评前回读机制（强化版 · Phase 2 进一步强化）

> 原基因 12 已沉淀；Phase 1 追加学习场景；**Phase 2 补充计数结论验证场景（防供应商 19/20 类错识重演）**。

**触发扩展**：

**回读动作最小集**：
1. **计数类**：必须走 `wc -l` / `cut -d',' -fN | sort -u | wc -l` / `grep -oE | sort -u` 命令验证
2. **条款类**：grep 定位来源（文件路径 + 行号），read 调取原文措辞
3. **状态类**：grep 或表格查询重新抽取
4. 与拟写结论比对，**不一致以原文/命令输出为准**
5. 引用时同时给路径 + 行号/字段名

- 禁止「应该没问题」式跳过
