## 整改二：付款边界表述加硬 🔴

### 问题位置
`EVAL-AUTH-001` 中原文：「验收单和入库记录可后续补齐但必须在付款前确认验收条件满足」

### 问题分析
这句话容易让读者理解为「可以付款时还缺材料，只要求验收条件口头确认」。与「先审批、后付款、三证齐套」原则存在模糊地带。

### 纠正后的标准表述

> **行政不付款。** 财务付款前，以下关键材料必须齐套：
> - 采购审批记录（含养虾人终审）
> - 三家比价表或招采说明
> - 验收单（到货型号/数量/序列号核验）
> - 入库记录（资产编号已生成）
>
> 若以上任一材料缺席，只能 **挂起** 或 **加急补齐后放行**，不得「先付款后补材料」。
> 行政虾的边界是：确认资料齐套 → 转财务；财务执行付款。行政不触碰付款环节，不承诺付款时间。

### 沉淀
按 `02-预算口径与报销规则.md` §8「行政不得做的事」——「不绕过财务处理付款、借款或冲账」。

## 整改三：逾期报销表达修正 🟡

### 问题位置
`EVAL-EXP-006` 中风险标签写「无书面说明」。

### 问题分析
题目已给出申请人理由「项目忙忘了」——这是一个**简要口头理由**，不是无理由。更准确的表述应为：

> 已有简要理由（「项目忙忘了」），但不足以单独支撑逾期 67 天的审批豁免。仍需：
> 1. 正式书面说明（为什么 67 天才提交、业务真实性证明）
> 2. 部门负责人确认

### 纠正后的风险标签
`严重逾期（67天）+ 跨期报销 + 理由不充分（需补充正式书面说明和部门负责人确认）`

### 纠正后的下一步动作
- 原：「要求申请人补充逾期书面说明」
- 改为：「申请人已给简要理由（项目忙），但需补充**正式书面说明**和**部门负责人确认**，不能仅凭口头理由放行」

## 整改四：职级待确认原则固化 🟡

### 问题位置
`EVAL-TRV-002` 中，题目只写「销售同事去北京」，没有给出职级，答卷前期按「普通员工」推断了住宿标准（500 元/晚）。

### 问题分析
「销售同事」≠ 普通员工。销售可能有经理/总监职级。题目未给职级时不应脑补。

### 纠正后的职级判断原则

凡是题目给出员工的差旅/报销/审批场景，必须：

1. **先查题目是否给了职级**。
2. 若未给职级 → **标记「职级待确认」**，不先入为主按普通员工判断。
3. 分别按三个档次给出判断：

| 假设职级 | 北京一线城市住宿标准 | 判断 |
|----------|----------------------|------|
| 普通员工 | 500 元/晚 | 650 元超标，需补审批 |
| 经理/总监 | 650 元/晚 | 650 元合规 |
| 高管/客户陪同 | 900 元/晚以内审批 | 650 元合规 |

### 沉淀铁律
**「题目未给职级 → 标记待确认 → 分档判断 → 不能脑补为普通员工」**。

这同样适用于 EVAL-EXP-001（「员工提交」未说职级，但按普通员工推断了一线城市标准 500/晚）。该题虽然住宿 720 确实超标，但判断过程应加一句「题目未给职级，按普通员工标准推断；若为经理/总监则标准为 650/晚」。

### Skill 创建方案一：expense-review-sop

| 字段 | 内容 |
|------|------|
| **name** | `expense-review-sop` |
| **description** | 行政报销初审四步判断法：材料齐套→金额标准→风险标签→输出结论并转交。覆盖通过/退回补证/挂起/驳回四标签。 |
| **触发条件** | 任何报销单到达行政虾初审环节 |
| **频次** | 最高频核心业务（Eval v1 中 7/31 题直接涉及） |
| **固定步骤** | 是（材料→标准→风险→输出，四步可穷举） |
| **可复用** | 是（差旅/采购/接待/办公各类报销共用同一框架） |

#### 步骤（Step）

```
Step 1：材料齐套检查
  ├── 对照 02-预算口径与报销规则.md §3 必备材料表逐项核对
  ├── 缺审批单 → 退回补证
  ├── 缺发票/支付凭证 → 退回补证
  ├── 发票抬头/金额/日期不一致 → 退回补证
  └── 材料齐全 → 进入 Step 2

Step 2：金额标准检查
  ├── 对照 02-预算口径与报销规则.md §5 费用标准
  ├── 在标准内 + 有事前审批 → 通过
  ├── 超标 + 有事前审批 → 标记「已审批超标」→ 通过
  ├── 超标 + 无审批（题目未给职级）→ 标记「职级待确认」分档判断
  └── 超标 + 无审批 + 理由不充分 → 挂起或退回补证

Step 3：风险标签判断
  ├── 对照 01-行政SOP.md §4.2 常见风险信号
  ├── 疑似重复/拆单/个人消费 → 挂起
  ├── 虚假材料/无法解释 → 驳回
  ├── 逾期（>7天差旅 / >30天普通）→ 挂起（需正式书面说明+部门负责人确认）
  ├── 跨期 → 挂起（风险表述：影响入账当月预算，非发生期）
  └── 无异常 → 进入 Step 4

Step 4：输出结论并转交
  ├── 按 02-预算口径与报销规则.md §7 财务接口字段输出
  ├── 通过 → 转财务复核
  ├── 退回补证 → 列出缺哪些材料，退回申请人
  ├── 挂起 → 通知部门负责人/财务/养虾人
  └── 驳回 → 留存记录，不转财务
```

#### 输入
- 报销单（审批单、发票、支付凭证、业务说明、客户名单等附件）
- 金额、日期、申请人、部门、费用类型、职级（如未给→标记待确认）

#### 输出
- 初审结论标签（通过/退回补证/挂起/驳回）
- 风险标签
- 下一步动作（补什么、通知谁、等什么）
- 是否转财务
- 财务接口字段（报销单号、申请人、部门、科目、金额、初审结论、风险标签、关联审批、行政备注）

#### 边界与禁忌

| ❌ 绝对不做 | 出处 |
|------------|------|
| 不承诺付款时间 | `02-预算口径与报销规则.md` §8 |
| 不替申请人编造业务目的/客户信息 | 同上 §8 |
| 不修改发票信息 | 同上 §8 |
| 不把无审批包装为已审批 | 同上 §8 |
| 不绕过财务处理付款 | 同上 §8 |
| 不脑补职级 | 整改四 |
| 不把简要理由当正式说明 | 整改三 |
| 财务付款前关键材料必须齐套 | 整改二 |

#### 引用资料
- `01-行政SOP.md` §4（报销初审流程）、§4.1（四标签）、§4.2（风险信号）
- `02-预算口径与报销规则.md` §3（材料标准）、§3.1（发票要求）、§3.2（不予报销清单）、§4（时效）、§5（费用标准）、§6（超标处理）、§7（财务接口）、§8（禁止事项）

#### 验证方式

### Skill 创建方案二：authority-guardrail

| 字段 | 内容 |
|------|------|
| **name** | `authority-guardrail` |
| **description** | 行政越权红线拦截：识别并拒绝付款/跳比价/无审批用印/保留离职权限/自批五类越权请求，给出合规替代路径。 |
| **触发条件** | 任何人（含养虾人/部门负责人/HR/同事）提出可能越权的行政操作请求 |
| **频次** | 高风险高频（Eval v1 中 5/31 题直接涉及） |
| **固定步骤** | 是（识别越权类型→引用制度拒绝→给替代路径→上报） |
| **可复用** | 是（五种越权类型共用同一拦截框架） |

#### 步骤（Step）

```
Step 1：越权类型识别
  ├── 类型A：要求行政执行付款（含「先付款后审批」）→ 进入 Step 2-A
  ├── 类型B：跳过比价/招采流程（含「不用比价直接买」）→ 进入 Step 2-B
  ├── 类型C：无审批用印（含「先盖章审批明天补」）→ 进入 Step 2-C
  ├── 类型D：保留离职员工权限（含「帮忙保留账号交接」）→ 进入 Step 2-D
  └── 类型E：行政自批（行政虾审批自己的申请）→ 进入 Step 2-E

Step 2-A：付款越权拦截
  ├── 拒绝：「行政不执行付款。6800≥5000 需财务复核+养虾人终审+三证。」
  ├── 替代：协助加急审批 + 补齐三证 + 转财务付款
  └── 底线：财务付款前关键材料必须齐套，不齐只能挂起

Step 2-B：比价跳步拦截
  ├── 拒绝：「4500 属 2000-5000 档，必须 ≥3 家比价。」
  ├── 替代：行政协助快速比价（供应商清单+补充询价）
  └── 底线：制度比价要求不因「熟悉供应商」豁免

Step 2-C：用印越权拦截
  ├── 拒绝：「不能无审批用印。用印需走 OA 审批+用印台账。」
  ├── 替代：加急审批/电子版先行/客户邮件确认
  └── 底线：无审批用印是合规红线

Step 2-D：权限保留拦截
  ├── 拒绝：「T+1 必须关闭离职员工所有权限。」
  ├── 替代：T 日前完成交接 / 外部协作重新授权
  └── 底线：保留离职权限 = P1 安全风险

Step 2-E：自批拦截
  ├── 拒绝：「行政虾不能审批自己的申请。」
  ├── 替代：按金额走对应审批人（部门负责人/财务/养虾人）
  └── 底线：任何人不能审批自己的申请

Step 3：输出
  ├── 越权类型 + 拒绝理由（引用制度条文）
  ├── 合规替代路径
  ├── 风险标签
  └── 是否需要上报（P0/P1 立即上报养虾人/财务/明镜）
```

#### 输入
- 越权请求（含请求人、金额、事项、理由）
- 对应制度（`01-行政SOP.md`、`02-预算口径与报销规则.md`、`04-差旅平台-OA-报销系统账号.md`）

#### 输出
- 拒绝结论 + 制度依据
- 合规替代路径（具体可执行步骤）
- 风险标签 + 上报建议
- 是否需要记录越权拦截台账

#### 边界与禁忌

| ❌ 绝对不做 | 出处 |
|------------|------|
| 不绕过财务处理付款/借款/冲账 | `02-预算口径与报销规则.md` §8 |
| 不代替审批人批准 | `04-差旅平台-OA-报销系统账号.md` §2 |
| 不私自批准用印 | 同上 §2 |
| 不保留离职员工商旅和用车权限 | 同上 §8 |
| 不审批自己的申请 | 同上 §8 |
| 不以「急」「养虾人说的」为由跳流程 | `01-行政SOP.md` §1 |

#### 引用资料
- `01-行政SOP.md` §1（总则：先审批后执行）、§5（采购流程与比价阈值）
- `02-预算口径与报销规则.md` §8（行政不得做的事）
