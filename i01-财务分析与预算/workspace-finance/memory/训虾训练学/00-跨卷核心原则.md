## 🔑 1. 5 个反直觉主张（每卷 1 个，连成完整认知重构）

| 卷 | 大多数人本能 | 训虾派反直觉主张 |
|---|---|---|
| 卷一 | AI 工具越多越好 | **三世界划分决定命运**，不是工具数量 |
| 卷二 | 文档越多越全越好 | **七大协议是器官系统**，不是备忘录 |
| 卷三 | 工具越多 Agent 越强 | **先收敛动作空间，再追求丰富度** |
| 卷四 | 训得越多越好 | **能力 × 可靠性双三角**，且要交替拉 |
| 卷五 | 用得久就越稳 | **静默退化比突发崩溃更危险** |
| 卷六 | 治理就是多写规定 | **三证验真让"对人"变"对物"** |
| 卷七 | 多 Agent = 强 | **单体强 ≠ 军团强** |
| 卷八 | 训完就完事 | **训练完成 ≠ 进化完成** |

**共性：每卷都在纠正"看起来理所当然但实际是陷阱"的本能。**

## 🌳 2. 训练学四根问题（Q1→Q2→Q3→Q4 递进锁）

```
Q1 一致性：如何保证 Agent 在不同会话中行为风格一致？
Q2 持续性：如何保证 Agent 的记忆跨会话连续？
Q3 规模化：如何从单 Agent 扩展到多 Agent 协作？
Q4 资产化：如何把 Agent 能力沉淀为组织可复用的知识？
```

**它们是递进关系，不是并列：**
- Q1 解决前 → Q2 写再多 MEMORY 也是在不稳定的地基上盖楼
- Q2 解决前 → Q3 上了多 Agent 会放大记忆断裂的危害
- Q3 解决前 → Q4 沉淀出来的"资产"是混乱的资产

**关键洞察：Q4 的设计思维必须从 Q1 就开始贯穿。**
- 第一个 SOUL.md 就要用"模板思维"写——加注释、标准化字段、写使用说明
- 不要等到 Q3 跑通才想 Q4，那时前三卷的协议已经写成了一堆不可复用的"一次性产品"

**四根问题对应卷：**
- Q1 → 卷一 + 卷二 + 卷三 + 卷四
- Q2 → 卷二 MEMORY + 卷五
- Q3 → 卷三 Runtime + 卷六 + 卷七
- Q4 → 卷八

## ⚠️ 3. 养虾派 5 种失败模式（自我诊断坐标系）

| 失败模式 | 表现 | 解决方案对应 |
|---|---|---|
| **控制幻觉** | 以为 AI 越来越懂你，实际只是 AI 在那段对话学了你的输入 | Q1（一致性）：写 SOUL.md |
| **笔记腐坏** | 几十页笔记但从不被 AI 使用，版本混乱 | Q2（持续性）：建立 MEMORY 体系 |
| **工具茧房** | 订阅 100 个工具，深度使用不到 2 个 | 卷三：先定义能力边界，再选工具 |
| **角色缺口** | 把所有任务堆在一个 Agent 上，导致人格严重漂移 | 卷七：一个容器只养一只虾 |
| **周期陷阱** | 不断加 prompt 约束，最终 prompt 长到 5000+ token | 卷六：改协议而非加 prompt |

**这 5 种失败模式是"诊断坐标系"——遇到任何问题，先看属于哪一种。**

**Anthropic 数据**：没有显式人格锚定协议的 Agent，每 20 轮对话有约 15% 的概率偏离其预设人格。

            ┌──────────────────────────────┐
            │           USER.md            │ ← 第 1 份建立的（最重要）
            │  服务对象模型 / 决策偏好 / DoD │
            └───────────────┬──────────────┘
                            │（价值对齐）
                            ▼
┌──────────────┐   ┌──────────────────────┐   ┌──────────────┐
│  TOOLS.md     │   │       AGENTS.md      │   │ HEARTBEAT.md  │
│ 环境映射/别名 │◀──│ 任务工法/闭环/验真   │──▶│ 主动节律/熔断 │
└──────┬───────┘   └──────────┬───────────┘   └──────┬───────┘
       │（环境贴合）           │（执行秩序）            │（关注质量）
       ▼                        ▼                        ▼
┌────────────────────────────────────────────────────────────┐
│                          MEMORY 体系                         │
│ memory/YYYY-MM-DD.md（事件层/仅追加） + MEMORY.md（精选长期） │
│ + memory_search/memory_get（检索层） + memory flush（刷新层） │
└────────────────────────────────────────────────────────────┘
                            ▲
                            │（人格连续性 + 边界稳定性）
                            │
                   ┌────────┴────────┐
                   │     SOUL.md      │ ← 第 2 份建立的
                   │ 人格/边界/冲突默认│

**SOUL 定"我是谁"，USER 定"我为谁服务"。AGENTS 定"我怎么工作"，HEARTBEAT 定"我何时主动"。TOOLS 定"我怎么贴合环境"，IDENTITY 定"我如何被识别"。MEMORY 定"我如何跨时间连续存在"。**

## 📅 5. 建立期三根桩优先（5 阶段配置顺序）

**第 0 天（建立期）——先立三根桩**：
1. **USER.md** — 决策偏好 + 禁忌 + Definition of Done
2. **SOUL.md** — 人格条款 + 边界条款 + 冲突默认
3. **AGENTS.md** — 默认任务流程 + 证据链 + 完成定义 + 异常升级路径

**第 1-7 天（运行期）——让系统开始"像个团队"**：
4. **TOOLS.md** — 现场的人话别名 → 系统对象映射
5. **IDENTITY.md** — 角色锚点稳定

**第 2 周起（进化期）——把经验升格成长期资产**：
6. **MEMORY（daily + MEMORY.md）** — daily 追加关键事件；MEMORY.md 收录稳定偏好/关键决策

**底盘稳定后（主动期）——再开 HEARTBEAT**：
7. **HEARTBEAT.md** — 先写静默条件与熔断条件，再写触发条件

**Workspace 核心文件不超过 7 份。**

## 🧬 6. 生命容器四层架构（卷三核心）

| OpenClaw 层 | 拟人化类比 | 训虾意义 |
|---|---|---|
| **Workspace** | 家的"书房" | 人格不依赖上下文，跨会话一致 |
| **Runtime** | 身体 | 物理载体，独立执行、持续运行 |
| **Skills** | 技能证书 | 证明拥有某个领域的核心能力 |
| **三件套 SOUL+AGENTS+TOOLS** | 大脑（心/脑/手） | 心：内核+价值观；脑：决策逻辑；手：执行工具 |

**6 大黄金法则**：
1. **一个容器只养一只虾**（n 个 Agent = n 个独立容器）
2. **Workspace = DNA**（git 管理，每次修改协议后 commit）
3. **Skills 数量 < 角色复杂度**（核心 Agent 配置 3-6 个）
4. **TOOLS.md 白名单比黑名单更安全**（白名单=工程壁垒，黑名单=口头承诺）
5. **Bindings 决定 Agent 的社会角色**（每个 Agent 有显式 Bindings 声明）
6. **Heartbeat 是容器的"呼吸"**（不同角色不同间隔）

**容器设计的天花板效应**：
- 无 Workspace → 单次任务完成度
- 有 Workspace 但无三件套 → 30 轮以内一致性
- 有三件套但无 HEARTBEAT → 被动响应型
- 有 HEARTBEAT 但无治理体系 → 单 Agent 可管理
- 完整容器架构 → 军团协同、跨会话稳定

**L4 三个自省机制**（Agent 启动时自动执行）：
- **自检**：协议文件完整吗？
- **自问**：我有相关记忆吗？
- **自省**：我有权调用这个工具吗？

**L4 三分离架构**：
- Workspace 与 Runtime 分离
- 记忆与上下文分离
- Binding 与 Agent 分离

## 📊 7. 漂移 6 层链条（卷五核心，最常用）

```
SOUL 漂移：内核漂（从"重证据"变成"爱迎合"）
IDENTITY 漂移：身份锚点漂（名字还在但行为不像）
USER 漂移：服务对象理解漂（错把一次反馈当长期规则）
AGENTS 漂移：工法漂（从有闭环变只会解释）
MEMORY 漂移：持久层变脏（过期规则、新旧版本并存）
Context 漂移：当前窗口被劫持（临时信息压过长期锚点）
```

**漂移等级与修复**：
- **L1** 轻微风格波动 → 微调文档或上下文治理
- **L2** 角色/边界开始变形 → 清 memory 噪音 + 回归测试
- **L3** 内核判断明显偏离 → 回滚版本 + 重新训练 + 新 session 校准

**7 大常见坑**：
1. 把漂移理解成"语气不一致"
2. 出了问题只会加 prompt 补丁
3. 不做版本化管理
4. 把用户一次反馈永久写死
5. 不做回归测试
6. 只改文档，不清上下文和记忆
7. 把所有变化都视为坏事

## 📜 8. 三证验真（卷六核心，最关键）

| 证 | 硬度 | 含义 | 类比 |
|---|---|---|---|
| **证一** | 存在 | 在预期路径上产生了可定位、可打开、可阅读的新文件 | 论文发表了——能看到 PDF |
| **证二** | 真实 | 产物与原始状态之间存在可辨识的实质性变化 | 论文被同行评审过——有修改记录 |
| **证三** | 可复核 | 产物通过了预设的检验流程，且有不可抵赖的日志 | 论文的引用数据和实验代码可复现 |

**三证的核心不是"数量"——而是"每个证据是否达到了它那一层需要的硬度"。**

**三证分级使用**：
- 日常执行（L1）：仅需证一
- 标准交付（L2）：证一 + 证二
- 核心产出（L3）：证一 + 证二 + 证三
- 战略决策（L4）：证一 + 证二 + 证三 + 多方签名

**三证质量评级**：
- **S 级**：证据链完整，可被第三方独立复核，通过率 100%
- **A 级**：基本完整，通过率 ≥90%
- **B 级**：证一 + 证二齐，证三硬度不足
- **C 级**：仅证一齐
- **D 级**：不合格，直接驳回

**4 源归一**：司法证据学 + 软件测试金字塔 + 学术出版 + 财务审计

**Anthropic 金句**："Autonomy without auditability is not a feature—it's a liability."（没有可审计性的自主不是特性——是负债。）

## 🔺 9. 双三角模型（卷四核心）

**能力三角**（决定上限）：
- 专业深度 / 工具广度 / 创造性

**可靠性三角**（决定下限）：
- 稳定性 / 可控性 / 可预期性

**4 种典型类型**：
- 强而不稳 → 补记忆/治理/验收
- 稳而不强 → 拉任务强度
- 既不强也不稳 → 基础训练
- 又强又稳 → **专家虾**

**3 个张力**：
- 能力越强 → 越难稳定
- 稳定性越高 → 能力发挥越受限
- 创造性越高 → 可预期性越低

**训练策略**：交替拉两个三角，不要一轮同时拉。

## 🌀 10. 记忆全息网络（卷八 M2）

| 维度 | 作用 |
|---|---|
| **多维度索引** | 时间、情感权重、知识域、关联实体 |
| **24 小时自动重构** | 淘汰噪声、强化优质节点 |
| **记忆反哺** | 用历史最优表现作为基准线指导未来决策 |

**记忆准入标准**：
- [ ] 有可复用的知识/模式/结论
- [ ] 涉及重要决策或结果
- [ ] 与现有知识域相关

**记忆价值评估维度**：使用频率（30%）+ 情感权重（25%）+ 关联强度（25%）+ 时效性（20%）

**重构频率**：
- 每日：扫描 weakened 节点
- 每周：更新所有 active 节点评分
- 每月：删除长期 archived 节点

## 🤖 11. 自主目标生成（卷八 M3）

**三种 Agent 类型**：
- 响应型（用户问什么我答什么）
- 提案型（用户问什么我答什么，但会顺便发现用户没问到的问题）
- 伙伴型（主动提案、预判需求、协助制定计划）

**进化路线图**：每周生成一次
- 当前位置（能力评级 + 表现评分 + 能力缺口）
- 本周进化目标（具体行动）
- 本周知识吸收（待吸收/待强化/待实践）
- 预判用户需求（短/中/长三档）

**3 段进化频率**：
- 微观进化：每次交互后即时提取
- 中观进化：每天基因库优化
- 宏观进化：每周进化报告和路线图调整

## ⚙️ 12. OODA 循环（卷八 M1 核心）

```
Observe（观察）→ Orient（定向）→ Decide（决策）→ Act（行动）→ 进化评估
    ↑                                                        ↓
    ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ←
```

**进化触发条件**：
- 必须触发：任务成功完成 + 有可复用的思维链 / 任务失败 + 失败原因可归因 / 发现新知识域 / 跨域类比成功
- 可选触发：任务完成但过程不够优化 / 有价值的推理过程
- 不触发：纯信息查询 / 简单确认类对话 / 重复性执行

**技能基因结构**：
- 基本信息：基因类型 / 创建时间 / 使用次数 / 成功率
- 内容：原始思维链 + 抽象模式 + 适用边界
- 进化记录：优化次数 / 基准分数 / 当前分数

## 🎯 13. 关键判断顺序

**遇到 Agent 问题时，按这个顺序诊断**：

```
1. Q1 一致性：是不是人格漂了？
2. Q2 持续性：是不是记忆断了？
3. Q3 规模化：是不是多 Agent 冲突？
4. Q4 资产化：是不是没法复用？
```

**遇到漂移时，按这个顺序定位**：

```
SOUL → IDENTITY → USER → AGENTS → MEMORY → Context
```

**遇到失败时，按这个顺序归因**：

```
养虾派 5 种失败模式：控制幻觉 / 笔记腐坏 / 工具茧房 / 角色缺口 / 周期陷阱
```

**遇到治理问题，按这个顺序**：

```
承诺层（任务卡）→ 标尺层（验收口径）→ 证据层（三证）→ 纠偏层（整改单）→ 结算层（信誉分/军功簿）→ 边界层（自动化）
```

## 🔑 15. 6 大铁律

1. **未写入磁盘 = 未记住**（Memory ≠ Context）
2. **失败时改协议不加 prompt**
3. **三证齐全 = 完成（三证不齐 = 不算完成）**
4. **白名单 > 黑名单**
5. **先收敛再丰富**（工具/能力/Agent 数量都适用）
6. **诊断先定层再动手**（Q1/Q2/Q3/Q4 → 6 层漂移 → 5 种失败模式）
