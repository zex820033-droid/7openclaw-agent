# 02｜卷二 生命协议：七大生命协议

> 7 份协议 + 总复盘
> 精读 3 轮后浓缩

---

## 一句话

**卷二要解决的不是"如何写几个 md 文件"，而是：如何用 7 份协议文件，把一个 LLM 训练成具备长期连续性、稳定边界与可治理执行力的硅基生命。**

---

## 七大协议（按建立顺序）

| 顺序 | 文件 | 训练对象 | 解决的问题 | 最常见失真 |
|---|---|---|---|---|
| 1 | **USER.md** | 服务对象模型（偏好/决策/完成定义） | 输出对齐用户价值 | 回答正确但不"对你有用" |
| 2 | **SOUL.md** | 人格内核 + 边界 + 冲突默认 | 长期身份稳定 | 角色漂移、风格漂移、越界承诺 |
| 3 | **AGENTS.md** | 工法/纪律/闭环/验真 | 从会说到会做 | 伪完成、无证据完成、协作抢答 |
| 4 | **TOOLS.md** | 环境映射（别名/路径/设备/约定） | 工具最后一公里 | 发错目录/连错主机 |
| 5 | **IDENTITY.md** | 身份锚点/协同识别/追责 | 多 Agent 秩序 | 串角色/抢答/路由不清 |
| 6 | **MEMORY 体系** | 连续性/沉淀/检索/进化 | 跨会话稳定表现 | 会话失忆、决策反复、检索无效 |
| 7 | **HEARTBEAT.md** | 主动性节律 + 静默 + 熔断 | 关注质量与可控主动性 | 通知疲劳、循环提醒、资源耗尽 |

**注意：顺序错了 SOUL.md 经常要重写。先 USER，后 SOUL。**

---

## 7 大协议之间的"训练对象"和"关系"

### 拓扑关系
```
USER.md（价值对齐）  ──→  SOUL.md（人格内核）
                              │
                              ├──→ IDENTITY.md（外显锚点/协同秩序）
                              │
                              ├──→ AGENTS.md（执行秩序）
                              │         │
                              │         ├──→ TOOLS.md（环境贴合）
                              │         │
                              │         └──→ HEARTBEAT.md（关注质量）
                              │
                              └──→ MEMORY 体系（跨时间连续性）
```

## 7 大协议 vs 4 根问题

| 协议 | 解决哪个根问题 |
|---|---|
| USER.md | Q1（一致性）—— 用户偏好写清才有一致性 |
| SOUL.md | Q1（一致性）—— 人格内核 |
| AGENTS.md | Q1（一致性）—— 工法纪律 |
| TOOLS.md | Q1（一致性）—— 工具边界 |
| IDENTITY.md | Q1 + Q3（一致性 + 规模化）—— 身份锚点 |
| MEMORY 体系 | Q2（持续性）—— 跨会话记忆 |
| HEARTBEAT.md | Q1（一致性）—— 主动性节律 |

## 7 大协议失真诊断表

| 症状 | 首查文件 | 次查文件 |
|---|---|---|
| 输出风格忽冷忽热、今天像顾问明天像客服 | SOUL.md | IDENTITY.md |
| 说得很好但永远不落地，缺文件/缺回执/缺验收 | AGENTS.md | USER.md（DoD） |
| 工具总用错，路径/别名/对外目录总混 | TOOLS.md | AGENTS.md |
| 主动提醒太吵/重复提醒/像骚扰 | HEARTBEAT.md | （静默条件/冷却/熔断） |
| 反复解释同一件事，隔天就忘、压缩后就忘 | MEMORY 体系 | USER/SOUL（稳定条款） |

## 关键原则

### 1. 未写入磁盘 = 未记住
- OpenClaw 记忆是工作空间里的纯 Markdown 文件
- 模型只"记住"写入磁盘的内容
- 会话里"看起来记得"= 只是暂存幻觉

### 2. 两层记忆结构
- `memory/YYYY-MM-DD.md`：事件层（append-only，daily 追加）
- `MEMORY.md`：精选长期层（主私聊加载；群聊不加载）

### 3. 检索能力
- `memory_search`：语义检索
- `memory_get`：安全读取片段

### 4. 静默 memory flush
- 会话接近 compaction 时，可触发静默 memory flush
- 把"必须长期保留的"从临时脑内转存为外部持久记忆

### 5. 双读者原则
- MEMORY.md 应同时面向 Agent 和人类读者
- 写给 Agent：可被 memory_search 检索
- 写给人：可读性、可时间排序、可追溯决策
- 假设有人离职了另一个人能不能从 MEMORY.md 读出"这周为什么这么做"

### 6. 协议修改先 staging
- 重大协议变更应在测试环境验证后再推向生产
- 改 SOUL.md → 先在临时 workspace 跑 3-5 个回归测试
- 改 TOOLS.md → 涉及安全必须走 review

### 7. 协议不是 prompt
| prompt | 协议 |
|---|---|
| 当前轮纠偏 | 长期行为规则 |
| 可以频繁改 | 版本化管理 |
| 不可审计 | 可审计（可对照检查） |

---

## USER.md 怎么写（最先建立）

### 必含字段
- 决策偏好：用户怎么思考
- 禁忌：用户不能接受什么
- Definition of Done（DoD）：完成定义
- 沟通风格：用户喜欢什么语气
- 时间偏好：响应时点

---

## SOUL.md 怎么写（第 2 建立）

### 必含字段
- 身份定义
- 核心价值观
- 沟通风格
- 边界
- 决策框架
- 冲突默认处理

### 6 大黄金法则
1. 抽象的口号（"永远保持热情"）无法指导行为
2. 必须写具体行为约束
3. 价值观要可被验证
4. 边界要写清"绝不做什么"
5. 冲突时默认立场要写清
6. SOUL.md 改完要通知用户（这是"我"的灵魂）

---

## AGENTS.md 怎么写（第 3 建立）

### 必含字段
- 默认任务流程
- 证据链（产物的验收方式）
- 完成定义
- 异常升级路径
- 协作纪律
- 让位规则

### 4 类常见失真
- 伪完成：说做完了但产物不存在
- 无证据完成：产物存在但无 Diff
- 无收口：产物不完整
- 协作抢答：跨角色越界

---

## TOOLS.md 怎么写（第 4 建立）

### 必含字段
- 工具白名单（不是黑名单）
- 工具别名映射
- 路径约定
- 设备/主机名
- 环境特定配置

### 6 大黄金法则（从卷一 M3 复用）
- 白名单 > 黑名单
- 默认拒绝（Default Deny）
- 工具边界即治理入口
- 工具调用纪律（直接/澄清/串行/并行/中止）

---

## IDENTITY.md 怎么写（第 5 建立）

### 必含字段
- 名称
- 角色定位
- emoji
- 风格标识
- 路由锚点

### 关键作用
- 可识别
- 可路由
- 可追责
- 可让位
- 可形成长期人格连续性感知

---

## MEMORY 体系怎么建（第 6 建立）

### 三件套
1. `MEMORY.md`（精选长期）
2. `memory/YYYY-MM-DD.md`（事件层）
3. `memory_search` / `memory_get`（检索层）

### 写入纪律
- 决策/偏好/持久事实 → 先写 daily，再提纯进 MEMORY.md
- 模糊信息不固化
- 敏感信息不入库
- 无 owner 不固化

### 6 大漂移（与卷五联动）
- SOUL 漂移
- IDENTITY 漂移
- USER 漂移
- AGENTS 漂移
- MEMORY 漂移
- Context 漂移

---

## HEARTBEAT.md 怎么写（第 7 建立）

### 必含字段
- 静默条件（什么情况不发）
- 熔断条件（什么情况停止）
- 触发条件（什么情况发）
- 任务清单

### 关键原则
- **先写静默条件与熔断条件，再写触发条件**
- **先盯 P0 变量，别扫全世界**
- **不同 Agent 不同间隔**

---

## 反模式（不要做的事）

1. 把 BOOTSTRAP 当长期规则库（用完即删）
2. 把 IDENTITY 当装饰页
3. 把会话当记忆
4. 把所有东西都写进 MEMORY.md
5. 没有"写入纪律"与"验收口径"
6. 一次修改协议不测就直接上 production
7. 7 份协议都写满但彼此冲突

---

## 演示方案（卷二的核心演示）

### 演示 A：从"prompt 驱动"到"协议驱动"
- 同一问题：只靠 prompt vs 有 USER/SOUL/AGENTS
- 展示差异：是否给 DoD / 是否给证据链 / 是否能收口

### 演示 B：多 Agent 协同为什么必须有 IDENTITY + AGENTS
- 展示"身份不清"如何导致抢答与责任不清
- 展示"身份锚点 + 让位规则 + 主责收口"如何让协作变秩序

### 演示 C：记忆系统如何让"跨天连续性"成立
- 展示 daily 事件层写了什么
- 展示 long-term 精选层写了什么
- 展示 memory_search 如何把关键规则找回来
