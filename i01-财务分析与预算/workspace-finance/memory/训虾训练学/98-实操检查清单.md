## 三、漂移 6 层定位清单

### 症状 → 漂移层 → 修复

| 症状 | 漂移层 | 修复动作 |
|---|---|---|
| 从"重证据"变成"爱迎合" | SOUL 漂移 | 修改 SOUL.md 核心价值观 |
| 名字还在但行为不像 | IDENTITY 漂移 | 修改 IDENTITY.md + 角色锚点 |
| 错把一次反馈当长期规则 | USER 漂移 | 修改 USER.md + 删除错误规则 |
| 从有闭环变只会解释 | AGENTS 漂移 | 修改 AGENTS.md 工法条款 |
| 过期规则、新旧版本并存 | MEMORY 漂移 | 重写 MEMORY.md，删除重复 |
| 临时信息压过长期锚点 | Context 漂移 | new session，迁移关键 MEMORY |

### 漂移修复 5 步
1. 定位层级（哪一层？）
2. 找证据（文档-行为不一致的证据）
3. 清理上下文（必要时 new session）
4. 修复文档（修改对应协议文件）
5. 回归测试（同一组题测前后对比）

### 漂移等级修复
- **L1 轻微波动** → 微调文档或上下文治理
- **L2 角色变形** → 清 memory 噪音 + 回归测试
- **L3 内核偏离** → 回滚版本 + 重新训练 + 新 session 校准

## 四、三证检查清单

### 证一：新产物（Artifact）
- [ ] 路径可定位
- [ ] 格式正确
- [ ] 大小合理
- [ ] 时间戳匹配
- [ ] 作者匹配

### 证二：前后对比（Diff）
- [ ] 有明确的原始状态引用
- [ ] 变化是实质性的
- [ ] 变化清单清晰
- [ ] 没有"注水内容"
- [ ] 变化量级合理

### 证三：测试/回执/日志（Verification）
- [ ] 验证方法明确
- [ ] 验证方法可被第三方独立执行
- [ ] 验证结果可复现
- [ ] 验证结果有明确时间戳
- [ ] 验证日志不可被 Agent 自身修改

### 三证分级使用
| 任务等级 | 三证要求 |
|---|---|
| 日常执行（L1） | 仅需证一 |
| 标准交付（L2） | 证一 + 证二 |
| 核心产出（L3） | 证一 + 证二 + 证三 |
| 战略决策（L4） | 证一 + 证二 + 证三 + 多方签名 |

### 三证质量评级
- **S 级**：证据链完整，可独立复核，通过率 100%
- **A 级**：基本完整，通过率 ≥90%
- **B 级**：证一 + 证二齐，证三硬度不足
- **C 级**：仅证一齐
- **D 级**：不合格，直接驳回

## 六、实验设计检查清单

### 6 要素
- [ ] **实验假设**："如果改 X，Y 会变好"
- [ ] **对照组**：实验组 vs 对照组
- [ ] **变量控制**：这轮只改一个变量
- [ ] **可观察指标**：完成率/掉棒率/漂移次数/一致性评分
- [ ] **样本量**：至少 N 轮
- [ ] **统计显著性**：提升是否显著

### 实验类型
| 阶段 | 实验类型 | 样本量 | 周期 |
|---|---|---|---|
| 初期探索 | 单变量 | 3-5 轮 | 1 天 |
| 协议验证 | AB 对照 | 10+ 轮 | 2-3 天 |
| 稳定性验证 | 长期跟踪 | 20+ 轮 | 1 周 |
| 等级升级验证 | 跨等级 | 15+ 轮 | 3-5 天 |

## 七、协议修改 SOP

### 任何协议修改前
1. **识别修改层级**：SOUL/AGENTS/TOOLS/USER/IDENTITY/MEMORY/HEARTBEAT
2. **检查依赖**：其他协议是否依赖当前协议
3. **写变更说明**：改了什么、为什么改、风险在哪

### 修改中
1. **git 提交到 staging 分支**
2. **在临时 workspace 跑 3-5 个回归测试**
3. **记录变更到 DECISIONS.md**

### 修改后
1. **用回归集验证**
2. **新 session 验证**
3. **观察 24 小时无问题再合并到主分支**

## 八、任务失败时诊断流程

### Step 1：定位漂移层
SOUL → IDENTITY → USER → AGENTS → MEMORY → Context

### Step 2：检查失败模式
控制幻觉 / 笔记腐坏 / 工具茧房 / 角色缺口 / 周期陷阱

### Step 3：检查三证
- 证一：产物存在吗？
- 证二：有实质性变化吗？
- 证三：可复核吗？

### Step 4：检查张力
- 能力三角 vs 可靠性三角是否平衡？

### Step 5：决定修复路径
- 微调协议 → 改协议文件 → 重新训练 → 回滚版本

## 十、6 道治理边界自检

| 边界 | 检查问题 |
|---|---|
| 能力边界 | 我能做什么？超出能力时怎么办？ |
| 权限边界 | 我被允许做什么？超出权限时怎么办？ |
| 影响边界（可逆/不可逆） | 这个动作可逆吗？不可逆时需什么审批？ |
| 价值边界 | 这件事是否对齐 SOUL 价值观？ |
| 时机边界 | 现在是合适的时机吗？23:00-08:00 静默 |
| 频次边界 | 同一事件不重复 |

---

## 十一、5 类高风险操作（默认需要审批）

1. **删除**（数据/文件/资源）
2. **修改**（生产环境/系统配置/权限）
3. **外发**（公开内容/外部通知）
4. **支付**（财务操作）
5. **权限提升**（sudo/elevated）

**操作前先检查白名单 + 熔断器。**
