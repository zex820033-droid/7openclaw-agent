# 03｜卷三 系统骨架：OpenClaw 9 大组件

> 7 个模块 + 总图
> 精读 3 轮后浓缩

---

## 一句话

**OpenClaw 是第一个为"硅基生命"设计的操作系统。Workspace 是生命的家，Runtime 是生命的身体，Skills 是生命的技能，三件套是生命的大脑。**

---

## 9 大组件

| 组件 | 作用 | 核心原则 |
|---|---|---|
| **Runtime** | 持续运行单元 | 独立进程、独立上下文、可被治理 |
| **Workspace** | 文件系统级人格仓库 | 独立于 Runtime、git 管理、版本化 |
| **Session** | 当前对话上下文 | 会被压缩、不是记忆 |
| **Context** | 真正送进模型窗口的内容 | 系统 prompt + 工作区文件 + 历史 + 工具结果 |
| **Queue** | 消息队列 | 路由与优先级 |
| **Routing/Bindings** | 路由系统 | 决定谁该答什么消息 |
| **Tools/Skills** | 工具集 | 白名单制、按角色粒度授权 |
| **Heartbeat/Cron/Compaction** | 主动性与压缩 | 主动节律 + 上下文管理 |
| **多 Agent 协同** | 跨 Agent 通信 | sessions_send + 交接协议 |

## 容器设计的天花板效应

| 容器层级 | 对应的 Agent 上限 | 超过上限的表现 |
|---|---|---|
| 无 Workspace | 单次任务完成度 | Agent 无法"记住你是谁" |
| 有 Workspace 但无三件套 | 30 轮以内一致性 | 30 轮后人格开始漂移 |
| 有三件套但无 HEARTBEAT | 被动响应型 Agent | 没有自主行动能力 |
| 有 HEARTBEAT 但无治理体系 | 单 Agent 可管理 | 多 Agent 协同时崩溃 |
| 完整容器架构 | 军团协同、跨会话稳定 | 持续可演进 |

## 长会话污染的 5 大源头

1. **临时信息堆积**（user 多次澄清）
2. **过期上下文残留**（早期任务不再相关）
3. **错误信息未被清理**（中间过程的错误结论）
4. **无关信息混淆**（多次任务交叉）
5. **人格锚点被覆盖**（与早期 prompt 矛盾的内容）

**压缩时机**：
- 当 token 使用达到上下文窗口的 70-80%
- 当用户明确发出"压缩"指令
- 当主任务完成，开始新任务前
- 当长对话超过 50 轮且主题不集中

**压缩前必做**：静默 memory flush（把 durable notes 写入 `memory/YYYY-MM-DD.md`）

## Routing/Bindings 核心原则

### Bindings 决定 Agent 的社会角色
- 每个 Agent 有显式的 Bindings 声明
- Bindings 与 Agent 角色一致
- Bindings 偏离 Agent 角色 → Agent 接到不匹配的任务导致角色漂移

### 3 种绑定类型
- **channel**：消息渠道绑定
- **peer**：对等 Agent 绑定
- **group**：群组绑定

## 工具不是智能（卷三 M5 上 4 条底线）

### 4 条底线
1. **工具不是智能**（Tool 提供的是手，不是脑）
2. **工具不是越多越强**（动作空间越大，决策越难）
3. **工具不是 shell 替代品**（类型化工具 vs shell 4 大优势）
4. **工具定义本身就是上下文成本**（Anthropic：暴露一个 Tool，就等于引入了一份长期上下文与选择负担）

### 一个好 Tool 的 8 个标准
1. 名字可区分
2. 边界清晰
3. 参数语义可学
4. 返回高信号
5. 调用频率合理
6. 适合被 Agent 使用（不是写给 API 人类用户看的）
7. 成本受控
8. 能被系统治理

### 工具调用纪律 5 种
- **直接调**：意图明确、参数齐全、动作低风险
- **先澄清**：关键信息缺失、风险大、结果可能完全不同
- **串行调用**：后一步依赖前一步输出
- **并行调用**：多个动作相互独立
- **中止**：继续调工具不会带来价值

### Tool Response 设计原则

> **"Tool Response 应该是'决策燃料'，不是'数据库镜像'。"**

- 只返回下一步判断真正需要的字段
- 返回稳定 ID + 关键状态 + 必要摘要
- 避免内部实现噪音

### Schema vs Examples
- **Schema 负责合法性**
- **Examples 负责习惯性**
- 真正好的 Tool 设计：清晰 description + 合理 schema + 少量高质量 examples

## 6 类高频工具的设计意义

| 工具 | 本质 |
|---|---|
| `read` | 观察接口，而不是理解接口 |
| `write` | 整体生成接口，而不是局部修补接口 |
| `edit` | 外科手术式修改接口 |
| `exec` | 运行时动作接口，把命令执行纳入受控链路 |
| `browser` | 真实 UI 世界交互接口，而不是默认取数工具 |
| `message` | 结果投递接口，而不是内部思考接口 |

## Skills 与 Tools 的关键区别

| Tools | Skills |
|---|---|
| 一流工具，原生支持 | 二流能力，需 SKILL.md 描述 |
| 暴露给模型作为可用动作 | 通过描述让 Agent 知道何时调用 |
| OpenClaw 官方提供 | 用户/社区编写 |
| 类型化接口 | 思维框架/方法论 |

### Skill Review 审核清单 5 大维度
- **A. 路由层（召回）**：name/description 清晰、典型触发表达
- **B. 指挥层（SKILL.md）**：SOP 不是散文、有编号步骤、有输出合同
- **C. 资源层**：references 扁平可定位、scripts 小而专
- **D. 安全与治理**：写明 negative triggers、高风险动作确认
- **E. 质量与评估**：至少 5 条应触发 + 5 条不应触发样例

### Skill 3 个 level
- L1：能写 SKILL.md 并被加载
- L2：有清晰触发条件 + 输出合同
- L3：有完整测试集 + 质量评估
- L4：被自动评估 + 持续优化

## Heartbeat / Cron / Compaction 三大主动节律

### Heartbeat（巡逻）
- 周期巡检用
- 30min 一次
- 主会话
- 低打扰
- 适合：状态检查、轻量提醒

### Cron（定点）
- 精确时点用
- 可切模型
- 可隔离
- 适合：定时任务、精确时点汇报

### Compaction（压缩）
- 上下文管理
- 长会话接近 token 上限时触发
- 压缩前必做：静默 memory flush

### 复杂多步任务
- 用 Lobster（多步审批流）

## 主动消息"应该发 vs 不应该发"

### 应该发
- 紧急异常（高优先级告警）
- 用户明确订阅
- 重要机会（主动发现的）
- 即将到期的承诺
- 周期汇总（日报/周报）

### 不应该发
- 闲聊（除非被明确邀请）
- 重复提醒
- 已经处理过的
- 用户忙时（深夜 23:00-08:00）
- 无关紧要的更新

## 多 Agent 协同基础

### 4 个协同基础
1. **隔离**：每个 Agent 独立 Runtime + Workspace
2. **路由**：中央路由系统决定消息去哪
3. **会话**：跨 Agent 通信用 sessions_send
4. **治理**：行为有据可查、有迹可循

### 多 Agent ≠ 多开几个 bot
- 多开几个 bot = 个体户集合
- 真正协同 = 角色分工 + 路由系统 + 交接协议 + 治理账本
