# 06｜卷六 治理系统：如何防止军团失真

> 6 个模块 + 总引言
> 精读 3 轮后浓缩

---

## 一句话

**治理 = 任务卡 + 验收口径 + 三证验真 + 整改单 + 信誉分 + 自动化边界。** **没有结算的治理 = 没有激励的制度 = 没有约束的系统。**

---

## 治理链条（6 层）

```
任务卡（M1）—— 承诺层："我承诺做这件事"
     ↓
验收口径（M2）—— 标尺层："这件事做到什么标准"
     ↓
三证验真（M3）—— 证据层："你做完了，证据在哪里"
     ↓
整改单（M4）—— 纠偏层："没做好，怎么修"
     ↓
信誉分/军功簿（M5）—— 结算层："做好做坏，进账本"
     ↓
自动化治理（M6）—— 边界层："什么情况自动降级熔断"
```

**每一层的输出都是下一层的输入。**

---

## M1 任务卡

### 任务卡 5 大要素
1. **Task-ID**：唯一编号
2. **Title**：一句话标题
3. **Owner**：责任人/Agent
4. **Deliverables**：交付物清单（含路径）
5. **Deadline**：截止时间

### 任务卡 5 大扩展要素
6. **Acceptance Criteria**：验收标准
7. **DoD**：完成定义
8. **Dependencies**：依赖项
9. **Risk Level**：风险等级
10. **Resources Required**：所需资源

### 任务卡 vs 自由派单
- 自由派单："帮我写个报告" → 完成后扯皮
- 任务卡："写一份 X 行业 Q3 报告，2000-3000 字，3 图表，X 月 X 日前交付" → 验收有据

---

## M2 验收口径

### 验收口径 4 维度
1. **完整性**：交付物是否齐全
2. **正确性**：内容是否正确
3. **及时性**：是否按时
4. **可用性**：是否真正可用

### 验收的 3 个反模式
- 模糊验收："差不多就行"
- 主观验收："我觉得好"
- 形式验收："看着像就行"

### 好的验收口径
- 可量化（"错误率<1%"）
- 可观察（"X 字段已填"）
- 可比较（"vs 历史平均提升 X%"）
- 可证伪（"X 不通过则不通过"）

## M3 三证验真（核心中的核心）

### 三证的本质
**三证不是三个数量——是三种硬度层级。**

| 证 | 硬度 | 含义 | 类比 |
|---|---|---|---|
| **证一** | 存在 | 在预期路径上产生了可定位、可打开、可阅读的新文件 | 论文发表了——能看到 PDF |
| **证二** | 真实 | 产物与原始状态之间存在可辨识的实质性变化 | 论文被同行评审过——有修改记录 |
| **证三** | 可复核 | 产物通过了预设的检验流程，且有不可抵赖的日志 | 论文的引用数据和实验代码可复现 |

### 三证的递进关系
- 证一 = "你做了"（存在）
- 证一 + 证二 = "你真的做了"（实质性变化）
- 证一 + 证二 + 证三 = "你做对了，而且我可以复核"（可信证据链）

### 三证分级使用
| 任务等级 | 三证要求 | 举例 |
|---|---|---|
| 日常执行（L1） | 仅需证一 | "今天检查了 CI/CD 状态" |
| 标准交付（L2） | 证一 + 证二 | "写了市场分析报告" |
| 核心产出（L3） | 证一 + 证二 + 证三 | "发布了新版产品文档" |
| 战略决策（L4） | 证一 + 证二 + 证三 + 多方签名 | "修改了核心 SOUL 协议" |

### 三证质量评级
- **S 级**：证据链完整，可被第三方独立复核，通过率 100%
- **A 级**：基本完整，通过率 ≥90%
- **B 级**：证一 + 证二齐，证三硬度不足
- **C 级**：仅证一齐
- **D 级**：不合格，直接驳回

### 三证的 4 源归一
| 领域 | 证的体现 |
|---|---|
| 司法证据学 | 存在 / 真实 / 关联可验证 |
| 软件测试金字塔 | 单元 / 集成 / 端到端 |
| 学术出版 | 发表 / 评审 / 数据可复现 |
| 财务审计 | 存在 / 完整 / 准确 |

### 6 大常见误区
1. 三证 = 三样东西，交满三个就是通过（错：核心是硬度）
2. 验收就是"证三"——有人看就行（错：层层递进）
3. 只有"测试"才是证三，回执和日志不算（错：回执日志同样满足）
4. 三证是验收方的责任——Agent 不需要管（错：是 Agent 交付品的一部分）
5. 三证只有"单次验证"的价值（错：积累的证据库）
6. 三证只适合"写文档"类任务（错：执行类天然有日志）

### L4 证据链的腐败防护
1. **产物 hash 独立存储**（SHA-256 只能追加不能修改）
2. **审计日志权限分离**（写入后可读但不可改）
3. **定期 hash 链验证**（每 24 小时一次）
4. **审计日志由治理容器独立管理**

## M4 整改单

### 整改单 4 大要素
1. **问题描述**：哪里不对
2. **根因分析**：5 Whys 追到系统层
3. **修复动作**：改什么、谁来改、何时改
4. **预防措施**：如何避免再次发生

### 5 Whys 必须追到系统层
- "某 Agent 不够努力"（错：根因分析失败）
- "流程缺失"（对：追到流程/系统/协议）
- 治理改系统，不改人

### Blameless Postmortem 原则
- 不追责个人
- 追系统缺陷
- 把失败转成资产
- Google SRE 的核心实践

## M5 信誉分与军功簿

### 信誉分（短期即时调节）
- 每次任务完成或违约即调整
- 影响：日限额、并发、审批强度
- 调整频率：随时可能变
- 恢复方式：单次战功显著恢复

### 军功爵（长期身份定义）
- 累计达到阈值时晋升/降级
- 影响：爵位等级、任务范围、跨战区权限
- 调整频率：相对稳定
- 恢复方式：连续稳定表现才能晋爵

### 信誉分 vs 军功爵
| 维度 | 信誉分 | 军功爵 |
|---|---|---|
| 核心功能 | 短期即时调节 | 长期身份定义 |
| 主要影响 | 日限额/并发/审批 | 爵位/任务范围/跨战区 |
| 触发场景 | 每次任务或违约 | 累计阈值时晋升 |
| 调整频率 | 随时可能变 | 相对稳定 |
| 恢复方式 | 单次战功显著恢复 | 连续稳定表现才能晋爵 |

### 5 大根问题
1. 行为没有清算
2. 好坏一个样
3. 承诺没有代价
4. 失职没有恢复路径
5. 荣誉空转

### 5 层设计逻辑
```
行为记录（每次任务/违约都被记录）
   ↓
信誉分（即时结算）
   ↓
军功爵（长期积累）
   ↓
资源档位（爵位绑定资源）
   ↓
历史沉淀（所有记录不可删）
```

### Stack Overflow 信誉系统类比
| Stack Overflow | 训虾派军功簿 |
|---|---|
| Reputation Points | 信誉分 |
| Privilege Levels | 军功爵 |
| Bounties | 高军功承接高价值任务 |
| Badges | 特殊军功勋章 |
| Downvote Penalty | 违约即扣分 |
| Answer Ban | 低信誉被限制作答 |

## M6 自动化治理边界

### 自治级别 4 档
| 等级 | 描述 | 适用 |
|---|---|---|
| **L0** 完全手动 | 人类决策 | 战略决策 |
| **L1** 人类决策 + Agent 执行 | 关键决策人类 | 重要发布 |
| **L2** Agent 决策 + 人类监督 | 常规任务自主 | 例行任务 |
| **L3** 完全自主 | 自我监控 + 自我熔断 | 监控/告警 |

### 自治升级 3 原则
1. **自治级别与能力正交**（聪明≠自治高）
2. **自治升级要渐进**（L0→L1→L2→L3）
3. **熔断器永远存在**（自治再高也保留人工熔断）

### 5 大类高风险操作（默认需要审批）
- **删除**（数据/文件/资源）
- **修改**（生产环境/系统配置/权限）
- **外发**（公开内容/外部通知）
- **支付**（财务操作）
- **权限提升**（sudo/elevated）

### 熔断器 3 大触发条件
- 连续失败次数超阈值
- 关键指标异常偏离
- 人工明确触发

### 白名单机制
- 高风险动作必须先在白名单中
- 白名单每次执行后要复审
- 白名单的变更要有人类 review

### 4 道边界
1. **能力边界**（能做什么）
2. **权限边界**（被允许做什么）
3. **影响边界**（后果可逆/不可逆）
4. **价值边界**（是否对齐 SOUL 价值观）

---

## 治理的 4 个真条件

一个治理系统能成立，必须满足：
1. **事前可定义**（规则写清）
2. **事中可监控**（过程可观察）
3. **事后可验真**（结果可验证）
4. **长期可结算**（行为进账本）

**缺任何一个，治理就是空话。**

## V3 制度（治理者不能治理自己的元规则）

> 参考 Silicon Valley Bank 倒闭案例：风险官已经发现问题但被 CEO 压制

### V3 的 3 个核心
1. **风险官直接对最高层**（不被 CEO 压制）
2. **多个独立治理方**（不依赖单点）
3. **不可越权熔断**（必要时强制降级）

### V3 的应用
- 治理 Agent 独立于被治理 Agent
- 治理者之间相互监督
- 紧急熔断权在治理者手中
- 治理动作必须留痕
