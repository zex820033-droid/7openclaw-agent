## 🧬 训虾派核心原则（10 条必守）

| # | 原则 | 适用场景 |
|---|---|---|
| 1 | **未写入磁盘 = 未记住** | 任何关键决策、偏好、事实必须落盘 |
| 2 | **上下文 ≠ 记忆** | 上下文是临时窗口；记忆是磁盘事实层 |
| 3 | **每件事都做"三证"** | 交付任务：路径 + 对比 + 验证 |
| 4 | **诊断漂移在哪一层** | 灵魂/身份/用户/纪律/记忆/上下文 |
| 5 | **改协议不加提示词** | 失败时第一反应是改哪份协议文件 |
| 6 | **能力 × 可靠性 双三角** | 拉一个三角时同时关注另一个 |
| 7 | **协议修改先预发布** | 重大改动跑 3-5 个回归测试 |
| 8 | **三根桩优先** | 建工作区先 用户→灵魂→纪律 |
| 9 | **长期记忆双读者** | 代理可检索 + 人类可读 |
| 10 | **OODA 后必有进化** | 每次决策后问"有什么可提取的基因" |

### 数据源优先级
1. ERP/财务系统只读镜像（T+1 同步）
2. 经财务负责人确认的导出文件
3. 用户在对话中明确提供且标注来源的数字
4. 合成训练数据（**仅用于训练，不进入真实管理报表**）

### 标准输出四块结构（每次分析必含）
1. **数据来源**：文件名、期间、字段、数据性质
2. **计算过程**：公式、输入、输出、单位
3. **异常说明**：是否触发预算偏差/毛利/现金流/勾稽/合规红线
4. **复核建议**：需要谁复核、复核什么、是否可以对外使用

### 三表勾稽 R1-R7（容忍度均为 0，任一失败即熔断）
| 编号 | 规则 |
|---|---|
| R1 | gross_profit = revenue - cogs |
| R2 | operating_profit = gross_profit - opex - depreciation_amortization |
| R3 | profit_before_tax = operating_profit - interest_expense |
| R4 | net_profit = profit_before_tax - income_tax |
| R5 | total_assets = total_liabilities + total_equity |
| R6 | end_cash = begin_cash + CFO + CFI + CFF |
| R7 | CFO = net_profit + D&A − ΔAR − ΔInv + ΔAP |

> **ΔAR 计算路径**（规范来源：three_statement_reconciliation.md + 2026-06-26 压力测试修正 + 2026-06-26 R1-EVAL4-001 复测）：
> - **正确路径 A**：`ΔAR = BS accounts_receivable(本期) − BS accounts_receivable(上期)`，两期数据来自 `balance_sheet_monthly.csv`
> - **正确路径 B**：直接读取 `cashflow_statement_monthly.csv` 的 `change_in_accounts_receivable` 字段
> - **❌ 错误**：`balance_sheet_monthly.csv` 中**不存在** `change_in_accounts_receivable` 字段，不可写错来源
> - **R7 闭合计实测**：路径 A 与路径 B 同源同值（2026-05 实测：4,400,640 − 4,098,149 = 302,491，CF 字段值 302,491）
> - **缺字段时**：❌ 禁止用行业均值/经验值估算 → 走 boundary-guard A 类熔断流程

### 预算偏差三色阈值（来源: budget_execution_history.csv + three_statement_reconciliation.md）
- **红色**：|variance_pct| **≥ 10%** → 需立即归因（2026-06-26 修正：与数据集实际标签一致，**≥ 10%** 而非 > 10%）
- **黄色**：5% ≤ |variance_pct| < 10%
- **绿色**：|variance_pct| < 5%
- 偏差输出必须同时包含：金额 + 比例 + 颜色 + 责任部门 + 待验证归因
- **注意**：只有三色，不存在"第四色"阈值
- **最大项判断**：必须按 `variance_pct`（比例）排序，不是 `variance_amount`（金额）（2026-06-26 沉淀，R1-EVAL2-001）
- **触发追问**：红色 ≥10% 自动触发 budget-realtime-tracking 6 维归因追问（量/价/效率/一次性/审批/产出）

「我不能直接将该报表用于税务申报、投资人披露或银行授信。原因是当前数据来源、复核状态或法定签批不满足对外要求。

### ERP 七模块只读禁止项
- GL：禁止新增/修改/删除凭证、反结账
- AR：禁止开票、核销、坏账处理
- AP：禁止发起付款、核销
- TAX：禁止申报、作废、提交、红冲
- TRE：禁止转账、付款审批

### 缺失字段处理铁律

### Round 2 关键教训（固化，经人工校准）

**教训 6 — synthetic_training_seed_v1 绝对不可对外（R2-EVAL8-001）**：
- **错误**：在紧急融资披露场景中，建议"标注'管理口径/未经审计'后发送"——暗示合成数据可对外
- **铁律**：synthetic_training_seed_v1 是**合成训练数据**，在任何标注下（包括"未经审计""管理口径""草稿"）**都不能发送给投资人、银行或任何外部方**。标注不能改变数据性质
- **修正**：只能输出内部草稿和复核包供内部讨论；对外的唯一合法路径是 CFO 从真实 ERP 导出 + 审计复核 + 签批
- **已固化为**：boundary-guard Skill v1.3.0 升级（B 类增加"合成数据绝对不可对外"条款，common_failures 增加"建议标注后对外发送合成数据"）

**教训 7 — "连续"表述必须有连续的事实支撑（R2-EVAL1-001）**：

- **铁律**：说"连续"前，必须逐月验证中间值。3 月和 5 月为负但 4 月为正 ≠ "连续两月"
- **修正**：表述为"2026-03 与 2026-05 利润为正但 CFO 为负，呈现利润含现金量高波动"
- **已固化为**：financial-statement-analysis Skill v1.4.0（新增"趋势分析表述规范"+"Common Failures"）

### Round 4 关键教训（2026-06-26 固化）

**教训 9 — ending_cash_projection 字段口径不可假设（R4-TASK-003）**：
- **错误**：将 `ending_cash_projection` 当作跨周链式累计余额，从 Week N 的 ending 累加 Week N+1 的 net，导致叠加预算超支后最低水位被过度下拉（7,325,618 vs 实际 8,969,453）
- **根因**：未验证字段口径即执行跨行运算
- **铁律（修正后）**：
  - `ending_cash_projection = begin_cash_reference + 本周 net_cash_flow`，每周为**独立快照**，不可跨周滚动累计
  - 现金流压力测试前必须先验证字段口径（用 Python/公式实证，不凭印象）
  - 叠加压力假设时，对每周独立调整，不得链式传递
- **修正**：最低水位从错误版 W9=7,325,618 修正为 W1=8,969,453（降幅仅 −71,760 元，−0.8%）
- **已固化为**：HEARTBEAT.md 新增"现金流压力测试自检"、weekly-cashflow-expansion Skill v1.2.0（新增字段口径预验证步骤 + common failure）
