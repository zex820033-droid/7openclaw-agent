# 数据诚信防护门禁 (Data Integrity Gate)

> **核心价值**：防止 Agent 在数据采集/统计/复用环节出现系统性偏差。
> **来源**：03_commander D-053/D-054/D-067 + 02_trainer D-069
> **复用范围**：所有需要 curl API / 统计数据 / 复用历史数据的 Agent

---

## 一、curl 参数铁律

GitHub API 数据采集的不可协商参数：

```bash
# ✅ 正确
curl "https://api.github.com/repos/{owner}/{repo}/pulls?state=closed&sort=created&direction=desc&per_page=100"

# ❌ 错误
# sort=updated → 返回历史PR而非近期PR（F-021）
# per_page=30 → 窗口太窄遗漏峰值（F-018）
```

### 采集后自检
```
curl返回后检查首条PR日期：
  首条日期 > 30天前 → 参数错误（sort=updated）→ 修正为 sort=created
  首条日期 = 今日/昨日 → ✅ 参数正确
```

## 二、缓存验证门禁

任何复用历史数据前必须执行：

```
□ 数据来源是上次curl的缓存？
   → 是 → 重新curl一次，对比差异
   → 重新curl失败 → 标注 [BLOCKED·缓存不可验证] + 不产出
□ 数据跨轮次引用？
   → 是 → 标注引用来源(轮次+日期+原始验证状态)
   → 原始数据曾被标记为虚构 → 绝对禁止引用
```

### 铁律
- API不可达 → `[BLOCKED]` + 零推测。禁止复用标记为虚构的旧数据。
- 缓存数据 ≠ 新鲜数据。每次关键决策前重新采集。

## 三、Σ 计数偏误修复

LLM 在生成表格和结论时是两个独立认知通道：
- 表格通道：逐个推理 → 精确
- 结论通道：印象整合 → 倾向于乐观舍入

### 检测方法
```
表格行数 ≠ 结论声称数 → Σ计数偏误确认

示例（Commander T3发布清单7+轮复发）:
  表格: 4✅ + 4⚠️ = 8项
  结论: "5/8✅" ❌ → 应为 "4/8✅"
```

### 修复机制
```
结论行必须从表内逐行计数再写入：
  1. 生成表格
  2. 逐行计数: grep -c '✅' / grep -c '⚠️' / grep -c '❌'
  3. 将计数结果写入结论
  4. 验证: 表内计数 = 结论声称数
```

## 四、数据虚构三连检测

跨轮次数据污染的典型模式：

```
第1轮: 参数错误 → 7/10（不自知）
第2轮: 单轮虚构 → 6/10（已标记但未验证）
第3轮: 跨轮复用已标记虚构数据 → 2/10（连锁污染）

检测信号:
  - 同一数据点在3+轮中数值完全一致（不可能）
  - W{N}数值逐轮复制而非重新采集
  - 趋势线 "加速🟢" 但实际数据 "回落📉"
```

## 五、反馈闭环检查表

收到训练师/审计方反馈后强制执行：

```
① 阅读反馈 → 逐条标注「已确认/争议/需澄清」
② 对每条 🔴 级反馈 → 立即定位错误源 → 立即修正
③ 修正后 → 回写确认「已修正于 [文件]:[行号]」
④ 下次同类任务执行前 → 回顾上次反馈清单 → 逐条确认「本次是否已避免」

⚠️ 「已反馈=已修复」= 幻觉 —— 反馈接收 ≠ 执行修正
```
