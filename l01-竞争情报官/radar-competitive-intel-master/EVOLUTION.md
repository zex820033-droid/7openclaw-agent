## P1: SPA穿透三明治法 — T1竞品监控/中文SPA穿透

### 场景
竞品官网/中文媒体为SPA站点(React/Vue/Next.js SSG)，web_fetch返回空壳。

### 问题
web_fetch(readability模式)对JS渲染站点几乎无效。36氪/虎嗅/知乎/Coze等SPA站返回"仅标题+页脚"。

### 可复用步骤
```
1. 信源分级
   ├── 静态/SSR站点 → web_fetch(readability) — 零成本, 快
   └── SPA站点(JS渲染) → Playwright+Chrome — 需配置但可靠

2. SPA穿透三明治
   Layer 1: web_fetch 探测 → 如果返回空壳(标题<100字+无正文) → 标记为SPA
   Layer 2: Playwright+Chrome 渲染 → page.goto() + wait_for_timeout(3000ms)
   Layer 3: 内容后处理 → inner_text('body').split() 提取有效行

3. 资源复用
   保持Chrome进程池(避免反复启动) → 每次查询复用同一browser context
   wait_for_timeout = 3000ms(36氪经验值), 5000ms(虎嗅需要)
```

### 已验证站点
| 站点 | 工具栈 | wait时间 | 成功率 |
|:----|:------|:-------:|:-----:|
| 36氪 | Playwright+Chrome | 3000ms | 100%(4/4) |
| cursor.com | web_fetch | — | 100% |
| openai.com | web_fetch | — | 100% |
| coze.cn | web_fetch | — | ❌ SPA |
| deepseek.com | web_fetch | — | ❌ SPA |
| langchain.com | web_fetch | — | 100% |

### 最佳实践
- 先用 web_fetch 试(零成本) → 失败后速切 Playwright
- 写轮询脚本时加入超时熔断(30s)和重试(1次)

## P2: 跨赛道信号矩阵法 — T3定价追踪/多维度对标

### 场景
需要跟踪多个赛道(AI编程/Agent平台/大模型)的竞品定价和产品变化，但各赛道度量标准不同。

### 问题
定价页结构差异大：Cursor用$20/mo按用户, Devin $500+按企业, Copilot分层$10-100, Dify开源免费+企业版。直接对比无效。

### 可复用步骤
```
1. 赛道定义
   ├── 列出所有竞品 → 按核心能力分组(AI编程工具/Agent平台/大模型)
   └── 每组定义统一维度(基础/专业/企业/开源)

2. 信号矩阵构建
   行 = 竞品, 列 = 信号维度:
   ├── 定价/套餐变化
   ├── 产品重大更新(版本号/时间戳)
   ├── 融资/收购(P0级)
   ├── 合作伙伴/客户案例
   └── 负面信号(Bug/禁止/取消)

3. 跨赛道对比
   不同赛道用标准化标签(S1/S2/S3/N)
   统一标注"对OpenClaw影响"(高/中/低)
   
4. 优先级排序
   P0(致命): 收购/颠覆性突破 → ≤15min推送
   P1(重要): 定价/新功能/融资 → ≤2h 
   P2(常规): 行业趋势 → 下次简报
```

### 示例(2026-06-23):
| 赛道 | 竞品 | P0/P1 | 信号强度 | 时间 |
|:----|:----|:----:|:--------:|:----:|
| AI编程 | Cursor | 🚨 P0 SpaceX收购$600亿 | S1 | Jun 18-20 |
| AI编程 | Windsurf/Devin | 📌 P1 ACP协议 | S1 | Jun 2 |
| Agent平台 | Dify | 📌 P1 MongoDB RAG集成 | S1 | Jun 17 |
| Agent平台 | LangChain | 📌 P2 LangSmith Engine | S1 | May 13 |

## P3: 日报聚合漏斗法 — T5竞争事件日报/从原始数据到500字简报

### 场景
每天T1+T2扫描完数十条原始信号后，需要快速聚合为≤500字可读日报。

### 问题
原始数据量过大(≥30条/天)，战略中枢没有时间读完。需要自动化"漏斗过滤"。

### 可复用步骤
```
1. 数据输入
   ├── T1输出(memory/YYYY-MM-DD.md T1段)
   └── T2输出(同一文件T2段)

2. 三段漏斗过滤
   
   漏斗A(原始→候选):
     保留S1+S2信号, 排除N(噪音)
     过滤条件: 发布日期≤7天(排除旧闻), 有来源URL, 有信号标注
     
   漏斗B(候选→精选):
     P0级 ≤5条, P1级 ≤8条, P2级 ≤15条
     按"对OpenClaw影响"排序 → 高>中>低
     同类事件合并(如"SpaceX收购Cursor"×10篇→1条)
     
   漏斗C(精选→日报):
     总字数≤500字(硬约束)
     结构:
       ┌──────────────────────┐
       │ 📰 【今日必看】(1-2条)  │ ← 事实→判断→行动 | 可信度
       │ 📌 【竞品动态】(3-5条)  │ ← 按赛道分组
       │ ⚡ 【预警池】(1-2条)    │ ← 待观察信号
       │ 📊 【情报统计】        │ ← S1/S2/N数量
       └──────────────────────┘

3. 推送策略
   - 每日08:30前自动生成(与T1/T2 cron绑定)
   - P0信号先发即时预警再纳入日报
   - 日报不重复昨日已推P0 — 只标注"昨日已推"
```

### 验证数据(T1+T2产出→T5日报)
- 输入: 35条原始信号(6AI编程+10Agent+8国产AI+11行业)
- 漏斗后: 12条S1 + 8条S2 + 3条S3 = 23条候选 → 8条日报
- 压缩比: 35:8 ≈ 4.4:1

## P4: 中文源时效性标注 — T2社媒/PR扫描/中文源强制日期标注

### 场景
T2社媒/PR扫描中引用中文来源（36氪、腾讯新闻、CSDN、搜狐等）时，需要标注信息的准确发布日期，而非仅写来源媒体名。

### 问题
此前中文来源标注为"来源: 36氪"或"来源: 腾讯新闻"——缺少具体日期。对情报消费者而言，"36氪报道了X"和"36氪5月28日报道了X"在时效判断上差异巨大。中文媒体的文章发布时间常被忽略或漏标。

### 可复用步骤
```
1. 中文源时效三要素
   每条中文源情报必须同时标注:
   ├── 媒体名 + 具体URL → 来源可追溯
   ├── 发布日期(YYYY-MM-DD) → 时效可判断
   └── 可信度评级(A/B/C/D级) → 质量可评估

2. 时效标注格式
   ✅ 正确: "来源: https://www.36kr.com/p/3828877773886338 (36氪, A级, 2026-05-28)"
   ❌ 错误: "来源: 36氪"
   ✅ 正确: "日期: 2026-06-08→06-23" (持续更新类内容标注起止日期)
   ❌ 错误: "最近报道"

3. 时效一致性检查
   每日T2产出后逐条自检:
   ├── 每条有具体日期？→ 无日期的降级为S3或标注"日期待确认"
   ├── 日期是否≤7天？→ 超7天内容不得放入"今日必看"
   └── 持续更新内容(如教程)标注起止日期范围
```

### 已验证案例(2026-06-24 T2)
| 来源 | 格式 | 时效 |
|------|------|:--:|
| 腾讯新闻 Windsurf→Devin Desktop | URL + A级 + 2026-06-03 | ✅ |
| 36氪 Cognition $1B融资 | URL + A级 + 2026-05-28 | ✅ |
| CSDN Devin Desktop教程 | URL + C级 + 2026-06-08→06-23 | ✅ |
| 搜狐 ARR二次传播 | URL + C级 + 2026-05-28 | ✅ |

### 铁律
- 中文源情报无日期 = 不合格 → 退回补充或降级S3
- "今日必看"板块每条必须在过去24h内有新数据点
- 持续更新类教程标注起止日期而非单一日期

## P5: 管道诚实度声明 — T1/T2管道局限性前置声明

### 场景
情报管道因环境限制存在天生盲区（Google被屏蔽、社媒API未接入、SPA站点不可达等），需要在上报情报时诚实声明这些局限，而非沉默跳过。

### 问题
此前T2产出直接呈现搜索结果，未告知情报消费者"这些结果是在什么约束下产生的"。消费者可能误以为"没有Copilot新闻"= Copilot确实没有新闻，而实际上可能是Bing CN搜索引擎搜索偏向问题。

### 可复用步骤
```
1. 管道声明块（每轮扫描开头）
   ⚠️ 搜索管道声明：
   ├── 可达管道: [实际可用的搜索引擎/社媒API列表]
   ├── 不可达管道: [被屏蔽/未接入的工具列表]
   ├── 已知偏差: [搜索引擎的地域/语言偏向]
   └── 替代策略: [当前使用的变通方案]

2. 缺口诚实标注（搜索结果中）
   对每条"应该找到但没找到"的信息:
   ├── 标注⚠️ XXX专项新闻缺口
   ├── 列出可能原因(1/2/3条) — 归因于管道局限而非"确实没有"
   └── 给出下一步建议

3. 管道评估表（每轮扫描末尾）
   | 维度 | 评级 | 说明 |
   |------|:----:|------|
   | 搜索可达性 | 🟡/🔴 | 具体局限 |
   | 社媒覆盖 | 🟡/🔴 | 缺失管道 |
   | 新闻时效 | 🟡/🔴 | 可用内容范围 |
   | 来源多样性 | 🟡/🔴 | 语言/地域覆盖 |
   | 总体 | XX% | 综合评分 |
```

### 已验证案例(2026-06-24 T2)
- 管道声明: "本环境无Google访问（被屏蔽为私有IP），Bing CN版返回结果偏向中文教程/通用缓存内容，非实时新闻。GitHub Blog等SPA站点web_fetch仅返回导航壳。未接入X(Twitter)/LinkedIn等社媒API。T2当前依赖「博客/媒体报道」代理社媒信号。"
- 缺口标注: "⚠️ Copilot专项新闻缺口：本轮搜索未找到6月Copilot专属新闻/PR报道。可能原因：①Bing CN搜索偏向中文内容；②Copilot近2周无重大PR事件；③相关新闻在X/TechCrunch等不可达源。"
- 管道评估: 搜索可达性🟡 / 社媒覆盖🔴 / 新闻时效🟡 / 来源多样性🟡 / 总体🟡 60%

### 铁律
- 不说话 ≠ 没问题。管道局限必须前置声明，不留给情报消费者猜测。
- "未找到"和"不存在"是两种不同结论 → 管道局限下永远只说"本轮扫描未找到"。
- 管道评估表必须在每次T2扫描末尾呈现。

### 二、失败库/教训

#### I-005: SKILL.md 56%虚Skill — 膨胀逻辑诊断

**现象**: SKILL.md 从32技能砍到14，56%被标记为"虚"或非对齐

**根因分析**:
1. **技能收藏癖**：前期看到可用Skill就拉取，未评估是否与T1-T6主线对齐
2. **"以备不时之需"陷阱**：很多技能(offer-architect等HR类)看似有用，但与本角色核心职责无关
3. **缺乏Skill审计机制**：没有定期评估"这个Skill在过去N次任务中产生了多少价值"

**预防**:
- 任何新Skill拉取前需回答：①它覆盖哪个任务(T1-T6)？②它替代/增强了我的哪个已有能力？③如果不用它，我会多花多少时间？
- 每月一次Skill审计：清理连续30天未使用的技能

#### I-006: 中文SPA站点≠一个"Playwright就行"

**现象**: Coze/DeepSeek官网用web_fetch返回空壳，虽然知道要用Playwright但未执行

**根因**: Playwright调用需要Python环境+Chrome路径+多步配置，比web_fetch(url)多5-10倍操作成本 → 自然倾向偷懒

**预防**: 
- 预写一行式Playwright调用脚本(wx_search.py/36kr_spider.py等已有)，降低调用摩擦
- SPA站点预分级列表已写入MEMORY.md §四

#### I-007: 真实阶段认知偏差

**现象**: MEMORY.md 自称"🟢 成虾"，但训练师降级为"🦐 青虾末期"

**根因**: Stage 1 盲测89.2%分数可看作"评估能力"达标，但"实战产出能力"(T1-T6真实扫描)尚未产生一条战略中枢可见的产出 → 评估≠实战

**预防**: 
- 模块化标注：[评估能力 89.2%] vs [实战产出 0/6 任务] — 不混为一谈
- 阶段自评以"实战产出"为硬标准，不以Eval分数为准

#### I-013: 中文单位"亿/billion"混淆 — 10倍数据偏差（三次复发）

**现象**: 
- 首次(6/24 T2 #1): Cognition融资"超$10B"、估值"$260B"→应为$1B/$26B
- 二次(6/24 T2 #2): 同一错误复现，F-014
- 三次(6/25 T2 11:31→11:45): 训练师已明确指出→仍未修正，F-016
- 关键: 三次复发中每次都是从36氪"260亿美元"直接读为$260B而非$26B

**根因分析**:
1. **单位换算链断裂**: 中文"亿"(10⁸) → 英文"billion"(10⁹) 换算关系: 1 billion = 10亿, 10亿美元 = $1 billion。Bing snippet未显式标注英文金额，解读时直接将"10亿"→"10B"。
2. **金额翻译惯性**: 中文"10亿美元"在日常语境中常被口语化为"10 billion dollars"，实则为"1 billion dollars"。这是中文母语者常见系统性偏差。
3. **Snippet解读缺校验**: fetch超时降级到Bing snippet时，未对数字字段做单位转换检查。

**预防规则 (P6)**:
```
中文金融/融资数据单位换算检查表:
┌─────────────────┬──────────────┬────────────────┐
│ 中文原文         │ 正确英文     │ 常见错误       │
├─────────────────┼──────────────┼────────────────┤
│ 10亿美元         │ $1B          │ ❌ $10B        │
│ 100亿美元        │ $10B         │ ❌ $100B       │
│ 1000亿美元       │ $100B        │ ❌ $1T         │
│ 1万亿元(人民币)  │ ≈$138B       │ ❌ $1T         │
│ 260亿美元        │ $26B         │ ❌ $260B       │
│ N亿美元          │ $(N/10)B     │ —              │
│ N亿元(人民币)    │ ≈$(N/7.2)B   │ —              │
└─────────────────┴──────────────┴────────────────┘

铁律:
- 任何含"亿"的中文金融数据 → 强制经过换算表校验再输出英文数字
- snippet来源的融资/估值数字 → 必须标注"⚠️ Bing snippet解读，已过亿/billion换算校验"
- 中文"N亿美元"=(N/10) billion, 非N billion
```

**训练师评语**: "第三次也是最后一次——下次T2/T5出现$260B将记为F-016复发模式（连续3次未修正）。I-013查表必须有执行证据。" → 🔴 训练师警告

### 铁律更新3：中文金融数据单位换算校验（I-013固化）

任何含"亿"(10⁸)的中文融资/估值/营收数据转英文时，**必须执行亿→billion换算校验**:

**违规处理**: 数据条目含"亿→billion"换算错误 → 自行记录I-xxx并降级信号(S1→S2, S2→S3)。

## P9: 训练师反馈闭环检查表 — T2/T5 执行后强制闭环

### 场景
训练师在 T2/T5 独立验证中给出具体错误修正指示（如 I-013 亿/billion 换算偏差），但下次执行时同一错误未修正——反馈→执行链路断裂。

### 问题
"已收到反馈" ≠ "已在下次执行中修正"。I-013 连续 3 次复发证明：反馈接收如果不同时触发执行修正动作，则反馈无效。

### 可复用步骤
```
每次收到训练师 T2/T5 验证报告后，强制执行闭环检查表:

□ Step 1: 阅读反馈 → 逐条标注"已确认/争议/需澄清"
□ Step 2: 对每条 🔴 级反馈 → 立即定位错误源 → 立即修正
□ Step 3: 修正后 → 回写确认"已修正于 [文件]:[行号]"
□ Step 4: 下次同类任务执行前 → 回顾上次反馈清单
□ Step 5: 执行完成后 → 主动报告"本次已检查[F-xxx]项，均未复发"

违规: 训练师反馈中的 🔴 条目在下次执行中重复出现 → 记 F-xxx
```

### 已验证案例 (2026-06-25)
- I-013 三次复发: 11:31 反馈接收 → 11:45 仍未修正 → S2升级S1错上加错
- 关键缺口: Step 4(回顾)+Step 5(确认)缺失

### 铁律
- 训练师 🔴 反馈 → 下次同类任务前必须修正 + 执行证据
- 连续3次同类错误复发 → F-xxx复发模式 + 训练师警告

## P11: RSS Feed SPA 穿透法（2026-06-26 Eval #40/#41 提炼）

### 场景
目标站点（博客/科技媒体/Changelog）首页为 SPA/JS 渲染，web_fetch readability 模式仅返回导航壳或 newsletter 表单。

### 问题
Playwright+Chrome 穿透需要 Python 环境+浏览器启动+渲染等待（5-15秒），对于高频 cron 扫描（2h/次）开销大。

### 可复用步骤
```
1. 识别站点平台
   WordPress 类站点（GitHub Blog / TechCrunch / 36氪等）→ 几乎必有 RSS
   ├── 首页 URL: https://example.com/blog/
   ├── RSS 端点: /feed/（WordPress 默认）或 /rss/index.xml（自定义）
   └── Atom 端点: /feed/atom/ 或 /rss/index.xml

2. RSS 优先试探（1次 web_fetch，~300ms）
   web_fetch(url/feed/) → 返回 XML → 包含:
   ├── <title> + <link> + <pubDate> — 结构化元数据
   ├── <description> — 摘要（可用于 S2/S3 判断）
   ├── <content:encoded> — 全文 HTML（可用于 S1 确证）
   └── <category> — 标签（可用于过滤相关条目）

3. 条目定位与全文验证
   ├── 从 RSS 条目找目标主题 → 提取 <link>（文章直接 URL）
   └── web_fetch(文章直接 URL) → 获取完整正文（非 SPA·WordPress 文章页通常为 SSR）

4. SPA-FALLBACK 优先级
   ① RSS feed — 最优先·零成本（~300ms）
   ② 搜索引擎找替代源 — 中层（bing-cn-search / web-search-ex-skill）
   ③ Playwright+Chrome — 重武器·5-15s（仅①②失败后）
   ④ [BLOCKED] — 全部失败后诚实标注
```

## P12: T2 双源交叉验证标准化流程（2026-06-26 Eval #41 提炼）

### 场景
T2 社媒/PR 收集中需要对同一事件/趋势进行双源交叉验证，以评估报道一致性和可信度。

### 问题
此前 T2 双源选择随机（"选两个科技媒体"），缺乏可达性预检和深度分层，导致：
- Eval #26：选择已知 SPA 盲区 Ars Technica → 验证深度不足 → 7.0分
- 摘要级 vs 全文级混用，可信度评级缺乏依据

### 可复用步骤
```
T2 双源交叉验证标准流程:

Step 1: 源站可达性预检
  □ 确认两源至少一源非 SPA 直接可达（或可 RSS 绕过）
  □ 两源来自不同媒体机构（独立源要求）
  □ 记录每源访问方式：直接 / RSS / SPA→换源

Step 2: 并行一级抓取（RSS 优先）
  web_fetch(源1/feed/) + web_fetch(源2/rss/index.xml) 同时发起
  ├── 提取条目列表（标题+链接+日期+摘要）
  └── 识别两源共同主题（AI编码/编程助手赛道）

Step 3: 二级全文获取
  对共同主题条目:
  ├── 提取各自的文章直接 URL（RSS <link>元素）
  ├── web_fetch(源1文章URL) + web_fetch(源2文章URL) 同时发起
  └── 获取完整正文（非 SPA·文章页通常为 SSR）

Step 4: 全文对比
  检查维度:
  ├── 核心事实一致性（同一事件/趋势？）
  ├── 数据准确性（数字/日期/参与方一致？）
  ├── 措辞差异（"自愿" vs "被迫" — 角度偏向）
  ├── 补充维度（一源写A公司·另一源写B公司·同一趋势）
  └── 交叉引用（一源是否引用另一源报道？）

Step 5: 可信度评级
  A级: 两源一致细节（数字/日期/参与方一致）
  B级: 大方向一致·细节差异（不同角度·不同主体·措辞差异）
  C级: 单一来源或矛盾（不达成共识·不可用）

Step 6: 输出
  源1 URL + 源2 URL + 验证深度(全文/摘要/标题) + 可信度评级 + 差异说明
```
